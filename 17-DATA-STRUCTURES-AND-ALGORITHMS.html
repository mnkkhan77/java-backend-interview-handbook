SECTION 1 - COMPLEXITY ANALYSIS

Q1. What Is Big O Notation?

ROI: ★★★★★ Frequency: Extremely High

Big O describes how an algorithm's running time (or memory) grows as the input grows, ignoring constants and lower-order terms. It is the worst-case upper bound, which is what interviewers usually mean by "complexity".

The point is hardware-independent comparison. An O(n) algorithm beats an O(n²) one for large n no matter the machine, because we care about the shape of growth, not exact milliseconds. Related notation: Θ (tight bound) and Ω (lower bound), but Big O dominates interviews.

Interview Gold Point

Always state both time and space. A common follow-up after a fast solution is "can you do it in O(1) extra space?" — which asks you to trade a HashMap for an in-place technique like two pointers.

Q2. How Do You Analyse the Complexity of Loops?

ROI: ★★★★★ Frequency: Very High

  • Single loop over n → O(n)
  • Nested loops, both over n → O(n²) (loops multiply)
  • Two separate (non-nested) loops → O(n) (loops add)
  • Range halves each step → O(log n)
  • Loop runs n times, doing O(log n) work each → O(n log n)
for (int i = 0; i < n; i++)          // O(n)
    for (int j = i+1; j < n; j++)     // still O(n^2): ~n^2/2 pairs, constants dropped
        check(a[i], a[j]);

Common Mistake

Sequential loops are added (O(n)+O(n)=O(n)); only nested loops multiply. And a loop from i to n nested in one from 0 to n is still O(n²) — halving the work drops the constant, not the class.

Q4. How Do You Analyse the Complexity of a Recursive Algorithm?

ROI: ★★★★☆ Frequency: High

Write a recurrence: how many recursive calls, and how much work per call. Then evaluate it — either with the recursion-tree intuition or the Master Theorem for divide-and-conquer T(n) = a·T(n/b) + O(nᴰ).

  • Merge sort: T(n) = 2T(n/2) + O(n) → O(n log n) (n work per level × log n levels)
  • Binary search: T(n) = T(n/2) + O(1) → O(log n)
  • Naive Fibonacci: T(n) = T(n-1) + T(n-2) → O(2ⁿ) (branching tree)

Don't forget the call-stack space: recursion depth d costs O(d) space. Balanced recursion (merge sort) is O(log n) stack; linear recursion is O(n) stack and can overflow.

Interview Gold Point

The recursion-tree picture is usually enough: "n work at each of log n levels = n log n". You rarely need the formal Master Theorem, but naming it signals depth.

Q5. What Is Amortized Complexity?

ROI: ★★★★☆ Frequency: Medium-High

Amortized complexity is the average cost per operation over a sequence, even when individual operations occasionally spike. The classic example is ArrayList.add: usually O(1), but when the backing array fills it copies everything to a larger array (O(n)). Because resizing happens rarely and doubles capacity, the cost spread across all adds is amortized O(1).

Interview Favorite

"Amortized O(1)" is not the same as "always O(1)" — a single add can be O(n). HashMap operations are also amortized O(1): occasional rehashing is the spike. Knowing the distinction is a senior signal.

SECTION 2 - ARRAYS & STRINGS

Q6. Two Sum — the Hashing Pattern

ROI: ★★★★★ Frequency: Extremely High

Naive is O(n²). The optimal uses a HashMap to remember seen values: for each number ask "have I seen target - current?" — O(1) per element. 

int[] twoSum(int[] nums, int target) {
    Map<Integer,Integer> seen = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int need = target - nums[i];
        if (seen.containsKey(need)) return new int[]{seen.get(need), i};
        seen.put(nums[i], i);
    }
    return new int[]{-1,-1};
}
// Time O(n), Space O(n)

Interview Gold Point

This "remember what you've seen in a map" idea collapses a huge family of O(n²) brute forces to O(n): subarray sums, anagrams, duplicates, first unique. When you see nested loops searching for a complementary value, reach for a map.

Q7. The Two-Pointer Technique

ROI: ★★★★★ Frequency: Very High

Two indices move through the array — from both ends, or at different speeds — replacing a nested loop with one pass. The canonical use is a pair summing to a target in a sorted array. 

boolean hasPair(int[] sorted, int target) {
    int lo = 0, hi = sorted.length - 1;
    while (lo < hi) {
        int sum = sorted[lo] + sorted[hi];
        if (sum == target) return true;
        if (sum < target) lo++; else hi--;
    }
    return false;
}
// Time O(n), Space O(1)

Interview Favorite

Triggers: "sorted array", "pair/triplet", "in place", "remove duplicates", "reverse", "palindrome", "merge two sorted arrays". The 3Sum problem is just "fix one element, then two-pointer the rest".

Q8. Sliding Window — Longest Substring Without Repeating Characters

ROI: ★★★★★ Frequency: Very High

A window [left, right] expands and shrinks instead of recomputing from scratch, turning O(n²) substring/subarray problems into O(n). 

int longestUnique(String s) {
    Set<Character> window = new HashSet<>();
    int left = 0, best = 0;
    for (int right = 0; right < s.length(); right++) {
        while (window.contains(s.charAt(right)))
            window.remove(s.charAt(left++));     // shrink
        window.add(s.charAt(right));             // expand
        best = Math.max(best, right - left + 1);
    }
    return best;
}
// Time O(n), Space O(k) distinct chars

Interview Gold Point

Trigger words: "longest/shortest/maximum/minimum contiguous subarray or substring", "at most K distinct", "window". Fixed-size windows (max sum of k elements) just slide without the inner shrink loop.

Q9. Maximum Subarray Sum (Kadane's Algorithm)

ROI: ★★★★☆ Frequency: High

Find the contiguous subarray with the largest sum. Kadane's insight: at each element, either extend the previous best-ending-here or start fresh at the current element. 

int maxSubArray(int[] nums) {
    int best = nums[0], cur = nums[0];
    for (int i = 1; i < nums.length; i++) {
        cur = Math.max(nums[i], cur + nums[i]);  // extend or restart
        best = Math.max(best, cur);
    }
    return best;
}
// Time O(n), Space O(1)

Interview Favorite

Kadane's is a 1D dynamic-programming problem in disguise: cur is "best sum ending at i". The restart-vs-extend decision is the recurrence. Handle the all-negative case — initialising to nums[0] (not 0) covers it.

Q10. Prefix Sums — Subarray Sum Equals K

ROI: ★★★★☆ Frequency: Medium-High

A prefix sum lets you compute any subarray sum in O(1). Combined with a HashMap of seen prefix sums, you count subarrays summing to k in one pass: if prefix - k was seen before, those starting points form valid subarrays. 

int subarraySum(int[] nums, int k) {
    Map<Integer,Integer> counts = new HashMap<>();
    counts.put(0, 1);                 // empty prefix
    int prefix = 0, result = 0;
    for (int n : nums) {
        prefix += n;
        result += counts.getOrDefault(prefix - k, 0);
        counts.merge(prefix, 1, Integer::sum);
    }
    return result;
}
// Time O(n), Space O(n)

Interview Gold Point

Prefix sums shine for repeated range-sum queries (precompute once, answer each in O(1)). The HashMap-of-prefixes trick generalises to "count/length of subarrays with sum/property X".

Q11. Merge Overlapping Intervals

ROI: ★★★★☆ Frequency: High

Sort by start, then sweep: if the current interval overlaps the last merged one (its start ≤ last end), extend the end; otherwise append a new interval. Sorting dominates at O(n log n). 

int[][] merge(int[][] intervals) {
    Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0]));
    List<int[]> out = new ArrayList<>();
    for (int[] iv : intervals) {
        if (!out.isEmpty() && iv[0] <= out.get(out.size()-1)[1])
            out.get(out.size()-1)[1] = Math.max(out.get(out.size()-1)[1], iv[1]);
        else
            out.add(iv);
    }
    return out.toArray(new int[0][]);
}
// Time O(n log n), Space O(n)

Interview Favorite

"Intervals" almost always means "sort first, then sweep". The same shape solves meeting-rooms, insert-interval, and interval intersection problems.

Q12. Sort Colors / Dutch National Flag (Three-Way Partition)

ROI: ★★★☆☆ Frequency: Medium

Sort an array of three values (0,1,2) in a single O(n) pass with three pointers: low, mid, high. It's the partition step behind 3-way quicksort. 

void sortColors(int[] a) {
    int low = 0, mid = 0, high = a.length - 1;
    while (mid <= high) {
        if (a[mid] == 0)      swap(a, low++, mid++);
        else if (a[mid] == 1) mid++;
        else                  swap(a, mid, high--);  // 2: don't advance mid
    }
}

Interview Gold Point

The subtlety: after swapping with high you do not advance mid, because the value swapped in is unexamined. Getting that right is the whole question.

Q13. Valid Anagram & Group Anagrams

ROI: ★★★★☆ Frequency: High

Two strings are anagrams if their character counts match. Compare via a frequency array (O(n)) or by sorting both (O(n log n)). To group anagrams, use the sorted string as a HashMap key. 

List<List<String>> groupAnagrams(String[] words) {
    Map<String, List<String>> groups = new HashMap<>();
    for (String w : words) {
        char[] c = w.toCharArray();
        Arrays.sort(c);
        groups.computeIfAbsent(new String(c), k -> new ArrayList<>()).add(w);
    }
    return new ArrayList<>(groups.values());
}
// Time O(n · k log k), Space O(n · k)

Interview Favorite

For lowercase-only input, a 26-length count array (turned into a key string) beats sorting and drops the per-word factor to O(k). Mentioning that optimisation is a strong signal.

SECTION 3 - HASHING & DESIGN

Q14. When Do You Reach for a HashMap or HashSet?

ROI: ★★★★★ Frequency: Very High

Use a HashSet for O(1) membership/de-duplication ("have I seen this?"). Use a HashMap for O(1) lookup of an associated value or a count ("how many times?", "what index?"). This converts countless O(n²) brute forces into O(n).

Frequency counting is the most common sub-pattern: map.merge(key, 1, Integer::sum) or map.getOrDefault(key, 0) + 1.

Interview Gold Point

Remember the cost of a custom key: your object needs correct equals() and hashCode() or the map breaks. Tie this back to the Collections chapter — this is exactly why those methods must agree.

Q15. First Non-Repeating Character

ROI: ★★★★☆ Frequency: Medium-High

Two passes: count frequencies, then scan again for the first character with count 1. A LinkedHashMap preserves insertion order so you can do it in one logical pass over the keys. 

int firstUniqChar(String s) {
    int[] count = new int[26];
    for (char c : s.toCharArray()) count[c - 'a']++;
    for (int i = 0; i < s.length(); i++)
        if (count[s.charAt(i) - 'a'] == 1) return i;
    return -1;
}
// Time O(n), Space O(1) (fixed 26)

Interview Favorite

A fixed-size count array is O(1) space and faster than a HashMap when the alphabet is bounded (ASCII/lowercase). Use a HashMap only when the key space is large or unknown.

Q16. Design an LRU Cache

ROI: ★★★★★ Frequency: High

An LRU (Least Recently Used) cache evicts the oldest-touched entry when full. The interview-grade solution combines a HashMap (O(1) lookup) with a doubly linked list (O(1) move-to-front and remove-from-tail). In Java you can lean on LinkedHashMap with access-order: 

class LRUCache<K,V> extends LinkedHashMap<K,V> {
    private final int capacity;
    LRUCache(int capacity) {
        super(capacity, 0.75f, true);   // true = access-order
        this.capacity = capacity;
    }
    @Override protected boolean removeEldestEntry(Map.Entry<K,V> eldest) {
        return size() > capacity;       // auto-evict oldest
    }
}
// get/put are O(1)

Interview Gold Point

Know both answers: the from-scratch HashMap + doubly-linked-list (interviewers often forbid LinkedHashMap to test fundamentals) and the production LinkedHashMap shortcut. Caffeine/Guava caches generalise this with TTL and size-based eviction — tie to the caching content in your System Design chapter.

SECTION 4 - LINKED LISTS

Q17. Reverse a Singly Linked List

ROI: ★★★★★ Frequency: Very High

Walk once, re-pointing each node's next to the previous node, tracking three references. 

Node reverse(Node head) {
    Node prev = null, curr = head;
    while (curr != null) {
        Node next = curr.next;  // save before overwriting
        curr.next = prev;       // reverse the link
        prev = curr;            // advance
        curr = next;
    }
    return prev;                // new head
}
// Time O(n), Space O(1)

Common Mistake

Forgetting to save curr.next before reassigning loses the rest of the list. The recursive version is elegant but O(n) stack space — mention the iterative O(1)-space version is preferred.

Q18. Fast & Slow Pointers — Cycle Detection and Middle Node

ROI: ★★★★★ Frequency: High

Floyd's two-speed pointers solve a family of list problems in O(1) space. Cycle: if fast (2 steps) ever meets slow (1 step), there's a loop. Middle: when fast reaches the end, slow is at the midpoint. 

boolean hasCycle(Node head) {
    Node slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next; fast = fast.next.next;
        if (slow == fast) return true;
    }
    return false;
}

Node middle(Node head) {
    Node slow = head, fast = head;
    while (fast != null && fast.next != null) { slow = slow.next; fast = fast.next.next; }
    return slow;
}

Interview Gold Point

To find the cycle's start: after the meeting point, reset one pointer to head and advance both one step at a time — they meet at the cycle entry (Floyd's second phase).

Q19. Merge Two Sorted Linked Lists

ROI: ★★★★☆ Frequency: High

Use a dummy head node to simplify edge cases, then splice the smaller current node each step. 

Node mergeTwo(Node a, Node b) {
    Node dummy = new Node(0), tail = dummy;
    while (a != null && b != null) {
        if (a.val <= b.val) { tail.next = a; a = a.next; }
        else                { tail.next = b; b = b.next; }
        tail = tail.next;
    }
    tail.next = (a != null) ? a : b;   // attach the remainder
    return dummy.next;
}
// Time O(n+m), Space O(1)

Interview Favorite

The dummy node trick removes special-casing for the head and is worth using in almost every linked-list build problem. Merging is also the core of merge sort and "merge K lists" (heap version below).

Q20. Remove the Nth Node From the End

ROI: ★★★★☆ Frequency: Medium-High

One pass with a gap: advance a fast pointer n nodes ahead, then move fast and slow together until fast hits the end — slow now sits just before the target. 

Node removeNthFromEnd(Node head, int n) {
    Node dummy = new Node(0); dummy.next = head;
    Node fast = dummy, slow = dummy;
    for (int i = 0; i < n; i++) fast = fast.next;   // create the gap
    while (fast.next != null) { fast = fast.next; slow = slow.next; }
    slow.next = slow.next.next;                      // unlink
    return dummy.next;
}
// Time O(n), Space O(1)

Interview Gold Point

The "two pointers with a fixed gap" idea generalises to any "kth from end" question without a length pre-pass. The dummy node handles the edge case of removing the head itself.

SECTION 5 - STACKS & QUEUES

Q21. Valid Parentheses — Why a Stack?

ROI: ★★★★☆ Frequency: High

Bracket matching is "last opened, first closed" — LIFO, exactly a stack. Push openers; on a closer, the top must be its matching opener. 

boolean isValid(String s) {
    Deque<Character> stack = new ArrayDeque<>();
    Map<Character,Character> pairs = Map.of(')','(', ']','[', '}','{');
    for (char c : s.toCharArray()) {
        if (pairs.containsValue(c)) stack.push(c);
        else if (stack.isEmpty() || stack.pop() != pairs.get(c)) return false;
    }
    return stack.isEmpty();
}
// Time O(n), Space O(n)

Interview Favorite

Use ArrayDeque, not the legacy Stack class (which extends Vector and is synchronised/slower). ArrayDeque is the modern stack and queue in Java.

Q22. Min Stack — O(1) getMin()

ROI: ★★★★☆ Frequency: Medium-High

Support push/pop/top and retrieving the minimum, all in O(1). Keep a second stack that tracks the minimum at each level (or store pairs of value + min-so-far). 

class MinStack {
    private Deque<Integer> stack = new ArrayDeque<>();
    private Deque<Integer> mins  = new ArrayDeque<>();
    void push(int x) {
        stack.push(x);
        mins.push(mins.isEmpty() ? x : Math.min(x, mins.peek()));
    }
    void pop()      { stack.pop(); mins.pop(); }
    int top()       { return stack.peek(); }
    int getMin()    { return mins.peek(); }
}

Interview Gold Point

The key realisation: you can't search for the min in O(1), so you must maintain it as you go. Carrying auxiliary state alongside the main structure is a reusable trick (also used in monotonic stacks below).

Q23. Monotonic Stack — Next Greater Element

ROI: ★★★★☆ Frequency: Medium-High

A monotonic stack keeps elements in increasing or decreasing order, solving "next greater/smaller element" in O(n) instead of O(n²). Pop everything smaller than the current value — the current value is their "next greater". 

int[] nextGreater(int[] nums) {
    int[] res = new int[nums.length];
    Arrays.fill(res, -1);
    Deque<Integer> stack = new ArrayDeque<>();   // holds indices
    for (int i = 0; i < nums.length; i++) {
        while (!stack.isEmpty() && nums[i] > nums[stack.peek()])
            res[stack.pop()] = nums[i];
        stack.push(i);
    }
    return res;
}
// Time O(n), Space O(n)

Interview Favorite

Monotonic stacks power "next greater/smaller", "daily temperatures", "largest rectangle in histogram", and "trapping rain water". Each element is pushed and popped at most once → O(n) despite the inner while loop.

Q24. Implement a Queue Using Two Stacks

ROI: ★★★☆☆ Frequency: Medium

Use an in stack for pushes and an out stack for pops. When out is empty, pour in into it (reversing order → FIFO). Each element moves at most once, so dequeue is amortized O(1). 

class MyQueue {
    private Deque<Integer> in = new ArrayDeque<>(), out = new ArrayDeque<>();
    void push(int x) { in.push(x); }
    int pop() {
        if (out.isEmpty()) while (!in.isEmpty()) out.push(in.pop());
        return out.pop();
    }
}

Interview Gold Point

This is a classic "amortized analysis" question: a single pop can be O(n) (when pouring), but across all operations each element is moved once, giving amortized O(1).

SECTION 6 - TREES & BINARY SEARCH TREES

Q25. Tree Traversals — DFS (Pre/In/Post) and BFS

ROI: ★★★★★ Frequency: Very High

DFS (recursion or explicit stack) has three orders: pre-order (node, left, right), in-order (left, node, right), post-order (left, right, node). BFS / level-order uses a queue. 

// In-order DFS (recursive)
void inorder(Node n, List<Integer> out) {
    if (n == null) return;
    inorder(n.left, out); out.add(n.val); inorder(n.right, out);
}

// BFS level-order
List<List<Integer>> levelOrder(Node root) {
    List<List<Integer>> res = new ArrayList<>();
    if (root == null) return res;
    Queue<Node> q = new LinkedList<>(); q.add(root);
    while (!q.isEmpty()) {
        int size = q.size();
        List<Integer> level = new ArrayList<>();
        for (int i = 0; i < size; i++) {
            Node n = q.poll(); level.add(n.val);
            if (n.left != null)  q.add(n.left);
            if (n.right != null) q.add(n.right);
        }
        res.add(level);
    }
    return res;
}

Interview Gold Point

In-order traversal of a BST yields sorted values — this single fact answers "validate BST", "kth smallest", and "two sum in BST". The level-size loop in BFS is the standard way to process the tree level by level.

Q26. Maximum Depth and Diameter of a Binary Tree

ROI: ★★★★☆ Frequency: High

Depth is a clean post-order recursion: 1 + max(left depth, right depth). Diameter (longest path between any two nodes) reuses depth — at each node the path through it is leftDepth + rightDepth. 

int maxDepth(Node n) {
    return n == null ? 0 : 1 + Math.max(maxDepth(n.left), maxDepth(n.right));
}

int diameter = 0;
int depth(Node n) {
    if (n == null) return 0;
    int l = depth(n.left), r = depth(n.right);
    diameter = Math.max(diameter, l + r);   // path through this node
    return 1 + Math.max(l, r);
}
// Both O(n) time, O(h) stack

Interview Favorite

Many tree problems are "compute something bottom-up and update a global best along the way" — diameter, max path sum, and balanced-tree checks all share this shape.

Q27. Validate a Binary Search Tree

ROI: ★★★★★ Frequency: Very High

The common bug is only comparing a node with its direct children. A BST requires every node be within a valid (min, max) range inherited from ancestors — thread the bounds down the recursion. 

boolean isValid(Node n, long min, long max) {
    if (n == null) return true;
    if (n.val <= min || n.val >= max) return false;
    return isValid(n.left, min, n.val) && isValid(n.right, n.val, max);
}
// call: isValid(root, Long.MIN_VALUE, Long.MAX_VALUE)
// Time O(n), Space O(h)

Interview Gold Point

Alternative: an in-order traversal of a valid BST is strictly increasing — track the previous value and fail if it ever doesn't increase. Use long bounds to avoid overflow when node values hit Integer.MIN/MAX_VALUE.

Q28. Lowest Common Ancestor (LCA)

ROI: ★★★★☆ Frequency: High

In a general binary tree, recurse: if a node is p or q, return it; otherwise recurse both sides. If both sides return non-null, this node is the LCA. In a BST it's simpler — walk down, going left/right until p and q split. 

// BST version - O(h)
Node lcaBST(Node root, Node p, Node q) {
    while (root != null) {
        if (p.val < root.val && q.val < root.val) root = root.left;
        else if (p.val > root.val && q.val > root.val) root = root.right;
        else return root;     // split point = LCA
    }
    return null;
}

Interview Favorite

Always ask "is it a BST?" first — the BST property turns an O(n) general-tree recursion into a simple O(h) walk. Recognising when extra structure simplifies the problem is a senior skill.

Q29. BST Search and Insert

ROI: ★★★★☆ Frequency: Medium-High

A BST gives O(h) search/insert — O(log n) when balanced, O(n) when degenerate (inserting sorted data builds a linked list). At each node, go left if smaller, right if larger. 

Node insert(Node root, int val) {
    if (root == null) return new Node(val);
    if (val < root.val) root.left  = insert(root.left, val);
    else                root.right = insert(root.right, val);
    return root;
}

Interview Gold Point

This is exactly why self-balancing trees exist: Java's TreeMap/TreeSet are Red-Black trees that guarantee O(log n) by rebalancing on insert/delete, avoiding the degenerate case. Reach for them when you need sorted keys with log-time operations.

SECTION 7 - HEAPS / PRIORITY QUEUE

Q30. What Is a Heap and When Do You Use a PriorityQueue?

ROI: ★★★★★ Frequency: High

A heap is a tree giving O(1) access to the min (or max) and O(log n) insert/remove. Java's PriorityQueue is a binary min-heap by default. Use it whenever you repeatedly need the smallest/largest of a changing set — "top K", "kth largest", "merge K", scheduling, Dijkstra. 

PriorityQueue<Integer> minHeap = new PriorityQueue<>();
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Comparator.reverseOrder());
minHeap.offer(5); minHeap.offer(1); minHeap.peek(); // 1, O(1)
minHeap.poll();                                      // remove min, O(log n)

Interview Gold Point

Heap vs sorting: if you only need the K best, a size-K heap is O(n log K), beating a full O(n log n) sort. "Top K / Kth" in the problem statement is the heap trigger.

Q31. Kth Largest Element & Top K Frequent

ROI: ★★★★★ Frequency: Very High

For the kth largest, keep a min-heap of size k: the heap's top is the kth largest once you've processed everything. For top-K-frequent, count with a HashMap, then heap by frequency. 

int findKthLargest(int[] nums, int k) {
    PriorityQueue<Integer> heap = new PriorityQueue<>();  // min-heap
    for (int n : nums) {
        heap.offer(n);
        if (heap.size() > k) heap.poll();   // drop smallest
    }
    return heap.peek();                     // kth largest
}
// Time O(n log k), Space O(k)

Interview Favorite

Counter-intuitive but key: to find the kth largest, use a min-heap of size k (so the smallest of the top-k sits on top, ready to be evicted). Quickselect gives an average O(n) alternative if asked to optimise further.

Q32. Merge K Sorted Lists

ROI: ★★★★☆ Frequency: High

Put the head of each list into a min-heap, then repeatedly poll the smallest and push its successor. This is O(N log k) where N is total nodes and k is the number of lists — far better than concatenate-and-sort. 

Node mergeKLists(Node[] lists) {
    PriorityQueue<Node> pq = new PriorityQueue<>((a,b) -> a.val - b.val);
    for (Node l : lists) if (l != null) pq.offer(l);
    Node dummy = new Node(0), tail = dummy;
    while (!pq.isEmpty()) {
        Node n = pq.poll();
        tail.next = n; tail = n;
        if (n.next != null) pq.offer(n.next);
    }
    return dummy.next;
}
// Time O(N log k), Space O(k)

Interview Gold Point

The heap holds only k nodes at a time (one per list), so space is O(k), not O(N). This same "k-way merge with a heap" powers external merge sort for data too big for memory.

SECTION 8 - GRAPHS

Q33. How Do You Represent a Graph?

ROI: ★★★★★ Frequency: High

Two standard representations:

  • Adjacency list — Map<Node, List<Node>> or array of lists. Space O(V+E), efficient for sparse graphs (most real graphs). The default choice.
  • Adjacency matrix — boolean[V][V]. O(1) edge lookup but O(V²) space; only for dense graphs or when you constantly test "is there an edge?".
Map<Integer, List<Integer>> graph = new HashMap<>();
graph.computeIfAbsent(u, k -> new ArrayList<>()).add(v);
graph.computeIfAbsent(v, k -> new ArrayList<>()).add(u);  // undirected

Interview Gold Point

Default to an adjacency list unless told the graph is dense. Many "grid" problems (islands, maze) are implicit graphs where neighbours are the four adjacent cells — no explicit structure needed.

Q34. Graph BFS and DFS — and Number of Islands

ROI: ★★★★★ Frequency: Very High

DFS (stack/recursion) goes deep; BFS (queue) explores level by level and finds the shortest path in an unweighted graph. Both need a visited set to avoid infinite loops on cycles. "Number of Islands" is a grid-DFS classic: each unvisited land cell starts a flood-fill. 

int numIslands(char[][] grid) {
    int count = 0;
    for (int r = 0; r < grid.length; r++)
        for (int c = 0; c < grid[0].length; c++)
            if (grid[r][c] == '1') { dfs(grid, r, c); count++; }
    return count;
}
void dfs(char[][] g, int r, int c) {
    if (r < 0 || c < 0 || r >= g.length || c >= g[0].length || g[r][c] != '1') return;
    g[r][c] = '0';                       // mark visited
    dfs(g,r+1,c); dfs(g,r-1,c); dfs(g,r,c+1); dfs(g,r,c-1);
}
// Time O(rows · cols), Space O(rows · cols) worst-case recursion

Interview Favorite

Forgetting the visited check is the #1 graph bug — it causes infinite loops. Use BFS when the question says "shortest" / "fewest steps"; use DFS for connectivity, flood-fill, and path-existence.

Q35. Topological Sort (and Cycle Detection in a DAG)

ROI: ★★★★☆ Frequency: Medium-High

A topological sort orders the nodes of a directed acyclic graph so every edge points forward — used for build/dependency ordering and course scheduling. Kahn's algorithm uses in-degrees and a queue; if you can't process all nodes, the graph has a cycle. 

int[] topoSort(int n, int[][] edges) {
    List<List<Integer>> adj = new ArrayList<>();
    int[] indeg = new int[n];
    for (int i = 0; i < n; i++) adj.add(new ArrayList<>());
    for (int[] e : edges) { adj.get(e[0]).add(e[1]); indeg[e[1]]++; }
    Queue<Integer> q = new LinkedList<>();
    for (int i = 0; i < n; i++) if (indeg[i] == 0) q.add(i);
    int[] order = new int[n]; int idx = 0;
    while (!q.isEmpty()) {
        int u = q.poll(); order[idx++] = u;
        for (int v : adj.get(u)) if (--indeg[v] == 0) q.add(v);
    }
    return idx == n ? order : new int[0];   // empty => cycle
}

Interview Gold Point

"Course schedule", "build order", "task dependencies" all map to topological sort. The fact that incomplete processing reveals a cycle is the standard way to detect cycles in a directed graph.

Q36. Dijkstra's Shortest Path (Weighted Graph)

ROI: ★★★★☆ Frequency: Medium

Dijkstra finds shortest paths from a source in a graph with non-negative edge weights. It's BFS with a min-heap (priority queue) ordering by cumulative distance — always expand the closest unfinalised node next. Complexity O((V+E) log V). 

int[] dijkstra(List<int[]>[] adj, int src, int n) {   // adj[u] = {v, weight}
    int[] dist = new int[n]; Arrays.fill(dist, Integer.MAX_VALUE);
    dist[src] = 0;
    PriorityQueue<int[]> pq = new PriorityQueue<>((a,b) -> a[1] - b[1]); // {node, dist}
    pq.offer(new int[]{src, 0});
    while (!pq.isEmpty()) {
        int[] cur = pq.poll(); int u = cur[0], d = cur[1];
        if (d > dist[u]) continue;                  // stale entry
        for (int[] e : adj[u]) {
            int v = e[0], nd = d + e[1];
            if (nd < dist[v]) { dist[v] = nd; pq.offer(new int[]{v, nd}); }
        }
    }
    return dist;
}

Interview Favorite

Dijkstra fails with negative weights — use Bellman-Ford there. For unweighted graphs, plain BFS already gives shortest paths (every edge weight is 1), so don't over-engineer.

Q37. Union-Find (Disjoint Set Union)

ROI: ★★★★☆ Frequency: Medium

Union-Find tracks connectivity between elements with near-O(1) find and union (with path compression + union by rank). It answers "are these connected?" and detects cycles in an undirected graph — the backbone of Kruskal's MST. 

class DSU {
    int[] parent, rank;
    DSU(int n) { parent = new int[n]; rank = new int[n];
        for (int i = 0; i < n; i++) parent[i] = i; }
    int find(int x) {
        if (parent[x] != x) parent[x] = find(parent[x]);  // path compression
        return parent[x];
    }
    boolean union(int a, int b) {
        int ra = find(a), rb = find(b);
        if (ra == rb) return false;                        // already connected (cycle)
        if (rank[ra] < rank[rb]) { int t=ra; ra=rb; rb=t; }
        parent[rb] = ra;
        if (rank[ra] == rank[rb]) rank[ra]++;
        return true;
    }
}

Interview Gold Point

Reach for Union-Find on "number of connected components", "redundant connection / detect cycle in undirected graph", and "accounts merge". With both optimisations, operations are effectively O(α(n)) — inverse-Ackermann, practically constant.

SECTION 9 - RECURSION & BACKTRACKING

Q38. What Makes a Correct Recursive Solution?

ROI: ★★★★☆ Frequency: Medium-High

Every recursion needs a base case that stops it and a recursive case that makes progress toward that base. Trust that the recursive call solves the smaller problem ("the leap of faith"), then combine. Miss the base case and you get a StackOverflowError.

Interview Gold Point

Java has no tail-call optimisation, so deep linear recursion overflows. For very deep inputs convert to iteration with an explicit stack, or restructure to balanced (divide-and-conquer) recursion.

Q39. Backtracking Template — Subsets, Permutations, Combinations

ROI: ★★★★★ Frequency: High

Backtracking explores a decision tree: choose → recurse → un-choose. It generates all subsets/permutations/combinations and solves constraint problems (N-Queens, Sudoku) by abandoning branches that can't lead to a solution ("pruning"). 

// All subsets (the power set)
List<List<Integer>> subsets(int[] nums) {
    List<List<Integer>> res = new ArrayList<>();
    backtrack(nums, 0, new ArrayList<>(), res);
    return res;
}
void backtrack(int[] nums, int start, List<Integer> path, List<List<Integer>> res) {
    res.add(new ArrayList<>(path));            // record current subset
    for (int i = start; i < nums.length; i++) {
        path.add(nums[i]);                     // choose
        backtrack(nums, i + 1, path, res);     // recurse
        path.remove(path.size() - 1);          // un-choose
    }
}
// Subsets: O(2^n) results; permutations: O(n!)

Interview Gold Point

The choose / recurse / un-choose skeleton is identical across all backtracking problems — only the choices, the base case, and the pruning condition change. Always copy the path (new ArrayList<>(path)) when recording, or every result points at the same mutating list.

SECTION 10 - DYNAMIC PROGRAMMING

Q40. How Do You Recognise and Approach a DP Problem?

ROI: ★★★★★ Frequency: Very High

DP applies when a problem has overlapping subproblems (the same smaller problems recur) and optimal substructure (the optimal answer builds from optimal sub-answers). Signals: "count the number of ways", "minimum/maximum cost", "can you reach", "longest/shortest".

Approach: (1) define the state, (2) write the recurrence/transition, (3) identify base cases, (4) choose memoization (top-down) or tabulation (bottom-up), (5) optimise space if each state only needs the previous row.

Interview Gold Point

Start by writing the brute-force recursion, notice the repeated calls, then add a cache — that's memoization, and it's the easiest way to derive the DP under pressure. Convert to bottom-up tabulation only if you need to drop the recursion stack.

Q41. 1D DP — Climbing Stairs / House Robber

ROI: ★★★★☆ Frequency: High

Many 1D DP problems reduce to "each state depends on the previous one or two". Climbing stairs (ways to reach step n taking 1 or 2 steps) is literally Fibonacci. House Robber (max sum, no two adjacent) is dp[i] = max(skip i, rob i + dp[i-2]). 

int rob(int[] nums) {
    int prev2 = 0, prev1 = 0;
    for (int n : nums) {
        int cur = Math.max(prev1, prev2 + n);   // skip vs rob
        prev2 = prev1; prev1 = cur;
    }
    return prev1;
}
// Time O(n), Space O(1)

Interview Favorite

When the recurrence only looks back a fixed number of steps, you can drop the array and keep two or three rolling variables — O(1) space. Demonstrating that optimisation after the O(n)-space version is a strong close.

Q42. 0/1 Knapsack

ROI: ★★★★☆ Frequency: High

Given items with weights and values and a capacity, maximise value without exceeding capacity; each item is taken or not (0/1). State: dp[i][w] = best value using the first i items within capacity w. Transition: skip the item, or take it (if it fits) and add its value. 

int knapsack(int[] wt, int[] val, int cap) {
    int n = wt.length;
    int[] dp = new int[cap + 1];                 // space-optimised to 1D
    for (int i = 0; i < n; i++)
        for (int w = cap; w >= wt[i]; w--)        // iterate weight DESC for 0/1
            dp[w] = Math.max(dp[w], dp[w - wt[i]] + val[i]);
    return dp[cap];
}
// Time O(n · cap), Space O(cap)

Interview Gold Point

The descending weight loop is what makes it 0/1 (each item used once); an ascending loop would allow reuse — that's the unbounded knapsack / coin-change variant. Knowing why the loop direction matters is the differentiator.

Q43. Coin Change & Longest Common Subsequence

ROI: ★★★★☆ Frequency: High

Coin Change (fewest coins to make an amount) is unbounded knapsack: dp[a] = 1 + min over coins of dp[a - coin]. LCS (longest common subsequence of two strings) is the canonical 2D string DP: if characters match, extend the diagonal; else take the better of dropping one char from either string. 

int coinChange(int[] coins, int amount) {
    int[] dp = new int[amount + 1];
    Arrays.fill(dp, amount + 1);                 // "infinity"
    dp[0] = 0;
    for (int a = 1; a <= amount; a++)
        for (int c : coins)
            if (c <= a) dp[a] = Math.min(dp[a], dp[a - c] + 1);
    return dp[amount] > amount ? -1 : dp[amount];
}
// Time O(amount · coins), Space O(amount)

Interview Favorite

LCS is the parent of edit distance, diff tools, and longest-common-substring. Recognising "two sequences, find a common/aligned structure" → 2D DP with an (i, j) grid is the reusable insight.

SECTION 11 - SEARCHING & SORTING

Q44. Binary Search and Its Variants

ROI: ★★★★★ Frequency: Very High

Binary search finds a target in a sorted array in O(log n) by halving the range. The variants — first/last occurrence, insertion point, search in a rotated array — are the same skeleton with adjusted boundary logic. Use lo + (hi - lo) / 2 to avoid overflow. 

int binarySearch(int[] a, int target) {
    int lo = 0, hi = a.length - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if (a[mid] == target) return mid;
        if (a[mid] < target) lo = mid + 1; else hi = mid - 1;
    }
    return -1;
}

Interview Gold Point

"Binary search the answer" is a powerful senior pattern: when the answer is monotonic (e.g. "minimum capacity to ship in D days"), binary-search over the answer space, not the array. Watch the <= vs < boundary and the overflow-safe midpoint — the two classic bugs.

Q45. How Do Quick Sort and Merge Sort Work?

ROI: ★★★★☆ Frequency: High

Quick sort picks a pivot, partitions elements around it, and recurses — average O(n log n), worst O(n²) (bad pivots), in-place, not stable. Merge sort splits in half, sorts each, and merges — guaranteed O(n log n), stable, but needs O(n) extra space.

AlgorithmAvgWorstStableSpace
QuickO(n log n)O(n²)NoO(log n)
MergeO(n log n)O(n log n)YesO(n)
HeapO(n log n)O(n log n)NoO(1)

Interview Gold Point

Java's Arrays.sort uses dual-pivot quicksort for primitives (speed; stability irrelevant) and Timsort for objects (a stable merge/insertion hybrid). Stability matters for objects (sorting by one field while preserving another's order), not primitives — that's why they differ.

SECTION 12 - GREEDY & BIT MANIPULATION

Q46. What Is a Greedy Algorithm? (Interval Scheduling)

ROI: ★★★★☆ Frequency: Medium-High

A greedy algorithm makes the locally optimal choice at each step, hoping it yields a global optimum. It's correct only when the problem has the greedy-choice property — e.g. activity selection: to fit the most non-overlapping intervals, always pick the one that ends earliest. 

int maxActivities(int[][] intervals) {
    Arrays.sort(intervals, (a,b) -> a[1] - b[1]);  // by end time
    int count = 0, lastEnd = Integer.MIN_VALUE;
    for (int[] iv : intervals)
        if (iv[0] >= lastEnd) { count++; lastEnd = iv[1]; }
    return count;
}

Interview Gold Point

Greedy is fast but often wrong — coin change with arbitrary denominations needs DP, not greedy. The senior move is to state why greedy is valid here (the exchange argument), or to recognise when it isn't and switch to DP.

Q47. Common Bit Manipulation Tricks

ROI: ★★★☆☆ Frequency: Medium

Bit tricks give O(1) operations and are occasionally the elegant answer:

  • x & 1 — is x odd?
  • x >> 1 / x << 1 — divide / multiply by 2
  • x & (x - 1) — clears the lowest set bit (count bits / check power of two)
  • a ^ b — XOR; x ^ x = 0, so XOR-ing all elements finds the single non-duplicated number
// "Single Number": every element appears twice except one
int singleNumber(int[] nums) {
    int x = 0;
    for (int n : nums) x ^= n;   // pairs cancel, the unique survives
    return x;
}
// Time O(n), Space O(1)
boolean isPowerOfTwo(int n) { return n > 0 && (n & (n - 1)) == 0; }

Interview Favorite

The XOR "single number" trick (O(1) space, no extra structure) and n & (n-1) for power-of-two / bit counting are the two that come up most. Don't force bit tricks where a HashMap is clearer — use them when O(1) space is required.

Q49. Common Java Gotchas in Coding Interviews

ROI: ★★★★☆ Frequency: High

  • Integer overflow — int sums/midpoints overflow silently; use long or lo + (hi-lo)/2.
  • == vs equals — == compares references for objects; use .equals(). Watch Integer auto-boxing caching (-128..127).
  • Array vs List — arrays use .length, Lists use .size(), Strings use .length().
  • char arithmetic — s.charAt(i) - 'a' indexes a 26-count array cleanly.
  • Custom keys — objects used as HashMap keys need consistent equals() + hashCode().

Interview Gold Point

Overflow is the most common silent bug, especially in binary search and sum problems. Saying "I'll use long here to be safe against overflow" unprompted is a small detail that signals real production experience.
