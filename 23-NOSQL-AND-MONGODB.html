NoSQL questions show up the moment an interviewer wants to test whether you can pick the right tool instead of forcing everything into a relational schema. For 4+ years of backend experience, knowing only SQL is no longer enough. NoSQL does not replace SQL knowledge (covered in the SQL & Database Design chapter) - it complements it. The relational world gives you strong schemas, joins and ACID; NoSQL gives you flexible schemas, horizontal scale and data models that match your access patterns. For 4+ years experience, interviewers expect: * Knowing why NoSQL emerged and the four families * When to choose NoSQL over a relational database * MongoDB document modelling (embedding vs referencing) * Consistency trade-offs (ACID vs BASE, eventual consistency) * Indexing and the aggregation pipeline * Scaling with replica sets and sharding * Spring Data MongoDB in practice They usually continue with deeper questions on data modelling, consistency and how you would scale the store under real production load.

SECTION 1 - NOSQL FUNDAMENTALS

Q1. What Is NoSQL And Why Did It Emerge?

ROI: ★★★★★ Frequency: Very High

Short Answer

NoSQL ("Not Only SQL") is a family of databases that relax the rigid relational model - fixed schemas, normalized tables and joins - in favour of flexible data models that scale horizontally across commodity machines.

It emerged in the mid-2000s when companies like Google, Amazon and Facebook hit the limits of a single relational server. The pain points were concrete:

  • Scale - relational databases scale vertically (bigger box); web-scale workloads needed to scale horizontally across many cheap nodes.
  • Schema rigidity - rapidly evolving products did not want a migration for every new field.
  • Unstructured / semi-structured data - JSON-like documents, logs and graphs do not map cleanly to tables.
  • Throughput - high write volumes where the cost of joins and strict ACID became a bottleneck.

The key insight: you trade some of the relational guarantees (rich joins, strict consistency) for scalability and a data model that fits your access patterns.

Interview Gold Point

Never say "NoSQL is faster than SQL". It is not inherently faster - it is differently optimized. The correct framing is: NoSQL optimizes for horizontal scale and access-pattern-shaped data at the cost of joins and strong consistency. Saying that signals senior judgement.

Q2. What Are The Four Types Of NoSQL Databases?

ROI: ★★★★★ Frequency: Very High

Short Answer

There are four classic NoSQL families, each suited to a different shape of data and access pattern.

TypeStores Data AsExample DBGood For
DocumentJSON / BSON documentsMongoDBFlexible records, content, catalogs
Key-ValueKey → opaque valueRedisCaching, sessions, counters
Column-Family (wide-column)Rows with dynamic columns grouped in familiesCassandraHuge write-heavy time-series / analytics
GraphNodes and edgesNeo4jRelationships - social, fraud, recommendations

A document is essentially a self-contained nested record. A key-value store is the simplest model - a giant distributed hash map. Wide-column stores look table-ish but each row can have different columns and they scale writes massively. Graph databases make traversing relationships ("friends of friends") cheap, which is painful with joins in SQL.

Interview Favorite

Interviewers love "give me a real example for each". Redis for a session cache (see the caching content in the System Design chapter), MongoDB for a product catalog, Cassandra for IoT/event ingestion, Neo4j for a "people you may know" feature. Concrete examples beat definitions.

Q3. SQL vs NoSQL - When Do You Choose Which?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

Choose based on data shape, access patterns, consistency needs and scale - not hype. Most real systems use both (polyglot persistence).

DimensionSQL (Relational)NoSQL
SchemaFixed, enforcedFlexible / dynamic
ScalingVertical (mostly)Horizontal (built-in)
JoinsFirst-classLimited / avoided by design
TransactionsStrong ACIDVaries (often eventual / limited)
Best whenComplex relationships, strong consistency, reportingHigh scale, evolving schema, access-pattern-shaped data

Pick SQL when you have well-defined relationships, need multi-table transactions and strong consistency - banking, orders, inventory ledgers. Pick NoSQL when the schema evolves quickly, you need to scale out, or the data is naturally a self-contained document or simple key lookup.

The relational contrast (normalization, joins, ACID) is covered in depth in the SQL & Database Design chapter - this chapter is the complement to it.

Common Mistake

Saying "use NoSQL because it scales" without naming the access pattern. NoSQL scales because it avoids joins and lets you shard - but only if your data model matches your queries. If you need ad-hoc joins and reporting, NoSQL becomes painful. Always tie the choice to the read/write pattern.

SECTION 2 - CONSISTENCY MODELS

Q4. What Is The Difference Between ACID And BASE?

ROI: ★★★★☆ Frequency: High

Short Answer

ACID is the relational guarantee model; BASE is the philosophy many distributed NoSQL systems adopt to favour availability and scale.

ACIDBASE
AtomicityBasically Available
ConsistencySoft state
IsolationEventually consistent
Durability(prioritizes availability over strict consistency)

ACID guarantees every transaction is all-or-nothing, leaves the database in a valid state, is isolated from other transactions and survives crashes. BASE accepts that in a large distributed system the data may be temporarily inconsistent ("soft state") but will converge ("eventually consistent"), in exchange for staying available and scalable.

Note: MongoDB is ACID-compliant at the single-document level by default, and supports multi-document ACID transactions since 4.0 (see Q16). So "NoSQL = no ACID" is outdated.

Interview Gold Point

The senior nuance: ACID vs BASE is a spectrum, not a binary. Modern databases let you tune it - in MongoDB you choose write concern and read preference per operation (Q16), effectively dialing where you sit between strong and eventual consistency.

Q5. How Does The CAP Theorem Apply To NoSQL?

ROI: ★★★★☆ Frequency: High

Short Answer

CAP states that during a network Partition a distributed system can guarantee only Consistency or Availability, not both. Since partitions are unavoidable in a distributed store, you are really choosing between CP and AP when the network splits.

  • CP (Consistency + Partition tolerance) - reject reads/writes that cannot be made consistent. MongoDB leans CP: during a partition the primary side stays consistent, the minority side cannot accept writes.
  • AP (Availability + Partition tolerance) - keep serving and reconcile later. Cassandra leans AP with tunable consistency.

CAP is covered in full in the Microservices chapter (it is central to the database-per-service pattern). For NoSQL the practical takeaway is: read the docs to learn where your store sits, because that decision is baked into its replication design.

Common Mistake

Treating CAP as "pick 2 of 3 always". Partition tolerance is not optional in a distributed system - you only sacrifice C or A while a partition is happening. The rest of the time you can have both. Stating it as a runtime trade-off, not a permanent one, is the senior answer.

Q6. What Is Eventual Consistency And What Are Its Trade-offs?

ROI: ★★★★☆ Frequency: High

Short Answer

Eventual consistency means that if no new writes occur, all replicas will eventually return the same value - but for a short window after a write, different replicas may return stale data.

Trade-offs:

  • Pro - high availability and low latency; reads can be served by any replica without coordination.
  • Con - stale reads. A user may update their profile and momentarily see the old value if they read from a lagging secondary.
  • Con - conflict resolution. Concurrent writes to different replicas need a strategy (last-write-wins, vector clocks, application merge).

You mitigate it where it matters: in MongoDB, read from the primary (read preference) or require majority write concern for the data that must be read-your-own-write consistent (e.g. payment status), while letting low-stakes data (e.g. view counts) stay eventually consistent.

Interview Favorite

The killer example: "Why is your Amazon order count sometimes off for a second?" - eventual consistency. Showing you know which data tolerates staleness (analytics, counters) versus which must be strongly consistent (balances, inventory) is exactly the judgement interviewers screen for.

SECTION 3 - MONGODB BASICS

Q7. What Are Documents, Collections And BSON In MongoDB?

ROI: ★★★★★ Frequency: Very High

Short Answer

These are MongoDB's core building blocks:

  • Document - a single record stored as a JSON-like structure with fields, nested objects and arrays. It is the unit of storage and roughly maps to a "row".
  • Collection - a group of documents, roughly maps to a "table". Collections are schema-flexible: documents in the same collection can have different fields.
  • BSON - "Binary JSON", the binary-encoded format MongoDB actually stores and transmits. It extends JSON with extra types (ObjectId, Date, Decimal128, Binary, 64-bit int) and is faster to traverse and more compact than text JSON.

Example document: 

{
  "_id": ObjectId("64f0a1c2e13b9a0012ab34cd"),
  "name": "Nasir Khan",
  "email": "nasir@growth-loop.io",
  "roles": ["ADMIN", "USER"],
  "address": {
    "city": "Bengaluru",
    "country": "IN"
  },
  "createdAt": ISODate("2026-06-16T10:00:00Z")
}

Notice the nested address object and the roles array embedded directly in the record - in a relational model these would be separate tables joined at query time.

Interview Gold Point

Be precise: JSON is what you read/write at the API level, BSON is what is stored on disk and on the wire. The driver converts between them. Mentioning that BSON adds typed values like Date and Decimal128 (not in plain JSON) signals you understand the storage layer.

Q8. What Is The _id Field In MongoDB?

ROI: ★★★★☆ Frequency: High

Short Answer

_id is the mandatory primary key of every MongoDB document. It is unique within a collection, automatically indexed, and immutable once set.

  • If you do not supply one, MongoDB generates an ObjectId - a 12-byte value containing a 4-byte timestamp, a 5-byte random value (machine/process) and a 3-byte incrementing counter.
  • Because the first 4 bytes are a timestamp, ObjectIds are roughly time-ordered - you can sort by _id to get near-insertion order and even extract the creation time.
  • You may use your own value (a UUID, an email, a natural key) as _id as long as it is unique.

Interview Favorite

The bonus point: an ObjectId is not sequential like an auto-increment integer. That is deliberate - sequential keys create a write hotspot when sharding (all inserts hit the same shard). ObjectId's randomness spreads writes. Connecting _id design to shard distribution is a strong senior signal.

Q9. How Does A MongoDB Document Map Versus A Relational Row?

ROI: ★★★★☆ Frequency: High

Short Answer

The terms map loosely, but the modelling philosophy is fundamentally different.

RelationalMongoDB
DatabaseDatabase
TableCollection
RowDocument
ColumnField
JoinEmbedding (or $lookup)
Foreign KeyReference (manual)

The big difference: a relational row is flat and references other tables; a Mongo document can nest related data inside itself. An order with its line items might be five rows across three relational tables, but one self-contained document in Mongo. You shape the document around how you read it, not around normalization rules.

Interview Gold Point

The one-liner: "In relational you model the data; in MongoDB you model the access pattern." That sentence captures the entire shift and almost always lands well.

SECTION 4 - DATA MODELING

Q10. Embedding vs Referencing - When Do You Denormalize?

ROI: ★★★★★ Frequency: Very High

Short Answer

This is the single most important MongoDB design decision. You either embed related data inside the parent document or reference it by storing its _id.

Embed WhenReference When
Data is read together (read locality)Data is read independently
One-to-few relationshipOne-to-many / many-to-many (unbounded)
Child has no life of its ownChild is shared or queried on its own
You want a single atomic read/writeDocument would grow unbounded (16MB limit)

Embed example (order + items together): 

{
  "_id": ObjectId("..."),
  "customer": "Nasir",
  "items": [
    { "sku": "A1", "qty": 2, "price": 19.99 },
    { "sku": "B7", "qty": 1, "price": 9.50 }
  ],
  "total": 49.48
}

Reference example (order points to a shared customer): 

{
  "_id": ObjectId("..."),
  "customerId": ObjectId("64ab..."),
  "items": [ /* ... */ ]
}

Denormalize (embed/duplicate) when reads dominate and the data is read together; normalize (reference) when data is large, shared, or grows without bound. Remember the hard limit: a single BSON document cannot exceed 16MB.

Interview Gold Point

The rule of thumb interviewers want: "embed for read performance, reference to avoid duplication and unbounded growth." Then mention the 16MB document cap as the hard constraint that forces referencing for true one-to-many relationships like comments on a viral post.

Q11. How Do You Model One-to-Many And Many-to-Many In MongoDB?

ROI: ★★★★☆ Frequency: High

Short Answer

One-to-many:

  • One-to-few (a person's few addresses) - embed as an array.
  • One-to-many (a blog post's comments) - embed if bounded, otherwise reference: store postId on each comment document in a separate collection.
  • One-to-squillions (server log lines for a host) - always reference from the "many" side; never embed an unbounded array in the parent.

Many-to-many (students ↔ courses): store an array of references on one or both sides, depending on which way you query most. 

// student document
{ "_id": 1, "name": "Ana", "courseIds": [101, 102] }

// course document
{ "_id": 101, "title": "Algorithms", "studentIds": [1, 5, 9] }

There is no join table. You hold arrays of _ids on the side you read from, and resolve them with a second query or $lookup in an aggregation.

Common Mistake

Embedding an unbounded "many" (e.g. all comments inside the post). It looks clean until the post goes viral, the document blows past 16MB, and every read drags megabytes over the wire. Match the cardinality to the strategy: few = embed, many = reference.

Q12. What Is The Schema-Design Mindset Shift From Relational?

ROI: ★★★★☆ Frequency: Medium

Short Answer

In relational design you normalize first (3NF), eliminate duplication, then write joins to assemble data. In MongoDB you do the opposite: you start from the queries and shape documents so the data you read together lives together.

  • Relational: "What is the cleanest normalized structure?" Schema first, queries follow.
  • MongoDB: "What are my read/write patterns?" Queries first, schema follows.
  • Duplication is acceptable (and sometimes desirable) if it saves a join on a hot read path - you trade storage and write complexity for read speed.
  • Schema is enforced by the application (or optional JSON Schema validation), not the engine.

This is why the SQL & Database Design chapter teaches normalization as the goal, while here denormalization is a deliberate, justified tool.

Interview Favorite

Say it crisply: "Relational optimizes for storage and integrity; MongoDB optimizes for the query." Add that you still validate at the application layer so flexibility does not become chaos - that balance is what separates a senior from someone who just dumps unstructured blobs.

SECTION 5 - QUERYING & PERFORMANCE

Q13. What Do Basic CRUD And Queries Look Like In MongoDB?

ROI: ★★★★☆ Frequency: High

Short Answer

MongoDB queries are expressed as documents - the filter is itself a JSON-like object. 

// Create
db.users.insertOne({ name: "Ana", age: 30, city: "Pune" })

// Read - find all users in Pune older than 25
db.users.find({ city: "Pune", age: { $gt: 25 } })

// Read - projection (only name + email), sorted, limited
db.users.find({ city: "Pune" }, { name: 1, email: 1, _id: 0 })
        .sort({ age: -1 })
        .limit(10)

// Update
db.users.updateOne(
  { name: "Ana" },
  { $set: { city: "Mumbai" }, $inc: { age: 1 } }
)

// Delete
db.users.deleteOne({ name: "Ana" })

Operators are prefixed with $: comparison ($gt, $lt, $in), logical ($and, $or), and update operators ($set, $inc, $push). The projection document selects fields (1 = include, 0 = exclude).

Interview Gold Point

A frequent gotcha: updateOne with $set updates only the listed fields, but passing a whole document without $set replaces the entire document (losing other fields). Knowing the difference between modify and replace semantics shows real hands-on experience.

Q14. How Do Indexes Work In MongoDB And Why Do They Matter?

ROI: ★★★★★ Frequency: Very High

Short Answer

Like in relational databases, indexes are B-tree structures that let MongoDB locate documents without scanning the whole collection. Without an index, a query does a COLLSCAN (full collection scan) - fine for small data, fatal at scale.

  • Single-field index - on one field: db.users.createIndex({ email: 1 }).
  • Compound index - on multiple fields, order matters: db.users.createIndex({ city: 1, age: -1 }). It supports queries on city, or city + age, but not age alone (the "prefix rule").
  • Other types - multikey (on array fields), text, geospatial, hashed (for sharding), TTL (auto-expire documents).
  • _id is always indexed automatically.

You diagnose query performance with explain(): an IXSCAN stage means the index is used; a COLLSCAN means it is not. 

db.users.find({ city: "Pune", age: { $gt: 25 } }).explain("executionStats")

Common Mistake

Forgetting the compound-index prefix rule and over-indexing. Indexes speed reads but slow writes and consume RAM - every write must update every index. The senior answer: index to match your actual query patterns (verified with explain()), not "just in case". This mirrors the indexing discussion in the SQL & Database Design chapter.

Q15. What Is The Aggregation Pipeline?

ROI: ★★★★★ Frequency: Very High

Short Answer

The aggregation pipeline is MongoDB's framework for data processing and analytics - the equivalent of SQL's GROUP BY, joins and transformations. Documents flow through an ordered array of stages, each transforming the stream and passing it to the next.

Common stages: $match (filter, like WHERE), $group (aggregate), $sort, $project (reshape, like SELECT), $lookup (left join to another collection), $unwind (flatten arrays), $limit.

Example - total revenue per city for completed orders: 

db.orders.aggregate([
  { $match: { status: "COMPLETED" } },
  { $group: {
      _id: "$city",
      totalRevenue: { $sum: "$total" },
      orderCount: { $sum: 1 }
  }},
  { $sort: { totalRevenue: -1 } },
  { $limit: 5 }
])

This is the SQL equivalent of SELECT city, SUM(total), COUNT(*) FROM orders WHERE status='COMPLETED' GROUP BY city ORDER BY 2 DESC LIMIT 5.

Interview Gold Point

Performance tip worth dropping: put $match and $sort as early as possible so they can use indexes and shrink the dataset before expensive stages like $group or $lookup. Stage ordering is to aggregation what query plans are to SQL.

SECTION 6 - SCALING & SPRING

Q16. Explain Replica Sets, Sharding, Transactions, Write/Read Concern And Spring Data MongoDB.

ROI: ★★★★★ Frequency: Very High

Short Answer

This bundle covers how MongoDB achieves high availability, scale and durability, plus how you use it from Spring.

Replica Set (high availability) - a group of mongod nodes holding the same data: one primary (takes all writes) and one or more secondaries (replicate the primary's oplog). If the primary fails, an automatic election promotes a secondary. This gives failover and redundancy. Secondaries can also serve reads (eventual consistency).

Sharding (horizontal scale) - partitions a collection across multiple shards by a shard key, so data and load spread across machines. Each shard is itself usually a replica set. Used when one server cannot hold the data or handle the write throughput. Choosing a good shard key (high cardinality, even distribution, matches query pattern) is critical to avoid hotspots.

Write concern & read preference - how you tune consistency vs latency:

  • Write concern - how many nodes must acknowledge a write. w:1 = primary only (fast, less durable); w:"majority" = majority of nodes (durable, survives failover).
  • Read preference - where reads go. primary (strong/fresh) vs secondaryPreferred (scale reads, may be stale).

Multi-document transactions - MongoDB always had atomic single-document writes, but multi-document ACID transactions arrived in 4.0 (2018) for replica sets (and 4.2 for sharded clusters). This was a big deal because it removed the last "but it's not ACID" objection - you can now update several documents atomically when you genuinely need it. The guidance is still to model so you rarely need them.

Spring Data MongoDB - gives you a repository abstraction just like Spring Data JPA (covered in the Spring Data JPA / Hibernate chapter). You extend MongoRepository and derived query methods are generated for you: 

@Document(collection = "users")
public class User {
    @Id
    private String id;          // maps to _id
    private String name;
    private String city;
    private int age;
    // getters / setters
}

public interface UserRepository extends MongoRepository<User, String> {
    List<User> findByCity(String city);
    List<User> findByCityAndAgeGreaterThan(String city, int age);
}

@Service
public class UserService {
    private final UserRepository repo;
    public UserService(UserRepository repo) { this.repo = repo; }

    public List<User> usersInCity(String city) {
        return repo.findByCity(city);   // generates the Mongo query
    }
}

For anything beyond derived queries you use MongoTemplate (criteria queries, aggregations, bulk ops).

Breadth (mention these for range): Redis is an in-memory key-value store, ideal for caching, sessions and rate limiting - the caching strategy details (cache-aside, TTL, eviction) are in the System Design chapter. Cassandra is a wide-column store built for massive write-heavy, multi-datacenter workloads with tunable (AP-leaning) consistency. Naming the right tool for each job shows polyglot-persistence maturity.

Interview Gold Point

Tie it together: "Replica sets give me availability, sharding gives me scale, write concern and read preference let me dial consistency per operation, and transactions are there for the rare cases my document model can't make atomic." That one sentence demonstrates you understand the whole operational picture, not just the API.
