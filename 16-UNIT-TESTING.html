16 - UNIT TESTING INTERVIEW HANDBOOK

Testing is one of the most reliable ways an interviewer separates a coder from an engineer. For a 4-year Senior Software Engineer role, you are expected to write code that is testable by design, not just code that "works on my machine". Interviewers use this topic to evaluate: * Whether you understand the difference between unit, integration and end-to-end tests * Whether you can isolate a unit using mocks and stubs * Whether you know JUnit 5 and Mockito well enough to use them daily * Whether you write tests that document behaviour, not just chase coverage numbers They rarely stop at "what is a unit test". They usually continue with: "How would you test a service that calls a repository and an external API?"

SECTION 1 - TESTING FUNDAMENTALS

Q1. What Is the Difference Between Unit, Integration and End-to-End Tests?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

A unit test verifies a single piece of logic (usually one class or method) in complete isolation, with all its collaborators replaced by mocks. It is fast and deterministic.

An integration test verifies that several real components work together correctly — for example a service talking to a real in-memory database, or a controller wired through the actual Spring context.

An end-to-end (E2E) test exercises the whole system from the outside, as a real user or client would: HTTP request in, response and side effects out, across every layer.

The Testing Pyramid

Level Scope Speed How Many
Unit One class, mocked deps Milliseconds Many (base)
Integration Few real components Seconds Some (middle)
End-to-End Whole system Slow Few (top)

The pyramid shape is the key insight: you want a large base of fast unit tests, fewer integration tests, and only a handful of slow E2E tests. Inverting this (mostly slow E2E tests) produces a suite that is brittle, slow and expensive to maintain.

Interview Gold Point

A strong candidate says: "Unit tests tell you where something broke; E2E tests tell you that something broke." You want both, but you lean on unit tests for fast feedback during development and reserve E2E tests for verifying critical user journeys.

Q2. What Makes a Good Unit Test?

ROI: ★★★★★ Frequency: Very High

The FIRST Principles

  • Fast — runs in milliseconds so developers actually run it
  • Isolated / Independent — does not depend on other tests or shared state
  • Repeatable — same result every run, on any machine
  • Self-validating — passes or fails automatically, no manual inspection
  • Timely — written close to the production code, ideally first

A good unit test follows the Arrange-Act-Assert (AAA) structure, tests one behaviour, and reads like documentation. The test name should describe the scenario and the expected outcome, so a failure tells you what broke without opening the code.

Example

@Test
void shouldReturnDiscountedPrice_whenCustomerIsPremium() {
    // Arrange
    PricingService service = new PricingService();
    Customer customer = new Customer(CustomerType.PREMIUM);

    // Act
    double price = service.priceFor(customer, 100.0);

    // Assert
    assertEquals(90.0, price);
}

Common Mistake

Tests that assert on many unrelated things at once. If a test breaks, you should know exactly which behaviour failed. One logical assertion per test (which may be several physical assertEquals on the same outcome) keeps failures meaningful.

SECTION 2 - JUNIT 5

Q3. JUnit 4 vs JUnit 5 — What Changed?

ROI: ★★★★☆ Frequency: High

JUnit 5 (also called Jupiter) is a complete rewrite, split into three modules: Platform (the engine that runs tests), Jupiter (the new programming model and annotations), and Vintage (backward compatibility to run old JUnit 4 tests).

Concept JUnit 4 JUnit 5
Run before each test @Before @BeforeEach
Run once before all @BeforeClass @BeforeAll
Ignore a test @Ignore @Disabled
Expected exception @Test(expected=...) assertThrows(...)
Extensions @RunWith / @Rule @ExtendWith

Interview Favorite

In JUnit 5, package-private test methods and classes are fine — you no longer need public. Also, the single extensible @ExtendWith model replaced JUnit 4's two competing mechanisms (Runners and Rules), which could not be combined.

Q4. Explain the JUnit 5 Lifecycle Annotations

ROI: ★★★★☆ Frequency: High

Lifecycle annotations control setup and teardown around your tests:

  • @BeforeAll — runs once before all tests (must be static by default); use for expensive shared setup
  • @BeforeEach — runs before every test; use to create fresh objects so tests stay isolated
  • @AfterEach — runs after every test; clean up per-test state
  • @AfterAll — runs once after all tests; release shared resources
class OrderServiceTest {

    static Database db;

    @BeforeAll
    static void initDb() { db = Database.connect(); }

    @BeforeEach
    void freshState() { db.clear(); }

    @Test
    void placesOrder() { /* ... */ }

    @AfterAll
    static void closeDb() { db.close(); }
}

Interview Gold Point

By default JUnit creates a new test instance per test method, which is why @BeforeAll must be static. If you annotate the class with @TestInstance(Lifecycle.PER_CLASS), one instance is reused and @BeforeAll can be non-static — useful but be careful about leaking state between tests.

Q5. How Do You Test That a Method Throws an Exception?

ROI: ★★★★☆ Frequency: High

Use assertThrows, which runs the lambda, asserts the exception type, and returns the caught exception so you can assert on its message too. 

@Test
void shouldThrow_whenAmountIsNegative() {
    Account account = new Account(100);

    IllegalArgumentException ex = assertThrows(
        IllegalArgumentException.class,
        () -> account.withdraw(-10)
    );

    assertEquals("Amount must be positive", ex.getMessage());
}

Why This Is Better Than JUnit 4

JUnit 4's @Test(expected = ...) could not pinpoint which line threw, and could not assert on the message. assertThrows scopes the expectation to exactly the call you care about, so a different line throwing the same exception will not cause a false pass.

Q6. What Are Parameterized Tests?

ROI: ★★★☆☆ Frequency: Medium

A parameterized test runs the same test logic with different inputs, avoiding copy-pasted near-identical tests. Use @ParameterizedTest with a source such as @ValueSource, @CsvSource or @MethodSource. 

@ParameterizedTest
@CsvSource({
    "2, 3, 5",
    "0, 0, 0",
    "-1, 1, 0"
})
void shouldAddCorrectly(int a, int b, int expected) {
    assertEquals(expected, calculator.add(a, b));
}

Interview Gold Point

Parameterized tests shine for boundary and equivalence-class testing. Mention that @MethodSource lets you supply complex objects or streams when a CSV string is not expressive enough.

SECTION 3 - MOCKING WITH MOCKITO

Q7. Mock vs Stub vs Spy vs Fake — What Is the Difference?

ROI: ★★★★★ Frequency: Very High

These are all "test doubles" — objects that stand in for real collaborators — but they differ in intent:

Double Purpose
Stub Returns canned answers to calls (state setup)
Mock A stub you also verify interactions on (behaviour)
Spy Wraps a real object; calls real methods unless stubbed
Fake A lightweight working implementation (e.g. in-memory DB)

The key distinction interviewers probe: a stub helps the test proceed by feeding it data, while a mock is used to assert that a particular interaction happened (for example, that emailService.send() was called exactly once).

Interview Gold Point

Prefer verifying outcomes (state) over verifying interactions (behaviour) where possible. Over-using verify() couples your test to the implementation, so any refactor breaks tests even when behaviour is unchanged.

Q8. Explain Mockito's when(), verify() and ArgumentCaptor

ROI: ★★★★★ Frequency: Very High

  • when(...).thenReturn(...) — stubs a method to return a value
  • verify(...) — asserts a method was called, optionally a specific number of times
  • ArgumentCaptor — captures the argument passed to a mock so you can assert on it
@Test
void shouldSaveUserWithEncodedPassword() {
    when(encoder.encode("raw")).thenReturn("hashed");

    userService.register("alice", "raw");

    ArgumentCaptor<User> captor = ArgumentCaptor.forClass(User.class);
    verify(userRepository, times(1)).save(captor.capture());

    assertEquals("hashed", captor.getValue().getPassword());
}

Interview Favorite

Know the difference between when().thenThrow() for stubbing exceptions and doThrow().when() — the latter is required for void methods, because you cannot pass a void call as an argument to when().

Q9. What Is the Difference Between @Mock and @InjectMocks?

ROI: ★★★★★ Frequency: Very High

@Mock creates a mock of a dependency. @InjectMocks creates a real instance of the class under test and injects the @Mock fields into it (via constructor, setter or field injection). 

@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    PaymentGateway gateway;       // dependency, faked

    @Mock
    OrderRepository repository;   // dependency, faked

    @InjectMocks
    OrderService orderService;    // real object, mocks injected

    @Test
    void placesOrderAndCharges() {
        when(gateway.charge(100)).thenReturn(true);

        orderService.place(new Order(100));

        verify(repository).save(any(Order.class));
    }
}

Interview Gold Point

@ExtendWith(MockitoExtension.class) is the JUnit 5 way to initialise these annotations. It also enables strict stubbing, which fails the test if you stub something that is never used — a great signal that a test is doing more setup than it needs.

SECTION 4 - TESTING SPRING BOOT

Q10. How Do You Unit Test a Spring Boot Service Layer?

ROI: ★★★★★ Frequency: Very High

A service-layer unit test should not load the Spring context at all. You construct the service directly (or via @InjectMocks), mock its repositories and clients, and assert on the business logic. Loading Spring would make it an integration test and slow it down by orders of magnitude.

Reserve @SpringBootTest for when you genuinely need the wired application context — it is the heaviest annotation and should be the exception, not the default.

Interview Gold Point

This is the most common real-world mistake: people sprinkle @SpringBootTest on every test "to be safe", turning a 2-second suite into a 5-minute one. A senior engineer mocks dependencies and keeps service tests pure.

Q11. When Do You Use @WebMvcTest vs @DataJpaTest vs @SpringBootTest?

ROI: ★★★★☆ Frequency: High

Spring Boot provides "test slices" that load only part of the context, keeping tests fast and focused:

Annotation Loads Use For
@WebMvcTest Web layer only (controllers, filters) Controller tests with MockMvc; services mocked
@DataJpaTest JPA layer + in-memory DB Repository / query tests
@SpringBootTest Full application context Integration / end-to-end tests 
@WebMvcTest(UserController.class)
class UserControllerTest {

    @Autowired MockMvc mockMvc;

    @MockBean UserService userService;  // service is mocked

    @Test
    void returns404_whenUserMissing() throws Exception {
        when(userService.find(1L)).thenReturn(Optional.empty());

        mockMvc.perform(get("/users/1"))
               .andExpect(status().isNotFound());
    }
}

Interview Favorite

Note @MockBean (Spring) vs @Mock (plain Mockito). @MockBean replaces a bean in the Spring context with a mock, which is exactly what you need inside a slice test like @WebMvcTest.

SECTION 5 - PRACTICES & PHILOSOPHY

Q12. Is 100% Code Coverage the Goal?

ROI: ★★★★☆ Frequency: High

No. Coverage measures which lines executed during tests, not whether the behaviour was actually asserted. You can reach 100% coverage with tests that assert nothing. Chasing a number leads to low-value tests on trivial getters while real edge cases go untested.

The healthy framing: coverage is a tool to find untested code, not a target to hit. Aim for meaningful tests on business logic and branches; a pragmatic team target is often 70–85%, with the gaps being deliberate (generated code, simple DTOs).

Interview Gold Point

Mention branch coverage vs line coverage. A single if can be 100% line-covered while only one branch is exercised. Branch coverage is the more honest metric for logic-heavy code.

Q13. What Is TDD and Have You Used It?

ROI: ★★★★☆ Frequency: High

Test-Driven Development follows the Red-Green-Refactor cycle:

  1. Red — write a failing test for the next small behaviour
  2. Green — write the minimum code to make it pass
  3. Refactor — clean up the code with the test as a safety net

The benefit is that it forces you to design testable, decoupled code and to think about the API from the caller's perspective before implementing it.

Honest Senior Answer

A credible answer admits TDD is not dogma: it shines for well-understood logic (algorithms, parsing, business rules) and is awkward for exploratory or UI work. Say you apply it where it adds value and write tests immediately after elsewhere — the non-negotiable is that the code ships with tests.

Q14. What Are Flaky Tests and How Do You Avoid Them?

ROI: ★★★★☆ Frequency: Medium-High

A flaky test passes sometimes and fails other times without any code change. They erode trust in the whole suite — once developers start re-running "just to be sure", the suite stops protecting you.

Common Causes & Fixes

  • Time / dates — inject a Clock instead of calling now()
  • Random data — seed the random generator
  • Shared state / order dependence — reset state in @BeforeEach
  • Real network / sleep-based timing — mock the dependency or use proper awaitility instead of Thread.sleep
  • Concurrency — avoid relying on thread scheduling for assertions

Interview Gold Point

Say you treat a flaky test as a bug, not a nuisance — quarantine it so it stops blocking the pipeline, but track and fix it. Silently re-running until green is how teams end up ignoring real failures.

Q15. How Do You Decide What to Test?

ROI: ★★★★☆ Frequency: Medium

Prioritise by risk and value:

  • Business logic and branches — the core rules and every decision path
  • Boundaries and edge cases — empty, null, min, max, off-by-one
  • Bug fixes — write a failing test that reproduces the bug first, so it never regresses
  • Public behaviour, not private methods — test through the public API; private methods get covered indirectly

What you generally don't unit-test: trivial getters/setters, framework code, and generated code — testing those adds maintenance cost with no real safety.

Interview Gold Point

"Test behaviour, not implementation." If your test names describe what the code should do (not how), you can refactor freely and the tests still guard the contract. That single principle is what keeps a test suite an asset rather than a liability.

SECTION 6 - INTEGRATION & ADVANCED TESTING

Q16. How Do You Write an Integration Test in Spring Boot?

ROI: ★★★★★ Frequency: Very High

Short Answer

An integration test loads a real (or partial) Spring context and verifies that components work together — controller, service, repository, and often a database. Use @SpringBootTest to boot the full context, and drive the app through HTTP with MockMvc (mock servlet) or TestRestTemplate/WebTestClient (real server when webEnvironment = RANDOM_PORT). 

@SpringBootTest(webEnvironment = WebEnvironment.RANDOM_PORT)
class OrderApiIntegrationTest {

    @Autowired TestRestTemplate rest;

    @Test
    void createsAndReturnsOrder() {
        ResponseEntity<Order> resp =
            rest.postForEntity("/orders", new Order(100), Order.class);
        assertEquals(HttpStatus.CREATED, resp.getStatusCode());
        assertNotNull(resp.getBody().getId());
    }
}

Interview Gold Point

Keep integration tests few and focused — they are slow because they boot the context. The test pyramid still applies: many fast unit tests at the base, a thin layer of integration tests verifying wiring and the critical happy paths.

Q17. What Is Testcontainers and Why Use It?

ROI: ★★★★★ Frequency: High

Short Answer

Testcontainers is a library that spins up real dependencies in throwaway Docker containers for your tests — a real Postgres, Kafka, or Redis — then tears them down afterwards. It replaces in-memory fakes (like H2) that behave subtly differently from production. 

@Testcontainers
@SpringBootTest
class RepositoryIT {

    @Container
    static PostgreSQLContainer<?> db =
        new PostgreSQLContainer<>("postgres:16");

    @DynamicPropertySource
    static void props(DynamicPropertyRegistry reg) {
        reg.add("spring.datasource.url", db::getJdbcUrl);
        reg.add("spring.datasource.username", db::getUsername);
        reg.add("spring.datasource.password", db::getPassword);
    }
    // tests now run against a REAL Postgres
}

Interview Favorite

The key argument: H2 tests pass but production breaks because H2 doesn't support Postgres-specific SQL, types, or constraints. Testcontainers gives "production parity" in tests at the cost of needing Docker available in CI. This is a strong senior-level testing practice to mention.

Q18. What Is Contract Testing?

ROI: ★★★★☆ Frequency: Medium-High

Short Answer

Contract testing verifies that two services agree on the shape of their API without running both together in a slow end-to-end test. The consumer defines its expectations as a contract; the provider's build runs tests generated from that contract, so a breaking change fails the provider's CI immediately.

Common tools: Spring Cloud Contract and Pact. This is "consumer-driven contracts" — the consumer's needs drive what the provider must honour.

Interview Gold Point

Contract tests solve the microservices problem of "I changed my API and silently broke three downstream services". They are far faster and more reliable than spinning up the whole system — tie this back to the Microservices chapter.

Q19. What Is Mutation Testing?

ROI: ★★★☆☆ Frequency: Medium

Short Answer

Mutation testing measures the quality of your tests, not just coverage. A tool (e.g. PIT / Pitest) introduces small bugs ("mutants") — flipping > to >=, removing a method call — then runs your tests. If your tests still pass, they failed to "kill" the mutant, revealing a weak assertion. The mutation score is the percentage of mutants killed.

Interview Favorite

This directly answers the "is 100% coverage meaningful?" trap: code can be fully covered yet have tests that assert nothing. Mutation testing catches exactly those hollow tests. Mention it when asked how you ensure tests are actually valuable.

Q20. How Do You Test Asynchronous or Time-Dependent Code?

ROI: ★★★★☆ Frequency: Medium-High

Short Answer

Never use Thread.sleep to "wait for" async work — it makes tests slow and flaky. Instead poll for the expected condition with a timeout using a library like Awaitility, and inject a Clock for time-based logic so you can control "now" deterministically. 

// Awaitility: poll until condition or timeout
await().atMost(5, SECONDS)
       .untilAsserted(() ->
           assertEquals(1, repository.count()));

Interview Gold Point

Injecting java.time.Clock instead of calling Instant.now() directly is the clean way to test schedulers, expiry, and timeouts — the test supplies a fixed clock. This ties to the flaky-test discussion: time and threading are the two biggest sources of flakiness.

QUICK REVISION SHEET

  • FIRST: Fast, Isolated, Repeatable, Self-validating, Timely
  • AAA: Arrange, Act, Assert
  • Pyramid: many unit, some integration, few E2E
  • JUnit 5: @BeforeEach / @BeforeAll / assertThrows / @ExtendWith
  • Doubles: Stub (data) vs Mock (verify) vs Spy (real+stub) vs Fake
  • Mockito: when().thenReturn(), verify(), ArgumentCaptor, @Mock + @InjectMocks
  • Spring slices: @WebMvcTest, @DataJpaTest, @SpringBootTest, @MockBean
  • Coverage is a tool, not a target — prefer branch coverage
  • Test behaviour, not implementation
  • Treat flaky tests as bugs
  • Integration: @SpringBootTest + MockMvc / TestRestTemplate (keep few)
  • Testcontainers: real DB/Kafka in Docker — production parity over H2
  • Contract testing (Spring Cloud Contract / Pact) catches API breakage between services
  • Mutation testing (PIT) measures test quality, not just coverage
  • Async tests: Awaitility (not Thread.sleep); inject Clock for time
