SECTION 1 - OBJECT ORIENTED PROGRAMMING

Q1. What Are The Four Pillars Of OOP?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

  • Encapsulation
  • Abstraction
  • Inheritance
  • Polymorphism

Detailed Answer

Object-Oriented Programming organizes software around objects, which are self-contained units that combine state (data) and behaviour (methods). Instead of writing procedures that operate on loosely related data, we model the problem domain as collaborating objects, which keeps large codebases understandable as they grow.

The four pillars are the foundational principles that make this model work. They exist to help us create software that is maintainable, reusable and scalable, and they directly map to the layered design we use every day in Spring Boot applications.

Encapsulation

Binding data and behavior together while hiding implementation details. State is kept private and exposed only through controlled methods, so internal changes do not ripple out to callers.

Abstraction

Showing only necessary information and hiding complexity. We program against an interface or contract rather than a concrete implementation, which reduces coupling.

Inheritance

Allowing one class to acquire properties of another so common behaviour can be shared. It models an "is-a" relationship between types.

Polymorphism

Same interface, different behavior. A single reference type can point to multiple concrete implementations, and the correct method is selected at runtime.

Project Example

In Spring Boot the request flows through layers: Controller → Service → Repository. Each layer abstracts implementation details from the others. The controller does not know how the service performs business logic, and the service does not know how the repository talks to the database. This separation is exactly the OOP pillars applied to a real architecture: encapsulation hides internal state, abstraction exposes clean interfaces, and polymorphism lets us swap implementations (for example a different repository) without touching callers.

Senior Engineer Perspective

Interviewers are not testing definitions. They are checking whether you can apply OOP principles in real systems. Expect them to follow up with "show me where you used abstraction in your project" or "give an example of polymorphism you actually wrote." Always anchor the pillars to concrete code you have shipped rather than reciting textbook definitions.

Q2. What Is Encapsulation?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

Encapsulation means wrapping data and methods together while restricting direct access to the internal state of an object.

Example


public class User {

    private String name;

    public String getName() {
        return name;
    }

    public void setName(String name) {
        this.name = name;
    }
}

Why Important?

  • Data Protection
  • Controlled Access
  • Maintainability
  • Flexibility

By keeping fields private and exposing only getters and setters, we control how state is read and modified. This lets us add validation, logging or computed values later without breaking any caller, because the public contract stays the same while the internals are free to change. It also protects invariants: a class can guarantee its data never enters an invalid state because no outside code can reach in and corrupt it directly.

Real Project Example

Entity fields are usually private, and access happens through methods rather than direct manipulation. In a JPA entity, for example, keeping fields private lets Hibernate manage them through accessors and lets you enforce rules (such as normalising an email to lowercase in the setter) in a single place. This is why exposing public mutable fields on domain objects is considered a design smell.

Q3. What Is Abstraction?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

Abstraction hides implementation details and exposes only essential behavior, letting callers depend on what an object does rather than how it does it.

Example


public interface PaymentService {

    void processPayment();
}

The caller does not know how payment is processed internally. It could call a credit-card gateway today and a wallet provider tomorrow, and as long as the contract stays the same, the calling code never changes. This is the core benefit of abstraction: it decouples the consumer of a capability from its implementation, which makes the system easier to extend and test.

Project Example

Your service layer exposes methods like applyJob(), saveJob() and deleteJob() while hiding database implementation details. The controller calls applyJob() without knowing whether the data is written to PostgreSQL, validated against business rules, or pushed to a cache. If you later introduce a new persistence strategy, the controller is untouched because it only depends on the abstraction. In Spring this is reinforced by coding to interfaces and letting dependency injection supply the concrete bean.

Q4. What Is Inheritance?

ROI: ★★★★★ Frequency: High

Short Answer

Inheritance allows one class to inherit properties and behavior from another, modelling an "is-a" relationship and letting subclasses reuse or specialise the parent's logic.

Example


class Animal {
    void eat() {}
}

class Dog extends Animal {
}

Advantages

  • Code Reuse
  • Reduced Duplication
  • Hierarchy Creation

Inheritance lets common behaviour live in one base type so subclasses do not repeat it. A Dog automatically gets eat() from Animal. This is useful for genuine type hierarchies, such as a base RuntimeException that several domain-specific exceptions extend.

Interview Trap

Inheritance is not always the best option. Modern systems often prefer composition because deep inheritance hierarchies create tight coupling: a change in the parent can silently break every subclass, and you inherit behaviour you may not actually want. A common interviewer follow-up is "when would you NOT use inheritance?" The strong answer is to use inheritance only for a true is-a relationship with a stable base, and otherwise favour composition for flexibility.

Q5. What Is Polymorphism?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

One interface with multiple implementations. The same method call can resolve to different behaviour depending on the actual object type at runtime.

Example


PaymentService payment =
    new CreditCardPayment();

payment.processPayment();

Same method call. Different behavior. The reference is typed as PaymentService, but the object is a CreditCardPayment, so the credit-card implementation runs. Swap in a WalletPayment and the same line of calling code now executes wallet logic. This is what lets frameworks like Spring inject any implementation of an interface and have your code work unchanged.

Types

  • Compile-Time Polymorphism
  • Runtime Polymorphism

Compile-time polymorphism is achieved through method overloading, where the compiler picks the right method based on argument types. Runtime polymorphism is achieved through method overriding and dynamic dispatch, where the JVM chooses the implementation based on the actual object at execution time. Most real-world design benefit comes from runtime polymorphism because it is the mechanism behind interfaces, abstract classes and dependency injection.

Q6. Difference Between Overloading And Overriding

ROI: ★★★★★ Frequency: Extremely High

Overloading Overriding
Same Method Name Same Method Name
Different Parameters Same Parameters
Compile Time Runtime
Same Class Child Class

Overloading is resolved by the compiler based on the method signature (number and types of parameters) and happens within the same class. Overriding is resolved by the JVM at runtime based on the actual object, and requires a subclass redefining a method it inherited with the same signature. In short: overloading is compile-time polymorphism, overriding is runtime polymorphism.

Example Overloading


void add(int a, int b)

void add(int a, int b, int c)

Example Overriding


class Animal {
   void sound(){}
}

class Dog extends Animal {

   @Override
   void sound(){}

}
Q7. Composition vs Inheritance

ROI: ★★★★★ Frequency: High

Answer

Inheritance models an IS-A relationship: a Dog IS-A Animal. Composition models a HAS-A relationship: a Car HAS-A Engine. The distinction matters because it changes how tightly your classes are bound together.

With inheritance, the subclass is locked to its parent's contract and exposes the parent's API. With composition, an object simply holds a reference to a collaborator and delegates to it, so you can swap that collaborator, mock it in tests, or change its behaviour without affecting the containing class.

Example


class Engine {}

class Car {

    private Engine engine;

}

Senior Engineer Perspective

Prefer composition over inheritance. Inheritance creates tight coupling and a rigid hierarchy, while composition provides flexibility and is far easier to test in isolation. This is one of the most quoted guidelines from Effective Java, and interviewers use it to separate engineers who memorise OOP from engineers who have felt the maintenance pain of deep inheritance trees in production.

Expected Follow-Up

Why does Spring use composition extensively?

Answer: Loose coupling and easier testing. Spring beans hold references to their collaborators (injected dependencies) rather than extending them, so any dependency can be replaced with a different implementation or a mock. This is what makes Spring code unit-testable and configurable without modifying source.

SECTION 2 - OBJECT CLASS

Q8. What Is Object Class?

ROI: ★★★★★ Frequency: High

Answer

Object is the root class of all Java classes. Every class directly or indirectly extends Object, even when you do not write "extends Object" explicitly. This is why every object in Java is guaranteed to provide a baseline set of methods such as equals(), hashCode() and toString().

This common root is what makes generic containers possible: a collection declared as holding Object can hold any reference type, and framework code can operate on any object through these inherited methods. Understanding the Object contract is the foundation for understanding how collections, equality and hashing behave.

Important Methods

  • equals()
  • hashCode()
  • toString()
  • clone()
  • wait()
  • notify()
  • notifyAll()
  • getClass()
Q9. Difference Between == and equals()

ROI: ★★★★★ Frequency: Extremely High

Short Answer

== compares references (whether two variables point to the same object in memory). equals() compares logical equality (whether two objects are meaningfully "the same" according to the class's definition).

Example


String a = new String("Java");

String b = new String("Java");

System.out.println(a == b);

System.out.println(a.equals(b));

Output


false

true

The two new String("Java") calls create two distinct objects on the heap, so == is false because the references differ. But String overrides equals() to compare character content, so a.equals(b) is true. For primitives there is no equals() and == compares actual values, which is why the rule "use equals() for objects, == for primitives" exists.

Project Example


if(user.getRole().equals("ADMIN"))

Never write: 


if(user.getRole() == "ADMIN")

The second form compares references, and although string literals are pooled, a role value built at runtime (from a database column or request) will not be the same reference as the literal "ADMIN", so the comparison can silently return false and cause a security or authorization bug. Always compare String content with equals(), and prefer "ADMIN".equals(user.getRole()) to avoid a NullPointerException when the role is null.

Follow-Up Questions

  • Why does String override equals()?
  • Why does hashCode() matter?
  • What happens in HashMap?

These follow-ups all probe the same theme: the equals/hashCode contract. The interviewer wants to see that you understand equals() defines logical equality, that hash-based collections rely on hashCode() to find the right bucket before calling equals(), and that breaking the contract corrupts HashMap and HashSet behaviour.

Q10. What Is hashCode()?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

hashCode() returns an integer representation of an object. It is used heavily by hash-based collections to decide which bucket an object belongs to, so that lookups can be performed in near constant time instead of scanning every element.

Important Contract

If equals() returns true for two objects, then their hashCode() values must be the same. The reverse is not required: two unequal objects may share a hash code (a collision), but two equal objects must never have different hash codes, or hash-based collections will fail to find them.

Example


user1.equals(user2)

== true

Then

user1.hashCode()

must equal

user2.hashCode()

Real Project Impact

An improper hashCode implementation can break HashMap, HashSet, caching and JPA entity behavior. A classic production bug is overriding equals() but forgetting hashCode(): two logically equal objects then land in different buckets, so a contains() or get() returns nothing even though the value was added. With JPA entities, an unstable hashCode (for example one based on a generated ID that is null before persistence) can cause entities to "disappear" from a HashSet after they are saved. The safe rule is to always override equals() and hashCode() together, using the same fields in both.

Strings are among the most frequently asked Java interview topics. Many interviewers start with simple String questions and gradually move into: * Memory Management * String Pool * Immutability * Thread Safety * Performance

Q11. Why Is String Special In Java?

ROI: ★★★★★ Frequency: Extremely High

Answer

String is special because:

  • Immutable
  • Stored in String Pool
  • Used heavily throughout Java ecosystem
  • Frequently used as HashMap keys
  • Security sensitive
  • Optimized by JVM

Unlike ordinary classes, String receives special treatment from both the language and the JVM. It has literal syntax built into the language, a dedicated pool for reusing identical literals, and immutability that the JVM relies on for caching and safe sharing across threads. Because Strings carry sensitive data like credentials, tokens and connection details, their immutability is also a security guarantee: once created, a String value cannot be altered by another part of the program.

Real Examples


JDBC URLs

JWT Tokens

Usernames

Roles

HTTP Headers

API Keys
Q12. What Is String Pool?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

The String Pool is a special memory area (inside the heap) where the JVM stores unique String literals so they can be reused instead of duplicated.

Example


String a = "Java";

String b = "Java";

Only one object is created. Both references point to the same pooled object. When the JVM encounters a literal, it first checks the pool; if an equal String already exists, it reuses it rather than allocating a new one. This saves memory and makes == comparisons between identical literals return true, because they really are the same object.

Memory View


String Pool

+--------+
| "Java" |
+--------+
   ^
   |
 a,b

Interview Trap


String a = "Java";

String b = new String("Java");

System.out.println(a == b);

Output: false. Because b points to a heap object created explicitly with new, not the pooled literal. The new keyword always forces a fresh object outside the pool, so even though the contents are equal, the references differ. Calling b.intern() would return the pooled reference and make a == b.intern() true. This is one of the most common String trick questions, so be ready to explain exactly which object lives where.

Q13. What Does intern() Do?

ROI: ★★★★☆ Frequency: Medium

Answer

intern() places the String into the String Pool if it is not already there, and returns the pooled reference. 


String a = new String("Java");

String b = a.intern();

After interning, b references the pooled String, so b == "Java" is true even though a was created with new. In practice intern() is rarely needed and should be used cautiously: interning a very large number of distinct strings can fill the pool and add overhead, so it is mainly useful when you have many duplicate values and want to reduce memory by collapsing them to a single shared instance.

Q14. Why Is String Immutable?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

For security, caching, thread safety and String Pool optimization. Once a String is created, its value can never change.

Detailed Answer

Reasons:

  • Security
  • Thread Safety
  • HashCode Caching
  • String Pool Efficiency
  • Predictable Behavior

Immutability is what makes the String Pool safe to share: if one reference could mutate a pooled String, every other reference pointing to it would change too. It also enables hashCode caching, because the value never changes the hash can be computed once and reused, which is a major performance win since Strings are so often used as map keys. And because the value is fixed, Strings are inherently thread-safe and can be passed freely between threads without synchronization.

Security Example


jdbc:mysql://prod-db

If Strings were mutable, one thread could modify connection details (or a username, file path or security check) after they had been validated but before they were used. Immutability closes this "time-of-check to time-of-use" gap, which is why sensitive values are passed around as Strings.

HashMap Example

String is often used as a key. If it were mutable, its hashCode could change after insertion, the entry would end up in the wrong bucket, and subsequent lookups would silently fail. Immutability guarantees the key's hash stays constant for the life of the map entry.

Senior Engineer Perspective

This question is often used to identify whether a candidate understands JVM internals rather than just reciting "String is immutable." A strong answer connects immutability to concrete consequences: pool safety, cached hashCode, thread safety, and security. If you can also mention that StringBuilder exists precisely because immutability makes heavy concatenation expensive, you signal genuine depth.

Q15. Difference Between String, StringBuilder and StringBuffer

ROI: ★★★★★ Frequency: Extremely High

Feature String StringBuilder StringBuffer
Mutable No Yes Yes
Thread Safe Yes No Yes
Performance Slow Fastest Slower

String is immutable, so every modification produces a new object. StringBuilder and StringBuffer are mutable character buffers designed for building strings incrementally. StringBuffer's methods are synchronized, making it thread-safe but slower, while StringBuilder drops synchronization for speed. String is considered "thread safe" only in the sense that an immutable object cannot be corrupted by concurrent access.

Example


StringBuilder sb = new StringBuilder();

sb.append("Java");
sb.append(" Spring");

Interview Favorite

Which should you use most often?

Answer: StringBuilder, because most string-building operations happen within a single thread (such as building a query, log message or response inside one method) where synchronization adds cost without benefit. Reach for StringBuffer only in the rare case where the same buffer is genuinely shared and mutated across threads, which in modern code is almost never the right design anyway.

Q16. Why Is String Concatenation Expensive?

ROI: ★★★★★ Frequency: High

Example


String s = "";

for(int i=0;i<1000;i++){

   s += i;

}

What Happens?

Each concatenation creates a new String object because String is immutable. In a loop, the existing characters are copied into a new buffer on every iteration, so the work grows quadratically with the number of iterations and many short-lived objects are produced that the garbage collector must later clean up. For a thousand iterations this is wasteful; for tens of thousands it becomes a real performance problem.

Better Approach


StringBuilder sb =
    new StringBuilder();

A StringBuilder reuses a single growable buffer and appends in place, so the loop runs in linear time with far less garbage. Note that a single concatenation expression outside a loop (a + b) is fine, because the compiler already optimises it using StringBuilder under the hood; the problem is specifically repeated concatenation in a loop.

SECTION 4 - IMMUTABLE CLASSES

Q17. What Is An Immutable Class?

ROI: ★★★★★ Frequency: Extremely High

Answer

An immutable object cannot be modified after creation. Every field is set once, in the constructor, and there is no way to change the object's state afterwards; any "modification" instead returns a new object.

Characteristics

  • Class final
  • Fields private final
  • No setters
  • Defensive copies when needed

The class is made final so it cannot be subclassed to add mutable behaviour. Fields are private and final so they are assigned exactly once. No setters are exposed. And when a field references a mutable object (like a Date or a collection), the constructor stores a defensive copy and getters return copies, so callers cannot reach in and mutate the internal state through the shared reference.

Example


public final class User {

   private final String name;

   public User(String name){
      this.name = name;
   }

   public String getName(){
      return name;
   }
}

Advantages

  • Thread Safety
  • Predictability
  • Easy Caching
  • No Side Effects

Immutable objects are inherently thread-safe because there is no state to corrupt, which removes the need for locking. They are safe to cache and reuse, safe to use as map keys, and they eliminate an entire class of bugs caused by code unexpectedly changing shared state. This is why value objects, DTOs and records are increasingly designed as immutable in modern Java.

SECTION 5 - FINAL KEYWORD

Q18. What Is final Keyword?

ROI: ★★★★★ Frequency: Extremely High

Usage

The final keyword applies a restriction depending on where it is used. It can be applied to:

  • Variables
  • Methods
  • Classes

In each case final means "this cannot change after it is set." On a variable the reference cannot be reassigned, on a method it cannot be overridden, and on a class it cannot be extended. The keyword communicates intent to other developers and lets the compiler and JVM make optimisations and safety guarantees.

Examples

Variable


final int age = 25;

Cannot be reassigned. Note that for a final reference to an object, the reference is fixed but the object's own internal state can still change unless the object itself is immutable.

Method


final void calculate(){}

Cannot be overridden by a subclass, which protects critical behaviour from being changed in derived classes.

Class


final class User {}

Cannot be extended. String is the most famous example of a final class, which is part of how its immutability is enforced.

Q19. Difference Between final, finally and finalize()

ROI: ★★★★★ Frequency: Extremely High

Keyword Purpose
final Restriction
finally Exception Cleanup
finalize() Garbage Collection Hook

These three are unrelated despite the similar spelling, which is exactly why interviewers like the question. final is a keyword that imposes a restriction on a variable, method or class. finally is a block attached to a try/catch that always runs to release resources. finalize() was a method on Object that the garbage collector could call before reclaiming an object.

Important Note

finalize() is deprecated. Do not use it in modern Java applications. It was unreliable (there is no guarantee when, or even whether, it runs), it hurt garbage-collection performance, and it could resurrect objects. The modern replacement for deterministic cleanup is try-with-resources together with the AutoCloseable interface, or the Cleaner API for native resources.

SECTION 6 - EXCEPTION HANDLING

Q20. What Is Exception Handling?

ROI: ★★★★★ Frequency: Extremely High

Answer

Exception handling allows applications to recover gracefully from errors instead of crashing. Java separates the normal flow of code from error-handling logic using try, catch, finally and throw, so that when something goes wrong the program can respond meaningfully rather than terminating abruptly.

Benefits

  • Prevents abrupt termination
  • Improves reliability
  • Allows meaningful error responses

In a Spring Boot API this is what lets you turn an internal failure into a clean HTTP response (for example a 400 with a clear message or a 500 with a correlation id) instead of leaking a stack trace to the client. Well-designed exception handling is a core part of building reliable, observable services.

Q21. Difference Between Checked And Unchecked Exceptions

ROI: ★★★★★ Frequency: Extremely High

Checked Unchecked
Compile Time Runtime
Must Handle Optional
Exception RuntimeException

Checked exceptions extend Exception (but not RuntimeException) and are enforced by the compiler: you must either catch them or declare them with throws. They represent recoverable conditions outside the program's control, such as I/O or database failures. Unchecked exceptions extend RuntimeException and are not enforced by the compiler; they typically signal programming errors like dereferencing null or indexing past an array bound.

Checked Example


IOException

SQLException

Unchecked Example


NullPointerException

ArithmeticException

IndexOutOfBoundsException

Senior Engineer Perspective

Most Spring Boot applications prefer custom runtime exceptions. Checked exceptions force boilerplate try/catch or throws clauses up the call stack and clutter service interfaces, so teams typically define unchecked domain exceptions (such as a ResourceNotFoundException extending RuntimeException) and handle them centrally with @RestControllerAdvice. Spring itself follows this philosophy by wrapping low-level checked exceptions like SQLException in its unchecked DataAccessException hierarchy.

In short, throw triggers the exception while throws advertises the possibility.

Q23. Can We Have Multiple Catch Blocks?

ROI: ★★★★☆ Frequency: High

Answer

Yes. A single try can be followed by multiple catch blocks, and more specific exception types should come first. The JVM evaluates catch blocks top to bottom and uses the first one whose type matches, so a broad type listed before a narrower one would intercept the exception and make the narrower block unreachable, which the compiler actually rejects as an error.

Correct


catch(IOException e)

catch(Exception e)

Wrong


catch(Exception e)

catch(IOException e)

The wrong version fails to compile because Exception is a supertype of IOException, so the IOException block can never be reached. Note that since Java 7 you can also combine unrelated exceptions in one block using the multi-catch syntax catch(IOException | SQLException e) to avoid duplicated handling code.

Q24. Does finally Always Execute?

ROI: ★★★★★ Frequency: High

Answer

Usually yes. The finally block runs whether the try block completes normally or an exception is thrown, which makes it the right place to release resources such as connections or locks. It even runs when there is a return statement inside the try or catch.

The exceptions are the rare cases where the JVM never gets a chance to run it:

  • System.exit()
  • JVM Crash
  • Power Failure

In modern code, try-with-resources is preferred over manual finally cleanup, because it closes AutoCloseable resources automatically and avoids the subtle bugs that arise when both the try and the finally block throw exceptions.

Q25. What Are Exception Handling Best Practices?

ROI: ★★★★★ Frequency: Senior-Level Favorite

Answer

  • Never swallow exceptions
  • Log properly
  • Use custom exceptions
  • Handle at correct layer
  • Avoid generic Exception
  • Provide meaningful messages
  • Use Global Exception Handlers

The unifying idea is that exceptions should carry enough context to diagnose a problem and should be handled at the layer that can actually do something useful. Swallowing an exception (an empty catch block) hides failures and makes production issues nearly impossible to debug. Catching the broad Exception type masks bugs you did not anticipate. Instead, define meaningful domain exceptions, log with enough context (including a correlation id), and translate them into appropriate responses at a single boundary.

Spring Boot Example


@RestControllerAdvice

@RestControllerAdvice is used for centralized exception handling. It lets you map each exception type to a consistent HTTP status and response body in one place, so controllers stay focused on the happy path and every error is reported uniformly across the API. This is the standard production pattern for turning thrown exceptions into clean, client-friendly responses.

SECTION 7 - GENERICS

Generics are one of the favorite topics used by senior interviewers. Most candidates know syntax. Few understand: * Why Generics exist * Type Safety * Type Erasure * Wildcards * PECS Principle

Q26. What Are Generics?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

Generics provide compile-time type safety by letting you parameterise classes and methods with the types they operate on, so type errors are caught by the compiler instead of surfacing as runtime ClassCastExceptions.

Without Generics


List list = new ArrayList();

list.add("Java");

String value =
(String) list.get(0);

With Generics


List list =
new ArrayList<>();

String value =
list.get(0);

Advantages

  • Type Safety
  • Compile-Time Checks
  • Cleaner Code
  • Reduced Casting

The generic version tells the compiler the list holds Strings, so it rejects adding the wrong type and removes the need for an explicit cast on retrieval. This makes APIs self-documenting (you can see exactly what a collection contains) and eliminates a whole category of casting bugs. Generics are why the entire Collections framework and Spring's typed repositories feel safe to use.

Q27. Why Were Generics Introduced?

ROI: ★★★★★ Frequency: High

Answer

Before Generics (Java 5), collections held raw Object references, so anything could be put in and casts were required to take things out: 


List list =
new ArrayList();

list.add("Java");

list.add(123);

Compilation succeeds, but runtime failures become possible because nothing stops mixing a String and an Integer in the same list; the error only appears later when code casts an element to the wrong type and a ClassCastException is thrown. Generics eliminate this entire category of problems by moving the check to compile time, where it is cheap to find and fix. They were a direct response to how error-prone and cast-heavy pre-Java-5 collection code had become.

Q28. What Is Type Erasure?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

Generic type information is removed during compilation. After the compiler has verified type safety, it strips the type parameters and the bytecode works with raw types plus casts.

Example


List

Becomes approximately: 


List

during runtime.

Why?

Backward compatibility with older Java versions. Erasure let generic code interoperate with pre-Java-5 libraries and run on the existing JVM without changing the bytecode format. The trade-off is that generic type information is not available at runtime, which is why you cannot do things like new T(), instanceof List<String>, or create an array of a generic type.

Interview Trap


List

List

At runtime both become just List, so they share the same Class object and you cannot distinguish them by reflection or instanceof. A classic trap is asking whether you can overload a method with parameters List<String> and List<Integer> — you cannot, because after erasure both have the identical signature List. Being able to explain this consequence of erasure signals real understanding rather than surface knowledge.

Q29. What Is Wildcard?

ROI: ★★★★★ Frequency: High

Answer

The wildcard represents an unknown type. It lets a method accept a collection of some type without caring which, which is useful for read-only operations that work regardless of the element type.

List

The catch is that, because the exact type is unknown, you cannot safely add anything (except null) to a List<?> — the compiler cannot guarantee what would be a valid element. You can still read elements as Object and call size(), iterate, or clear. Wildcards are how you write flexible APIs without locking them to one concrete type parameter.

Q30. Difference Between and

ROI: ★★★★★ Frequency: Senior Favorite

Extends

This is an upper-bounded wildcard: the type is Number or any subtype. It is read-only (a producer) — you can safely read elements as Number, but you cannot add anything because the compiler does not know the exact subtype the list actually holds.

Super

This is a lower-bounded wildcard: the type is Integer or any supertype. It is write-safe (a consumer) — you can safely add Integers, but when you read you only get back Object, because the exact supertype is unknown.

PECS Principle

Producer Extends, Consumer Super. If a parameter produces values for you to read, use extends; if it consumes values you provide, use super. This is the mnemonic from Effective Java that tells you which bound to choose, and it is exactly how the JDK declares methods like Collections.copy(dest, src).

Interview Gold Point

Many experienced interviewers specifically ask about PECS. Being able to state the rule and explain why extends is read-only while super is write-only (with the underlying type-safety reasoning) is a strong differentiator, because most candidates can use generics but cannot articulate the variance rules behind wildcards.

SECTION 8 - COMPARABLE AND COMPARATOR

Q31. Difference Between Comparable and Comparator

ROI: ★★★★★ Frequency: Extremely High

Comparable Comparator
Natural Ordering Custom Ordering
compareTo() compare()
Inside Class External Class

Comparable defines a single natural ordering inside the class itself via compareTo(), so the class "knows" how to sort itself (for example, sorting users by id). Comparator defines an ordering externally via compare(), so you can create many different orderings without modifying the class. Use Comparable for the one obvious default order, and Comparator whenever you need alternative or multiple sort strategies.

Comparable Example


public class User
implements Comparable{

   @Override
   public int compareTo(User u){
      return this.id - u.id;
   }
}

Comparator Example


Comparator byName =
(a,b) ->
a.getName()
.compareTo(b.getName());
Q32. When Would You Use Comparator?

ROI: ★★★★★ Frequency: High

Answer

Use a Comparator when multiple sorting strategies are needed for the same type. For example, you may need to sort a User by:

  • Name
  • Salary
  • Joining Date
  • Department

A class can only define one natural ordering through Comparable, so Comparator is the right tool when the desired order varies by context. It avoids modifying entity classes, keeps sorting logic out of your domain model, and composes nicely: you can chain orderings with Comparator.comparing(...).thenComparing(...) and reverse them with reversed(), which is far cleaner than baking every possible order into compareTo().

SECTION 9 - CLONING

Q33. Difference Between Shallow Copy And Deep Copy

ROI: ★★★★★ Frequency: High

Shallow Copy

A shallow copy duplicates the top-level object but copies references to nested objects, so the original and the copy share those nested objects. Changing a nested object through one reference is visible through the other.

Deep Copy

A deep copy recursively duplicates the object and everything it references, producing completely independent copies. Mutating the copy never affects the original.

Example


User

 -> Address

Shallow Copy: both Users share the same Address object, so editing one user's address changes the other's. Deep Copy: each User gets its own new Address, so they are fully isolated.

Project Impact

This matters when handling DTOs and mutable objects, especially anything shared across threads or stored in a cache. A subtle production bug is mutating a cached object that was only shallow-copied, which silently corrupts the cached value for every other caller. When isolation matters, prefer a deep copy (often via a copy constructor or a mapping library) rather than relying on shallow copies.

Q34. What Is Cloneable?

ROI: ★★★★☆ Frequency: Medium

Answer

Cloneable is a marker interface (it declares no methods) used to signal that a class permits object cloning via Object.clone(). If a class does not implement Cloneable and calls clone(), the JVM throws CloneNotSupportedException. 


implements Cloneable

Senior Engineer Perspective

Cloneable is rarely used in modern enterprise applications, and it is widely considered a flawed design: clone() performs a shallow copy by default, bypasses constructors, and the contract is awkward to implement correctly. Copy constructors or static factory methods (and for DTOs, mapping libraries like MapStruct) are generally preferred because they are explicit, type-safe and let you control exactly what gets copied.

SECTION 10 - SERIALIZATION

Q35. What Is Serialization?

ROI: ★★★★★ Frequency: High

Answer

Serialization is the process of converting an object into a byte stream so it can be stored or transmitted, and deserialization reconstructs the object from that byte stream. In Java a class opts in by implementing the Serializable marker interface.

Uses

  • Caching
  • Messaging
  • Network Transfer
  • Persistence

Serialization is what lets an object cross a boundary — written to disk, placed in a distributed cache like Redis, sent over a message queue, or transferred between JVMs. In modern services, JSON serialization (via Jackson) is far more common than Java's native binary serialization, which has known security risks during deserialization of untrusted data and is increasingly avoided.

Q36. What Is serialVersionUID?

ROI: ★★★★★ Frequency: High

Answer

serialVersionUID is a unique version identifier for a Serializable class. It is written into the serialized byte stream and checked during deserialization to verify that the class loading the data is compatible with the class that produced it. 


private static final long
serialVersionUID = 1L;

It is used to verify compatibility during deserialization. If you do not declare one, the JVM generates it from the class structure, so any change to the class (even adding a field) changes the generated value and causes an InvalidClassException when reading older data. Declaring it explicitly gives you control over versioning and lets a class evolve while still deserializing previously stored objects.

Q37. What Is transient Keyword?

ROI: ★★★★★ Frequency: High

Answer

The transient keyword marks a field to be skipped during serialization. When the object is serialized, transient fields are not written, and on deserialization they are restored to their default value (null for objects, 0 for numbers). 


private transient String password;

Real Example

Passwords and other sensitive data should not be serialized, because the serialized bytes could be written to disk, a cache or a log and leak the secret. transient is also used for derived fields that can be recomputed, or for non-serializable references you do not want to persist. Marking such fields transient is both a security and a correctness practice.

SECTION 11 - REFLECTION

Q38. What Is Reflection?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

Reflection allows a program to inspect and modify classes, methods and fields at runtime, even ones it had no knowledge of at compile time. It is the JVM's introspection API.

Capabilities

  • Read Class Metadata
  • Access Methods
  • Access Fields
  • Create Objects Dynamically

Through reflection you can discover a class's annotations, invoke methods by name, read or set fields (even private ones), and instantiate objects without knowing their type at compile time. This dynamic capability is what makes frameworks possible: they can wire up and operate on your classes without you writing any glue code.

Q39. Why Does Spring Use Reflection?

ROI: ★★★★★ Frequency: Senior-Level Favorite

Answer

Spring uses reflection extensively for:

  • Dependency Injection
  • Bean Creation
  • AOP
  • Annotations
  • Proxy Generation

At startup Spring scans your classes, reads their annotations, instantiates beans and injects dependencies — all through reflection, because it has no compile-time knowledge of your specific classes. It also uses reflection (and dynamic proxies) to apply cross-cutting concerns like transactions and security around your methods. This is the mechanism that lets you write a plain annotated class and have the framework manage its lifecycle for you.

Example

@Autowired works because Spring discovers the annotated fields and constructors at runtime via reflection and supplies the matching beans. You never call a setter yourself; Spring reflects over the bean, finds the injection points, and fills them in.

Q40. What Are Drawbacks Of Reflection?

ROI: ★★★★★ Frequency: High

Answer

  • Slower Performance
  • Reduced Type Safety
  • Security Concerns
  • Harder Debugging

Reflective calls are slower than direct calls because they bypass many JVM optimisations and perform lookups at runtime. They sidestep compile-time type checking, so errors surface only at runtime. They can break encapsulation by accessing private members, raising security concerns. And because the code does not reference symbols directly, IDE navigation, refactoring and debugging all become harder.

Senior Engineer Perspective

Reflection should be used carefully. Frameworks like Spring and Hibernate use it heavily and amortise the cost by caching reflective metadata, but ordinary business code usually should not. If you find yourself reaching for reflection in application logic, it is often a sign that a cleaner abstraction (an interface, a strategy, or a small bit of polymorphism) would serve better. Knowing when to use it, and when to avoid it, is what interviewers are probing for.

SECTION 12 - ANNOTATIONS

Q41. What Are Annotations?

ROI: ★★★★★ Frequency: Extremely High

Answer

Annotations are metadata attached to Java elements such as classes, methods, fields and parameters. They do not change code behaviour by themselves; instead they carry information that the compiler, tools or frameworks read and act upon.

Examples


@Override

@Service

@Repository

@Entity

Annotations provide information to the compiler and frameworks. @Override is read by the compiler to verify a method really overrides a superclass method. @Service, @Repository and @Entity are read at runtime (via reflection) by Spring and JPA to register beans and map classes to tables. Annotations have a retention policy (SOURCE, CLASS or RUNTIME) that determines whether they survive to runtime, which is why framework annotations must be RUNTIME-retained.

Q42. Difference Between @Component, @Service and @Repository

ROI: ★★★★★ Frequency: Extremely High

Answer

Technically all three are Spring stereotype annotations and are themselves meta-annotated with @Component, so all three cause the class to be detected by component scanning and registered as a bean. The difference is primarily semantic: they communicate the role each class plays in the architecture.

  • @Component → Generic Bean
  • @Service → Business Layer
  • @Repository → Data Layer

Using the specific stereotype makes the layering explicit and self-documenting, and lets Spring apply layer-specific behaviour where relevant.

Interview Gold Point

@Repository also enables exception translation. Spring wraps that bean with a post-processor that converts low-level, vendor-specific persistence exceptions (like SQLException or Hibernate exceptions) into Spring's consistent, unchecked DataAccessException hierarchy. Many candidates know the three annotations are "basically the same" but miss this concrete functional difference, so calling it out signals real Spring depth.

SENIOR ENGINEER RAPID FIRE

Q43. Can We Override Static Methods?

Answer

No. Static methods belong to the class, not to an instance, so they are not subject to runtime dynamic dispatch. If a subclass declares a static method with the same signature, it hides the parent's method rather than overriding it — this is called method hiding.

The practical difference is that which static method runs is decided at compile time by the reference type, not at runtime by the actual object. So if you call a hidden static method through a parent-typed reference pointing to a child object, the parent's version runs. This is the opposite of how overridden instance methods behave, which is exactly why interviewers ask it.

Q44. Can Constructor Be Final?

Answer

No. The final modifier on a method means "cannot be overridden," but constructors are never inherited and therefore can never be overridden, so applying final to a constructor is meaningless and the compiler rejects it.

Each class defines its own constructors; a subclass invokes a parent constructor with super(...) rather than inheriting and overriding it. Because there is no overriding to prevent, the keyword simply does not apply.

Q45. Can We Have Abstract Constructor?

Answer

No. abstract means "declared without an implementation, to be implemented later by a subclass," but a constructor's whole purpose is to initialise an object, which requires a body. There is nothing to defer to a subclass, so an abstract constructor is a contradiction and is not allowed.

Note that an abstract class can still have constructors — they run when a concrete subclass is instantiated, via the super() chain — but those constructors themselves are concrete, not abstract.

Q46. Can Interface Have Methods?

Answer

Yes, and what an interface may contain has expanded significantly over recent Java versions. Originally interfaces held only abstract method declarations and constants.

  • Java 8+: default methods (with a body, so implementations inherit behaviour) and static methods.
  • Java 9+: private methods, used to share code between default methods without exposing it.

default methods were added so that interfaces could evolve — new methods could be added to widely-implemented interfaces (like the Collection API) without breaking every existing implementation. This is a popular follow-up because it tests whether you have kept up with modern Java.

Q47. Can Private Methods Be Overridden?

Answer

No. Private methods are not inherited by subclasses and are not visible to them, so there is nothing for a subclass to override. If a subclass declares a method with the same signature, it is an entirely new, independent method, not an override.

Because private methods are resolved statically at compile time rather than through dynamic dispatch, this is also why calling a private method from within a parent class always invokes the parent's version even when the object is a subclass instance.

Q48. Can We Instantiate Abstract Class?

Answer

No. An abstract class may contain abstract methods with no implementation, so creating an instance directly would leave those methods undefined. You must create a concrete subclass that implements the remaining abstract methods, and instantiate that.

A common nuance worth mentioning is that you can use an anonymous subclass to instantiate "on the fly" (new AbstractType() { ... }), which is really creating a concrete subclass inline rather than instantiating the abstract class itself.

Q49. Why Is Java Pass-By-Value?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

Java always passes copies of variables, never the variables themselves. For primitives, the copied value is the actual value. For objects, the copied value happens to be the reference (the pointer to the object), not the object itself. This is what confuses many developers and leads them to wrongly call it "pass by reference."

The key consequence: because the reference is copied, reassigning the parameter inside a method (param = new Object()) does not affect the caller's variable — they now point to different objects. However, mutating the object the reference points to (param.setName("X")) is visible to the caller, because both copies of the reference point to the same object on the heap.

A clean way to prove it in an interview: a method that sets a parameter to a new object has no effect on the caller's variable, which would be impossible if Java were truly pass-by-reference. The reference is passed by value.

Q50. What Happens When You Create An Object?

ROI: ★★★★★ Frequency: Extremely High

Answer

  1. Class Loaded
  2. Memory Allocated
  3. Fields Initialized
  4. Constructor Executed
  5. Reference Returned

When you write new User(), the JVM first ensures the class is loaded, linked and initialised (this happens only once per class). It then allocates memory on the heap for the new instance and zeroes the fields to their defaults. Next, instance initialisers and field initialisers run, followed by the constructor body (which first chains to its superclass constructor via super()). Finally a reference to the fully constructed object is returned and assigned to your variable.

Interview Follow-Up

Which memory area is the object stored in? The heap. Where does the reference variable itself live? On the stack (for a local variable). This split is the basis for understanding garbage collection: the object on the heap is eligible for collection once no live reference on any stack (or static field) points to it anymore. Expect the interviewer to push from here into the JVM memory model and garbage collection.
