22 - MESSAGING & EVENT STREAMING INTERVIEW HANDBOOK

Once a system grows past a single service, the way components talk to each other becomes a design decision in its own right. Senior backend interviews spend a lot of time here because asynchronous messaging is the backbone of decoupled, resilient, scalable systems. Many engineers have "used Kafka" but cannot explain when a queue beats a stream, what delivery guarantees actually mean, or how to stop duplicate processing. For a Senior role you should be able to confidently discuss: * Why asynchronous messaging exists (decoupling, buffering, resilience) * The difference between point-to-point queues, publish/subscribe, and event streaming * Apache Kafka and Apache RabbitMQ, and when each one fits * Managed options like AWS SQS and SNS * Delivery guarantees: at-most-once, at-least-once, exactly-once * Idempotency, deduplication, ordering, dead letter queues and retries * Consumer lag, backpressure, and integrating all of this with Spring This chapter deliberately keeps the deep Kafka internals short, since the Microservices chapter already covers Kafka, the outbox pattern, the saga pattern and event-driven architecture in depth. Here we focus on breadth: the alternatives, the comparison, the delivery semantics and the reliability patterns. Interviewers usually continue with follow-ups on exactly-once semantics, ordering guarantees, and how you would design a reliable consumer that never loses or double-processes a message.

SECTION 1 - FUNDAMENTALS

Q1. Why Use Asynchronous Messaging At All?

ROI: ★★★★★ Frequency: Very High

Short Answer

Asynchronous messaging lets a producer hand off work to a broker and continue immediately, instead of blocking until a downstream service responds. The three classic reasons are decoupling, buffering and resilience.

Decoupling. The producer does not need to know who consumes the message, how many consumers there are, or whether they are even online right now. It publishes to a topic or queue and walks away. New consumers can be added later without touching the producer. This temporal and spatial decoupling is the single biggest architectural benefit.

Buffering (load levelling). A broker absorbs traffic spikes. If 10,000 orders arrive in one second but the payment service can only process 500 per second, the queue holds the backlog and the consumer drains it at its own pace. Without a buffer, that spike would either crash the consumer or force you to over-provision for peak load you rarely hit.

Resilience. If the consumer is down, messages wait durably in the broker rather than being lost. When the consumer recovers it picks up where it left off. A synchronous REST call to a dead service simply fails; a message to a durable queue survives the outage.

Interview Gold Point

Frame it as "smoothing and surviving." Async messaging smooths spiky load and survives downstream failure. The trade-off you should always mention unprompted is added complexity and eventual consistency: the caller no longer gets an immediate answer, so you must design for "the work will happen, just not right now."

Q2. Synchronous (REST) vs Asynchronous Communication

ROI: ★★★★★ Frequency: Very High

Short Answer

In synchronous communication the caller sends a request and blocks until it receives a response, as with a REST or gRPC call. The two services are coupled in time: both must be available, and the caller's latency is the sum of every downstream hop. It is simple, gives an immediate result, and is the right choice for a query where the user is waiting for an answer (for example "fetch this user's profile").

In asynchronous communication the caller publishes a message and moves on; the work happens later on a separate thread of execution in another service. The two services are decoupled in time, which buys resilience and scalability but costs you immediacy and simplicity. It is the right choice for commands and events that can happen in the background (for example "send a welcome email" or "order placed").

Aspect Synchronous (REST) Asynchronous (Messaging)
Coupling Caller waits for callee Decoupled via broker
Availability Both must be up Consumer may be down
Latency felt by caller Sum of all hops Just the publish
Result Immediate Eventual
Best for Queries needing an answer Background work, events

Common Mistake

Treating it as "async is better." It is not better, it is different. A request the user is staring at (login, search) should be synchronous. The senior answer is "synchronous for queries that need an answer now, asynchronous for side effects and fan-out work that can be eventually consistent." Reaching for messaging where a simple REST call would do just adds operational overhead.

Q3. Queue (Point-to-Point) vs Pub/Sub vs Event Streaming

ROI: ★★★★★ Frequency: Very High

Short Answer

These are three different delivery models and the distinction is a favourite interview question.

Point-to-point (queue). One message is delivered to exactly one consumer among many. If you run five workers on the same queue, the broker load-balances messages across them and each message is processed once. This is the classic "work queue" / competing consumers pattern (RabbitMQ queues, AWS SQS). It is ideal for distributing tasks.

Publish/subscribe (pub/sub). One message is delivered to every interested subscriber. The producer publishes an event and N independent subscribers each get their own copy (RabbitMQ fanout exchange, AWS SNS, JMS topics). This is for fan-out: one "OrderPlaced" event notifies billing, inventory and analytics simultaneously.

Event streaming. Events are appended to a durable, ordered, replayable log that is retained for a long time, and consumers read at their own position (offset). Multiple independent consumer groups can each read the entire history, and a new consumer can replay events from the beginning (Apache Kafka, AWS Kinesis). The key difference from pub/sub is that the broker does not delete a message once delivered; it keeps the log, so the data is a re-readable source of truth, not a transient notification.

Interview Favorite

The crisp one-liner: "A queue deletes a message after one consumer handles it; pub/sub copies it to every subscriber and then forgets it; a stream keeps the whole ordered log so anyone can replay it." Streaming = durable log + replay + per-consumer offset. That replay-from-history capability is what makes Kafka fundamentally different from a traditional message queue.

SECTION 2 - APACHE KAFKA

Q4. Kafka Core Concepts Recap (Topic, Partition, Offset, Consumer Group, Broker)

ROI: ★★★★★ Frequency: Very High

Short Answer

Kafka is an event streaming platform built on a distributed, append-only commit log. The Microservices chapter goes deep on Kafka, so this is just the vocabulary you must be able to rattle off.

A topic is a named stream of events (for example orders). A topic is split into partitions, which are the unit of parallelism and ordering. Each message in a partition gets a monotonically increasing offset, which is just its position in that partition's log. A broker is a Kafka server; a cluster of brokers stores the partitions and replicates them for fault tolerance. Producers write to partitions; consumers read by tracking offsets.

A consumer group is a set of consumers that cooperate to read a topic. Kafka assigns each partition to exactly one consumer within a group, so adding consumers (up to the partition count) scales throughput, and each group gets its own independent view of the offsets. Two different groups reading the same topic each receive the full stream, which is how Kafka does pub/sub on top of a log.

Term One-line meaning
Topic Named stream of events
Partition Ordered shard of a topic; unit of parallelism
Offset Position of a message within a partition
Broker A Kafka server holding partitions
Consumer Group Consumers sharing partitions; own their offsets

Interview Gold Point

The relationship to nail: ordering is guaranteed only within a partition, not across a topic. Parallelism within a group is capped at the number of partitions, so partition count is a capacity decision you make up front. Modern Kafka (KRaft mode) no longer needs ZooKeeper, which is a good detail to drop if asked about cluster coordination.

Q5. How Partitioning Enables Scale & Ordering

ROI: ★★★★☆ Frequency: High

Short Answer

Partitioning is the single mechanism behind both Kafka's horizontal scalability and its ordering guarantee, and explaining how the two interact separates seniors from juniors.

Scale. Because a topic is split into N partitions that can live on different brokers, producers and consumers can work on them in parallel. With 12 partitions, a consumer group can have up to 12 active consumers, each owning a partition, so throughput grows nearly linearly until you hit the partition count.

Ordering. Kafka guarantees ordering only within a single partition. The producer chooses a partition by hashing the message key, so all messages with the same key land in the same partition and are therefore processed in order. If you need every event for a given order processed in sequence, you use the order id as the key. 

// Same key -> same partition -> strict ordering for that key
producer.send(new ProducerRecord<>(
        "orders",
        order.getId(),        // key: drives partition selection
        order.toJson()));     // value

Common Mistake

Claiming "Kafka guarantees ordering." It does not, globally. The tension is real: more partitions means more parallelism but coarser ordering. The senior framing is "choose the partition key to match the entity whose order matters, and accept that ordering across keys is undefined." If you need total ordering, you are forced down to a single partition and lose parallelism.

Q6. Log Retention & Replay

ROI: ★★★★☆ Frequency: High

Short Answer

Unlike a traditional queue, Kafka does not delete a message when it is consumed. Messages stay in the partition log according to a retention policy, and consumers track their own position. This is what makes replay possible.

Retention can be time-based (keep 7 days), size-based (keep 100 GB per partition), or compacted (keep only the latest value per key, useful for changelog/state topics). Because the data is still there, a consumer can reset its offset and re-read history. That powers reprocessing after a bug fix, bootstrapping a brand-new consumer (for example a new analytics pipeline), and event sourcing where the log is the source of truth. 

# Reset a consumer group to replay everything from the start
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --group analytics --topic orders \
  --reset-offsets --to-earliest --execute

Interview Favorite

"The consumer owns its offset, not the broker." In RabbitMQ the broker removes a message once acked; in Kafka the broker keeps the log and each consumer group decides where to read. This decoupling of storage from consumption is exactly why Kafka can serve replay, multiple independent consumers and event sourcing, none of which a plain queue does well.

Q7. When Is Kafka the Right Choice?

ROI: ★★★★☆ Frequency: High

Short Answer

Reach for Kafka when you need high-throughput event streaming, durable retention, replay, and multiple independent consumers reading the same data. Concretely:

* Very high volume (hundreds of thousands to millions of events per second).
* You want events retained as a re-readable log (event sourcing, replay, audit).
* Many independent consumer groups need the same stream (one event feeds billing, search indexing and analytics).
* Stream processing and aggregations (Kafka Streams, ksqlDB, Flink).
* Ordering per key matters and you can partition by that key.

Kafka is a poor fit when you mainly need flexible per-message routing, complex priorities, or simple low-volume task queues with per-message acknowledgement and short-lived messages. That is RabbitMQ territory. It is also operationally heavier than a managed queue like SQS, so for low volume a managed service is often the pragmatic choice.

Interview Gold Point

Tie it back to the Microservices chapter: Kafka is the typical backbone for event-driven architecture, the transactional outbox relay target, and the event bus behind choreographed sagas. The decision rule to say out loud: "Kafka for streams of events I want to keep and replay; a queue for transient work I want to route and process once."

SECTION 3 - RABBITMQ

Q8. The RabbitMQ / AMQP Model (Exchange, Queue, Binding, Routing Key)

ROI: ★★★★☆ Frequency: High

Short Answer

RabbitMQ implements AMQP, and its defining feature is that producers never publish directly to a queue. They publish to an exchange, and the exchange decides which queues receive the message based on routing rules. This indirection is what gives RabbitMQ its flexible routing.

The four pieces fit together like this. A producer sends a message with a routing key to an exchange. A binding is a rule that connects an exchange to a queue, optionally matching on the routing key. The exchange evaluates the bindings and copies the message into every matching queue. Consumers then read from queues and send an ack once they have processed a message; if they do not ack (crash, nack), the message is requeued or dead-lettered. 

Producer --(routing key)--> Exchange --(binding)--> Queue --> Consumer (ack)

Interview Gold Point

The mental model to state: "Kafka is a log you read from; RabbitMQ is a router that pushes messages into queues." RabbitMQ's per-message ack and smart routing make it excellent for work distribution; its messages are typically transient and removed once acked, which is the opposite of Kafka's retained log.

Q9. RabbitMQ Exchange Types (Direct / Topic / Fanout)

ROI: ★★★☆☆ Frequency: Medium

Short Answer

The exchange type controls how routing keys map to queues, and being able to pick the right one is a common practical question.

Direct. Routes to queues whose binding key exactly matches the message's routing key. Use it for precise, point routing, for example send order.created to a single queue bound with that exact key.

Topic. Routes on wildcard pattern matching of dotted routing keys, where * matches one word and # matches zero or more. A queue bound with order.*.eu receives order.created.eu but not order.created.us. This is the most flexible and most used type for category-based fan-out.

Fanout. Ignores the routing key entirely and broadcasts to every bound queue. This is straight pub/sub: every subscriber gets a copy. (There is also a fourth, headers, which routes on message header attributes instead of the routing key, but direct/topic/fanout are what interviewers ask about.)

Exchange Routing rule Typical use
Direct Exact routing-key match Targeted point routing
Topic Wildcard pattern (* and #) Category-based fan-out
Fanout Broadcast to all bound queues Pub/sub notifications

Interview Favorite

If asked "how would you broadcast an event to many services in RabbitMQ?" the answer is a fanout exchange (or a topic exchange with a broad pattern). If asked "how do I route by category?" say topic. The default (nameless) exchange is a direct exchange that routes by queue name, which is why convertAndSend("queueName", msg) just works in simple demos.

Q10. When RabbitMQ Fits Better Than Kafka

ROI: ★★★★☆ Frequency: High

Short Answer

RabbitMQ is the better fit when your problem is task distribution and routing rather than a high-volume replayable stream. Prefer RabbitMQ when:

* You need flexible, complex routing (topic/header based) to direct messages to specific consumers.
* You need per-message acknowledgement and immediate requeue/retry semantics for a work queue.
* You want priorities, TTLs, delayed delivery, or a built-in dead-letter mechanism out of the box.
* Volume is moderate and messages are transient commands ("resize this image", "send this email") rather than a permanent event log.
* You want a simpler operational footprint than running a Kafka cluster for streaming.

Choose Kafka instead when you need very high throughput, long retention, replay, ordering per key at scale, or many independent consumers reading the same stream.

Interview Gold Point

The one-sentence rule of thumb: "RabbitMQ for smart routing of transient tasks (a message broker); Kafka for high-throughput retained event streams you can replay (a streaming log)." Saying both can do pub/sub but solve different problems, and naming the routing-vs-replay axis, signals real experience rather than buzzword matching.

SECTION 4 - COMPARISON

Q11. Kafka vs RabbitMQ vs AWS SQS/SNS - How To Choose

ROI: ★★★★★ Frequency: Very High

Short Answer

These three (four) cover most of what a backend team will reach for. Kafka is a self-managed event streaming log. RabbitMQ is a feature-rich message broker. AWS SQS is a fully managed queue (point-to-point) and SNS is a fully managed pub/sub notification service; the common AWS pattern is SNS fan-out to multiple SQS queues.

Dimension Kafka RabbitMQ AWS SQS / SNS
Model Streaming log Broker (exchange/queue) Managed queue (SQS) / pub-sub (SNS)
Ordering Per partition Per queue (best effort) FIFO queues only; standard = best effort
Retention Long (days, replayable) Until acked (transient) Up to 14 days, then deleted
Throughput Very high (millions/s) High (tens of thousands/s) High, auto-scaled by AWS
Replay Yes (offset reset) No No
Ops burden Highest (run a cluster) Medium Lowest (fully managed)
Best use case Event streaming, replay, analytics Routing, task/work queues Simple decoupling on AWS

Interview Favorite

The decision framework to verbalise: start from requirements, not tools. Need replay and high-volume streams? Kafka. Need complex routing and per-message control of transient tasks? RabbitMQ. Already on AWS and just need to decouple two services without running infrastructure? SQS, with SNS in front when you need fan-out. The senior signal is choosing the simplest option that meets the requirement, often a managed queue, rather than defaulting to Kafka for everything.

SECTION 5 - DELIVERY GUARANTEES & RELIABILITY

Q12. Delivery Guarantees: At-Most-Once, At-Least-Once, Exactly-Once

ROI: ★★★★★ Frequency: Very High

Short Answer

Every messaging system gives you one of three guarantees, determined mostly by when you acknowledge or commit.

At-most-once. A message is delivered zero or one times; it may be lost but never duplicated. You get this by committing the offset (or acking) before processing. Fast and simple, acceptable only when occasional loss is tolerable (for example metrics samples).

At-least-once. A message is delivered one or more times; never lost but possibly duplicated. You get this by committing/acking after successful processing. If the consumer crashes after processing but before committing, it reprocesses on restart. This is the default and most common guarantee, and it is why idempotency matters.

Exactly-once. Each message is effectively processed once, with neither loss nor duplication. This is the hardest because a consumer's two actions, processing the message and recording that it was processed, are not naturally atomic across a network. Kafka offers exactly-once within Kafka via idempotent producers and transactions (atomic read-process-write across topics plus offset commit). But the moment your side effect is an external system (charge a card, send an email), end-to-end exactly-once is generally not achievable, so you settle for at-least-once plus idempotent consumers.

Guarantee Duplicates? Loss? How
At-most-once No Possible Commit before processing
At-least-once Possible No Commit after processing
Exactly-once No No Transactions / dedup; hard with external systems

Interview Gold Point

The line that wins this question: "In practice I aim for at-least-once delivery plus idempotent consumers, which gives effectively-once processing without the cost and limits of true exactly-once." Then add: Kafka's exactly-once is real but only end-to-end within Kafka; it cannot make an external API call exactly once.

Q13. Idempotent Consumers & Deduplication

ROI: ★★★★★ Frequency: Very High

Short Answer

Because at-least-once delivery can deliver the same message twice, the consumer must be safe to run more than once with the same input. That property is idempotency: processing the same message N times produces the same result as processing it once.

There are two main techniques. First, make the operation naturally idempotent: an UPSERT by a business key, or "set status = SHIPPED" rather than "increment count," gives the same end state on retry. Second, when the operation is not naturally idempotent (charge a card, send an email), deduplicate using a unique message id. The consumer records each processed id in a store (a database table with a unique constraint, or Redis with a TTL) and skips the message if the id is already present, ideally in the same transaction as the side effect. 

@KafkaListener(topics = "payments", groupId = "billing")
@Transactional
public void handle(PaymentEvent event) {
    // dedup table has UNIQUE(message_id)
    if (processedRepo.existsById(event.getMessageId())) {
        return; // already handled -> skip duplicate
    }
    processedRepo.save(new Processed(event.getMessageId()));
    paymentService.charge(event); // safe: runs at most once per id
}

Common Mistake

Relying on the broker's "exactly-once" setting and skipping dedup. Even with Kafka transactions, the moment you touch an external system you are back to at-least-once. The robust design is always: at-least-once delivery + idempotent consumer. Note also that the dedup write and the side effect should share a transaction (or use the outbox pattern), otherwise a crash between them reopens the duplicate window.

Q14. Message Ordering

ROI: ★★★★☆ Frequency: High

Short Answer

Ordering is harder than it looks because parallelism and strict ordering pull in opposite directions. The general rule across systems: you can only guarantee ordering within a single sequential channel.

In Kafka, ordering holds within a partition. Route related events to the same partition using a partition key (for example the aggregate id), and a single consumer in the group will process them in order. You lose global ordering across partitions, which is the price of parallelism.

In RabbitMQ, a single queue with a single consumer preserves order, but the moment you add competing consumers for throughput, messages can be processed out of order (and requeue-on-failure can reorder too). In SQS, standard queues are best-effort ordering; only FIFO queues guarantee order, and they do it per message-group id at reduced throughput.

Interview Favorite

The phrase to use: "Order by key, not globally." Identify the entity whose events must stay in sequence, key/group by that, and accept that unrelated entities are processed concurrently. If a candidate demands global total ordering, push back: it forces a single partition / single consumer and destroys scalability, so it is almost never the right requirement.

Q15. Dead Letter Queues, Retries & Poison Messages

ROI: ★★★★★ Frequency: Very High

Short Answer

A poison message is one the consumer can never process successfully (malformed payload, references a deleted record, hits a permanent bug). If you blindly retry it forever, it blocks the partition or queue and starves every message behind it. You need a strategy to retry the transient failures and quarantine the permanent ones.

Retries with exponential backoff. Transient failures (a downstream service briefly down, a timeout) should be retried, but with increasing delays (1s, 2s, 4s, 8s...) plus jitter to avoid hammering the dependency and to avoid a thundering herd. Cap the number of attempts.

Dead Letter Queue (DLQ). After the retry budget is exhausted, route the message to a separate dead-letter queue or topic instead of dropping it or looping forever. The DLQ preserves the message and its failure context so the main flow keeps moving, and a human or an automated job can inspect, fix and replay the dead-lettered messages later. RabbitMQ supports DLX (dead-letter exchanges) natively; SQS has a redrive policy with a maxReceiveCount; Spring Kafka provides a DeadLetterPublishingRecoverer. 

// Spring Kafka: retry with backoff, then route to <topic>.DLT
@Bean
DefaultErrorHandler errorHandler(KafkaTemplate<?, ?> template) {
    var recoverer = new DeadLetterPublishingRecoverer(template);
    // 3 retries, 1s initial, x2 multiplier, max 10s
    var backoff = new ExponentialBackOffWithMaxRetries(3);
    backoff.setInitialInterval(1000L);
    backoff.setMultiplier(2.0);
    backoff.setMaxInterval(10000L);
    return new DefaultErrorHandler(recoverer, backoff);
}

Interview Gold Point

Distinguish transient vs permanent failure out loud: retry the transient with backoff and jitter, dead-letter the permanent. The DLQ is not a graveyard; you should monitor and alert on its depth, because a growing DLQ is an early warning of a bug or a bad deploy. Replaying from the DLQ after a fix is the recovery story interviewers want to hear.

SECTION 6 - PATTERNS & SPRING

Q16. Consumer Lag, Backpressure & Spring Integration

ROI: ★★★★☆ Frequency: High

Short Answer

Consumer lag is how far a consumer group is behind the latest produced offset, in other words the size of the backlog it has not yet read. It is the single most important health metric for a streaming consumer: steadily rising lag means producers are outpacing consumers and you must scale out consumers (up to the partition count), optimise processing, or add partitions. Tools like the consumer-groups CLI, Burrow, or your monitoring stack expose it.

Backpressure is the mechanism that stops a fast producer or broker from overwhelming a slow consumer. A pull-based system like Kafka has natural backpressure: the consumer fetches at its own pace and the durable log buffers the rest, so you bound in-flight work via max.poll.records and concurrency rather than dropping data. Push-based systems like RabbitMQ use a prefetch limit (QoS) so the broker only sends a bounded number of unacked messages per consumer. The buffering property from Q1 is exactly what gives the consumer room to apply backpressure instead of falling over.

The dual-write problem and the transactional outbox. A classic trap is updating the database and publishing an event in two separate steps: if the process crashes between them, you either lose the event or publish one that the DB never committed. The transactional outbox pattern fixes this by writing the event into an outbox table in the same DB transaction as the business change, then a relay (or CDC tool like Debezium) reads the outbox and publishes to Kafka reliably. This is covered in depth in the Microservices chapter alongside the saga pattern; for messaging interviews you just need to name the dual-write problem and that the outbox solves it.

Spring integration. Spring for Apache Kafka gives you @KafkaListener and KafkaTemplate; Spring AMQP gives you @RabbitListener and RabbitTemplate. Both turn a method into a message consumer with container-managed threading, acknowledgement and error handling. 

// Spring Kafka consumer
@KafkaListener(topics = "orders", groupId = "fulfilment", concurrency = "3")
public void onOrder(OrderEvent event, Acknowledgment ack) {
    fulfilmentService.process(event); // idempotent
    ack.acknowledge();                // manual commit AFTER processing
}

// Spring AMQP (RabbitMQ) consumer
@RabbitListener(queues = "orders.queue")
public void onOrder(OrderEvent event) {
    fulfilmentService.process(event);  // ack on normal return
}

Interview Favorite

Mention manual acknowledgement: commit/ack after processing to get at-least-once, never before. Tie the whole chapter together: at-least-once delivery + idempotent consumer + DLQ for poison messages + outbox to avoid dual writes is the standard recipe for a reliable event-driven service, and it connects directly to the Kafka, outbox and saga material in the Microservices chapter.

QUICK REVISION SHEET

  • Async messaging = decoupling + buffering (load levelling) + resilience; trade-off is eventual consistency
  • Sync (REST) for queries needing an answer now; async for background work and fan-out
  • Queue = one consumer, deleted after; Pub/Sub = copy to all, then forgotten; Stream = durable replayable log with per-consumer offsets
  • Kafka: topic → partitions → offsets; consumer group shares partitions; ordering only within a partition
  • Partition key (e.g. entity id) gives per-key ordering; parallelism capped at partition count
  • Kafka keeps the log (time/size/compaction retention) → replay via offset reset; consumer owns its offset
  • Kafka for high-volume retained replayable streams; RabbitMQ for smart routing of transient tasks
  • RabbitMQ: producer → exchange → (binding + routing key) → queue → consumer (ack)
  • Exchanges: direct (exact key), topic (wildcards * #), fanout (broadcast)
  • SQS = managed queue (FIFO for order); SNS = managed pub/sub; managed = lowest ops burden
  • At-most-once (commit before, may lose) | at-least-once (commit after, may dup) | exactly-once (hard; only within Kafka)
  • Default is at-least-once → build idempotent consumers (UPSERT or dedup by message id in same txn)
  • Order by key, not globally; competing consumers can reorder; SQS FIFO orders per message-group
  • Retry transient failures with exponential backoff + jitter; quarantine poison messages in a DLQ and replay after fix
  • Consumer lag = backlog size, the key health metric; Kafka pull + max.poll.records, RabbitMQ prefetch for backpressure
  • Dual-write problem → transactional outbox (event + business change in one DB txn, relay/CDC to broker); see Microservices chapter
  • Spring: @KafkaListener / KafkaTemplate and @RabbitListener / RabbitTemplate; manual ack AFTER processing
