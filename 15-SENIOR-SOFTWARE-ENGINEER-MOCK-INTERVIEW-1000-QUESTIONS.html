Q27. What Is Exception Propagation?

ROI: ★★★★★ Frequency: High

Answer

If an exception is not handled, it propagates up the call stack until handled.

Each method on the stack is given a chance to catch the exception; if it does not, the JVM unwinds that frame and passes control to the caller. If the exception reaches the top of the stack with no handler, the thread terminates and the JVM prints the stack trace. This is why the stack trace is read top-down to find where the exception originated and bottom-up to see how execution reached that point.

Q38. What Is Thread Safety?

ROI: ★★★★★ Frequency: Extremely High

Answer

Code is thread-safe when multiple threads can execute it concurrently without causing incorrect behavior.

Q43. What Is Intrinsic Lock?

ROI: ★★★★★ Frequency: High

Answer

Every Java object has an internal monitor lock. synchronized uses this lock.

This is why you do not declare a separate lock object for basic synchronization: the lock is built into the object header itself. When a thread enters a synchronized method or block it acquires that monitor, and any other thread attempting to acquire the same monitor blocks until it is released. The lock is reentrant, so the same thread can re-enter synchronized code guarded by a lock it already holds without deadlocking itself.

Q44. Static Synchronization

ROI: ★★★★★ Frequency: High

Answer

Locks the Class object instead of an instance.

Because the lock is held on the single Class object rather than on a per-instance basis, a static synchronized method serializes access across all instances of that class, not just one. This matters when the shared state being protected is itself static. A common trap is mixing instance and static synchronization on the same data, since they use different monitors and therefore do not mutually exclude each other. 


synchronized static method

Q52. AtomicInteger vs synchronized Counter

ROI: ★★★★★ Frequency: High

Interview Answer

AtomicInteger usually performs better under moderate contention because it avoids heavyweight locking.

Q77. What Is ReentrantLock?

ROI: ★★★★★ Frequency: High

Interview Ready Answer

A thread can acquire the same lock multiple times without deadlocking itself.

ReentrantLock keeps a hold count that increments on each acquire and decrements on each unlock, releasing the lock only when the count returns to zero. Beyond reentrancy it gives capabilities that intrinsic synchronized locks lack: tryLock with a timeout, interruptible lock acquisition, and an optional fairness policy. The trade-off is that you must release it in a finally block, otherwise a thrown exception can leave the lock permanently held.

Q79. Semaphore

ROI: ★★★★★ Frequency: High

Interview Ready Answer

Controls access to a limited number of resources.

A Semaphore holds a fixed number of permits; a thread calls acquire to take a permit before using a resource and release to return it, blocking when no permits remain. Unlike a lock, permits are not tied to ownership, so one thread can acquire while another releases, which makes it well suited to bounding concurrency, such as throttling the number of simultaneous calls to a downstream service.

Q719. What Is Backpressure?

Answer

Mechanism that prevents producers from overwhelming consumers. SECTION 130 - DATABASE CONCURRENCY & LOCKING

Q846. What Is Optimistic Locking?

ROI: Guaranteed ★★★★★

Answer

Assumes conflicts are rare. Detects conflicts during update.

Q847. How Is Optimistic Locking Implemented In JPA?

ROI: Maximum ★★★★★

Answer

Optimistic locking is implemented using a @Version field on the entity. JPA reads the version with the row, includes it in the WHERE clause of the update, and increments it on every write. If two transactions read the same row and both try to save, the second update finds a stale version and fails, which is how conflicts are detected without holding a database lock.

Q848. What Happens During Optimistic Lock Conflict?

ROI: Maximum ★★★★★

Answer

When the version no longer matches, the update affects zero rows and JPA throws an OptimisticLockException. The losing transaction is rolled back, and the application is expected to reload the latest state and retry. This guarantees lost updates are never silently committed, pushing the conflict-resolution decision back to the business logic.

Q849. What Is Pessimistic Locking?

ROI: Guaranteed ★★★★★

Answer

Pessimistic locking has the database lock the records immediately when they are read, blocking any other transaction from modifying them until the lock is released. It prevents conflicts outright rather than detecting them afterwards, which is appropriate for high-contention writes where retries would be expensive, at the cost of reduced concurrency and the risk of deadlocks.

SECTION 131 - DISTRIBUTED TRANSACTIONS

Q851. What Is A Distributed Transaction?

Interview Ready Answer

Q853. What Is Two-Phase Commit (2PC)?

ROI: Maximum ★★★★★

Answer

Two-Phase Commit is a protocol that coordinates a transaction commit across multiple participants through a central coordinator. In the prepare phase every participant votes whether it can commit, and only if all vote yes does the coordinator issue the commit. It guarantees atomicity but blocks resources while waiting for votes, which hurts availability.

Q855. What Is A Partial Commit Problem?

ROI: Maximum ★★★★★

Answer

A partial commit happens when one service commits its part of the work while another participant fails, leaving the overall operation half-done and the system in an inconsistent state. This is the core risk distributed transactions exist to address, and it is why compensating actions or atomic coordination are required.

Q856. What Is Eventual Consistency?

ROI: Guaranteed ★★★★★

Answer

Eventual consistency means the system does not guarantee all replicas or services agree immediately, but it guarantees they converge to a consistent state over time once updates propagate. It trades immediate correctness for higher availability and scalability, and it is the default model for most large distributed systems.

Q857. Why Is Eventual Consistency Popular?

ROI: Maximum ★★★★★

It improves scalability and availability because services can accept writes and respond without waiting for global coordination. Under the CAP theorem, choosing availability and partition tolerance naturally leads to eventual consistency, which is acceptable for the vast majority of use cases that do not require strict, immediate global agreement.

Q858. What Is Compensation?

ROI: Maximum ★★★★★

Answer

Compensation is an action that semantically reverses a previously completed operation when a later step in a workflow fails. Because you cannot roll back a committed local transaction in another service, you instead issue a new transaction that undoes its effect. This is the foundation of how Sagas maintain consistency.

SECTION 132 - SAGA PATTERN

Q861. What Is Saga Pattern?

ROI: Maximum ★★★★★

Interview Ready Answer

A Saga is a sequence of local transactions coordinated through events or an orchestrator, where each step commits independently. If any step fails, previously completed steps are undone with compensating transactions. It replaces a single distributed transaction with a series of smaller, recoverable ones, achieving consistency without distributed locking.

Q862. Why Use Saga Instead Of 2PC?

ROI: Architect Favorite ★★★★★

Answer

Sagas offer better scalability and reduced coupling because each service commits locally without holding resources while waiting on a coordinator. Unlike 2PC, there is no blocking lock phase and no single point of failure in a coordinator, which makes Sagas a far better fit for high-throughput microservice architectures.

Q863. Choreography Saga

ROI: Maximum ★★★★★

Answer

In a choreography Saga, services communicate through events with no central coordinator. Each service listens for events, performs its local transaction, and emits the next event. This gives loose coupling, but the overall workflow becomes implicit and harder to trace as the number of steps grows.

Q864. Orchestration Saga

ROI: Maximum ★★★★★

Answer

In an orchestration Saga, a central coordinator explicitly drives the workflow, telling each service when to act and invoking compensations on failure. This gives clear visibility and centralized control of the business process, at the cost of introducing a component that must itself be reliable.

Q866. Saga Failure Handling

When a step fails, compensating transactions are executed in reverse order to restore consistency by semantically undoing the already-committed steps. Because the system never had a global lock, this is the only way to recover, which is why every forward step in a Saga must have a defined compensating action.

Q871. What Is CQRS?

ROI: Maximum ★★★★★

Interview Ready Answer

CQRS, Command Query Responsibility Segregation, separates the write model (commands that change state) from the read model (queries that return data). The two sides can use different data structures, stores and scaling strategies, which lets each be optimized independently for its workload.

Q872. Why Use CQRS?

ROI: Maximum ★★★★★

  • Performance Optimization
  • Complex Domains
Q873. Command Side

ROI: High ★★★★☆

The command side handles state-changing operations such as create, update and delete. It enforces business rules and validation, and is optimized for correctness and consistency rather than fast reads, often emitting events that update the read model.

Q874. Query Side

ROI: High ★★★★☆

The query side handles read operations and is optimized purely for fast retrieval. It frequently uses denormalized or precomputed views tailored to specific screens, so reads stay fast even when the write model is highly normalized.

Q875. CQRS Drawback

ROI: Architect Favorite ★★★★★

Answer

The main drawback is additional complexity. You maintain two models, often two stores, and must handle the synchronization and eventual consistency between them. This overhead is only justified when the domain or scaling needs genuinely demand it.

Q876. When Should You Avoid CQRS?

ROI: Maximum ★★★★★

ROI: Maximum ★★★★★

Q878. Biggest CQRS Mistake

ROI: Maximum ★★★★★

The biggest mistake is using CQRS without a clear business need. Applied to a simple domain it introduces eventual consistency, duplicate models and extra failure modes for no payoff, turning a straightforward system into a hard-to-maintain one.

Q879. CQRS Interview Tip

ROI: Maximum ★★★★★

Q880. CQRS Golden Rule

ROI: Architect Favorite ★★★★★

Q890. What Is A Kafka Partition?

ROI: Guaranteed ★★★★★

Answer

A partition is an ordered, append-only log segment within a topic. Messages within a single partition are strictly ordered and identified by an offset, and partitions are the unit of parallelism and distribution across brokers and consumers.

Q891. Why Partitions Matter?

ROI: Maximum ★★★★★

Partitions enable parallelism and scalability because each one can be consumed independently by a different consumer in a group. More partitions mean more consumers can process a topic concurrently, so partitioning is the primary lever for scaling Kafka throughput.

Q892. What Is Consumer Group?

ROI: Guaranteed ★★★★★

A consumer group is a set of consumers that cooperate to process a topic, with each partition assigned to exactly one consumer in the group at a time. This lets you scale processing horizontally while ensuring every message is handled once per group.

Q893. Why Can Ordering Break?

ROI: Architect Favorite ★★★★★

Ordering is guaranteed only within a partition, not across the whole topic. If related messages are spread across multiple partitions they can be processed out of order, which is why you key messages so that all events for the same entity land in the same partition.

Q894. What Is Rebalancing?

ROI: Maximum ★★★★★

Rebalancing is the redistribution of partitions among consumers in a group, triggered when a consumer joins, leaves or fails. It keeps the workload evenly assigned, but during a rebalance partition ownership changes hands and processing is briefly paused.

Q895. Why Is Rebalancing Expensive?

ROI: Maximum ★★★★★

It is expensive because consumers temporarily stop processing while partitions are reassigned, causing a pause known as a stop-the-world rebalance. Frequent rebalances add latency and lag, which is why stable consumer membership and tuned session timeouts matter in production.

Q896. What Is Kafka Retention?

ROI: Maximum ★★★★★

Retention is the policy controlling how long messages remain stored in a topic, configured by time or by size. Messages are kept until the retention limit is reached regardless of whether they have been consumed, which is what allows replay and multiple independent consumers.

Q897. Kafka Interview Trap: Does Kafka Delete Messages After Consumption?

ROI: Interview Trap ★★★★★

Correct Answer

No. Kafka does not delete a message just because a consumer read it. Messages are retained according to the retention policy (time or size based), and consumers simply advance their offset. This is what lets multiple consumer groups read the same data independently and replay history.

Interview Trap

Many engineers assume Kafka behaves like a traditional message queue that removes a message once it is acknowledged. It does not. Kafka is a distributed commit log, and offsets are tracked per consumer group while the underlying log stays until retention expires. Saying messages disappear after consumption is a classic giveaway of queue thinking applied to a log.

Q898. Kafka Golden Rule

ROI: Architect Favorite ★★★★★

Design consumers to be idempotent. Kafka guarantees at-least-once delivery by default, so the same message can be reprocessed after a retry or rebalance. Idempotent consumers ensure reprocessing a message produces no duplicate side effects, which is essential for correctness.

Q955. What Is N+1 Query Problem?

ROI: Ultra High ★★★★★

Answer

One query loads parent data. Additional queries load child data repeatedly.

So loading N parents triggers N extra queries for their children, turning one logical fetch into N+1 round trips and crushing performance. The fix is to load the children eagerly with a join fetch or a batched query rather than lazily per parent.

Q956. Why Is JWT Logout Difficult?

ROI: Ultra High ★★★★★

Answer

JWT is stateless. Immediate revocation requires extra mechanisms.

Because the server does not store the token, it remains valid until it expires, so simply deleting it on the client does not invalidate it. True immediate logout requires a server-side blocklist or short-lived tokens with refresh, which reintroduces some state.

Q959. Why Is Idempotency Important?

Answer

Retries should not create duplicate side effects.
