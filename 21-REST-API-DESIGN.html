21 - REST API DESIGN & DOCUMENTATION INTERVIEW HANDBOOK

API design is one of the most under-rated topics in senior backend interviews. Almost every candidate has "built REST APIs", but very few can defend their design choices under pressure. Interviewers use this topic to separate engineers who copy controller boilerplate from those who understand the HTTP contract they are exposing to other teams. Expect questions that probe whether you can: * Explain what actually makes an API RESTful (not just "it returns JSON") * Pick the correct HTTP method and status code without hesitation * Reason about idempotency, safety and retries * Design clean, versioned, evolvable resource URIs * Handle pagination, filtering and consistent error contracts * Defend cross-cutting concerns: caching, rate limiting, idempotency keys * Auto-generate documentation and know when REST is the wrong tool A good answer ties design decisions to consumer impact: backward compatibility, network cost, retry safety and developer experience. A weak answer is just a list of annotations. Once you state a rule, interviewers usually continue with "and how would you handle it when...", pushing you toward versioning, partial failure, concurrency and real production trade-offs.

SECTION 1 - REST FUNDAMENTALS

Q1. What is REST and what are its core architectural constraints?

ROI: ★★★★★ Frequency: Very High

Short Answer

REST (Representational State Transfer) is an architectural style, defined by Roy Fielding, for building distributed systems on top of HTTP. It is not a protocol and not a framework. A system is RESTful when it obeys a set of constraints; in return you get scalability, evolvability and visibility.

The six constraints are:

  • Client-Server - separation of concerns; UI and data storage evolve independently.
  • Stateless - each request carries everything the server needs; the server stores no client session between requests.
  • Cacheable - responses must declare whether they can be cached (via headers like Cache-Control).
  • Uniform Interface - the cornerstone: resources identified by URIs, manipulated through representations, self-descriptive messages, and HATEOAS.
  • Layered System - a client cannot tell whether it talks to the origin server or an intermediary (gateway, proxy, load balancer).
  • Code on Demand (optional) - the server may ship executable code (e.g. JavaScript) to extend the client.

Statelessness is the one most worth stressing: because no server holds session state, any node can serve any request, which is exactly what lets you scale horizontally behind a load balancer. State that must persist (the user's auth) travels in a token (see the JWT chapter), not a server-side session.

Interview Gold Point

Many candidates confuse "stateless" with "no database". Application state (resource data) absolutely lives on the server and in the DB; what must not live there is client session state. Saying this distinction out loud signals seniority.

Q2. What is the Richardson Maturity Model, and what actually makes an API "RESTful"?

ROI: ★★★★☆ Frequency: Medium

Short Answer

The Richardson Maturity Model grades how closely an HTTP API follows REST, in four levels:

  • Level 0 - The Swamp of POX: one endpoint, one verb (usually POST), RPC-over-HTTP (e.g. SOAP-style).
  • Level 1 - Resources: distinct URIs per resource, but still one verb everywhere.
  • Level 2 - HTTP Verbs: proper use of GET/POST/PUT/DELETE and meaningful status codes. This is where most "REST" APIs in industry sit, and it is genuinely good enough.
  • Level 3 - Hypermedia Controls (HATEOAS): responses include links telling the client what it can do next.

Pragmatically, an API is considered RESTful when it reaches Level 2: noun-based resource URIs, correct verbs, correct status codes, statelessness and cacheability. Level 3 is the academic ideal but rare in practice because client tooling rarely consumes the links.

Interview Favorite

A strong line: "Most production APIs, including those at large companies, are Level 2. HATEOAS is theoretically pure but adds payload weight and complexity that few clients use - so I treat it as optional rather than a failure." This shows judgement, not dogma.

SECTION 2 - HTTP METHODS & STATUS CODES

Q3. Explain the semantics of GET, POST, PUT, PATCH and DELETE.

ROI: ★★★★★ Frequency: Very High

Short Answer

MethodPurposeBody?Typical success code
GETRead a resource or collection; no side effectsNo200
POSTCreate a subordinate resource / trigger a processYes201 (or 200/202)
PUTFull replace of a resource at a known URI (or create-at-id)Yes (full)200 / 204 / 201
PATCHPartial update of a resourceYes (partial/diff)200 / 204
DELETERemove a resourceNo204 (or 200)

Key distinctions: PUT replaces the entire resource - omitted fields are typically cleared - whereas PATCH modifies only the supplied fields. PUT is also used to create a resource when the client controls the identifier (e.g. PUT /users/42); POST is used when the server assigns the id (e.g. POST /users returning Location: /users/42). 

@RestController
@RequestMapping("/api/v1/orders")
public class OrderController {

    @PostMapping
    public ResponseEntity<OrderDto> create(@Valid @RequestBody CreateOrderRequest req) {
        OrderDto created = orderService.create(req);
        URI location = URI.create("/api/v1/orders/" + created.id());
        return ResponseEntity.created(location).body(created); // 201 + Location header
    }

    @GetMapping("/{id}")
    public ResponseEntity<OrderDto> get(@PathVariable Long id) {
        return ResponseEntity.ok(orderService.findById(id)); // 200
    }

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        orderService.delete(id);
        return ResponseEntity.noContent().build(); // 204
    }
}

Common Mistake

Using POST for everything (a Level 0 anti-pattern) or using GET with side effects. A GET must never mutate state - crawlers, prefetchers and proxies will replay GETs freely, and that has caused real production data loss.

Q4. What is the difference between safe and idempotent methods?

ROI: ★★★★★ Frequency: High

Short Answer

  • Safe = read-only; calling it does not change server state. (GET, HEAD, OPTIONS)
  • Idempotent = calling it N times has the same effect on server state as calling it once. The response may differ, but the resulting state must not.
MethodSafeIdempotent
GET / HEADYesYes
OPTIONSYesYes
PUTNoYes
DELETENoYes
POSTNoNo
PATCHNoNot guaranteed

Why it matters: idempotency is what makes safe retries possible. If a client times out on a PUT or DELETE, it can resend with no harm. POST is not idempotent - two POSTs create two resources - which is exactly why idempotency keys exist (see Q13). PATCH is idempotent only if its semantics are a full-value set (e.g. JSON Merge Patch with absolute values); a relative operation like {"op":"increment","by":1} is not.

Interview Gold Point

"DELETE is idempotent even though the second call returns 404." Deleting twice still ends in the same state (resource gone). The differing status code does not break idempotency - idempotency is about state, not the response code. Saying this catches a subtlety most candidates miss.

Q5. Walk through the HTTP status code families and when you use them.

ROI: ★★★★★ Frequency: Very High

Short Answer

FamilyMeaningCommon codes
2xxSuccess200 OK, 201 Created, 202 Accepted, 204 No Content
3xxRedirection / caching301 Moved Permanently, 304 Not Modified
4xxClient error (the caller must fix the request)400, 401, 403, 404, 409, 422, 429
5xxServer error (not the caller's fault)500, 502, 503, 504

The senior signal here is precision: never return 200 with an {"error": ...} body, and never blanket-return 500 for validation failures. The status line is part of your API contract - clients, gateways and monitoring all branch on it. A spike of 5xx pages the on-call engineer; a spike of 4xx points at a misbehaving client. Conflating them poisons observability.

Common Mistake

Returning 200 OK for failures "to keep the frontend simple". This breaks caching, retries, alerting and any standard HTTP client. The status code is data - use it correctly.

Q6. When do you use 201 vs 200, 400 vs 422, 401 vs 403, and 404 vs 409?

ROI: ★★★★★ Frequency: High

Short Answer

  • 201 vs 200: 201 Created when a request created a new resource (return a Location header). 200 OK for a successful read or an update that returns a body. Use 204 No Content when the update/delete succeeds but there is nothing to return.
  • 400 vs 422: 400 Bad Request = the request is malformed or unparseable (broken JSON, wrong type, missing required field). 422 Unprocessable Entity = the syntax is fine but the request fails business/semantic validation (e.g. "end date before start date", "email already in a pending state"). Many teams use 400 for both; using 422 for semantic rules is the cleaner, more senior choice.
  • 401 vs 403: 401 Unauthorized = you are not authenticated (no/invalid credentials) - "who are you?". 403 Forbidden = you are authenticated but not allowed - "I know who you are, you still can't". (See the Spring Security chapter.)
  • 404 vs 409: 404 Not Found = the resource does not exist. 409 Conflict = the request conflicts with current state - duplicate creation, or an optimistic-lock version mismatch (stale update). 409 is the natural code for "this changed under you, refetch and retry".

Interview Favorite

On security, some teams deliberately return 404 instead of 403 to avoid leaking the existence of a resource to unauthorized users (resource enumeration). Mentioning this trade-off - correctness vs information disclosure - is a strong senior touch.

SECTION 3 - RESOURCE & URI DESIGN

Q7. What are the conventions for naming resources and designing URIs?

ROI: ★★★★☆ Frequency: High

Short Answer

  • Use nouns, not verbs. The verb is the HTTP method. GET /users/42, not GET /getUser?id=42.
  • Use plural collection names. /orders, /orders/42, /orders/42/items.
  • Nest to express ownership, but stop at one or two levels. /users/42/orders is fine; /users/42/orders/9/items/3/reviews is a smell - reference the leaf directly via /reviews/3.
  • Lowercase, hyphen-separated path segments: /purchase-orders, not /purchaseOrders or /purchase_orders.
  • Query params for filtering/sorting/paging, not for identity: /orders?status=PAID&sort=-createdAt.
  • Use sub-resources for relationships and reserve a few "controller" style endpoints for non-CRUD actions when truly needed (e.g. POST /orders/42/cancel).
GET    /api/v1/customers              # list customers
POST   /api/v1/customers              # create a customer
GET    /api/v1/customers/42           # read one
PUT    /api/v1/customers/42           # replace
PATCH  /api/v1/customers/42           # partial update
DELETE /api/v1/customers/42           # delete
GET    /api/v1/customers/42/orders    # that customer's orders

Common Mistake

Action verbs in the path (/createUser, /deleteOrder). The only widely accepted exception is non-CRUD domain actions that do not map to a verb, modelled as a POST to a sub-resource: POST /accounts/42/transfers.

Q8. What are the API versioning strategies and their trade-offs?

ROI: ★★★★★ Frequency: Very High

Short Answer

Three mainstream approaches:

StrategyExampleProsCons
URI path/api/v1/ordersExplicit, easy to test in a browser/curl, easy routing & cachingNot "pure" REST (URI should identify a resource, not a version); duplicates routes
Custom / Accept headerAccept: application/vnd.acme.v2+jsonClean URIs; version negotiation is content-drivenHarder to test/debug; easy to forget; weaker cache keys
Query parameter/orders?version=2Simple to addPollutes the query string; clumsy caching; easy to omit

URI versioning is the most common in industry because it is unambiguous and cache-friendly. The purist choice is media-type (header) versioning, which keeps the resource URI stable. Whatever you pick, the real goal is backward compatibility: prefer additive, non-breaking changes (new optional fields, new endpoints) so you rarely need to bump the major version at all. Bump the version only on breaking changes, and support N-1 for a deprecation window.

Interview Gold Point

"The best version bump is the one you never make." Adding fields, never removing or repurposing them, and treating unknown fields as ignorable on the client lets a v1 API evolve for years. This shows you understand evolution, not just syntax.

SECTION 4 - PAYLOADS & ERROR HANDLING

Q9. How do you design pagination - offset vs cursor/keyset?

ROI: ★★★★★ Frequency: High

Short Answer

Offset/limit pagination (?page=3&size=20 or ?offset=40&limit=20) is simple and allows jumping to any page. But it has two problems at scale: it is slow on large offsets (OFFSET 1000000 still scans and discards a million rows), and it suffers from drift - if rows are inserted/deleted while paging, items get skipped or repeated.

Cursor / keyset pagination passes an opaque cursor that encodes the last seen sort key (e.g. ?after=eyJpZCI6MTIzfQ&limit=20). The query becomes WHERE (created_at, id) < (:lastTs, :lastId) ORDER BY created_at DESC, id DESC LIMIT 20, which uses the index directly - O(limit), not O(offset). It is stable under inserts and the right default for large, frequently-changing or infinite-scroll datasets. The cost: you cannot jump to "page 47".

AspectOffsetCursor/Keyset
Random page accessYesNo (next/prev only)
Performance on deep pagesDegradesConstant
Stable under writesNo (drift)Yes
Best forSmall, admin tables, page numbersFeeds, logs, large/live data

Interview Favorite

The "drift" point wins this question. Saying "offset pagination double-counts or skips rows when the underlying data changes between pages, so for a live feed I default to keyset" demonstrates you have actually paginated production data.

Q10. How do you design filtering and sorting query parameters?

ROI: ★★★☆☆ Frequency: Medium

Short Answer

  • Filtering: field-named query params - GET /orders?status=PAID&customerId=42. For ranges, use suffixes or paired params: ?createdAfter=2026-01-01&amountGte=100.
  • Sorting: a sort param with a sign or direction - ?sort=-createdAt,name (descending createdAt, then ascending name). Spring Data's Pageable accepts ?sort=createdAt,desc natively.
  • Field selection / sparse fieldsets: ?fields=id,name,total to trim payload size.
  • Whitelist every filterable and sortable field server-side. Never feed a raw client field directly into a SQL ORDER BY or dynamic query - that is an injection vector (see the SQL chapter).
@GetMapping("/orders")
public Page<OrderDto> search(
        @RequestParam(required = false) OrderStatus status,
        @PageableDefault(size = 20, sort = "createdAt",
                         direction = Sort.Direction.DESC) Pageable pageable) {
    return orderService.search(status, pageable);
}

Common Mistake

Allowing arbitrary sort/filter fields straight from the client. Always map to an allow-list of known columns; reject unknown fields with 400. Convenience here becomes a SQL-injection or performance footgun in production.

Q11. How do you design a consistent error response format?

ROI: ★★★★★ Frequency: High

Short Answer

Every error from your API should share one machine-readable shape so clients can parse failures uniformly. The standard is RFC 7807 (Problem Details for HTTP APIs), served as Content-Type: application/problem+json: 

{
  "type": "https://api.acme.com/errors/insufficient-funds",
  "title": "Insufficient funds",
  "status": 422,
  "detail": "Account 42 balance 10.00 is below the requested 50.00",
  "instance": "/api/v1/accounts/42/transfers",
  "traceId": "a1b2c3d4"
}

Spring Boot 3 ships ProblemDetail and supports this natively. Centralize error mapping in a single @RestControllerAdvice (see the Spring Boot chapter for @ControllerAdvice) so you never scatter error formatting across controllers: 

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(InsufficientFundsException.class)
    public ProblemDetail handle(InsufficientFundsException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(
                HttpStatus.UNPROCESSABLE_ENTITY, ex.getMessage());
        pd.setTitle("Insufficient funds");
        pd.setType(URI.create("https://api.acme.com/errors/insufficient-funds"));
        pd.setProperty("traceId", MDC.get("traceId"));
        return pd;
    }
}

Always include a correlation/trace id so a support ticket maps to a log line (see distributed tracing in the Microservices chapter). Never leak stack traces or SQL to clients.

Interview Gold Point

Mentioning RFC 7807 / application/problem+json by name immediately signals you have designed real public APIs. Bonus: note that error type should be a stable, documented URI clients can branch on - more reliable than parsing the human-readable detail string.

Q12. How do you handle request validation in a REST API?

ROI: ★★★★☆ Frequency: Medium

Short Answer

Validate at the edge, declaratively, and return a structured 400/422 listing every field error at once (not one at a time). In Spring, use Bean Validation (Jakarta Validation) annotations on the request DTO plus @Valid on the controller parameter: 

public record CreateUserRequest(
        @NotBlank String name,
        @Email String email,
        @Min(18) int age) {}

@PostMapping("/users")
public ResponseEntity<UserDto> create(@Valid @RequestBody CreateUserRequest req) { ... }

A failed @Valid throws MethodArgumentNotValidException, which you map in the advice to a problem+json body whose errors array lists each rejected field and message. Distinguish two layers: syntactic/format validation (annotations, 400) and semantic/business validation (service layer, often 422), e.g. "this coupon is expired" - a rule annotations cannot express.

Common Mistake

Returning only the first validation error. Clients (and forms) want all errors in one response. Aggregate field errors into a list so the user fixes everything in one round trip.

SECTION 5 - ADVANCED API DESIGN

Q13. What are idempotency keys and why are they needed for POST?

ROI: ★★★★★ Frequency: High

Short Answer

POST is not idempotent, so a client that times out and retries a "create payment" call risks charging the customer twice. The fix is an idempotency key: the client generates a unique key (e.g. a UUID) and sends it in a header on the POST: 

POST /api/v1/payments
Idempotency-Key: 7c9e6679-7425-40de-944b-e07fc1f90ae7

The server stores the key together with the result of the first successful request (typically in Redis or a dedicated table). On a retry with the same key, it skips re-processing and replays the original response. This is exactly how Stripe and other payment APIs make POST safely retryable.

Key implementation points:

  • Persist key -> (status, response body) with a TTL (e.g. 24h).
  • While the first request is in flight, lock the key so concurrent retries get 409 or wait, not a double-process.
  • Optionally hash the request body and reject reuse of the same key with a different body (400) - protects against accidental key reuse.

Interview Gold Point

Tie it back: "GET/PUT/DELETE are idempotent by HTTP semantics, so retries are inherently safe; POST is not, so I add an idempotency key for any create that has financial or side-effecting impact." That sentence connects Q4 to a real production pattern and is a strong senior answer.

Q14. How do you implement rate limiting and what does the API return?

ROI: ★★★★☆ Frequency: Medium

Short Answer

Rate limiting protects the service from abuse and noisy neighbours and enforces quotas. Common algorithms (covered in the System Design chapter) are token bucket, leaking bucket, fixed window and sliding window; token bucket is the usual default because it allows bursts up to capacity.

When a client exceeds its quota, return 429 Too Many Requests, and tell the client how to behave: 

HTTP/1.1 429 Too Many Requests
Retry-After: 30
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1718553600
  • Retry-After - seconds (or a date) to wait before retrying; well-behaved clients honour it with backoff.
  • X-RateLimit-Limit / -Remaining / -Reset - let clients self-throttle before hitting the limit.

Enforce it at the gateway/edge where possible (API Gateway, NGINX, a filter), backed by a shared store like Redis so the limit is global across instances - not per-node in-memory, which a horizontally scaled service would multiply.

Interview Favorite

Stress the shared counter: "In-memory rate limiting breaks the moment you scale to N pods, because each enforces the limit independently - the effective limit becomes N times the intended one. I keep the counter in Redis."

Q15. Explain HATEOAS, HTTP caching (ETag) and content negotiation.

ROI: ★★★★☆ Frequency: Medium

Short Answer

HATEOAS (Hypermedia As The Engine Of Application State) is REST Level 3: responses embed links describing valid next actions, so the client discovers transitions instead of hard-coding URLs. 

{
  "id": 42,
  "status": "PENDING",
  "_links": {
    "self":   { "href": "/orders/42" },
    "cancel": { "href": "/orders/42/cancel" },
    "pay":    { "href": "/orders/42/payment" }
  }
}

Caching with ETag: the server returns an ETag (a hash/version of the representation). The client caches it and on the next read sends If-None-Match. If unchanged, the server responds 304 Not Modified with no body, saving bandwidth. For state-changing requests, ETag + If-Match gives optimistic concurrency: a stale write fails with 412 Precondition Failed (or 409). Cache-Control (e.g. max-age, no-store, private) governs how long and where a response may be cached. 

GET /orders/42
If-None-Match: "v7"          --> 304 Not Modified (empty body) if still version 7

Content negotiation: the client states what it wants via Accept (e.g. application/json vs application/xml) and describes its body via Content-Type; the server selects a representation or returns 406 Not Acceptable. This is also the mechanism behind media-type API versioning (Q8).

Interview Gold Point

The ETag + If-Match concurrency angle is gold: "ETags do double duty - 304 for caching on reads, and 412/409 optimistic locking on writes, so two clients can't silently clobber each other." This links REST to the optimistic-locking discussion in the JPA chapter.

SECTION 6 - DOCUMENTATION & ALTERNATIVES

Q16. How do you document a REST API, and how do REST, GraphQL and gRPC compare?

ROI: ★★★★★ Frequency: High

Short Answer

Documentation: the standard is the OpenAPI Specification (formerly Swagger) - a machine-readable contract for endpoints, schemas, status codes and auth. In Spring Boot, springdoc-openapi generates this automatically from your controllers and DTOs and serves an interactive Swagger UI: 

<dependency>
    <groupId>org.springdoc</groupId>
    <artifactId>springdoc-openapi-starter-webmvc-ui</artifactId>
    <version>2.6.0</version>
</dependency>

# then:
#   /v3/api-docs        -> the OpenAPI JSON
#   /swagger-ui.html    -> interactive docs

@Operation(summary = "Create an order", responses = {
    @ApiResponse(responseCode = "201", description = "Order created"),
    @ApiResponse(responseCode = "422", description = "Validation failed")
})
@PostMapping("/orders")
public ResponseEntity<OrderDto> create(@Valid @RequestBody CreateOrderRequest req) { ... }

Generating docs from code keeps them in sync; you can also drive client SDK generation from the OpenAPI file. A "contract-first" variant writes the OpenAPI spec first and generates the server stubs.

REST vs GraphQL vs gRPC:

AspectRESTGraphQLgRPC
Transport / formatHTTP/1.1+ , JSONHTTP, JSONHTTP/2, Protobuf (binary)
Data fetchingFixed per endpoint; over/under-fetchClient picks exact fields; one round tripFixed, defined by .proto contract
ContractOpenAPIGraphQL schemaProtobuf IDL (strongly typed, codegen)
StreamingLimited (SSE)SubscriptionsFirst-class bidirectional streaming
CachingNative HTTP caching (great)Hard (mostly POST)Not HTTP-cacheable
Best forPublic APIs, CRUD, broad client supportRich frontends with varied data needs, aggregationLow-latency internal service-to-service

Rule of thumb: REST for public-facing CRUD APIs where HTTP caching and ubiquity matter; GraphQL when diverse clients need to shape their own queries and avoid over-fetching; gRPC for high-throughput, low-latency internal microservice communication (see the Microservices chapter), where the binary Protobuf payload and HTTP/2 multiplexing win.

Interview Favorite

"They are not mutually exclusive." A mature system commonly exposes REST/GraphQL at the edge for clients and uses gRPC internally between services. Saying you choose per-boundary, not per-religion, signals architectural maturity.

QUICK REVISION SHEET

  • REST constraints (6): Client-Server, Stateless, Cacheable, Uniform Interface, Layered System, Code-on-Demand (optional). Statelessness enables horizontal scaling.
  • Richardson Maturity: L0 RPC -> L1 resources -> L2 verbs+status (industry norm) -> L3 HATEOAS.
  • Methods: GET read, POST create (server id), PUT full replace / create at known id, PATCH partial, DELETE remove.
  • Safe = no state change (GET/HEAD/OPTIONS). Idempotent = same state after N calls (GET, PUT, DELETE, HEAD). POST & PATCH not guaranteed.
  • Status codes: 200 OK, 201 Created (+Location), 204 No Content, 304 Not Modified, 400 malformed, 401 not authenticated, 403 not allowed, 404 missing, 409 conflict/stale version, 422 semantic validation, 429 rate limited, 5xx server.
  • URIs: plural nouns, lowercase-hyphenated, shallow nesting, query params for filter/sort/page, methods carry the verb.
  • Versioning: URI (common, cache-friendly), header/media-type (purist), query (avoid). Best version bump = none; prefer additive changes.
  • Pagination: offset (random access, drifts, slow deep pages) vs cursor/keyset (stable, O(limit), no jump) - keyset for live/large data.
  • Errors: RFC 7807 problem+json, central @RestControllerAdvice, include traceId, never leak stack traces.
  • Validation: Bean Validation + @Valid at the edge (400/422), return all field errors at once.
  • Idempotency key: header (UUID) stored with first result; replays make POST safely retryable (payments).
  • Rate limit: 429 + Retry-After + X-RateLimit-* headers; shared (Redis) counter, enforce at the edge.
  • Caching/concurrency: ETag + If-None-Match -> 304; ETag + If-Match -> 412/409 optimistic locking. Cache-Control governs lifetime.
  • Content negotiation: Accept / Content-Type; 406 if unsupported; basis of media-type versioning.
  • Docs: OpenAPI/Swagger via springdoc-openapi - auto-generated, interactive, drives SDK codegen.
  • REST vs GraphQL vs gRPC: REST for public CRUD + caching; GraphQL for flexible client queries; gRPC for fast internal service-to-service. Often combined per boundary.
