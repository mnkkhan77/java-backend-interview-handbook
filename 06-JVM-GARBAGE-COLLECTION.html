SECTION 1 - JVM ARCHITECTURE

Q1. What Is JVM?

ROI: ★★★★★ Frequency: Extremely High

Answer

The JVM (Java Virtual Machine) is an abstract computing machine that provides the runtime environment in which Java bytecode executes. It is the component that makes Java's "write once, run anywhere" promise real: the same compiled .class file runs unchanged on Windows, Linux or macOS because each platform ships a JVM that knows how to turn portable bytecode into native machine instructions for that hardware.

JVM (Java Virtual Machine) is responsible for:

  • Loading Classes
  • Executing Bytecode
  • Managing Memory
  • Garbage Collection
  • Security
  • Optimization

Beyond simply interpreting bytecode, a modern JVM such as HotSpot includes a Just-In-Time (JIT) compiler that profiles your running code and compiles hot paths into optimised native code, which is why a long-running Java service often gets faster after warming up. It also owns memory management and garbage collection, enforces bytecode verification for security, and abstracts the operating system away from your application code.

Flow


.java

↓

javac

↓

.class

↓

JVM

↓

Machine Code

The flow above is the lifecycle of Java code: you write .java source, javac compiles it into platform-independent .class bytecode, and the JVM loads, verifies and executes that bytecode, ultimately producing native machine code. Understanding this pipeline is the foundation for every other JVM and garbage-collection topic in this handbook.

Q2. JVM vs JRE vs JDK

ROI: ★★★★★ Frequency: Extremely High

JDK JRE JVM
Development Execution Virtual Machine
Contains JRE Contains JVM Runs Bytecode

These three are nested layers, not alternatives. The JVM is the engine that actually runs bytecode. The JRE (Java Runtime Environment) wraps the JVM together with the core class libraries needed to run an application, but it has no compiler. The JDK (Java Development Kit) is the full developer toolkit: it contains the JRE plus development tools such as javac, jar, javadoc and the diagnostic utilities (jmap, jstack).

A useful way to remember it: JDK is for building applications, JRE is for running them, and the JVM is the abstract machine inside the JRE that executes the bytecode. Note that since Java 11 the standalone JRE is no longer shipped separately by Oracle, so in practice teams install the JDK everywhere, but the conceptual distinction is still asked in interviews.

SECTION 2 - JVM MEMORY AREAS

Q3. JVM Memory Structure

ROI: ★★★★★ Frequency: Extremely High 


JVM Memory

├── Heap

├── Stack

├── Metaspace

├── PC Register

└── Native Method Stack

The JVM divides its runtime memory into distinct regions, each with a clear responsibility. The Heap is the large, shared area where all objects and arrays live and where garbage collection operates. Each thread gets its own Stack, which stores method call frames, local variables and partial results. Metaspace holds class metadata and lives in native memory outside the heap.

The remaining two are per-thread and smaller: the PC (Program Counter) Register tracks the address of the bytecode instruction the thread is currently executing, and the Native Method Stack supports native (non-Java) method calls made through JNI. Knowing which structures are shared across threads (Heap, Metaspace) versus per-thread (Stack, PC Register, Native Method Stack) is exactly the distinction that explains thread-safety and memory-error behaviour later in this file.

Q4. Heap vs Stack

ROI: ★★★★★ Frequency: Interview Favorite

Heap Stack
Objects Method Frames
Shared Per Thread
GC Managed Automatic Cleanup
Large Smaller

The Heap stores every object instance and is shared across all threads, which is why object access can require synchronization and why the heap is the domain of the garbage collector. The Stack is private to each thread and holds method frames containing local variables and the operand stack for that method call. Stack memory is reclaimed automatically and deterministically: when a method returns, its frame is popped, so there is no garbage collection involved.

This difference drives the two classic memory errors. Exhausting the heap produces an OutOfMemoryError, while exhausting a thread's stack (typically through deep or infinite recursion) produces a StackOverflowError. The stack is also far smaller than the heap by design, since each thread needs its own copy.

Example


User user = new User();

Reference → Stack Object → Heap

In the line above, the new User() object itself is allocated on the heap, while the local variable user — which merely holds a reference (a pointer) to that object — lives in the current method's stack frame. When the method returns, the reference disappears from the stack, and once no live reference points to the object, it becomes eligible for garbage collection on the heap.

Q5. What Happens When Object Is Created?

ROI: ★★★★★ Frequency: Extremely High

Steps

  1. Class Loaded
  2. Memory Allocated In Heap
  3. Fields Initialized
  4. Constructor Executes
  5. Reference Returned

When you call new, the JVM first ensures the class is loaded, linked and initialized (this happens only once per class). It then allocates a contiguous block of memory on the heap large enough for the object's fields, and zeroes that memory so every field starts at its default value (0, false or null). Next the constructor chain runs — instance initializers and the constructor body assign the real values — and finally a reference to the fully constructed object is returned and stored in the variable on the stack.

This ordering matters in interviews because it explains why fields have safe defaults before the constructor runs, and why an object can briefly exist in a partially-initialized state. In modern JVMs the actual heap allocation is extremely cheap thanks to thread-local allocation buffers (TLABs), which let each thread bump-allocate in its own slice of Eden without contending for a global lock.

Follow-Up

Where is reference stored? Answer: Stack

The object lives on the shared heap, but the reference to it is stored wherever the variable lives — for a local variable that means the current thread's stack frame. If the reference is instead an instance field of another object, it is stored inside that object on the heap, and if it is a static field it is reachable from the class metadata. The interviewer is checking that you do not confuse the location of the object with the location of the reference pointing to it.

Q6. What Is Metaspace?

ROI: ★★★★★ Frequency: High

Answer

Stores class metadata. Introduced in Java 8. Replaced PermGen.

Metaspace is the region where the JVM keeps class-level information — the runtime representation of each loaded class, including its methods, field layouts and constant pool. The critical change in Java 8 is that Metaspace lives in native memory rather than inside the Java heap, so it is no longer bounded by the heap size and grows dynamically as classes are loaded.

This move solved the chronic java.lang.OutOfMemoryError: PermGen space failures that plagued earlier versions, especially in application servers that repeatedly loaded and reloaded classes. It is still possible to exhaust Metaspace (you would see OutOfMemoryError: Metaspace), typically caused by classloader leaks or dynamically generating an unbounded number of classes, so it can be capped with -XX:MaxMetaspaceSize.

Contains

  • Class Definitions
  • Method Metadata
  • Runtime Information
Q7. PermGen vs Metaspace

ROI: ★★★★★ Frequency: High

PermGen Metaspace
Java 7 Java 8+
Fixed Size Dynamic
Inside JVM Heap Native Memory

PermGen (Permanent Generation) was the pre-Java-8 home for class metadata, and it had two painful limitations: it was a fixed-size region carved out of the Java heap, and its default size was small. Applications that loaded many classes — or worse, leaked classloaders during hot redeploys — would hit OutOfMemoryError: PermGen space even when plenty of heap remained.

Metaspace replaces it by moving metadata into native memory that grows automatically by default. This removed the most common reason developers had to tune -XX:MaxPermSize, but it shifted the risk: an unbounded Metaspace leak now consumes native OS memory instead. The practical senior takeaway is that you rarely tune Metaspace, but you should still set a ceiling in long-lived services to catch classloader leaks before they exhaust the machine.

SECTION 3 - GARBAGE COLLECTION BASICS

Q8. What Is Garbage Collection?

ROI: ★★★★★ Frequency: Extremely High

Answer

Automatic memory management process. Removes objects that are no longer reachable.

Garbage collection is the JVM's automatic memory-management mechanism. Instead of forcing developers to manually allocate and free memory the way C and C++ require, the JVM tracks which objects are still in use and reclaims the memory of objects that can no longer be reached by the running program. This eliminates entire categories of bugs — dangling pointers, double frees and most manual memory leaks — that dominate lower-level languages.

Benefit

Developers do not manually free memory.

The trade-off is that the developer gives up precise control over when memory is released and pays a runtime cost for the GC itself, including occasional pauses while it works. Understanding that GC is automatic but not free is the foundation for the performance and tuning discussions later in this handbook: a senior engineer's job is not to avoid GC but to write code that produces less garbage and to pick and tune the right collector.

Q9. How Does JVM Identify Garbage?

ROI: ★★★★★ Frequency: High

Answer

Using Reachability Analysis.

If an object cannot be reached from GC Roots, it becomes eligible for collection.

The JVM does not use reference counting (which cannot handle cyclic references). Instead it performs reachability analysis: starting from a set of known-live anchors called GC Roots, it traces every reference chain and marks every object it can reach as live. Anything not reached during this trace is, by definition, unreachable by the program and therefore garbage. This is why two objects that reference only each other but are not reachable from any root are still collected — the cycle is unreachable as a whole.

GC Roots

  • Local Variables
  • Static Variables
  • Active Threads
  • JNI References

GC Roots are the entry points the collector trusts as always live: local variables and parameters on active stack frames, static fields of loaded classes, live thread objects, and references held by native code through JNI. This list also explains the most common Java memory leaks — a static collection or a long-lived ThreadLocal keeps an object reachable from a root forever, so even though the object is logically useless, the GC correctly refuses to collect it.

Q10. Is Eligible For GC Same As Guaranteed GC?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

No. Eligibility only means JVM may collect it. There is no guarantee when GC will occur.

Eligibility and collection are two separate things. An object becomes eligible the moment it is no longer reachable from any GC Root, but the actual collection happens only when the JVM decides to run a GC cycle — typically when a memory region fills up. The object could be reclaimed milliseconds later, or it could sit in memory for a long time if there is no allocation pressure.

This is also why calling System.gc() is only a hint: the JVM is free to ignore it, and relying on it in production code is an anti-pattern. The senior-level point interviewers are listening for is that you never write code whose correctness depends on when an object is collected — that is exactly the mistake the old finalize() method encouraged, and why deterministic cleanup belongs in try-with-resources instead.

SECTION 4 - HEAP GENERATIONS

One of the most common JVM interview topics. Interviewers often ask: "Where does a newly created object go?" "Why is Young Generation needed?"

Q11. Heap Memory Structure

ROI: ★★★★★ Frequency: Extremely High 


Heap

├── Young Generation
│   ├── Eden
│   ├── Survivor S0
│   └── Survivor S1
│
└── Old Generation

The heap is split into generations based on a powerful empirical observation known as the weak generational hypothesis: most objects die young, and the few that survive tend to live a long time. The Young Generation is where all new objects are born, and it is subdivided into Eden (where objects are first allocated) and two Survivor spaces, S0 and S1, that hold objects which have survived at least one collection.

The Old Generation (also called the Tenured generation) holds objects that have survived enough Young Generation collections to be considered long-lived. This separation lets the JVM collect the Young Generation very frequently and cheaply while touching the Old Generation only rarely, which is the core idea behind generational garbage collection.

Q12. What Is Young Generation?

ROI: ★★★★★ Frequency: Extremely High

Answer

Newly created objects are allocated here.

The Young Generation is the nursery of the heap: nearly every object you create with new starts its life here, in Eden. It is sized to be collected often and quickly, because the generational hypothesis tells us most of these objects will become garbage almost immediately.

Reason

Most objects die young. Examples: Request DTOs Temporary Collections Method Variables

In a typical Spring Boot service this pattern is everywhere: a request DTO, the intermediate collections you build while processing it, and the local variables inside a method are all created to serve one request and are discarded the moment the response is sent. Because these short-lived objects dominate allocation, concentrating them in a small, frequently-collected region makes garbage collection fast — a Young Generation collection only has to copy out the handful of survivors and can discard everything else wholesale.

Q13. What Is Eden Space?

ROI: ★★★★★ Frequency: High

Answer

New objects are initially created in Eden.

Eden is the specific slice of the Young Generation where brand-new objects are allocated. Allocation here is extremely fast because it is essentially a pointer bump — the JVM simply advances a pointer to carve out the next chunk of memory, often within a thread-local allocation buffer so threads do not contend. When Eden fills up, a Minor GC is triggered.

Flow


New Object

↓

Eden

↓

Minor GC

↓

Survivor

↓

Old Generation

The flow above traces an object's possible journey: it is born in Eden, and if it is still reachable when a Minor GC runs, it is copied into a Survivor space rather than discarded. Each time it survives another collection its "age" counter increments, and once that age crosses a tenuring threshold the JVM promotes it to the Old Generation, assuming it has proven itself long-lived. The vast majority of objects never make it past the first step — they are simply reclaimed when Eden is collected.

Q14. What Are Survivor Spaces?

ROI: ★★★★★ Frequency: High

Answer

Temporary holding areas for objects that survive Minor GC. 


S0

S1

Objects move between survivor spaces before promotion.

There are two Survivor spaces, S0 and S1, and at any moment one is empty while the other holds the current survivors. During each Minor GC the JVM uses a copying algorithm: it copies all live objects from Eden and the in-use Survivor space into the previously empty Survivor space, then wipes the now-vacated regions clean. The two spaces simply swap roles on each cycle.

This ping-pong design serves a purpose — it compacts live objects together (eliminating fragmentation) and lets the JVM track how many collections each object has survived. Once an object's survival count reaches the tenuring threshold, instead of being copied to the other Survivor space it is promoted to the Old Generation. Having one Survivor space always empty is also why you should not be alarmed to see one of them showing zero usage in monitoring tools.

Q15. What Is Old Generation?

ROI: ★★★★★ Frequency: Extremely High

Answer

Stores long-lived objects.

The Old Generation (Tenured space) is where objects end up once they have survived enough Young Generation collections to be deemed durable. It is collected far less frequently than the Young Generation, but those collections are more expensive because the region is larger and the objects in it are densely packed with live data.

Examples

  • Application Caches
  • Singleton Beans
  • Large Data Structures
  • Long-Lived Sessions

The examples here are the kinds of objects that genuinely should live a long time: an in-memory cache that serves the whole application, the singleton Spring beans created at startup and held for the lifetime of the context, large reference data structures, and long-lived session state. Because these legitimately occupy the Old Generation, a healthy service shows a stable Old Generation; an Old Generation that keeps climbing over time is a classic memory-leak signature worth investigating.

Q16. What Is Object Promotion?

ROI: ★★★★★ Frequency: High

Answer

Moving surviving objects from Young Generation to Old Generation.

Why?

Object survived multiple garbage collection cycles. JVM assumes it is long-lived.

Promotion (also called tenuring) happens when an object's survival count crosses the tenuring threshold, signalling that it has outlived the assumption that "most objects die young." Rather than keep copying it back and forth between Survivor spaces on every Minor GC, the JVM moves it to the Old Generation where collections are rare, sparing the cost of repeatedly relocating an object that clearly is not going away.

A senior-level nuance worth mentioning is premature promotion: if the Young Generation or Survivor spaces are too small, objects get promoted before they have had a chance to die, polluting the Old Generation with short-lived garbage and triggering more frequent, expensive Old Generation collections. This is one of the most common reasons GC tuning involves resizing the Young Generation.

SECTION 5 - GARBAGE COLLECTION TYPES

Q17. What Is Minor GC?

ROI: ★★★★★ Frequency: Extremely High

Answer

GC performed on Young Generation.

A Minor GC collects only the Young Generation (Eden plus the active Survivor space). It is triggered when Eden fills up, and because it uses a copying algorithm that only needs to relocate the small set of survivors, it is fast. Importantly, even a Minor GC is a stop-the-world event — all application threads are briefly paused — but the pause is usually short enough to be imperceptible.

Characteristics

  • Frequent
  • Fast
  • Usually Low Impact

Minor GCs are frequent precisely because the Young Generation is small and high-churn, but each one is cheap, so in a healthy application they barely register. The pattern to watch for in production is not the frequency of Minor GCs but whether each one is promoting too much into the Old Generation — that is the early warning sign of a heap that is mis-sized or an application that is retaining objects longer than it should.

Q18. What Is Major GC?

ROI: ★★★★★ Frequency: High

Answer

GC performed on Old Generation.

A Major GC collects the Old Generation, where long-lived objects accumulate. Because the Old Generation is larger and its objects are mostly live (they earned their place by surviving), reclaiming space there involves more work — marking live objects, then sweeping or compacting — and therefore causes longer pauses than a Minor GC.

Characteristics

  • Less Frequent
  • More Expensive
  • Longer Pauses

The terminology can be ambiguous in interviews, so it is worth being precise: in many JVMs a Major GC effectively coincides with a Full GC because cleaning the Old Generation often requires touching the whole heap. The practical concern is the same regardless of the label — Major and Full collections are the ones that produce the noticeable application pauses, so reducing how often objects reach the Old Generation is a key tuning goal.

Q19. What Is Full GC?

ROI: ★★★★★ Frequency: Extremely High

Answer

Collection of entire heap.

A Full GC collects the entire heap — Young Generation, Old Generation, and typically Metaspace as well — in a single, comprehensive pass. It is the heaviest collection the JVM performs and almost always involves a long stop-the-world pause during which every application thread is frozen.

Impact

Can cause significant application pauses.

During a Full GC the application does no useful work, so requests queue up and latency spikes. A Full GC is triggered by conditions such as the Old Generation filling up, Metaspace pressure, or an explicit (and discouraged) System.gc() call. In a latency-sensitive service even a single multi-second Full GC can cause request timeouts, health-check failures and cascading retries.

Interview Gold Point

Frequent Full GC is often a production problem.

If you see Full GCs happening repeatedly, it is rarely a tuning detail — it is usually a symptom of a real defect. The most common root causes are a memory leak slowly filling the Old Generation, an undersized heap for the workload, or excessive premature promotion stuffing short-lived objects into Old. The strong interview answer is to treat recurring Full GC as a signal to capture GC logs and a heap dump and find what is being retained, rather than to simply enlarge the heap and hope.

Q20. Why Is Full GC Dangerous?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

  • Long Stop-The-World Pauses
  • Reduced Throughput
  • API Latency Spikes
  • User Experience Impact

The danger of a Full GC is that it is a "stop-the-world" event: the JVM suspends every application thread while it walks and cleans the entire heap. For the duration of that pause your service is effectively offline — it accepts no work, completes no requests, and answers no health checks. In a system with tight latency SLAs, a Full GC pause of even a second or two translates directly into a visible reliability incident.

Production Symptom

Spring Boot APIs suddenly become slow.

In practice this shows up as a Spring Boot service whose response times are normally fast but periodically spike, often in a sawtooth pattern that lines up exactly with Full GC events in the GC logs. Downstream, load balancers may mark the instance unhealthy during the pause and clients may retry, amplifying the load. The senior engineer's instinct is to correlate the latency spikes with GC logs, confirm Full GC is the cause, and then attack the underlying reason — a leak, an undersized heap, or the wrong collector — rather than masking the symptom.

SECTION 6 - GARBAGE COLLECTORS

Q21. Common Garbage Collectors

ROI: ★★★★★ Frequency: High

  • Serial GC
  • Parallel GC
  • CMS (Deprecated)
  • G1 GC
  • ZGC
  • Shenandoah

The JVM ships with several collectors, each optimised for a different balance of throughput, latency and footprint. Serial GC uses a single thread and is fine only for small, single-core or low-memory environments. Parallel GC (the "throughput collector") uses multiple threads to maximise total work done, at the cost of longer pauses, which suits batch jobs. CMS (Concurrent Mark Sweep) historically reduced pause times by doing most marking concurrently, but it has been removed in recent JDKs.

G1 GC is the modern default and aims for a predictable balance of low pauses and good throughput on multi-gigabyte heaps. ZGC and Shenandoah are the newest generation of concurrent collectors, designed to keep pauses in the sub-millisecond range even on very large heaps. The senior takeaway is that there is no universally "best" collector — the right choice depends on whether your application prioritises throughput (Parallel) or low and predictable latency (G1, ZGC, Shenandoah).

Q22. What Is G1 Garbage Collector?

ROI: ★★★★★ Frequency: Extremely High

Answer

G1 (Garbage First) is the default collector in modern JVMs.

G1, the Garbage-First collector, has been the default since Java 9 and is designed for server applications running on multi-core machines with large heaps. Its defining innovation is that it divides the heap into many equally-sized regions and dynamically labels each region as Eden, Survivor or Old, rather than using fixed contiguous generations. It then collects the regions that contain the most garbage first — hence "Garbage First" — to reclaim the most memory for the least effort.

Features

  • Low Pause Times
  • Region-Based Heap
  • Predictable Performance
  • Large Heap Friendly

The headline feature is that G1 lets you specify a soft pause-time goal (via -XX:MaxGCPauseMillis), and it sizes its collection work to try to meet that target, which gives far more predictable latency than the older collectors. Because it works region by region and does much of its marking concurrently with the application, it handles large heaps gracefully without the long, monolithic pauses that Parallel GC would incur on the same heap.

Interview Gold Point

G1 divides heap into regions instead of fixed generations.

This is the single sentence that signals you actually understand G1 rather than just naming it. In the older collectors the Young and Old generations are fixed, contiguous blocks of memory; in G1 the generations are logical labels applied to a grid of uniform regions, and a region can be reassigned from one role to another between collections. This region model is what enables incremental collection, compaction without a full-heap pass, and the tunable pause-time targets that make G1 the sensible default for most enterprise services.

Q23. Why G1 GC Is Popular?

ROI: ★★★★★ Frequency: High

Answer

Balances:

  • Latency
  • Throughput
  • Memory Efficiency

Suitable for most enterprise applications.

G1 is popular because it is the sensible "general-purpose" default: it delivers low and predictable pause times without sacrificing too much throughput, and it scales comfortably from a few gigabytes up to very large heaps. For the overwhelming majority of Spring Boot services and enterprise web applications, G1 gives good behaviour out of the box with little or no tuning, which is exactly why it was made the default.

The honest senior framing is that G1 wins on being a strong all-rounder rather than being the best at any single metric. If you needed absolute maximum throughput for a batch workload you might choose Parallel GC, and if you needed sub-millisecond pauses on a huge heap you would reach for ZGC or Shenandoah — but for typical request/response services, G1's balance is hard to beat, so it is the safe choice you should be prepared to justify.

Q24. What Is ZGC?

ROI: ★★★★★ Frequency: Increasingly Common

Answer

Low-latency garbage collector designed for very large heaps.

ZGC is a scalable, concurrent collector built specifically to keep pause times extremely low regardless of heap size. It achieves this by performing almost all of its work — including the expensive relocation of live objects — concurrently with the running application, using techniques such as coloured pointers and load barriers so that threads almost never have to stop. It scales from small heaps up into the terabyte range.

Goal

Keep pause times extremely low.

ZGC's design goal is for pause times to stay in the sub-millisecond range and, crucially, to remain flat as the heap grows — a 10 GB heap and a 1 TB heap should see similar pauses. This is fundamentally different from older collectors whose pause times grow with the amount of live data, and it is the property that makes ZGC attractive for systems where even occasional tens-of-milliseconds GC pauses are unacceptable.

Typical Use Cases

  • Large Scale Systems
  • Trading Systems
  • High Throughput APIs

The natural fit for ZGC is latency-critical workloads on large heaps: financial trading platforms where microseconds matter, real-time bidding and gaming backends, and high-throughput APIs with strict tail-latency requirements. The trade-off to mention is that this near-pauseless behaviour costs some throughput and additional memory overhead compared with G1 or Parallel, so ZGC is the right tool when latency is the dominant requirement rather than a default for every service.

SECTION 7 - MEMORY PROBLEMS

Q25. What Is Memory Leak In Java?

ROI: ★★★★★ Frequency: Extremely High

Important Point

Java can still have memory leaks.

A common misconception is that garbage collection makes memory leaks impossible. It does not. Garbage collection only reclaims objects that are unreachable — so if your code unintentionally keeps a reference to an object you no longer need, the GC will faithfully keep it alive forever. In Java a memory leak is therefore a logical leak: the memory is still referenced but is no longer useful.

How?

Objects remain reachable but no longer useful.

Examples

  • Static Collections
  • Cache Misuse
  • Listener Leaks
  • ThreadLocal Leaks

The classic culprits all share the pattern of an object staying reachable from a GC Root indefinitely. A static collection lives for the entire application lifetime, so anything added and never removed accumulates forever. A cache without an eviction policy grows without bound. Registered listeners or callbacks that are never deregistered keep their target objects alive. And a ThreadLocal on a pooled thread (such as a Tomcat request thread) can pin objects long after the request ends because the thread itself is never garbage collected. Recognising these patterns — and removing entries, bounding caches, and cleaning up ThreadLocals in a finally block — is the practical defence.

Q26. Example Of Memory Leak

ROI: ★★★★★ Frequency: High 


private static List users
    = new ArrayList<>();

Objects continuously added. Never removed. Memory grows forever.

This is the textbook Java leak. Because the list is static, it is reachable from a GC Root (the class) for the entire life of the application, and every User it holds is therefore reachable too. If code keeps appending users — for example caching every request's user without ever evicting them — the list grows unbounded, the Old Generation steadily fills, Full GCs become more frequent and longer, and eventually the JVM throws an OutOfMemoryError.

The fix depends on intent: if the collection is meant to be a cache, give it a bounded, evicting implementation (such as a size- or time-limited cache rather than a raw ArrayList); if entries become obsolete, remove them explicitly; and be especially wary of static collections, since their root-level reachability is exactly what turns an ordinary list into a leak.

Q27. What Is OutOfMemoryError?

ROI: ★★★★★ Frequency: Extremely High

Answer

JVM cannot allocate required memory.

An OutOfMemoryError is thrown when the JVM cannot allocate memory for a new object and the garbage collector cannot free enough space to satisfy the request. It is an Error, not an Exception, signalling that the situation is generally not recoverable by ordinary application logic. The most common variant is OutOfMemoryError: Java heap space, but you can also see Metaspace, GC overhead limit exceeded (when the JVM spends almost all its time in GC reclaiming almost nothing), and unable to create new native thread.

Common Causes

  • Memory Leak
  • Huge Collections
  • Insufficient Heap
  • Infinite Object Creation

The causes fall into two buckets: the heap is genuinely too small for a legitimate workload (just raise -Xmx), or — far more often in production — something is retaining objects that should have been released, i.e. a leak. The professional response is to run the JVM with -XX:+HeapDumpOnOutOfMemoryError so a heap dump is captured automatically when it happens, then analyse that dump in a tool like Eclipse MAT to find the dominator objects and reference chains holding the memory, rather than blindly increasing the heap and merely delaying the crash.

Q28. What Is StackOverflowError?

ROI: ★★★★★ Frequency: High

Answer

Occurs when call stack exceeds available stack memory.

A StackOverflowError happens when a thread's call stack runs out of space. Every method call pushes a new frame (holding parameters, local variables and the return address) onto the thread's stack, and that stack has a fixed maximum size. When the nesting of calls goes too deep, the stack limit is breached and the JVM throws this error. Unlike OutOfMemoryError, it concerns the per-thread stack rather than the shared heap.

Common Cause


Infinite Recursion

By far the most common cause is unbounded recursion — a method that calls itself (directly or through a cycle) without a correct base case, so the frames pile up until the stack is exhausted. Less commonly it can result from extremely deep but legitimate recursion on a problem that should have been solved iteratively. The fix is to add or correct the base case, convert deep recursion to an iterative loop, and only as a last resort increase the thread stack size with -Xss.

Q29. OutOfMemoryError vs StackOverflowError

OOM SOE
Heap Related Stack Related
Memory Exhaustion Deep Recursion

The two errors look similar in name but originate in completely different memory regions, which is exactly why the comparison is asked. OutOfMemoryError is about the shared heap (or related native areas like Metaspace) running out of allocatable space, typically due to a memory leak, oversized data structures, or an undersized heap. StackOverflowError is about a single thread's call stack overflowing, almost always due to runaway recursion.

Both extend Error rather than Exception, so neither should normally be caught and "handled" in business logic. The clean mental model to state in an interview is: heap problem versus stack problem, and memory-retention bug versus recursion bug. Getting that distinction right immediately shows you understand the JVM memory model from Section 2.

SECTION 8 - PRODUCTION TROUBLESHOOTING

Q30. Spring Boot Service Memory Keeps Growing. What Do You Check?

ROI: ★★★★★ Frequency: Senior Favorite

Answer Framework

  1. Heap Usage
  2. GC Logs
  3. Heap Dumps
  4. Large Collections
  5. Caches
  6. ThreadLocals
  7. Static References

This is a leak investigation, and interviewers want to hear a systematic process rather than guesses. Start by confirming the symptom with monitoring: watch heap usage over time and check whether the Old Generation trends upward across GC cycles instead of returning to a stable baseline — a steadily rising post-GC floor is the signature of a leak rather than normal churn. Then enable and read the GC logs to see whether Full GCs are becoming more frequent and reclaiming less each time.

Once a leak is confirmed, capture a heap dump and analyse it to find what is being retained. The usual suspects map directly to the list above: unbounded or unevicted caches, large collections that only ever grow, static references holding objects for the application's lifetime, and ThreadLocal values left on pooled request threads. Walking the reference chain from the biggest retained objects back to a GC Root tells you precisely which code is holding on.

Tools

  • jmap
  • jstack
  • VisualVM
  • JMC
  • MAT

Each tool has a role: jmap captures a heap dump (or prints a histogram of live objects), jstack dumps thread stacks to spot stuck or runaway threads, VisualVM and Java Mission Control (JMC) give live monitoring of heap, GC and threads, and Eclipse Memory Analyzer (MAT) is the gold standard for offline heap-dump analysis with its leak-suspect reports and dominator tree. Naming these and knowing what each is for is what distinguishes an engineer who has actually debugged a production leak from one who has only read about them.

Q31. How Do You Analyze Heap Dumps?

ROI: ★★★★★ Frequency: High

Answer

Look for:

  • Largest Objects
  • Retained Memory
  • Leak Suspects
  • Reference Chains

Heap-dump analysis (typically in Eclipse MAT) starts by sorting objects by retained size rather than shallow size. Shallow size is just the object itself; retained size is all the memory that would be freed if that object were collected — including everything reachable only through it. The objects with the largest retained size are the "dominators," and they are where the memory is actually trapped.

From those dominators you follow the reference chain back to a GC Root using a "path to GC Roots" view, which reveals exactly which field, static reference or ThreadLocal is keeping the memory alive. Tools like MAT also generate an automatic Leak Suspects report that flags unusually large retained sets. The whole exercise is about answering one question: what is still holding a reference to memory that should have been released, and from where?

Q32. How Would You Reduce GC Pressure?

ROI: ★★★★★ Frequency: Senior Favorite

  • Reduce Object Creation
  • Reuse Objects Carefully
  • Optimize Collections
  • Remove Memory Leaks
  • Tune Heap Size
  • Choose Proper GC

GC pressure is the rate at which your application creates garbage; the more short-lived objects you allocate, the more often the collector must run. The highest-leverage fix is almost always to allocate less in hot paths — avoid creating throwaway objects inside tight loops, prefer streaming over materialising huge intermediate collections, and size collections like ArrayList and HashMap with their expected capacity so they do not repeatedly resize and discard backing arrays.

"Reuse objects carefully" is deliberately cautious advice: object pooling can help for genuinely expensive resources, but pooling cheap objects often does more harm than good because modern Young Generation collection of short-lived objects is extremely fast, and pooling can instead create leaks or false sharing. After reducing allocation, eliminate any real leaks (they force premature promotion and Full GCs), then tune heap sizing and finally select the collector that matches your latency-versus-throughput goal — for example G1 for balanced services or ZGC for ultra-low-pause requirements.
