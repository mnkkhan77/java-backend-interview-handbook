SECTION 1 - PROCESS VS THREAD

Q1. Difference Between Process And Thread

ROI: ★★★★★ Frequency: Extremely High

A process is an independent program in execution with its own dedicated address space, while a thread is a unit of execution that lives inside a process and shares that process's memory with sibling threads. Because each process is isolated, communication between processes requires explicit inter-process mechanisms, whereas threads can simply read and write the same heap objects.

That shared memory is precisely what makes threads both powerful and dangerous. It is why a single backend service can run hundreds of request-handling threads cheaply, but it is also why concurrency bugs such as race conditions and visibility problems exist. The entire rest of this handbook is essentially about controlling that shared state safely.

Process Thread
Independent Execution Unit Lightweight Execution Unit
Own Memory Shared Memory
Heavy Lightweight
Slower Context Switch Faster Context Switch

Example

Chrome Browser is a process. Each browser tab can have multiple threads handling rendering, networking and scripting. The process owns the memory; the threads divide the work inside it.

Q2. Why Use Multithreading?

ROI: ★★★★★ Frequency: Extremely High

Benefits

Multithreading exists to make better use of modern multi-core hardware and to keep applications responsive while work happens in the background. Instead of processing tasks strictly one after another, a service can run independent units of work in parallel, which raises overall throughput and reduces latency for end users.

  • Better CPU Utilization
  • Higher Throughput
  • Responsiveness
  • Concurrent Task Execution

The key distinction a senior engineer should make is between CPU-bound and IO-bound work. IO-bound tasks (database calls, HTTP requests) benefit massively from concurrency because threads spend most of their time waiting; CPU-bound tasks are limited by the number of cores. Threading is not free either, since context switching and contention add overhead, so it should be applied deliberately.

Project Examples

  • Email Sending
  • Report Generation
  • Notification Processing
  • Parallel API Calls

In a typical Spring Boot backend, these are exactly the tasks we push off the request thread. When a user submits a job application, we return the HTTP response immediately and let a separate worker thread send the confirmation email and trigger notifications, so the user never waits on slow downstream systems.

SECTION 2 - THREAD CREATION

Q3. Ways To Create Thread

ROI: ★★★★★ Frequency: Extremely High

Java offers several approaches to create and run threads, ranging from low-level (extending Thread) to high-level managed execution (ExecutorService). The first two define what work to run, while the later approaches focus on how the work is scheduled and managed.

Method 1


class MyThread
extends Thread

Method 2


class Task
implements Runnable

Method 3


Callable

Method 4


ExecutorService

Implementing Runnable or Callable is preferred over extending Thread because Java allows only single inheritance, so extending Thread wastes the one superclass slot and tightly couples the task to a thread. Runnable and Callable cleanly separate the unit of work from the mechanism that runs it, which is also what lets us hand the same task to a thread pool.

Senior Engineer Perspective

Direct Thread creation is rarely used in enterprise systems. Creating raw threads gives you no control over how many run at once, which can exhaust the operating system under load. In production we almost always use an ExecutorService or a framework-managed pool, because it bounds resource usage, reuses threads and centralises lifecycle and error handling.

Q4. Runnable vs Callable

ROI: ★★★★★ Frequency: Extremely High

Both Runnable and Callable represent a task to be executed on a thread, but Callable was introduced later to fix two limitations of Runnable: it can return a result and it can throw checked exceptions. This makes Callable the natural choice when you need to retrieve a computed value or propagate failures back to the caller through a Future.

Runnable Callable
No Return Value Returns Value
No Checked Exception Can Throw Exception

Examples

() -> sendEmail()

Use Runnable for fire-and-forget side effects such as sending an email where the caller does not need a result. Use Callable when the work produces a value the caller will consume, such as fetching a user, because the returned Future lets you both retrieve that value and observe any exception that was thrown during execution.

Q5. start() vs run()

ROI: ★★★★★ Frequency: Interview Classic

This is a classic trap because the two methods look interchangeable but behave completely differently. The distinction is whether a new thread of execution is actually created, or whether the code simply runs on the current thread like any ordinary method call.

run()

Calling run() directly is a normal method call. The code executes synchronously on the same thread that made the call, so no concurrency happens at all.

start()

Calling start() asks the JVM to create a new thread, register it with the scheduler, and then invoke run() on that new thread. This is the only way to achieve true parallel execution.

Example


thread.run();

No new thread. The body of run() executes on the calling thread. 


thread.start();

New thread created. A common follow-up is what happens if you call start() twice on the same Thread object: it throws IllegalThreadStateException, because a thread cannot be restarted once it has run.

Q6. Thread Lifecycle

ROI: ★★★★★ Frequency: Very High

A Java thread moves through a well-defined set of states managed by the JVM, exposed through the Thread.State enum. A thread starts in NEW, becomes RUNNABLE once start() is called, and the operating system scheduler decides when it actually gets CPU time to run. It may temporarily move to WAITING or BLOCKED, and finally reaches TERMINATED when run() completes. 


NEW

↓

RUNNABLE

↓

RUNNING

↓

WAITING /
BLOCKED

Interview Favorite

A favourite follow-up is the difference between WAITING and BLOCKED. A WAITING thread has voluntarily suspended itself (for example by calling wait() or join()) and will resume only when another thread signals it. A BLOCKED thread is involuntarily stuck trying to enter a synchronized region because another thread currently owns the monitor lock. The key nuance: WAITING is cooperative and self-induced, while BLOCKED is contention on a lock.

BLOCKED: the thread is waiting to acquire a monitor lock held by another thread.

Q7. sleep() vs wait()

ROI: ★★★★★ Frequency: Extremely High

Although both pause a thread, they belong to different classes and serve different purposes. sleep() is a static method on Thread used purely for timed pauses, while wait() is defined on Object and is part of the inter-thread coordination mechanism that works together with notify() and notifyAll().

sleep() wait()
Thread Class Object Class
Keeps Lock Releases Lock
Timed Pause Synchronization

Interview Gold Point

The decisive difference is lock behaviour. wait() releases the monitor lock it holds so other threads can enter the synchronized block and change state, then re-acquires the lock when it wakes. sleep() never releases any lock it is holding, so other threads stay blocked the entire time. A second gotcha: wait() must be called from inside a synchronized block, otherwise it throws IllegalMonitorStateException.

Q8. notify() vs notifyAll()

ROI: ★★★★★ Frequency: High

Both methods wake threads that are waiting on an object's monitor, but they differ in how many they wake. notify() wakes a single arbitrary waiting thread, while notifyAll() wakes every waiting thread so they can all re-compete for the lock.

notify notifyAll
One Thread All Waiting Threads

Senior Recommendation

notifyAll() is generally safer. With notify() you cannot control which thread is woken, so if multiple threads wait on the same monitor for different conditions, the JVM may wake one whose condition is not satisfied while leaving the right one asleep forever. This is the lost-wakeup problem. notifyAll() avoids it by letting every thread re-check its condition, which is why waiting threads should always re-test their guard in a while loop rather than an if.

SECTION 4 - SYNCHRONIZATION

Q9. What Is Synchronization?

ROI: ★★★★★ Frequency: Extremely High

Answer

Synchronization is the mechanism that controls access to shared mutable state so that only one thread can execute a protected section at a time. By serialising access, it prevents race conditions where overlapping reads and writes from multiple threads would otherwise corrupt the data.

Beyond mutual exclusion, synchronization also provides a visibility guarantee. When a thread exits a synchronized block it flushes its changes to main memory, and when another thread enters it reads the latest values. So synchronized gives you both atomicity and visibility, which is why it is the most fundamental concurrency tool in Java.

Example


synchronized void increment(){

}
Q10. Why Synchronization Needed?

ROI: ★★★★★ Frequency: Extremely High

Race Condition Example

Synchronization is needed because many operations that look atomic in source code are not atomic at the bytecode or CPU level. The classic example is incrementing a counter, which the developer reads as a single step but the machine performs as three. 


count++;

This is not atomic. It actually compiles to a read-modify-write sequence: 


Read

Modify

Write

If two threads interleave between the read and the write, both can read the same starting value, both increment it, and both write back the same result, so one update is silently lost. This is the textbook race condition, and it is why an unsynchronised counter under load reports fewer increments than actually occurred.

Q11. What Is Intrinsic Lock?

ROI: ★★★★★ Frequency: High

Answer

Every Java object has an associated internal monitor, also called the intrinsic lock. When a thread enters a synchronized method or block, it must first acquire this monitor, and it releases it on exit. Only one thread can hold a given object's monitor at a time, which is what enforces mutual exclusion.

For a synchronized instance method the lock is the object itself (this); for a static synchronized method it is the Class object. Understanding which object's monitor is being used matters, because two synchronized methods guarding the same data only protect it if they lock on the same object.

Q12. Synchronized Method vs Block

ROI: ★★★★★ Frequency: High

Both forms use an object's intrinsic lock, but they differ in how much code they protect. A synchronized method locks for its entire duration, while a synchronized block lets you lock only the critical section and choose exactly which object to lock on.

Method


synchronized void save(){}

The entire method body is locked on the instance.

Block


synchronized(lock){

}

Only the specific section inside the block is locked, and you can use a dedicated private lock object.

Senior Engineer Perspective

Prefer the smallest possible lock scope, because the more code you hold a lock for, the longer other threads are blocked, which reduces throughput and increases contention. Locking on a dedicated private final object rather than this is also safer, since it prevents external code from accidentally acquiring your lock and causing unexpected blocking or deadlocks.

SECTION 5 - VOLATILE

Q13. What Is volatile?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

The volatile keyword guarantees visibility: when one thread writes to a volatile field, every other thread is guaranteed to see the new value immediately rather than a stale cached copy. It also prevents the compiler and CPU from reordering instructions around the volatile access.

Without volatile, each thread may cache a field in a CPU register or core-local cache, so a change made by one thread can go unnoticed by another indefinitely. The most common real-world use is a boolean flag that signals a background thread to stop, which can loop forever if the flag is not volatile.

Example


private volatile boolean running;

Changes to running become visible to all threads immediately, so a worker loop checking it will reliably observe the stop signal.

Q14. volatile vs synchronized

ROI: ★★★★★ Frequency: Extremely High

These two are often confused because both relate to thread safety, but they solve different problems. volatile only guarantees visibility of a single variable and adds no locking, whereas synchronized guarantees both visibility and atomicity of an entire block by acquiring a lock.

volatile synchronized
Visibility Visibility + Atomicity
No Lock Uses Lock
Faster Slower

Interview Trap

The trap is assuming volatile makes compound operations thread safe. 


count++

This is still not thread safe even if count is volatile, because the increment is a read-modify-write sequence and volatile only protects each individual read or write, not the whole operation. For atomicity you need synchronized or an Atomic class. Use volatile only when a single thread writes and others merely read, such as a status flag.

Q15. Can volatile Solve Race Conditions?

ROI: ★★★★★ Frequency: Interview Favorite

Answer

No. volatile provides visibility but not atomicity. It ensures all threads see the latest value of a variable, but it does nothing to make a multi-step operation indivisible, so any operation that reads a value and then writes a new one based on it remains vulnerable to interleaving.

A race condition occurs when correctness depends on the timing of multiple threads accessing shared state, and fixing it requires guaranteeing that the read-and-update happens as one atomic unit. volatile cannot do that, so you must reach for synchronized, an explicit Lock, or an Atomic class instead.

Example


volatile int count;

count++;

Still unsafe, because count++ is three operations and two threads can interleave and lose an update despite the field being volatile.

SECTION 6 - ATOMIC CLASSES

Atomic classes are among the most misunderstood concurrency topics. Interviewers frequently ask: "If volatile is not enough, what should we use?"

Q16. What Are Atomic Classes?

ROI: ★★★★★ Frequency: Extremely High

Answer

Atomic classes, found in the java.util.concurrent.atomic package, provide variables that can be updated atomically without explicit locking. Operations like incrementing or compare-and-set execute as a single indivisible step, so multiple threads can update the same value concurrently without losing updates.

They are described as lock-free because instead of acquiring a monitor they rely on low-level CPU compare-and-swap instructions combined with volatile semantics. This makes them faster than synchronized under moderate contention and avoids the blocking and context-switching overhead of locks, which is exactly the answer to "if volatile is not enough, what do we use?"

Examples


AtomicInteger

AtomicLong

AtomicBoolean

AtomicReference
Q17. Why AtomicInteger Instead Of int?

ROI: ★★★★★ Frequency: Extremely High

A plain int cannot be incremented safely from multiple threads, even if marked volatile, because the increment is not atomic. AtomicInteger replaces that broken pattern with methods such as incrementAndGet() that perform the whole read-modify-write as a single atomic operation.

Unsafe


count++;

This actually performs three separate steps: 


Read

Increment

Write

Multiple threads can interleave between these steps and overwrite each other's updates.

Safe


AtomicInteger count =
new AtomicInteger();

count.incrementAndGet();

incrementAndGet() guarantees the read, increment and write happen atomically, so no update is lost no matter how many threads call it concurrently.

Q18. How AtomicInteger Works Internally?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

Internally, AtomicInteger holds a volatile int and updates it using compare-and-swap (CAS) rather than locks. Each update reads the current value, computes the new value, and then atomically swaps it in only if the current value still matches what it read; if another thread changed it in the meantime, the operation retries in a loop.

  • CAS (Compare And Swap)
  • CPU Instructions
  • Volatile Memory Semantics

The CAS itself maps directly to a single hardware instruction (such as CMPXCHG on x86), and the volatile field guarantees visibility of the result to other threads. Because there is no explicit synchronization, threads never block on each other; under heavy contention, however, the retry loop can spin, which is a trade-off worth mentioning.

SECTION 7 - CAS (COMPARE AND SWAP)

Q19. What Is CAS?

ROI: ★★★★★ Frequency: Extremely High

Concept

Compare-and-swap is an atomic CPU operation that takes three inputs: a memory location, an expected old value, and a new value. It updates the location to the new value only if the location currently holds the expected value; otherwise it does nothing and reports failure. The entire compare-then-set happens as one indivisible hardware instruction.

This single primitive is the foundation of lock-free programming. Algorithms use it in a retry loop: read the current value, compute the desired new value, then CAS; if it fails because another thread intervened, simply re-read and try again. This is how Atomic classes and many concurrent collections achieve thread safety without locks.

Example


Expected = 5

Current = 5

New = 6

Update Success
Q20. Advantages Of CAS

ROI: ★★★★★ Frequency: High

The core advantage of CAS is that it achieves thread safety without blocking. Because a thread never waits to acquire a lock, it cannot be suspended by the scheduler mid-operation, which avoids the expensive context switches and the risk of deadlock that come with lock-based approaches.

  • No Blocking
  • Better Performance
  • Reduced Context Switching
  • Lock-Free Operations

The trade-off, worth raising in a senior interview, is that CAS performs best under low to moderate contention. When many threads hammer the same variable, the retry loop can spin repeatedly and waste CPU, in which case a lock or a different data structure may actually perform better.

Q21. What Is ABA Problem?

ROI: ★★★★★ Frequency: Senior-Level Favorite

Scenario

The ABA problem is a subtle flaw in naive CAS usage. A value changes from A to B and then back to A between a thread's read and its CAS. The CAS succeeds because it only checks that the value still equals A, yet the underlying state may have changed in ways that matter, so the thread proceeds on a false assumption. 


A → B → A

CAS sees the same value A and assumes nothing changed, but the value actually changed twice in between. This is dangerous in structures like lock-free stacks, where a node could be popped, freed and recycled, leaving a dangling reference that CAS happily accepts.

Solution


AtomicStampedReference

The fix is to attach a version stamp or counter to the value, as AtomicStampedReference does, so CAS compares both the value and its stamp. Even if the value returns to A, the stamp will have advanced, so the stale CAS correctly fails.

SECTION 8 - EXECUTOR FRAMEWORK

Q22. Why Executor Framework?

ROI: ★★★★★ Frequency: Extremely High

Problem

Creating a new thread for every task is expensive: each thread consumes memory for its stack and requires operating system bookkeeping, and creation and teardown have real cost. Worse, if tasks arrive faster than they finish, unbounded thread creation can exhaust memory and crash the application.

Solution

The Executor framework solves this by decoupling task submission from task execution. You submit work, and a managed thread pool reuses a fixed set of threads to run it, queueing tasks when all threads are busy. This bounds resource usage, amortises thread creation cost, and gives you a clean API for lifecycle management, scheduling and result handling.

Q23. What Is ExecutorService?

ROI: ★★★★★ Frequency: Extremely High

Answer

ExecutorService is the central interface of the Executor framework for managing a pool of worker threads. It accepts tasks via execute() and submit(), schedules them onto pooled threads, and provides lifecycle methods such as shutdown() and awaitTermination() to drain and close the pool gracefully.

You typically obtain an implementation from the Executors factory, though in production it is often better to construct a ThreadPoolExecutor directly so you can control the queue type, pool sizes and rejection policy explicitly rather than relying on the defaults.

Example


ExecutorService executor =

Executors.newFixedThreadPool(10);
Q24. Benefits Of Thread Pools

ROI: ★★★★★ Frequency: High

A thread pool maintains a set of pre-created worker threads that are reused across many tasks, which delivers the following benefits. The recurring theme is that you trade unbounded, unpredictable thread usage for controlled, reusable capacity.

  • Reuse Threads
  • Lower Creation Cost
  • Better Resource Management
  • Improved Throughput

The most important benefit operationally is resource management: a bounded pool acts as a natural backpressure mechanism. If work arrives faster than it can be processed, tasks queue up instead of spawning unlimited threads, which keeps the service stable under load instead of crashing.

Q25. Fixed Thread Pool vs Cached Thread Pool

ROI: ★★★★★ Frequency: High

A fixed thread pool keeps a constant number of threads and queues any excess tasks, giving predictable, bounded resource usage. A cached thread pool creates new threads on demand and reuses idle ones, which is flexible for bursty, short-lived tasks but has no upper bound on thread count.

Fixed Cached
Fixed Threads Dynamic Threads
Predictable Potentially Unbounded

Interview Tip

Fixed thread pools are generally safer for production. A cached pool can spin up thousands of threads under a traffic spike and bring down the JVM with out-of-memory errors. Senior engineers often avoid the Executors factory entirely and configure a ThreadPoolExecutor with an explicit bounded queue and a sensible rejection policy, so overload degrades predictably rather than catastrophically.

Q26. submit() vs execute()

ROI: ★★★★★ Frequency: High

Both methods hand a task to the executor, but they differ in what they accept and what they return. execute() takes only a Runnable and returns nothing, while submit() accepts a Runnable or Callable and returns a Future you can use to retrieve the result or check for completion.

execute submit
No Future Returns Future
Runnable Runnable/Callable

There is an important exception-handling difference too. With execute(), an uncaught exception propagates to the thread's UncaughtExceptionHandler. With submit(), the exception is captured and stored in the Future, so it stays silent until you call future.get() and it is rethrown wrapped in an ExecutionException. Forgetting this is a common source of swallowed errors.

Q27. What Is Future?

ROI: ★★★★★ Frequency: High

Answer

A Future represents the eventual result of an asynchronous computation. When you submit a task to an executor you get a Future immediately, and you can later call get() to retrieve the result, isDone() to poll for completion, or cancel() to attempt cancellation.

The crucial behaviour to understand is that get() blocks the calling thread until the result is ready. This makes Future useful for kicking off work in parallel and then collecting results, but it also means careless use can reintroduce the very blocking you were trying to avoid.

Example


Future future =
executor.submit(task);
Q28. Problems With Future

ROI: ★★★★★ Frequency: Senior Favorite

Future was a first step toward async programming, but it is limited. Its biggest weakness is that the only way to read the result is the blocking get(), which ties up a thread and makes it awkward to build non-blocking pipelines.

  • Blocking get()
  • No Chaining
  • No Combination
  • Poor Composition

You also cannot register a callback to run when a Future completes, chain a transformation onto its result, or combine several Futures into one. CompletableFuture, introduced in Java 8, solves all of this with methods like thenApply, thenCompose and allOf, enabling fully non-blocking, composable async workflows.

SECTION 9 - LOCKS

Q29. synchronized vs ReentrantLock

ROI: ★★★★★ Frequency: Extremely High

synchronized is built into the language and acquires and releases the lock implicitly at block boundaries, which is simple and hard to misuse. ReentrantLock is a class-based lock that you acquire and release explicitly, giving you far more control at the cost of having to manage the unlock yourself.

synchronized ReentrantLock
Implicit Lock Explicit Lock
Less Flexible More Features
Auto Release Manual Unlock

ReentrantLock Features

  • tryLock()
  • Fair Locking
  • Interruptible Locking

These extra capabilities are why senior engineers reach for ReentrantLock in advanced scenarios: tryLock() lets a thread give up rather than block forever (useful for deadlock avoidance), fair locking grants the lock in request order to prevent starvation, and interruptible acquisition lets a blocked thread be cancelled. The catch is that you must always release in a finally block, otherwise a thrown exception leaks the lock permanently.

Q30. What Is ReentrantLock?

ROI: ★★★★★ Frequency: High

Answer

ReentrantLock is reentrant, meaning a thread that already holds the lock can acquire it again without blocking itself. The lock maintains a hold count that increments on each acquisition and decrements on each release, and the lock is only truly freed when the count returns to zero.

This property is essential whenever a synchronized or locked method calls another method that needs the same lock, such as recursion or one guarded method delegating to another. Without reentrancy, a thread would deadlock against itself. Note that Java's intrinsic synchronized locks are also reentrant for the same reason.

Example

Method A acquires the lock, then calls Method B which acquires the same lock. Because the lock is reentrant, this is allowed; the hold count simply rises to 2 and falls back as each method returns.

Q31. What Is ReadWriteLock?

ROI: ★★★★★ Frequency: High

Answer

A ReadWriteLock maintains a pair of locks: a shared read lock and an exclusive write lock. Many threads may hold the read lock simultaneously as long as no thread holds the write lock, but the write lock is exclusive and blocks all readers and other writers while held.

  • Readers
  • Writers

Benefit

This is a major optimisation for read-heavy workloads such as caches or configuration that is read constantly but updated rarely. Allowing concurrent reads dramatically improves throughput compared with a plain mutual-exclusion lock that would serialise every access. A caveat to mention is writer starvation: under a constant stream of readers, a waiting writer may be delayed, which is why ReentrantReadWriteLock offers a fair mode.

SECTION 10 - DEADLOCKS

Q32. What Is Deadlock?

ROI: ★★★★★ Frequency: Extremely High

Answer

A deadlock is a situation where two or more threads are each waiting for a resource the other holds, so none of them can ever proceed. Each thread is blocked indefinitely, and because no thread will release its lock until it acquires the one it is waiting for, the system is permanently stuck.

Example


Thread 1

Lock A

Waiting For Lock B

Thread 2

Lock B

Waiting For Lock A

Thread 1 holds A and wants B; Thread 2 holds B and wants A. Neither will give up what it has, so both wait forever and the affected part of the system hangs. In production this typically manifests as requests timing out while threads sit blocked, which you can confirm with a thread dump that shows the circular wait.

Q33. Deadlock Conditions

ROI: ★★★★★ Frequency: High

A deadlock can only occur when all four of the Coffman conditions hold simultaneously. This matters practically because breaking any single one of them is enough to prevent deadlock entirely, which is the basis of every prevention strategy.

  1. Mutual Exclusion
  2. Hold And Wait
  3. No Preemption
  4. Circular Wait

In short: a resource can be held by only one thread (mutual exclusion), threads hold resources while waiting for more (hold and wait), resources cannot be forcibly taken away (no preemption), and there is a cycle of threads each waiting on the next (circular wait). Circular wait is the easiest condition to break in practice, usually by imposing a global lock ordering.

Q34. How To Prevent Deadlocks?

ROI: ★★★★★ Frequency: High

Deadlock prevention works by ensuring at least one Coffman condition can never hold. The techniques below each target one of those conditions, and the most reliable in everyday code is consistent lock ordering.

  • Consistent Lock Ordering
  • Reduce Lock Scope
  • Use tryLock()
  • Avoid Nested Locks

If every thread always acquires locks in the same global order, a circular wait becomes impossible. Where ordering is hard to guarantee, tryLock() with a timeout lets a thread back off and retry instead of blocking forever, breaking the hold-and-wait condition. Keeping critical sections small and avoiding holding multiple locks at once further reduces the surface area for deadlock.

Q35. What Is Livelock?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

In a livelock, threads are not blocked but keep responding to each other's actions in a way that prevents any of them from making progress. They are busy and consuming CPU, repeatedly retrying or yielding, yet no useful work ever completes. The classic analogy is two people in a hallway who keep stepping the same direction to let each other pass.

Difference

Deadlock: the threads are completely stopped and idle, waiting on locks.

Livelock: the threads remain active and keep changing state, but cannot advance. Livelock often arises from naive deadlock-avoidance logic where threads release and retry in lockstep; adding randomised backoff usually breaks the symmetry and lets one proceed.

Q36. What Is Starvation?

ROI: ★★★★★ Frequency: High

Answer

Starvation occurs when a thread is perpetually denied the resources or CPU time it needs to make progress, because other threads keep getting priority. Unlike deadlock, the system as a whole keeps working; it is one or more specific threads that are continually passed over.

Common causes include unfair locks that always favour certain threads, thread priorities that let high-priority threads monopolise the CPU, or a steady stream of readers starving a writer in a read-write lock. The usual remedy is fairness policies, which grant access in roughly the order it was requested so no thread waits indefinitely.

SECTION 11 - THREAD SAFETY

Q37. What Is Thread Safety?

ROI: ★★★★★ Frequency: Extremely High

Answer

Code is thread safe if it continues to behave correctly when accessed by multiple threads at the same time, regardless of how the runtime schedules those threads, and without requiring extra coordination from the callers. Correctness here means the data stays consistent and the program's invariants hold no matter how operations interleave.

The need for thread safety arises specifically around shared mutable state. Stateless code and immutable objects are inherently thread safe because there is nothing to corrupt; the danger appears only when multiple threads can read and write the same changing data, which is what synchronization, locks and atomics exist to manage.

Q38. Ways To Achieve Thread Safety

ROI: ★★★★★ Frequency: Extremely High

There are several complementary strategies for thread safety, and a good engineer chooses based on the access pattern rather than reaching for the heaviest tool every time. They range from explicit locking to designs that avoid shared mutable state altogether.

  • Synchronization
  • Locks
  • Atomic Classes
  • Immutable Objects
  • Concurrent Collections

synchronized and explicit Locks provide mutual exclusion for arbitrary critical sections; Atomic classes give lock-free safety for single variables; concurrent collections like ConcurrentHashMap handle the locking internally for you; and immutable objects sidestep the problem entirely. The general preference is to avoid shared mutable state first, then use the lightest mechanism that the situation allows.

Q39. Why Are Immutable Objects Thread Safe?

ROI: ★★★★★ Frequency: Extremely High

Answer

Immutable objects are thread safe because their state is fixed at construction and can never change afterward. Since there are no writes after the object is published, there is nothing for concurrent threads to corrupt, and every thread always sees a consistent value without any synchronization.

This is why immutability is one of the most powerful and underused concurrency tools: it eliminates the entire class of race conditions by design rather than guarding against them at runtime. Once safely published, an immutable object can be shared freely across any number of threads, which also makes it ideal for caching and as map keys.
