SECTION 1 - FUNCTIONAL INTERFACES

Q1. What Is Functional Interface?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

A functional interface is an interface that contains exactly one abstract method. That single abstract method (often called the SAM, "Single Abstract Method") gives the compiler a target type to which a lambda expression or method reference can be assigned.

Example


@FunctionalInterface
public interface Calculator {

   int add(int a,int b);

}

Important Point

A functional interface may still declare any number of default and static methods; those do not count against the single-abstract-method rule because they already have an implementation. It may also redeclare public methods of Object (such as equals or toString) without breaking the contract.

The @FunctionalInterface annotation is optional but recommended. It tells the compiler to fail the build if someone accidentally adds a second abstract method, which protects the interface's role as a lambda target over time.

Q2. Why Functional Interfaces?

ROI: ★★★★★ Frequency: High

Answer

Functional interfaces are the bridge between Java's object-oriented type system and the functional programming style introduced in Java 8. Because the compiler can infer the single target method, behaviour can be passed around as a value rather than wrapped in a verbose anonymous class.

They enable:

  • Lambda Expressions
  • Method References
  • Functional Programming

Without functional interfaces there is no lambda support, because a lambda is simply a concise implementation of a functional interface's one abstract method. This is also why the entire Stream API and the java.util.function package are built on top of functional interfaces.

Q3. Common Functional Interfaces

ROI: ★★★★★ Frequency: Extremely High

Interface Method
Predicate test()
Function apply()
Consumer accept()
Supplier get()

These four are the most heavily used members of the java.util.function package. Predicate takes a value and returns a boolean (ideal for filtering), Function transforms an input of one type into an output of another, Consumer accepts a value and returns nothing (a side effect such as logging), and Supplier takes no input but produces a value (useful for lazy or deferred creation).

The same package also provides specialized variants such as BiFunction, BinaryOperator, UnaryOperator and the primitive forms like IntPredicate that avoid autoboxing in hot paths.

Examples

n -> n > 10;

String::length;

SECTION 2 - LAMBDA EXPRESSIONS

Q4. What Is Lambda Expression?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

A lambda expression is a concise way of providing the implementation of a functional interface inline, without the ceremony of an anonymous class. It captures only behaviour: the parameter list, an arrow token, and a body.

Before Java 8


new Runnable() {

   public void run() {

   }

};

Java 8


() -> {

}

Benefits

  • Less Boilerplate
  • Readable Code
  • Functional Style

Beyond brevity, lambdas have an important semantic difference from anonymous classes: they do not create a new scope for this. Inside a lambda, this refers to the enclosing instance, whereas inside an anonymous class this refers to the anonymous class itself. The compiler also implements lambdas via the invokedynamic bytecode instruction rather than generating a synthetic class file per lambda, which keeps startup and class-loading cost lower.

Q5. Can Lambda Access Local Variables?

ROI: ★★★★★ Frequency: High

Answer

Yes, a lambda can read local variables from its enclosing method, but only if the variable is:

  • final
  • effectively final

"Effectively final" means the variable is never reassigned after initialization, even if it is not explicitly marked final. The restriction exists because the lambda may outlive the stack frame that created it (for example when passed to another thread), so Java captures the value rather than a live reference. Allowing mutation would create confusing concurrency semantics and is therefore rejected at compile time. Note that instance and static fields are not subject to this rule and can be freely mutated from inside a lambda.

Example


int value = 10;

Runnable r =
() -> System.out.println(value);

SECTION 3 - METHOD REFERENCES

Q6. What Are Method References?

ROI: ★★★★★ Frequency: High

Answer

A method reference is a shorthand syntax for a lambda that does nothing but call an existing method. When a lambda's only job is to forward its arguments to another method, the method reference reads more clearly and signals intent directly.

Example


list.forEach(
System.out::println
);

Types

  • Static Method
  • Instance Method
  • Constructor Reference

There are four recognised forms: a reference to a static method (Integer::parseInt), a reference to an instance method of a particular object (system.out::println), a reference to an instance method of an arbitrary object of a particular type (String::length, where the first stream element becomes the receiver), and a constructor reference (User::new). Functionally a method reference is identical to the equivalent lambda; it is purely a readability improvement.

SECTION 4 - STREAM API

Q7. What Is Stream API?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

The Stream API is a functional, declarative approach for processing sequences of elements. Instead of describing how to iterate (the imperative loop), you describe what transformation you want, and the pipeline takes care of the mechanics.

Benefits

  • Readable
  • Declarative
  • Parallelizable

A stream is not a data structure; it does not store elements. It is a pipeline of operations over a source (a collection, array, I/O channel, or generator). Pipelines are built from intermediate operations (which are lazy and return a new stream) and a single terminal operation (which triggers execution and produces a result or side effect). Because the pipeline is described declaratively, switching to parallel execution is often a one-word change.

Example


users.stream()
     .filter(User::isActive)
     .collect(Collectors.toList());
Q8. Difference Between Collection And Stream

ROI: ★★★★★ Frequency: High

Collection Stream
Stores Data Processes Data
Reusable Single Use
Eager Lazy

A collection is an in-memory data structure focused on storage and access; you can iterate it as many times as you like. A stream is a one-shot view focused on computation. Once a terminal operation has consumed a stream it is closed, and attempting to reuse it throws IllegalStateException; you must obtain a fresh stream from the source.

The eager-versus-lazy distinction matters for performance. A collection materializes all its elements; a stream pulls elements through the pipeline only as the terminal operation demands them, which enables short-circuiting and processing of conceptually infinite sequences.

Q9. What Is Lazy Evaluation?

ROI: ★★★★★ Frequency: High

Answer

Lazy evaluation means intermediate operations such as filter and map are not executed when they are declared. They merely record what should happen. Nothing flows through the pipeline until a terminal operation is attached.

This design unlocks two important optimizations. First, the runtime can fuse operations so each element is passed through the whole chain once rather than creating an intermediate collection at every step. Second, short-circuiting operations like findFirst or limit can stop early, so an expensive map may run on only a handful of elements instead of the entire source.

Example


stream.filter(...)
      .map(...)

Nothing executes until a terminal operation appears: 


collect()

count()

findFirst()
Q10. filter() vs map()

ROI: ★★★★★ Frequency: Extremely High

filter()

filter takes a Predicate and keeps only the elements that satisfy it; it can reduce the number of elements but never changes their type. The stream that comes out contains a subset of the same objects that went in. 


.filter(age -> age > 18)

map()

map takes a Function and transforms each element into exactly one new element, possibly of a different type. The count of elements is unchanged, but the contents are converted, for example turning a stream of User into a stream of their names. 


.map(User::getName)
Q11. map() vs flatMap()

ROI: ★★★★★ Frequency: Extremely High

map()

map is a one-to-one transformation: each input element produces exactly one output element. If you apply map to a stream of lists you end up with a stream of lists, not a stream of their contents.

flatMap()

flatMap is a one-to-many transformation. Each input element is mapped to a stream, and flatMap then flattens all of those inner streams into a single continuous stream. This is exactly how you collapse nested structures into one level.

Example


List>

Convert into: 


List

Interview Favorite

Many interviewers use flatMap to evaluate genuine stream understanding, because candidates who only memorised syntax tend to reach for map and then get confused by the resulting Stream<Stream<T>>. A clean answer is: use map when each element becomes one element, and flatMap when each element expands into zero, one, or many elements that you want merged into a flat result.

Q12. Distinct Operation

ROI: ★★★★★ Frequency: High 


stream.distinct()

distinct removes duplicate elements from the stream, preserving the first occurrence and the encounter order. It relies on equals() and hashCode() to decide whether two elements are the same.

This is a common trap with custom objects: if your class does not override equals() and hashCode() correctly, distinct falls back to identity comparison and "obvious" duplicates are not removed. In a parallel stream, distinct on an ordered stream is also more expensive because order has to be reconciled across threads.

Q13. sorted()

ROI: ★★★★☆ Frequency: High 


stream.sorted()



stream.sorted(
Comparator.comparing(
User::getName
))

sorted() with no argument sorts using the elements' natural ordering, which requires them to implement Comparable; otherwise it throws ClassCastException at runtime. The overload that accepts a Comparator lets you sort by any key, and comparators chain cleanly with thenComparing and reversed.

It is worth knowing that sorted is a stateful intermediate operation: it must buffer every element before it can emit the first sorted one, so it defeats laziness and short-circuiting up to that point. On large datasets it is usually cheaper to sort at the database level than in a stream.

Q14. limit() and skip()

ROI: ★★★★★ Frequency: High

Example


stream.skip(10)
      .limit(10)

skip(n) discards the first n elements and limit(n) truncates the stream after n elements. Combined they implement an offset/page-size window. limit is a short-circuiting operation, so it allows the pipeline to stop pulling elements once the quota is met.

Real Project Example

A typical use is custom in-memory pagination over an already-loaded collection. In a real backend, however, paginating with skip/limit only makes sense for small in-memory lists; for database-backed data you should page at the query level (LIMIT/OFFSET or Spring Data Pageable) so you never load the whole table into memory just to throw most of it away.

Q15. Terminal Operations

ROI: ★★★★★ Frequency: High

Examples

  • collect()
  • count()
  • findFirst()
  • findAny()
  • forEach()
  • reduce()

A terminal operation is what actually triggers the pipeline and consumes the stream. After it runs, the stream is closed and cannot be reused. Terminal operations fall into a few categories: reductions that fold elements into a single value (reduce, count, sum), collectors that gather results into a container (collect), side-effecting consumers (forEach), and short-circuiting searches (findFirst, findAny, anyMatch).

Q16. findFirst() vs findAny()

ROI: ★★★★★ Frequency: High

findFirst findAny
First Element Any Element
Ordered Optimized For Parallel

Both return an Optional and both short-circuit, but they differ in their ordering guarantee. findFirst always returns the first element in encounter order, even in a parallel stream, which can force extra coordination between threads. findAny is free to return any matching element, so in a parallel pipeline it can return whichever element a thread finds first, avoiding the coordination cost.

On a sequential stream they typically behave identically. The distinction matters only when you go parallel: prefer findAny when you genuinely do not care which match you get.

Q17. reduce()

ROI: ★★★★★ Frequency: Extremely High

Purpose

reduce combines all elements of a stream into a single result by repeatedly applying an associative accumulator function. It is the general-purpose fold operation behind specialized reductions like sum, min and max.

The three-argument form (identity, accumulator, combiner) is the one to understand for senior rounds. The identity must be a true neutral element, the accumulator must be associative, and the combiner is used to merge partial results when the stream runs in parallel. Violating associativity is a classic source of subtle bugs that only appear under parallel execution.

Example


int sum =

numbers.stream()

       .reduce(
           0,
           Integer::sum
       );
Q18. collect()

ROI: ★★★★★ Frequency: Extremely High

Examples


collect(Collectors.toList())

collect(Collectors.toSet())

collect(Collectors.toMap())

collect is a mutable reduction: instead of folding into a single immutable value, it accumulates elements into a mutable container such as a List, Set, Map or String. It is driven by a Collector, which bundles a supplier, accumulator and combiner so the operation works correctly even in parallel.

A frequent pitfall is Collectors.toMap() throwing IllegalStateException when two elements map to the same key; the overload that accepts a merge function avoids this. The Collectors class is also composable, letting you nest collectors (for example groupingBy with a downstream counting) for powerful aggregations.

SECTION 5 - OPTIONAL

Optional was introduced to reduce NullPointerExceptions and make code more expressive. Many developers misuse Optional. Interviewers frequently ask both theory and practical usage questions.

Q22. What Is Optional?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

Optional is a container object that may or may not hold a non-null value. It encodes the possibility of absence directly in the type system, so a method signature like Optional<User> documents that "no user" is a legitimate outcome.

Example


Optional user =
userRepository.findById(id);

Benefit

The real benefit is intent. A method returning Optional tells the caller, at compile time, that they must handle the empty case, instead of discovering it at runtime through a NullPointerException. It also provides a fluent API (map, filter, orElse, orElseThrow) for transforming or defaulting the contained value without explicit null checks.

Q23. Why Was Optional Introduced?

ROI: ★★★★★ Frequency: High

Before Optional


User user =
findUser(id);

if(user != null){

}

Problem

Before Optional, the only signal that a method might not return a value was a comment or convention. Callers had to remember to write defensive null checks everywhere, and forgetting one produced the most common bug in Java history, the NullPointerException. Tony Hoare famously called the null reference his "billion-dollar mistake."

After Optional


findUser(id)
.orElseThrow();

With Optional the absence becomes part of the return type, so the compiler nudges the caller toward handling it. The code also reads as a pipeline of intent ("find the user, or throw") rather than a branch on a possibly-null reference.

Q24. of() vs ofNullable() vs empty()

ROI: ★★★★★ Frequency: Extremely High

Method Behavior
of() Null Not Allowed
ofNullable() Null Allowed
empty() Empty Optional

Use Optional.of(value) only when you are certain the value is non-null; passing null throws NullPointerException immediately, which is actually useful as a fail-fast guard. Use Optional.ofNullable(value) when the value might be null, and it safely yields an empty Optional in that case. Optional.empty() explicitly creates an empty container, typically as a return value meaning "nothing found."

Examples


Optional.of(user)



Optional.ofNullable(user)



Optional.empty()
Q25. orElse() vs orElseGet()

ROI: ★★★★★ Frequency: Senior Favorite

Example


optional.orElse(
expensiveOperation()
);

With orElse, the argument is an ordinary method call that Java evaluates before orElse even runs. That means expensiveOperation() executes every time, regardless of whether the Optional already contains a value. The result is simply discarded when a value is present. 


optional.orElseGet(
() -> expensiveOperation()
);

With orElseGet, you pass a Supplier (a lambda), and that supplier is invoked only when the Optional is empty. The expensive work is deferred and skipped entirely on the happy path. As a rule of thumb: use orElse for cheap constants and orElseGet whenever the default value is computed, fetched, or has side effects.

Interview Gold Point

Many developers incorrectly assume the two methods behave identically. The eager evaluation of orElse is a real production bug source, for example calling orElse(repository.findDefault()) hits the database on every call even when the value is present. Pointing this out distinguishes a candidate who has actually read the contract from one who has only used the happy path.

Q26. Optional Best Practices

ROI: ★★★★★ Frequency: High

  • Use as return type
  • Avoid Optional fields
  • Avoid Optional parameters
  • Use orElseThrow()
  • Use map() effectively

Optional was designed specifically as a return type for methods that might not produce a result. Using it as a field is discouraged because it is not Serializable and adds a wrapper object per instance; a plain nullable field is cleaner. Using it as a method parameter is also an anti-pattern, since the caller can already pass null or the method can be overloaded, and it forces an awkward wrapping at every call site.

Inside a chain, prefer map and filter to transform the contained value, and finish with orElseThrow to surface a meaningful domain exception when the value is missing. This keeps null-handling logic declarative and centralised.

Spring Example


userRepository
.findById(id)
.orElseThrow(
() -> new UserNotFoundException()
);

SECTION 6 - ADVANCED COLLECTORS

Q27. What Is Collectors Utility Class?

ROI: ★★★★★ Frequency: High

Collectors is a utility class of factory methods that produce ready-made Collector implementations for the collect() terminal operation. Rather than writing the supplier/accumulator/combiner triple by hand, you compose these building blocks to gather stream results into the shape you need.

Examples


toList()

toSet()

toMap()

groupingBy()

partitioningBy()

joining()

counting()

The power of Collectors comes from composition: many of these accept a downstream collector. For instance groupingBy(classifier, counting()) groups elements and then reduces each group, giving you a histogram in one declarative statement.

Q28. groupingBy()

ROI: ★★★★★ Frequency: Extremely High

Example


employees.stream()

.collect(
Collectors.groupingBy(
Employee::getDepartment
));

groupingBy applies a classifier function to each element and builds a Map from each distinct classifier value to the list of elements that produced it. It is the streams equivalent of a SQL GROUP BY. The two- and three-argument overloads let you supply a downstream collector (to count, sum, or map within each group) and even choose the map implementation.

Output


IT -> Employees

HR -> Employees

Finance -> Employees

Real Project Usage

In real backends this shows up constantly: grouping orders by status to render a dashboard, grouping jobs by category for a listing page, or grouping users by role for an admin summary. When the dataset is large, prefer grouping at the database level, but for already-loaded collections groupingBy is concise and readable.

Q29. partitioningBy()

ROI: ★★★★★ Frequency: High

Answer

partitioningBy is a specialized form of grouping that always produces exactly two groups, keyed by the boolean results true and false, based on a predicate. Both keys are always present in the resulting map even if one partition is empty, which makes it safer than groupingBy when you always expect both buckets.

Example


employees.stream()

.collect(
Collectors.partitioningBy(
e -> e.getSalary() > 50000
));
Q30. joining()

ROI: ★★★★☆ Frequency: Medium

Example


names.stream()

.collect(
Collectors.joining(",")
);

joining concatenates the stream of CharSequence elements into a single String. The overloads let you specify a delimiter and optional prefix and suffix, which is handy for building comma-separated lists or simple SQL IN clauses. Internally it uses a StringJoiner, so it is far more efficient than manually concatenating strings in a loop.

Output


John,Mike,David
Q31. counting()

ROI: ★★★★☆ Frequency: Medium

Example


employees.stream()

.collect(
Collectors.counting()
);

counting() returns the number of elements as a Long. On its own it is equivalent to stream.count(), but its real value is as a downstream collector inside groupingBy or partitioningBy, where it tallies how many elements fall into each group.

SECTION 7 - PARALLEL STREAMS

Q32. What Is Parallel Stream?

ROI: ★★★★★ Frequency: High

Answer

A parallel stream splits its source into multiple chunks and processes them concurrently on multiple threads, then merges the partial results. Under the hood it uses the common ForkJoinPool, whose default size is one less than the number of available processor cores.

Example


list.parallelStream()

Benefit

For CPU-bound work over a large, easily splittable source (such as an ArrayList), parallelism can deliver near-linear speedups on multi-core machines. The win comes only when the per-element work is substantial enough to outweigh the cost of splitting, scheduling, and merging.

Q33. Should We Always Use Parallel Streams?

ROI: ★★★★★ Frequency: Senior-Level Favorite

Answer

No. Parallel streams introduce real overhead from splitting the source, coordinating threads, and combining results, and they may degrade performance for:

  • Small Data Sets
  • I/O Operations
  • Database Calls

The deeper problem in a server context is that the default parallel stream uses the shared common ForkJoinPool. Blocking that pool with I/O or database calls starves every other parallel stream in the JVM, including library code you do not control. Parallel streams are best for pure, CPU-bound, stateless computation over large in-memory data, and even then only after benchmarking.

Interview Gold Point

Parallel streams are not a universal performance optimization. A strong answer notes the shared common pool, the cost of boxing and merging, and the requirement that operations be stateless and associative. Saying "I would benchmark before and after rather than assume parallel is faster" signals senior judgement.

Q34. Parallel Stream vs CompletableFuture

ROI: ★★★★★ Frequency: High

Parallel Stream CompletableFuture
Collection Processing Async Workflow
Less Control More Control
ForkJoinPool Custom Executors

They solve different problems. Parallel streams are for data parallelism, splitting one large dataset across cores for CPU-bound computation. CompletableFuture is for task parallelism and asynchronous orchestration, running independent operations (often I/O-bound calls to other services) concurrently and composing their results.

Crucially, CompletableFuture lets you supply your own Executor, so you can size and isolate a thread pool appropriate for blocking I/O instead of polluting the shared ForkJoinPool. That control is why CompletableFuture is the right tool for aggregating remote API calls.

SECTION 8 - COMPLETABLEFUTURE

Q35. Why CompletableFuture?

ROI: ★★★★★ Frequency: Extremely High

Problem

The original Future API from Java 5 was limited. The only way to read its result was the blocking get() call, and there was no way to attach a callback, chain a transformation, or combine two futures. As a result, asynchronous code degenerated into blocking waits or callback spaghetti.

  • No Chaining
  • No Combination
  • No Easy Async Pipelines

Solution

CompletableFuture (Java 8) implements both Future and CompletionStage, adding a rich, fluent API for chaining (thenApply), composing (thenCompose), combining (thenCombine), and handling errors (exceptionally, handle) without ever blocking a thread until you truly need the value.

Q36. What Is CompletableFuture?

ROI: ★★★★★ Frequency: Extremely High

Answer

CompletableFuture is an asynchronous computation framework introduced in Java 8 that represents a value which will be available at some point in the future. Unlike a plain Future, it can be explicitly completed and, more importantly, it lets you register dependent stages that run automatically when the result arrives.

By default the async factory methods run on the common ForkJoinPool, but every async method has an overload that accepts a custom Executor, which is the recommended choice for I/O-heavy backend workloads.

Example


CompletableFuture future =

CompletableFuture
.supplyAsync(
() -> "Hello"
);
Q37. runAsync() vs supplyAsync()

ROI: ★★★★★ Frequency: High

runAsync supplyAsync
No Return Returns Value
Runnable Supplier

Use runAsync when the asynchronous task produces no result, only a side effect such as sending an email or writing a log; it accepts a Runnable and yields a CompletableFuture<Void>. Use supplyAsync when the task computes and returns a value; it accepts a Supplier and yields a CompletableFuture of that value's type, which downstream stages can then transform.

Examples


CompletableFuture.runAsync(
() -> sendEmail()
);



CompletableFuture.supplyAsync(
() -> fetchUser()
);
Q38. thenApply()

ROI: ★★★★★ Frequency: High

Purpose

thenApply transforms the result of a completed stage by applying a Function, producing a new CompletableFuture with the transformed value. It is the asynchronous analogue of Stream.map: one value in, one value out. Use it when the transformation is a quick, synchronous computation; if the transformation itself returns a future, use thenCompose instead.

Example


future.thenApply(
String::length
);
Q39. thenAccept()

ROI: ★★★★☆ Frequency: High

Purpose

thenAccept consumes the result with a Consumer and returns a CompletableFuture<Void>. Use it for a terminal side effect that needs the produced value, for example logging the result or pushing it onto a queue, when no further transformation is required.

Q40. thenRun()

ROI: ★★★★☆ Frequency: High

Purpose

thenRun executes a plain Runnable once the previous stage completes. It neither receives the result nor produces one, so it is appropriate for cleanup or notification steps that simply need to know the prior stage finished, such as releasing a resource or signalling completion.

Q41. thenCompose()

ROI: ★★★★★ Frequency: Senior Favorite

Purpose

thenCompose chains two dependent asynchronous operations where the second depends on the result of the first and itself returns a CompletableFuture. It flattens the nested CompletableFuture<CompletableFuture<T>> that thenApply would produce in this situation into a single CompletableFuture<T>.

Comparable To


flatMap()

The mental model is exactly flatMap in Streams: use thenApply when your function returns a plain value, and thenCompose when your function returns another future, for example "fetch the user, then use that user's id to asynchronously fetch their orders."

Q42. thenCombine()

ROI: ★★★★★ Frequency: High

Purpose

thenCombine waits for two independent futures to both complete and then merges their results with a BiFunction. Because the two futures do not depend on each other, they run concurrently, and the combine step fires only when both are done, making the total latency roughly the slower of the two rather than their sum.

Example


userFuture

+

orderFuture

A typical use is fetching a user and their orders from two separate services in parallel and then assembling a combined response object.

Q43. allOf() vs anyOf()

ROI: ★★★★★ Frequency: High

allOf anyOf
Wait All Wait One

allOf returns a future that completes only when every supplied future has completed; it is how you fan out N parallel calls and wait for all of them. Its result type is CompletableFuture<Void>, so you typically call join() on each individual future afterwards to read the results. anyOf completes as soon as the first of the supplied futures finishes, which is useful for racing redundant sources and taking whichever responds first.

Q44. Exception Handling In CompletableFuture

ROI: ★★★★★ Frequency: High

Methods


exceptionally()

handle()

whenComplete()

exceptionally runs only on the failure path and lets you supply a fallback value, leaving the success path untouched. handle runs on both paths, receiving either the result or the exception, so you can transform either outcome into a single value. whenComplete also runs on both paths but is meant for side effects (logging, metrics, cleanup); it does not transform the result and re-propagates any exception.

Without one of these handlers, an exception inside an async stage is captured in the future and only surfaces when you call get() or join(), wrapped in a CompletionException. That silent swallowing is a common production trap.

Example


future.exceptionally(
ex -> "fallback"
);
Q45. Real Project Use Case

ROI: ★★★★★ Frequency: Extremely High

Scenario

The classic use case is an API aggregator (sometimes called a backend-for-frontend) that must gather data from several downstream services to build one response. 


Fetch User

Fetch Orders

Fetch Payments

Instead of calling the three services sequentially (where total latency is the sum of all three), each call is launched with supplyAsync on a dedicated executor and the results are merged with allOf or thenCombine. The overall response time then drops to roughly the slowest single call, dramatically improving the page's perceived performance.

Senior Engineer Perspective

This aggregation pattern is one of the most common real-world uses of CompletableFuture in microservice backends. A senior answer adds that each call should use a bounded custom executor (not the common pool), should set timeouts via orTimeout or completeOnTimeout, and should degrade gracefully with exceptionally so one slow or failing dependency does not take down the whole response.

Q47. What Is Java Module System?

ROI: ★★★★☆ Frequency: Medium

Answer

The Java Platform Module System (JPMS, "Project Jigsaw") introduced a unit of code above the package: the module. A module explicitly declares which packages it exports and which other modules it requires, giving the platform strong, reliable encapsulation that the classpath never provided.

Before Java 9

Before modules, applications were large, monolithic JARs on a flat classpath. Any public class in any JAR was reachable by any other code, so there was no real encapsulation boundary, and missing or duplicated JARs only failed at runtime ("classpath hell").

After Java 9


module-info.java

A module-info.java descriptor declares dependencies and exports explicitly, so the compiler and runtime can verify the module graph up front and hide internal packages even if they are public. In practice many enterprise applications still run on the classpath, but the JDK itself was modularized, which is why awareness matters even if you do not author modules.

Q48. Factory Methods For Collections

ROI: ★★★★☆ Frequency: Medium

Example


List names =
List.of("Java","Spring");

List.of, Set.of and Map.of create compact, immutable collections in a single expression, replacing the verbose pre-Java-9 idioms or the noisy Collections.unmodifiableList(new ArrayList<>(...)) pattern. Two gotchas worth mentioning in interviews: the returned collections are truly immutable, so any mutation throws UnsupportedOperationException, and they reject null elements with a NullPointerException.

For many simple service-to-service calls it removes the need for a third-party client library entirely.

Q53. What Are Records?

ROI: ★★★★★ Frequency: Extremely High

Problem Before Records


private final String name;

private final int age;

Constructors

Getters

equals()

hashCode()

toString()

A simple immutable data carrier traditionally required dozens of lines of boilerplate: private final fields, an all-args constructor, an accessor per field, plus correct equals(), hashCode() and toString(). That boilerplate is error-prone (forgetting a field in equals causes subtle bugs) and obscures the fact that the class is really just a tuple of values.

Solution


public record User(
String name,
int age
){}

Automatically Generates

  • Constructor
  • Getters
  • equals()
  • hashCode()
  • toString()

A record declares its state once in the header and the compiler synthesises a canonical constructor, an accessor per component (named name() rather than getName()), and value-based equals(), hashCode() and toString(). Records are implicitly final and their components are immutable, which makes them inherently thread-safe. You can still add a compact constructor to validate or normalise inputs.

Project Example

Records are an excellent fit for DTOs, API request and response objects, and other value-holding types where immutability is desirable and behaviour is minimal. They are not meant to replace JPA entities, which require a no-arg constructor and mutable, proxied state.

Q54. Records vs POJOs

ROI: ★★★★★ Frequency: High

POJO Record
Verbose Compact
Mutable Possible Immutable
Manual Methods Generated

A traditional POJO is flexible: it can be mutable, can extend another class, and you control every method. A record trades that flexibility for conciseness and correctness, it is final, immutable, and its core methods are generated and guaranteed consistent. Choose a record when the type's identity is defined entirely by its data; choose a POJO or entity when you need mutability, inheritance, or framework requirements such as JPA's no-arg constructor.

Q55. What Are Sealed Classes?

ROI: ★★★★★ Frequency: High

Answer

A sealed class (or interface) explicitly restricts which other classes are permitted to extend or implement it, using the permits clause. Inheritance becomes a deliberate, closed set rather than something any class anywhere can join.

Example


public sealed class Payment

permits

CardPayment,
UPIPayment

{}

Only the permitted subclasses can extend Payment, and each of those subclasses must itself be declared final, sealed, or non-sealed to state whether the hierarchy is closed or reopened below it. This gives the author control over the type hierarchy while still allowing it to be public.

Q56. Why Sealed Classes?

ROI: ★★★★★ Frequency: High

Benefits

  • Better Domain Modeling
  • Controlled Inheritance
  • Improved Safety
  • Exhaustive Pattern Matching

Sealing lets you model a domain as a fixed set of known variants, for example a Payment that is only ever a CardPayment or a UPIPayment. Because the compiler knows the complete set of subtypes, a switch over a sealed type can be checked for exhaustiveness: if you add a new variant and forget to handle it, the code fails to compile rather than slipping through at runtime. Combined with records, sealed types give Java the algebraic data types that make pattern matching safe and expressive.

Q57. Enhanced Switch Expression

ROI: ★★★★☆ Frequency: Medium

Old


switch(day){

case MONDAY:
break;

}

Modern


String result = switch(day){

case MONDAY -> "Work";

case SUNDAY -> "Rest";

default -> "Unknown";

};

The switch expression returns a value, so it can be assigned directly to a variable. The arrow form (case X ->) eliminates fall-through, which removes the classic bug of forgetting a break, and each branch handles exactly one case. When used as an expression a switch must be exhaustive, so the compiler forces you to cover every case (or add a default), and the yield keyword lets a multi-statement branch produce a value.

Q59. What Are Virtual Threads?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

Virtual threads are lightweight threads scheduled by the JVM rather than the operating system. Many virtual threads are multiplexed onto a small pool of OS-level "carrier" threads, so you can create millions of them at negligible memory cost.

Problem With Platform Threads

A traditional platform thread maps one-to-one to an OS thread and carries a large (often ~1 MB) stack, so an application can realistically support only a few thousand. In the common "thread-per-request" server model, threads spend most of their life blocked on I/O, wasting that expensive resource and capping throughput.

Solution

With virtual threads, when a virtual thread blocks on I/O the JVM detaches it from its carrier thread and lets another virtual thread run, then reattaches it when the I/O completes. The familiar, easy-to-read blocking style is preserved, but the scalability ceiling jumps from thousands to millions of concurrent tasks.

Example


Thread.startVirtualThread(
() -> {

}
);
Q60. Benefits Of Virtual Threads

ROI: ★★★★★ Frequency: Extremely High

  • Lower Memory Usage
  • Higher Concurrency
  • Simpler Programming Model
  • Better Resource Utilization

The biggest practical win is that you can write straightforward, blocking, sequential code, which is far easier to read and debug than reactive or callback-based code, and still scale to enormous concurrency. This lets teams achieve the throughput benefits of asynchronous frameworks without adopting their complexity.

Interview Gold Point

Virtual threads are not faster threads; a single task does not run any quicker. They allow many more concurrent tasks by not tying up an OS thread while blocked. A strong answer also mentions the caveats: they are designed for I/O-bound, not CPU-bound work, and you should avoid pinning carrier threads inside synchronized blocks (prefer ReentrantLock) and stop pooling threads, since virtual threads are cheap to create per task.

Q61. Virtual Threads vs Traditional Threads

ROI: ★★★★★ Frequency: High

Platform Thread Virtual Thread
OS Managed JVM Managed
Heavy Lightweight
Limited Count Massive Scale

A platform thread is a thin wrapper over an OS thread: heavy, scheduled by the kernel, and limited to a few thousand instances. A virtual thread is scheduled by the JVM, has a small resizable stack on the heap, and can exist in the millions. The key insight is that both implement the same Thread API, so existing code that uses threads or executors can often adopt virtual threads with a single configuration change rather than a rewrite.

Q62. What Is Structured Concurrency?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

Structured concurrency treats a group of related concurrent tasks as a single unit of work with a clear scope, much as a block of code groups sequential statements. Tasks are forked within a scope, and the scope does not return until all of its child tasks have finished (or been cancelled), so their lifetimes are bounded by the enclosing block.

Benefits

  • Better Error Handling
  • Simpler Cancellation
  • Improved Readability

The structured model eliminates leaked threads and thread "orphans." If one subtask fails, the scope can automatically cancel the siblings (shutdown-on-failure); if you only need the first success, it can cancel the rest (shutdown-on-success). Because the concurrency follows the code's block structure, error propagation and cancellation behave predictably instead of being scattered across detached futures.

Use Case

A natural example is calling a user service, order service, and payment service simultaneously to assemble a response. With structured concurrency the three calls are forked in one scope; if any one fails the others are cancelled immediately, and the scope returns only once all results are in, giving clean all-or-nothing semantics.

SECTION 13 - SENIOR ENGINEER INTERVIEW TRAPS

Q63. Are Streams Always Faster?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

No. Streams are primarily a readability and expressiveness improvement, not a performance optimization. For small collections or tight numeric loops, a plain for loop is often faster because it avoids the per-element overhead of the stream pipeline, lambda invocation, and (for object streams) autoboxing.

Streams shine when the declarative pipeline makes complex transformations clearer, or when you can genuinely benefit from parallelism on large datasets. The senior takeaway is to choose streams for clarity and measure before assuming any performance difference either way.

Q64. Should Optional Be Used Everywhere?

ROI: ★★★★★ Frequency: High

Answer

No. Optional was designed for one job: as a method return type that signals a value may be absent. Use it:

  • Fields - avoid (not serializable, extra wrapper per object)
  • Parameters - avoid (callers must wrap; overloading is cleaner)
  • Entity Attributes - avoid (frameworks like JPA expect plain fields)

Wrapping fields, parameters, and entity attributes in Optional adds allocation and complexity without buying the safety it was meant to provide. Keep it at the boundary where callers consume a result, and let internal data stay as plain nullable fields handled with normal guards.

Q65. Should Parallel Streams Always Be Used?

ROI: ★★★★★ Frequency: High

Answer

No. Parallel streams can easily hurt performance, and in a server they can do harm beyond their own pipeline by saturating the shared common ForkJoinPool. They only help when the work is CPU-bound, the source splits cheaply (arrays and ArrayList are good; LinkedList is poor), the per-element cost is high, and the operations are stateless and associative.

The disciplined answer is always benchmark before adopting a parallel stream, and never use one for blocking I/O or database calls.

Q66. Is CompletableFuture A Replacement For Microservices?

ROI: ★★★★★ Frequency: Advanced

Answer

No. The two operate at completely different levels. CompletableFuture is an in-process concurrency tool for running and composing asynchronous tasks within a single JVM. Microservices are an architectural style for decomposing a system into independently deployable, network-separated services.

They are complementary rather than alternatives: you might well use CompletableFuture inside one microservice to call several other microservices in parallel. Conflating them in an interview signals a gap in understanding the difference between concurrency and architecture.

SECTION 14 - TEXT BLOCKS & PATTERN MATCHING

Q67. What Are Text Blocks?

ROI: ★★★☆☆ Frequency: Medium

Short Answer

Text blocks (finalised in Java 15) are multi-line string literals delimited by triple quotes """. They remove the need for \n escapes and string concatenation, which makes embedded JSON, SQL, and HTML far more readable. 

String json = """
    {
        "name": "Alice",
        "role": "engineer"
    }
    """;

String sql = """
    SELECT id, name
    FROM users
    WHERE active = true
    """;

Interview Gold Point

The compiler strips "incidental" leading whitespace based on the closing delimiter's indentation, so you can indent the block to match your code without it appearing in the value. Text blocks are still plain String objects — no new type, no runtime cost — purely a readability feature.

Q68. What Is Pattern Matching (instanceof and switch)?

ROI: ★★★★☆ Frequency: Medium-High

Short Answer

Pattern matching lets you test a type and bind a variable in one step, removing the classic test-then-cast boilerplate. It started with pattern matching for instanceof (Java 16) and extended to switch with type and record patterns (finalised in Java 21). 

// Old: test then cast
if (obj instanceof String) {
    String s = (String) obj;
    System.out.println(s.length());
}

// instanceof pattern (Java 16+): bind in one step
if (obj instanceof String s) {
    System.out.println(s.length());
}

// switch pattern matching (Java 21)
String describe(Object obj) {
    return switch (obj) {
        case Integer i -> "int: " + i;
        case String s  -> "string of length " + s.length();
        case null      -> "null";
        default        -> "unknown";
    };
}

Interview Favorite

Combined with sealed classes and record patterns, switch pattern matching gives exhaustiveness checking — the compiler verifies every permitted subtype is handled, so you can drop the default branch. This brings algebraic-data-type style modelling to Java and pairs naturally with records for clean, type-safe domain logic.
