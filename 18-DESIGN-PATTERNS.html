18 - DESIGN PATTERNS INTERVIEW HANDBOOK

Design patterns are proven, reusable solutions to recurring design problems. For a Senior Software Engineer they matter for two reasons: interviewers test them directly, and — more importantly — the entire Spring framework is built on them, so understanding patterns explains why Spring works the way it does. Patterns fall into three families: * Creational — how objects are created (Singleton, Factory, Builder, Prototype) * Structural — how objects are composed (Adapter, Decorator, Proxy, Facade) * Behavioural — how objects interact (Strategy, Observer, Template Method, Command) A strong candidate does not just recite definitions — they say where the pattern appears in the JDK or Spring, and when NOT to use it. Patterns are tools, and over-engineering with patterns you do not need is a real anti-pattern.

SECTION 1 - CREATIONAL PATTERNS

Q1. What Are Design Patterns and Why Use Them?

ROI: ★★★★☆ Frequency: High

A design pattern is a named, reusable solution to a problem that recurs across many designs. They are not code you copy — they are templates for how to structure classes and objects.

Their value is communication and proven structure: saying "use a Strategy here" instantly conveys intent to any engineer, and the pattern comes with known trade-offs so you avoid reinventing (and re-breaking) the wheel.

Interview Gold Point

The mature answer adds a caveat: patterns add indirection and abstraction, which is cost. Apply them when the flexibility is actually needed, not pre-emptively. "Make it work, make it right, make it fast" — patterns belong to the "make it right" step, not premature design.

Q2. Explain the Singleton Pattern and How to Make It Thread-Safe

ROI: ★★★★★ Frequency: Extremely High

A Singleton ensures a class has exactly one instance with a global access point — used for shared, stateless resources like configuration or connection pools. The challenge is making creation thread-safe without paying a lock cost on every access.

Double-Checked Locking

public class Config {
    private static volatile Config instance;
    private Config() {}

    public static Config getInstance() {
        if (instance == null) {                 // 1st check (no lock)
            synchronized (Config.class) {
                if (instance == null)           // 2nd check (locked)
                    instance = new Config();
            }
        }
        return instance;
    }
}

The volatile keyword is essential — without it, another thread could see a partially constructed object due to instruction reordering.

Interview Gold Point

The cleanest thread-safe Singleton in Java is an enum: enum Config { INSTANCE; }. The JVM guarantees a single instance and handles serialization and reflection attacks for free. Also note: a Spring bean is a singleton per container by default — a different, framework-managed concept.

Q3. Factory vs Abstract Factory

ROI: ★★★★☆ Frequency: High

The Factory Method hides object creation behind a method, so callers ask for a product by type without using new directly. The Abstract Factory goes one level up: it creates families of related objects (e.g. a UI toolkit that produces matching buttons, checkboxes and menus for a given theme). 

// Simple factory method
interface Notification { void send(String msg); }

class NotificationFactory {
    static Notification create(String type) {
        return switch (type) {
            case "EMAIL" -> new EmailNotification();
            case "SMS"   -> new SmsNotification();
            default -> throw new IllegalArgumentException(type);
        };
    }
}

Interview Favorite

The benefit is decoupling: callers depend on the Notification interface, not concrete classes, so adding a new channel does not touch existing call sites. Spring's BeanFactory is a large-scale factory.

Q4. Explain the Builder Pattern

ROI: ★★★★★ Frequency: Very High

The Builder constructs complex objects step by step, avoiding the "telescoping constructor" problem (constructors with many parameters, half of them optional). It produces readable, immutable objects. 

User user = User.builder()
    .name("Alice")
    .email("alice@example.com")
    .age(30)
    .build();

Interview Gold Point

Mention real-world Builders you have used: Lombok's @Builder, StringBuilder, Java's Stream.Builder, and the fluent builders all over Spring (e.g. HttpSecurity, WebClient.builder()). It shines when an object has many optional fields and you want it immutable once built.

Q5. What Is the Prototype Pattern?

ROI: ★★★☆☆ Frequency: Medium

The Prototype pattern creates new objects by cloning an existing instance rather than constructing from scratch — useful when object creation is expensive, or when you want a configured template copied many times.

In Java this maps to Cloneable/clone(), though copy constructors or copy factory methods are usually clearer and avoid clone()'s well-known pitfalls (shallow copy, broken contracts).

Interview Favorite

Spring's prototype scope is named after this pattern: each request for the bean yields a fresh instance, unlike the default singleton scope.

SECTION 2 - STRUCTURAL PATTERNS

Q6. Explain the Adapter Pattern

ROI: ★★★★☆ Frequency: High

An Adapter converts the interface of one class into another that a client expects — the software equivalent of a travel power-plug adapter. Use it to make incompatible or third-party code work with your existing interfaces without modifying either side.

Interview Gold Point

JDK examples: Arrays.asList() adapts an array to a List, and the old InputStreamReader adapts a byte stream to a character stream. The key signal is "I have class A but my code needs interface B".

Q7. Explain the Decorator Pattern

ROI: ★★★★☆ Frequency: High

A Decorator adds behaviour to an object at runtime by wrapping it in another object that implements the same interface. It is the composition-based alternative to subclass explosion: instead of BufferedGzipEncryptedStream, you wrap layers. 

InputStream in = new BufferedInputStream(
                     new GZIPInputStream(
                         new FileInputStream("data.gz")));

Interview Favorite

The entire java.io stream library is the textbook Decorator example. Each wrapper adds one capability (buffering, compression) while keeping the same InputStream interface, so they compose freely.

Q8. Explain the Proxy Pattern

ROI: ★★★★★ Frequency: Very High

A Proxy is a stand-in that controls access to a real object — adding lazy loading, access control, caching, or cross-cutting logic without the caller knowing. The proxy implements the same interface as the real subject.

Interview Gold Point

This is the most important pattern for Spring developers: Spring AOP, @Transactional, @Cacheable, and lazy-loaded Hibernate entities all use proxies (JDK dynamic proxies for interfaces, CGLIB for classes). This is also why calling an @Transactional method from within the same class bypasses the proxy — the famous "self-invocation" problem.

Q9. Explain the Facade Pattern

ROI: ★★★☆☆ Frequency: Medium

A Facade provides a single, simplified interface to a complex subsystem. Clients talk to the facade instead of wiring together many low-level classes themselves, which reduces coupling.

Interview Favorite

A typical Spring @Service often acts as a facade: one placeOrder() method hides the coordination of repositories, payment gateways and notification senders behind a clean API.

SECTION 3 - BEHAVIOURAL PATTERNS

Q10. Explain the Strategy Pattern

ROI: ★★★★★ Frequency: Very High

The Strategy pattern defines a family of interchangeable algorithms behind a common interface and lets you swap them at runtime. It replaces sprawling if/else or switch blocks with polymorphism. 

interface PaymentStrategy { void pay(double amount); }

class PaymentService {
    void checkout(PaymentStrategy strategy, double amount) {
        strategy.pay(amount);   // behaviour chosen by caller
    }
}
// new PaymentService().checkout(new PaypalStrategy(), 99.0);

Interview Gold Point

In Spring you rarely hand-code this — inject a List<PaymentStrategy> or a Map<String, PaymentStrategy> and Spring wires every implementation automatically. That combination of Strategy + dependency injection is a very strong senior answer.

Q11. Explain the Observer Pattern

ROI: ★★★★☆ Frequency: High

The Observer pattern lets one subject notify many dependent observers automatically when its state changes — a publish/subscribe relationship. Observers can be added or removed without changing the subject.

Interview Favorite

Spring's ApplicationEvent / ApplicationListener (and @EventListener) is Observer in action. At an architectural scale, Kafka pub/sub is the same idea distributed across services — tie this back to the event-driven content in your Microservices chapter.

Q12. Explain the Template Method Pattern

ROI: ★★★★☆ Frequency: Medium-High

The Template Method defines the skeleton of an algorithm in a base class, deferring specific steps to subclasses. The overall sequence is fixed; only the variable steps are overridden.

Interview Gold Point

Spring's many *Template classes — JdbcTemplate, RestTemplate, TransactionTemplate — embody this: the boilerplate (open connection, handle errors, close) is fixed, and you supply only the variable bit (the SQL, the row mapper). That is also why they are called "templates".

SECTION 4 - PATTERNS IN PRACTICE

Q13. Where Does Spring Use Design Patterns?

ROI: ★★★★★ Frequency: Extremely High

This is a favourite "do you really understand the framework" question. Strong answer:

Pattern Where in Spring
Singleton Default bean scope
Factory BeanFactory / ApplicationContext
Proxy AOP, @Transactional, @Cacheable, lazy entities
Template Method JdbcTemplate, RestTemplate
Strategy Injected interface implementations
Observer ApplicationEvent / @EventListener
Builder HttpSecurity, WebClient.builder()

Interview Gold Point

Tie it together: "Spring's whole programming model is patterns — dependency injection is the Inversion of Control principle, beans are managed singletons, and AOP is built on proxies." This shows you understand the framework, not just its annotations.

Q14. What Are the SOLID Principles?

ROI: ★★★★★ Frequency: Very High

SOLID is five object-oriented design principles that lead to maintainable, loosely-coupled code — the foundation that makes patterns work:

  • S — Single Responsibility: a class should have one reason to change
  • O — Open/Closed: open for extension, closed for modification
  • L — Liskov Substitution: subtypes must be usable through their base type
  • I — Interface Segregation: many small interfaces beat one fat one
  • D — Dependency Inversion: depend on abstractions, not concretions

Interview Gold Point

Connect D directly to Spring: dependency injection is the Dependency Inversion Principle in practice — your classes depend on interfaces, and the container supplies the concrete implementation. Giving a concrete example for each letter beats reciting the acronym.

Q15. When Should You NOT Use a Design Pattern?

ROI: ★★★★☆ Frequency: Medium-High

When the flexibility it buys is not needed. Patterns add layers of abstraction and indirection, which makes simple code harder to read and debug. Introducing a Strategy for a value that will never change, or a Factory for a single concrete type, is over-engineering.

The senior mindset: start with the simplest code that works, and refactor toward a pattern only when a real, repeated need for flexibility appears — not in anticipation of one that may never come (YAGNI: "You Aren't Gonna Need It").

Interview Gold Point

Demonstrating restraint impresses senior interviewers more than naming twenty patterns. The strongest candidates know patterns deeply and know when plain, direct code is the better engineering choice.

QUICK REVISION SHEET

  • Three families: Creational, Structural, Behavioural
  • Singleton → one instance; prefer enum or double-checked + volatile
  • Factory hides creation; Abstract Factory creates families
  • Builder → complex/immutable objects with many optional fields
  • Adapter converts interfaces; Decorator adds behaviour by wrapping
  • Proxy controls access — the heart of Spring AOP / @Transactional
  • Strategy → swap algorithms (great with DI); Observer → pub/sub events
  • Template Method → fixed skeleton, variable steps (JdbcTemplate)
  • SOLID: SRP, Open/Closed, Liskov, Interface Segregation, Dependency Inversion
  • Dependency Injection = Dependency Inversion in practice
  • Don't over-engineer — patterns are tools, not goals (YAGNI)
