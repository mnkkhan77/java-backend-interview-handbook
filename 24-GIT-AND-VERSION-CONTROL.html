SECTION 1 - GIT FUNDAMENTALS

Q1. What is Git and how does a distributed VCS differ from a centralized one?

ROI: ★★★★☆ Frequency: High

Short Answer

Git is a distributed version control system (DVCS). Every clone is a full copy of the repository, including the entire history, not just the latest snapshot. This is the key difference from a centralized VCS like SVN or Perforce, where there is one canonical server and your checkout holds only the current files.

Because each developer has the full history locally, almost everything is fast and offline: commits, diffs, branching, viewing log and blame all happen without a network round-trip. The "remote" (GitHub, GitLab) is just another clone that the team agrees to treat as the source of truth.

AspectCentralized (SVN)Distributed (Git)
History locationCentral server onlyEvery clone has full history
CommitNeeds server connectionLocal, offline
BranchingExpensive, server-sideCheap, local pointer
Single point of failureYes (the server)No — any clone can restore

Interview Gold Point

Stress that in Git a "commit" and a "push" are separate steps. Centralized-VCS thinking assumes saving = publishing; the senior insight is that you commit locally as often as you like and only push when the work is shareable.

Q2. Explain the three areas: working directory, staging area, and repository.

ROI: ★★★★★ Frequency: Very High

Short Answer

Git tracks changes across three areas. Understanding them explains almost every confusing Git moment.

  • Working directory — the actual files on disk you edit.
  • Staging area (index) — a "draft" of your next commit. git add moves a snapshot of a change here.
  • Repository (the .git directory) — the committed history. git commit takes whatever is staged and records it permanently. HEAD is a pointer to the commit you currently have checked out.
# See all three at once
git status

# working dir -> staging area
git add UserService.java

# staging area -> repository
git commit -m "Add user lookup by email"

# inspect what is staged vs not staged
git diff            # working dir vs staging
git diff --staged   # staging vs last commit (HEAD)

Interview Favorite

The staging area is what lets you craft a clean commit from a messy working directory. Mention git add -p (patch mode) to stage only selected hunks of a file — a strong signal that you curate commits deliberately rather than blindly git add . everything.

Q3. What is a commit really? What does it store?

ROI: ★★★★☆ Frequency: Medium

Short Answer

A commit is an immutable snapshot of the entire tracked tree at a point in time — not a diff. Each commit object stores a pointer to a tree (the directory/file structure), the author and committer metadata, the message, and a pointer to its parent commit(s). A merge commit has two or more parents.

The full history is therefore a directed acyclic graph (DAG) of commits linked by parent pointers. Each commit is identified by a SHA-1 (now SHA-256) hash of its content, which is why history is tamper-evident: change anything and every downstream hash changes.

git cat-file -p HEAD
# tree 9f3a...        <- snapshot of files
# parent 4b1c...      <- previous commit
# author Nasir <n@x.io> 1718...
# committer ...
#
# Add user lookup by email

Common Mistake

Saying "a commit stores the changes/diff." Git stores full snapshots and computes diffs on demand. It deduplicates with content-addressable storage, so unchanged files just reuse the same blob — efficient, but conceptually still a snapshot.

Q4. What is the difference between git fetch and git pull?

ROI: ★★★★★ Frequency: Very High

Short Answer

git fetch downloads new commits and refs from the remote into your local remote-tracking branches (e.g. origin/main) but does not touch your working branch. It is safe and non-destructive — you can inspect before integrating.

git pull is effectively git fetch followed by an automatic merge (or rebase) of the remote branch into your current branch. It changes your working branch and can trigger conflicts immediately.

git pull                 # = git fetch + git merge origin/<branch>
git pull --rebase        # = git fetch + git rebase origin/<branch>

# Safer habit: look before you leap
git fetch origin
git log --oneline HEAD..origin/main   # what's new upstream
git merge origin/main                 # integrate when ready

Interview Gold Point

Many teams set pull.rebase = true to avoid noisy "Merge branch main" commits on every sync. Mentioning that you prefer fetch + review, or a rebasing pull, shows you care about keeping history linear and readable.

SECTION 2 - BRANCHING & INTEGRATION

Q5. What is a branch in Git, and why is it so cheap?

ROI: ★★★★☆ Frequency: High

Short Answer

A branch is simply a movable pointer to a commit — a 41-byte file under .git/refs/heads/ containing a SHA. Creating a branch does not copy any files; it just writes a new pointer. That is why branching is instant and weighs almost nothing, unlike older centralized systems.

When you commit on a branch, the branch pointer advances to the new commit. HEAD normally points at the current branch, which in turn points at the latest commit.

git branch feature/email-login   # create pointer at current commit
git switch feature/email-login   # move HEAD onto it (modern syntax)
git switch -c feature/email-login  # create + switch in one step

cat .git/refs/heads/feature/email-login   # just a SHA

Interview Favorite

Prefer git switch / git restore (introduced in Git 2.23) over the overloaded git checkout. Knowing that checkout historically did three unrelated jobs — switch branch, restore files, detach HEAD — and that the team split it for clarity, signals you keep current.

Q6. Merge vs rebase: what is the difference and when do you use each?

ROI: ★★★★★ Frequency: Very High

Short Answer

Merge combines two branches by creating a new merge commit with both branches as parents. It preserves the true history exactly as it happened, including the divergence — but produces a non-linear graph.

Rebase replays your commits on top of another branch's tip, rewriting them as new commits with new hashes. The result is a clean, linear history as if you had branched from the latest main all along.

# Bring my feature up to date with main, cleanly:
git switch feature/email-login
git rebase main          # replays feature commits onto main's tip

# Integrate finished work, preserving history:
git switch main
git merge feature/email-login
Use rebase whenUse merge when
Updating your local feature branch with mainIntegrating a feature into a shared branch
Cleaning up local commits before a PRYou want to preserve the real branch topology
You want linear, readable historyThe branch is shared / already pushed

Interview Gold Point

The golden rule: never rebase commits that others have already pulled (public/shared history). Rebasing rewrites hashes; teammates' history then diverges and they get nasty conflicts on the next pull. Rebase private/local work freely; merge to integrate shared work.

Q7. Fast-forward vs three-way merge — what is the difference?

ROI: ★★★★☆ Frequency: Medium

Short Answer

A fast-forward merge happens when the target branch has not diverged — the branch you are merging is strictly ahead. Git simply moves the pointer forward; no merge commit is created.

A three-way merge happens when both branches have new commits since they diverged. Git uses three inputs — the two branch tips and their common ancestor (merge base) — to compute the result, and records a merge commit with two parents.

# Force a merge commit even when a fast-forward is possible
git merge --no-ff feature/email-login

# Refuse to merge unless it can fast-forward (keeps history linear)
git merge --ff-only feature/email-login

Interview Favorite

Many teams use --no-ff for feature merges so each feature appears as one identifiable bubble in history (easy to revert as a unit), while using --ff-only on pulls to guarantee a linear mainline. Explaining the trade-off, not just the flags, is what scores points.

Q8. How do you resolve a merge conflict?

ROI: ★★★★★ Frequency: Very High

Short Answer

A conflict occurs when two branches change the same lines (or one edits a file the other deletes) and Git cannot auto-merge. Git pauses, marks the conflicting regions in the file with markers, and waits for you to decide.

You open each conflicted file, pick the correct combined result, remove the markers, stage the file, and continue the merge or rebase.

<<<<<<< HEAD
int timeout = 30;        // your change
=======
int timeout = 60;        // incoming change
>>>>>>> feature/email-login

# Workflow:
git status               # list conflicted files
# edit files, resolve, delete the <<< === >>> markers
git add UserService.java
git merge --continue     # or: git rebase --continue
git merge --abort        # bail out and return to pre-merge state

Interview Gold Point

Mention git rerere ("reuse recorded resolution") for repetitive rebase conflicts, and that the best conflict is the one you avoid — small PRs and frequent rebasing onto main keep conflicts tiny. Senior engineers manage conflict surface area, not just resolve conflicts.

SECTION 3 - HISTORY MANAGEMENT

Q9. Explain git reset --soft, --mixed, --hard versus git revert.

ROI: ★★★★★ Frequency: Very High

Short Answer

git reset moves the current branch pointer (and optionally the index/working tree) to a different commit — it rewrites history and is for local, unpushed work. git revert creates a new commit that undoes a previous commit's changes — it is safe for shared history because it adds rather than rewrites.

CommandMoves HEADStaging areaWorking dirUse for
reset --softYesKept (staged)KeptRecommit / squash last commits
reset --mixed (default)YesResetKeptUnstage but keep changes
reset --hardYesResetDiscardedThrow work away (dangerous)
revertNo (adds commit)——Undo a pushed commit safely
git reset --soft HEAD~1     # uncommit last commit, keep changes staged
git reset --mixed HEAD~1    # uncommit + unstage, keep file edits
git reset --hard HEAD~1     # destroy last commit AND its changes
git revert <sha>            # safe undo on a shared branch

Common Mistake

Using reset --hard on a branch others share. The rule: reset for private history, revert for public history. Also note git restore is the modern, focused way to discard working-directory changes without touching commits.

Q10. How do you squash commits and use interactive rebase?

ROI: ★★★★☆ Frequency: High

Short Answer

Interactive rebase lets you rewrite a series of local commits — reorder, combine, edit messages, or drop them — before sharing. Squashing collapses several work-in-progress commits ("wip", "fix typo", "oops") into one clean, meaningful commit for review.

git rebase -i HEAD~4

# An editor opens with the last 4 commits:
pick   a1b2c3  Add email login endpoint
squash d4e5f6  wip
squash 7890ab  fix validation
reword cdef01  add tests

# Commands you can use:
#   pick   = keep commit as-is
#   reword = keep changes, edit the message
#   squash = merge into the previous commit, combine messages
#   fixup  = like squash but discard this commit's message
#   drop   = delete the commit entirely

Interview Favorite

Tie this back to the rebase golden rule: squash before you push or merge, never on commits others already have. Also mention many platforms offer "Squash and merge" on the PR button, which keeps mainline clean without you rewriting locally.

Q11. What makes a good commit message, and how do tags / semantic versioning work?

ROI: ★★★★☆ Frequency: Medium

Short Answer

A good commit message has a concise (~50 char) imperative subject line ("Add retry to payment client"), a blank line, then a body explaining the why, not the what (the diff already shows what). Many teams adopt Conventional Commits (feat:, fix:, chore:) to drive automated changelogs and version bumps.

Tags mark a specific commit, typically a release. An annotated tag stores metadata and is the right choice for releases; a lightweight tag is just a pointer. Semantic Versioning is MAJOR.MINOR.PATCH: breaking change bumps MAJOR, backward-compatible feature bumps MINOR, bug fix bumps PATCH.

git tag -a v2.3.0 -m "Release 2.3.0: add OAuth login"
git push origin v2.3.0
git tag                       # list tags

# Conventional Commits map cleanly to SemVer:
#   fix:   -> PATCH (2.3.0 -> 2.3.1)
#   feat:  -> MINOR (2.3.1 -> 2.4.0)
#   feat!: / BREAKING CHANGE -> MAJOR (2.4.0 -> 3.0.0)

Interview Gold Point

Explain that disciplined commit messages and SemVer tags are not vanity — they let tools auto-generate changelogs, let consumers reason about upgrade risk, and make git bisect and incident forensics dramatically faster.

SECTION 4 - WORKFLOWS

Q12. Compare Git Flow, GitHub Flow, and trunk-based development.

ROI: ★★★★★ Frequency: High

Short Answer

These are branching strategies, and the right one depends on your release cadence and team maturity. The trend in high-performing teams is toward short-lived branches and frequent integration (trunk-based), which is what continuous delivery and DORA research favor.

StrategyBranchesBest fitTrade-off
Git Flowmain, develop, feature/*, release/*, hotfix/*Scheduled releases, versioned products, multiple supported versionsHeavy; many long-lived branches; merge pain
GitHub Flowmain + short feature branches via PRsWeb apps deploying continuously from mainAssumes strong CI and fast review
Trunk-basedmain + very short-lived branches; feature flagsCI/CD, mature teams, high deploy frequencyRequires flags, strong tests, discipline

Interview Gold Point

The senior answer is "it depends on release cadence." For a SaaS deploying many times a day, Git Flow's develop/release branches add overhead with little benefit — trunk-based with feature flags is the better fit. Knowing why long-lived branches cause integration pain (merge debt grows with divergence) is the differentiator.

Q13. Walk me through a pull request and code review workflow.

ROI: ★★★★☆ Frequency: High

Short Answer

A pull request (PR) is a request to merge a branch, plus a place for review, CI checks, and discussion. The flow: branch from main, commit focused changes, push, open a PR with a clear description and linked ticket, let CI run, address review comments, then merge (often squash-merge) and delete the branch.

git switch -c feature/JIRA-481-email-login
# ... commit work ...
git push -u origin feature/JIRA-481-email-login
# open PR in GitHub/GitLab UI

# keep the PR current without a messy merge commit:
git fetch origin
git rebase origin/main
git push --force-with-lease   # safe force: refuses if remote moved
  • Keep PRs small and single-purpose — easier to review, faster to merge.
  • Write a description: what, why, how to test.
  • Reviewers focus on correctness, design, edge cases — not style (let a linter do that).
  • Require green CI and at least one approval before merge.

Interview Favorite

Use git push --force-with-lease, never plain --force, after a rebase. --force-with-lease refuses to overwrite if someone else pushed in the meantime, protecting against clobbering a teammate's work — a hallmark of someone who has been burned and learned.

Q14. How do you undo common mistakes and recover with reflog?

ROI: ★★★★★ Frequency: Very High

Short Answer

Most "I broke Git" panics are recoverable because Git rarely deletes anything immediately. The reflog records every move of HEAD — even after a hard reset or a deleted branch — so you can find the lost commit's SHA and restore it.

# Undo the last commit but keep the changes
git reset --soft HEAD~1

# Unstage a file (keep edits)
git restore --staged config.yml

# Discard working-directory changes to a file
git restore config.yml

# Recover a "lost" commit or deleted branch
git reflog                       # find the SHA before the mistake
git reset --hard HEAD@{2}        # jump back to that state
git switch -c rescued <sha>      # rebuild a branch from a lost commit

Interview Gold Point

The line that lands: "Git almost never loses committed work — reflog and dangling objects keep it for ~30 days by default." Knowing the reflog exists separates people who fear Git from people who use it confidently.

SECTION 5 - PRACTICAL

Q15. Explain git stash, cherry-pick, .gitignore, hooks, and handling secrets / large binaries.

ROI: ★★★★☆ Frequency: High

Short Answer

These are the everyday practical tools that show day-to-day fluency.

  • git stash — shelve uncommitted changes to switch context (e.g. an urgent hotfix), then restore them.
  • git cherry-pick — apply a single commit from one branch onto another, e.g. backporting a fix to a release branch.
  • .gitignore — patterns for files Git should never track (build output, IDE files, secrets, dependencies like target/ or node_modules/).
  • Git hooks — scripts that fire on events; pre-commit can run linters/tests/secret-scanners before a commit is allowed.
git stash push -m "wip login form"
git switch hotfix/payments
# ... fix and commit ...
git switch feature/login
git stash pop                    # restore shelved work

git cherry-pick <sha>           # backport one commit to a release branch

# .gitignore example
target/
*.log
.env
.idea/

Secrets & large binaries: never commit secrets — once pushed, they live in history forever and must be treated as compromised (rotate the key, then scrub with git filter-repo or BFG). Use .env + a secrets manager and a pre-commit secret scanner. For large binaries, avoid bloating the repo (every clone pays for them forever) — use Git LFS or an artifact store instead.

Common Mistake

Thinking git rm --cached or a later commit "removes" a leaked secret. It is still in history and on every clone. The only correct response is: rotate the credential immediately, then rewrite history. Saying this out loud signals real security maturity.

QUICK REVISION SHEET

  • Git is distributed — every clone has full history; commit (local) and push (publish) are separate steps
  • Three areas: working directory → staging/index (git add) → repository (git commit); HEAD = current commit
  • A commit is an immutable snapshot + parent pointer(s), identified by a content hash — not a diff
  • fetch = download only (safe); pull = fetch + merge/rebase (touches your branch)
  • A branch is a movable pointer (~41 bytes) — branching is instant and cheap
  • Merge preserves history (merge commit); rebase rewrites for a linear history — never rebase shared/public history
  • Fast-forward = no divergence (just move pointer); three-way merge = both diverged, uses common ancestor + merge commit
  • Conflicts: edit markers <<< === >>>, git add, then --continue; small PRs reduce conflict surface
  • reset rewrites history (--soft keep staged, --mixed unstage, --hard destroy) — local only; revert = safe undo for pushed commits
  • Interactive rebase (git rebase -i): pick/reword/squash/fixup/drop to clean up commits before sharing
  • Good commits: imperative subject + why in body; annotated tags for releases; SemVer MAJOR.MINOR.PATCH
  • Workflows: Git Flow (scheduled releases), GitHub Flow (continuous deploy), trunk-based (CI/CD + feature flags)
  • PRs: small, single-purpose, green CI, one approval; rebase + push --force-with-lease (never plain --force)
  • reflog recovers lost commits/branches (~30 days); restore unstages/discards; reset --soft uncommits
  • stash (shelve work), cherry-pick (one commit), .gitignore (never track), pre-commit hooks; never commit secrets — rotate + scrub; use Git LFS for binaries
