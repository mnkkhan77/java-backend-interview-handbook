SECTION 1 - URL SHORTENER

Q1. Design TinyURL

ROI: ★★★★★ Frequency: Extremely High

Functional Requirements

  • Create Short URL
  • Redirect To Original URL
  • Track Clicks

High-Level Design


Client

↓

API

↓

URL Service

↓

Database

↓

Redis Cache

A URL shortener is a read-heavy system. For every short URL created, the same link may be resolved millions of times, so the dominant traffic pattern is redirects, not writes. This shapes every decision: we optimise the read path aggressively with caching while keeping writes simple and durable in the primary database.

The write path stores the mapping (short code → original URL) in the database. The read path first checks Redis; on a hit it returns immediately, on a miss it loads from the database and back-fills the cache. Because short codes are immutable once created, cache invalidation is trivial here, which is what makes a URL shortener a popular warm-up question.

On scale estimation, a senior answer mentions rough numbers: if we expect 100 million new URLs per year and a 10:1 read-to-write ratio, we plan capacity and storage around the read traffic and choose a cache hit ratio target (for example 90%+) to keep database load bounded.

Q2. How To Generate Short Code?

ROI: ★★★★★ Frequency: Extremely High

Options

  • Base62 Encoding
  • Hashing
  • Random Generation

Preferred

Base62(ID)

The preferred approach is to take a unique numeric ID (typically an auto-increment or a distributed ID such as Snowflake) and encode it into Base62 (a-z, A-Z, 0-9). Base62 produces short, URL-safe codes and guarantees uniqueness because it is a deterministic encoding of an already-unique number, so there are zero collisions by construction.

Hashing (for example MD5 truncated) and random generation both risk collisions, which forces a uniqueness check or retry on every insert and adds latency. Base62 over a guaranteed-unique ID avoids that problem entirely. The trade-off is that sequential IDs make codes predictable; if that matters, you can shuffle the ID space or add a random offset before encoding.

At scale the ID source itself must be discussed. A single auto-increment column becomes a write bottleneck and a single point of failure, so distributed systems use ID ranges per node, a ticket server, or Snowflake-style IDs that embed timestamp and machine bits.

Q3. Why Redis?

ROI: ★★★★★ Frequency: High

Frequently accessed URLs can be served without hitting the database. A small percentage of links typically account for the majority of redirect traffic (the classic hot-key / Pareto distribution), so caching those hot URLs in Redis removes the heaviest load from the primary database.

Redis is in-memory, so reads return in sub-millisecond time compared with a disk-backed relational lookup. Because the redirect path is the most latency-sensitive and highest-volume part of the system, serving it from memory directly improves both response time and overall throughput.

The cache-aside pattern fits naturally: on a miss we read the database and store the result with a TTL. Since short-code mappings are immutable, stale data is not a concern, which keeps the caching layer simple and reliable.

Q4. Potential Bottlenecks

ROI: ★★★★★ Frequency: High

  • Database Reads
  • Hot URLs
  • Cache Misses
  • Redirect Traffic

Database reads become a bottleneck when the cache hit ratio drops, because every miss falls through to disk-backed storage. The mitigation is a higher cache hit ratio, read replicas, and indexing the short-code column so lookups are O(log n) rather than full scans.

Hot URLs concentrate traffic on a single cache key, which can overload one Redis shard. Strategies include replicating hot keys across nodes, using a local in-process cache in front of Redis for the hottest entries, and load balancing redirect traffic across application instances.

Cache misses are most damaging right after a deployment or cache flush, when many requests stampede the database simultaneously. This is the cache stampede problem, addressed later in this file with distributed locks, cache warming, and randomised TTLs.

SECTION 2 - RATE LIMITER

Q5. What Is Rate Limiting?

ROI: ★★★★★ Frequency: Extremely High

Rate limiting restricts the number of requests a client can make within a time window. It is enforced per identity, typically per user, per API key, or per IP address, and rejects or delays requests once the configured threshold is exceeded.

It is a protective control rather than a feature: it keeps a small number of clients from consuming a disproportionate share of capacity. In a Spring Boot system it usually lives at the API gateway or in a filter so the limit is applied before any expensive business logic runs.

Q6. Why Rate Limiting?

ROI: ★★★★★ Frequency: Extremely High

  • Prevent Abuse
  • Prevent DDoS
  • Protect Resources
  • Fair Usage

The primary motivation is protecting shared resources. Without limits, a single misbehaving client (an aggressive script, a retry storm, or a malicious actor) can exhaust database connections, thread pools, or downstream quotas and degrade service for everyone.

Rate limiting also enforces fair usage across tenants and supports tiered pricing, where different plans get different limits. As a defence layer it blunts brute-force and denial-of-service attempts, though it is one part of a broader strategy that also includes WAFs and load shedding.

Q7. Common Algorithms

ROI: ★★★★★ Frequency: Extremely High

  • Fixed Window
  • Sliding Window
  • Token Bucket
  • Leaky Bucket

Fixed Window counts requests per discrete interval (for example per minute). It is the simplest, but it allows bursts at window boundaries: a client can send the full quota at the end of one window and again at the start of the next, briefly doubling the intended rate.

Sliding Window smooths that boundary problem by tracking requests over a rolling time range, at the cost of more state. Token Bucket allows controlled bursts by refilling tokens at a fixed rate, while Leaky Bucket enforces a strictly smooth output rate by draining a queue at a constant pace. The right choice depends on whether bursts are acceptable.

Q8. Most Common Production Choice?

ROI: ★★★★★ Frequency: High

Answer

Token Bucket.

Token Bucket is the most common production choice because it balances two competing needs: a steady average rate plus tolerance for short bursts. Tokens are added to a bucket at a fixed refill rate up to a maximum capacity, and each request consumes a token; when the bucket is empty, requests are throttled.

This maps well to real user behaviour, which is bursty rather than perfectly even. It is also cheap to implement with two values per client (token count and last refill timestamp), and those values can be stored and updated atomically in Redis, making it work across a distributed fleet of application servers.

Q9. Why Redis For Rate Limiting?

ROI: ★★★★★ Frequency: Extremely High

  • Fast
  • Atomic Operations
  • Distributed

The core challenge in a distributed system is that the limit must be enforced globally, not per server. If each of ten application instances kept its own in-memory counter, a client could effectively get ten times the intended limit. A shared store solves this, and Redis is the standard choice.

Redis is fast because it is in-memory, and critically it offers atomic operations (INCR, EXPIRE) and Lua scripting so the read-modify-write of a counter happens without race conditions. Multiple application instances can safely increment the same counter, giving a single consistent view of usage across the whole fleet.

SECTION 3 - NOTIFICATION SYSTEM

Q10. Design Notification System

ROI: ★★★★★ Frequency: Extremely High

Channels

  • Email
  • SMS
  • Push Notification

A notification system is fundamentally an asynchronous, fan-out problem: one business event (an application submitted, a password reset) may produce several messages across different channels. The design separates the event source from delivery so that slow or failing providers never block the user-facing request.

Each channel has its own delivery semantics and third-party provider (an email gateway, an SMS gateway, a push service such as FCM/APNs). A good design abstracts these behind a common interface so channels can be added without changing the core, and routes events to the correct channel based on user preferences.

Q11. Architecture

ROI: ★★★★★ Frequency: High 


User Action

↓

Kafka

↓

Notification Service

↓

Email Provider

SMS Provider

Push Provider

The user action publishes an event to Kafka and returns immediately, so the user is never made to wait for an email or SMS to be sent. The Notification Service consumes those events asynchronously, decoupling event production from the slower, failure-prone work of contacting external providers.

The Notification Service determines which channels apply, formats the message, and calls the relevant provider. Because the providers sit behind a queue, a provider outage causes a backlog rather than a user-facing failure, and the service can retry on its own schedule.

Q12. Why Kafka?

ROI: ★★★★★ Frequency: Extremely High

Notification delivery should not block user requests. Sending an email or hitting an SMS provider can take hundreds of milliseconds or fail outright, and we never want that latency or unreliability to leak into the synchronous request the user is waiting on.

Kafka provides durable, ordered, replayable event storage. If the Notification Service crashes or a provider is down, events remain in the topic and are processed once consumers recover, so nothing is lost. This durability plus the ability to add new consumers (analytics, audit) without touching producers is what makes Kafka the standard choice over a simple in-memory queue.

It also enables horizontal scaling: partitioned topics let multiple consumer instances process notifications in parallel, so throughput grows by adding consumers rather than rewriting the producer.

Q13. Important Design Considerations

ROI: ★★★★★ Frequency: Senior Favorite

  • Retries
  • Idempotency
  • Dead Letter Queue
  • Monitoring

Retries handle transient provider failures, but they must be bounded and use exponential backoff so a struggling provider is not hammered. This is exactly where idempotency becomes critical: a retry must not send the same notification twice, so each notification carries a unique key that the service deduplicates on.

A Dead Letter Queue captures messages that fail repeatedly so they can be inspected and reprocessed instead of silently lost or endlessly retried. Monitoring and alerting on delivery rates, failure rates, and DLQ depth turn a black-box system into an operable one.

Senior Engineer Perspective

Idempotency is the answer interviewers are listening for here. At-least-once delivery is the default in Kafka, which means duplicates are a normal occurrence, not an edge case. A senior engineer designs the consumer to be idempotent (dedupe by message ID) so that re-delivery is safe rather than embarrassing.

SECTION 4 - CHAT SYSTEM

Q14. Design Chat Application

ROI: ★★★★★ Frequency: High

Requirements

  • Send Messages
  • Receive Messages
  • Online Status
  • Message History

A chat application is defined by its real-time requirement: messages must reach the recipient with low latency and without the recipient polling. This pushes the design toward persistent connections rather than request-response, and toward a clear split between the live delivery path and the durable history store.

Online status (presence) and message history are separate concerns. Presence is volatile, high-churn state that suits an in-memory store like Redis, while message history must be durably persisted and queryable, which suits a database optimised for write-heavy, time-ordered data.

Q15. REST Or WebSocket?

ROI: ★★★★★ Frequency: Extremely High

Answer

WebSocket.

Chat requires real-time communication. REST is request-response: the client must ask the server for new messages, so delivering messages instantly would require constant polling, which is wasteful and still adds latency. WebSocket provides a persistent, bidirectional connection so the server can push a message to the recipient the moment it arrives.

The trade-off is that WebSocket connections are stateful and long-lived, which complicates load balancing and scaling: each connection is pinned to a specific server, so you need a way (often Redis pub/sub or Kafka) to route a message to whichever server holds the recipient's connection. REST is still used for non-real-time operations like loading message history or user profiles.

Q16. Chat Architecture

ROI: ★★★★★ Frequency: High 


Client

↕

WebSocket Server

↓

Kafka

↓

Message Service

↓

Database

The client holds a WebSocket connection to a WebSocket Server, which is responsible only for connection management and pushing messages. When a message is sent, the server publishes it to Kafka rather than writing to the database synchronously, keeping the live path fast.

The Message Service consumes from Kafka, persists the message to the database for history, and routes it to the recipient's WebSocket server. This decoupling lets the delivery layer and the persistence layer scale independently, and ensures messages are durably stored even under bursts, because Kafka absorbs the spike.

SECTION 5 - FILE STORAGE SYSTEM

Q17. Design Resume Upload Feature

ROI: ★★★★★ Frequency: Extremely High

Relevant for job portals. A resume upload feature is a classic object-storage problem: the file itself is large, immutable binary content, while the application only needs to reference it. The design therefore separates the bytes (stored in object storage) from the metadata (stored in the relational database).

A senior answer also covers the upload path: clients should upload directly to object storage using pre-signed URLs rather than streaming bytes through the application server. This keeps large file transfers off the application tier, reduces its memory and bandwidth load, and improves scalability.

Q18. Where Should Files Be Stored?

ROI: ★★★★★ Frequency: Extremely High

Avoid


Database BLOB

Preferred


AWS S3

Cloud Storage

Object storage such as AWS S3 is purpose-built for large, durable, cheaply replicated binary content. It scales effectively without limit, offers high durability guarantees, integrates with CDNs for fast delivery, and supports lifecycle policies for archival, none of which a relational database provides efficiently.

Storing files as database BLOBs bloats the database, slows backups and restores, consumes connection and buffer-pool resources meant for transactional queries, and makes replication heavier. The standard pattern is to keep only the file URL and metadata in the database and the bytes in S3.

Q19. Why Not Store Large Files In Database?

ROI: ★★★★★ Frequency: High

  • Database Growth
  • Backup Issues
  • Performance Problems

Large binary files cause uncontrolled database growth, and since the database is usually the most expensive and hardest-to-scale tier, filling it with bytes that never participate in queries is wasteful. Backups and restores become slow and costly because every byte must be copied with the transactional data.

Performance also suffers: BLOB reads and writes compete for the same I/O, buffer pool, and connections as real queries, and a single large-file fetch can evict useful query data from cache. Object storage isolates this load entirely and is far cheaper per gigabyte.

Q20. Metadata Storage

ROI: ★★★★★ Frequency: High

Store In Database


file_name

url

size

uploaded_by

created_at

The database stores only the lightweight metadata that the application needs to query and authorise access: the file name, the storage URL or key, the size, who uploaded it, and when. This keeps rows small and queries fast while the heavy bytes live in S3.

Keeping uploaded_by and created_at lets you enforce ownership (a user can only download their own resume) and apply retention policies. The url or key links the metadata row to the object in storage, so the two systems stay consistent through a single reference.

SECTION 6 - JOB PORTAL DESIGN

Q21. Design A Job Portal

ROI: ★★★★★ Frequency: Extremely High

Services


User Service

Job Service

Application Service

Notification Service

Search Service

The job portal decomposes into services around clear business domains. The User Service owns identity and profiles, the Job Service owns postings, the Application Service owns the apply workflow, the Notification Service handles asynchronous messaging, and the Search Service serves fast, full-text job discovery.

This separation follows domain boundaries rather than technical layers, so each service can be scaled and reasoned about independently: search is read-heavy and benefits from a dedicated index, while applications are write-heavy and need strong consistency. For a 4-year engineer it is worth noting that this can start as a modular monolith and be split into services only when the team or load justifies it.

Q22. Database Choice

ROI: ★★★★★ Frequency: High

Answer


PostgreSQL

Strong consistency. Complex filtering support.

Job portal data is highly relational: users, jobs, recruiters, and applications reference each other through foreign keys, and operations like "prevent duplicate applications" need transactional guarantees. PostgreSQL gives ACID transactions, mature indexing, and a powerful query planner for the complex multi-column filters job search demands.

A NoSQL store would offer easier horizontal scaling but would push consistency and join logic into the application, which is the wrong trade-off for data with strong relational integrity requirements. PostgreSQL handles the expected scale comfortably with read replicas and indexing.

Q23. Search Optimization

ROI: ★★★★★ Frequency: Senior Favorite

Answer

Use Elasticsearch/OpenSearch.

Not PostgreSQL LIKE queries at scale.

Full-text job search with relevance ranking, fuzzy matching, faceted filters, and typo tolerance is exactly what a search engine like Elasticsearch or OpenSearch is built for. It maintains inverted indexes that make text queries fast even over millions of documents, where a relational LIKE scan would degrade badly.

The pattern is to keep PostgreSQL as the source of truth and asynchronously sync changes into the search index via an event stream such as Kafka. This keeps the two systems decoupled, accepts eventual consistency on the search side (a new job appears in search a moment after it is created), and lets each system do what it is best at.

Q24. Prevent Duplicate Applications

ROI: ★★★★★ Frequency: Extremely High

Answer


UNIQUE(user_id, job_id)

A composite unique constraint on (user_id, job_id) enforces "one application per user per job" at the database level, which is the only place the guarantee can be made reliably. Even if two requests race in (a double-click or a retry), the database rejects the second insert, so correctness does not depend on application-level checks.

Relying on a "check then insert" in application code is unsafe under concurrency: two threads can both read "no existing application" and both insert. The unique constraint closes that race, and the application simply catches the constraint-violation exception and returns a friendly "already applied" response.

Q25. Caching Strategy

ROI: ★★★★★ Frequency: High

  • Popular Jobs
  • Company Profiles
  • Frequently Viewed Jobs

The right things to cache are read-heavy, slowly-changing, and frequently requested: popular and featured jobs, company profiles, and frequently viewed listings. These are read far more often than they change, so a cache delivers a high hit ratio and removes substantial load from the database.

The cache-aside pattern with a sensible TTL fits well here. Because job and company data does change occasionally, the design must handle invalidation (or accept a short staleness window via TTL) so users do not see outdated postings. Highly personalised or write-heavy data is generally a poor caching candidate.

SECTION 7 - LOAD BALANCING

Almost every scalable system requires load balancing. Interviewers frequently ask: "How would you handle increasing traffic?"

Q26. What Is Load Balancing?

ROI: ★★★★★ Frequency: Extremely High

Answer

Distributing traffic across multiple servers.

A load balancer sits in front of a pool of identical application instances and spreads incoming requests across them. This is the foundation of horizontal scaling: instead of buying a bigger server (vertical scaling, which has a ceiling), you add more commodity servers behind the balancer and traffic is shared among them.

It also provides a single stable entry point and performs health checks, automatically removing unhealthy instances from rotation so failures are masked from users. Load balancers operate at Layer 4 (TCP) or Layer 7 (HTTP), with Layer 7 enabling routing decisions based on URL paths, headers, or cookies.

Q27. Why Load Balancing?

ROI: ★★★★★ Frequency: Extremely High

  • High Availability
  • Scalability
  • Fault Tolerance
  • Better Resource Utilization

Load balancing delivers high availability and fault tolerance because if one instance fails its health check, the balancer routes around it and users never notice. There is no single server whose failure takes down the service, which directly attacks the single-point-of-failure problem.

It is also what makes horizontal scaling possible and keeps resource utilisation even, so no single server is overwhelmed while others sit idle. This combination of scalability and resilience is why nearly every production system places a load balancer in front of its application tier.

Q28. Common Load Balancers

ROI: ★★★★★ Frequency: High

  • Nginx
  • HAProxy
  • AWS ALB
  • AWS NLB

Nginx and HAProxy are widely used software load balancers that you run and configure yourself, giving fine-grained control. In cloud environments, managed options are common: the AWS Application Load Balancer (ALB) operates at Layer 7 for HTTP routing, while the Network Load Balancer (NLB) operates at Layer 4 for raw TCP throughput and ultra-low latency.

The choice comes down to the layer you need and the operational model: ALB for content-based HTTP routing and managed scaling, NLB for high-performance TCP, and Nginx/HAProxy when you want full control or are running outside a cloud provider.

Q29. Round Robin Load Balancing

ROI: ★★★★★ Frequency: High 


Request 1 → Server A

Request 2 → Server B

Request 3 → Server C

Round Robin cycles through the server pool in order, sending each new request to the next server. It is simple, stateless, and works well when all servers have similar capacity and all requests are roughly equal in cost.

Its limitation is that it ignores actual load: if one request is expensive and ties up a server while the others are cheap, round robin will still keep handing that busy server more work. When request cost varies significantly, a load-aware strategy like least connections is a better fit.

Q30. Least Connections Strategy

ROI: ★★★★★ Frequency: High

Send request to server with fewest active connections.

Least Connections routes each new request to whichever backend currently has the fewest open connections, which approximates routing to the least busy server. This adapts to uneven request costs far better than round robin, because a server stuck handling slow requests accumulates connections and stops receiving new ones until it catches up.

It is the preferred strategy when request durations vary, such as a mix of quick lookups and long-running operations, because it naturally balances real load rather than just request count.

Q31. Sticky Sessions

ROI: ★★★★★ Frequency: Senior Favorite

Answer

Same user routed to same server.

Sticky sessions (session affinity) pin a given user to the same backend, usually via a cookie, so that server-side session state held in memory remains accessible across that user's requests. It is a workaround for applications that keep state on the server.

Problem

Uneven traffic distribution.

Stateless services are preferred.

Stickiness undermines load balancing: traffic can pile up unevenly, and if a server dies, every user pinned to it loses their session. It also blocks clean horizontal scaling, because new instances cannot pick up existing users' sessions.

The senior-level answer is to make services stateless: store session state externally (in Redis or a JWT held by the client) so any instance can serve any request. Then the load balancer is free to use round robin or least connections, and instances can be added or removed at will.

Senior Engineer Perspective

When an interviewer raises sticky sessions, they are usually probing whether you understand stateless design. The strong answer pivots immediately to externalising state. Stateless services are the foundation of cloud-native scaling, blue-green deployments, and auto-scaling groups, all of which break the moment a server holds irreplaceable in-memory state.

SECTION 8 - REDIS

Q32. What Is Redis?

ROI: ★★★★★ Frequency: Extremely High

Answer

In-memory key-value datastore.

Redis stores data in RAM rather than on disk, which is why it serves reads and writes in sub-millisecond time. It supports rich data structures beyond simple strings, and offers optional persistence (snapshots and append-only logs) so data can survive restarts, though it is most often used as a fast volatile layer.

It is single-threaded for command execution, which guarantees that individual commands are atomic, a property heavily relied on for counters, locks, and rate limiting. It scales out through replication and clustering (sharding by key).

Q33. Why Redis?

ROI: ★★★★★ Frequency: Extremely High

  • Fast
  • Low Latency
  • Simple
  • Distributed

Redis is chosen because in-memory access gives consistently low latency, which is exactly what caching, session lookups, and rate-limit checks need on the hot path. Its API is simple and its atomic operations make concurrency-sensitive patterns straightforward.

It is also distributed-friendly: it acts as a single shared state store that many application instances can read and write consistently, which is essential for things like a global rate-limit counter or a distributed lock that must be honoured across the whole fleet.

Q34. Common Redis Use Cases

ROI: ★★★★★ Frequency: Extremely High

  • Caching
  • Rate Limiting
  • Session Storage
  • Distributed Locks
  • Leaderboards

Caching is the headline use case: Redis sits in front of the database to absorb read traffic. Rate limiting uses its atomic counters, and session storage externalises user sessions so application servers stay stateless and horizontally scalable.

Distributed locks (using SET NX with an expiry, or Redlock) coordinate access to a shared resource across servers, and leaderboards leverage the sorted-set structure to maintain ranked data efficiently. Each use case exploits a specific Redis strength: speed, atomicity, or a purpose-built data structure.

Q35. Redis Data Structures

ROI: ★★★★★ Frequency: Extremely High

  • String
  • Hash
  • List
  • Set
  • Sorted Set

Strings are the basic value type, ideal for cached blobs and counters. Hashes store field-value maps, perfect for representing an object like a user session without serialising the whole thing. Lists are ordered and support queue/stack operations.

Sets store unique unordered members for membership tests and deduplication, while Sorted Sets keep members ordered by a score, which is what makes them the natural choice for leaderboards, priority queues, and sliding-window rate limiters. Choosing the right structure is what makes Redis solutions both fast and simple.

Q36. What Is Redis TTL?

ROI: ★★★★★ Frequency: High

Answer

Time To Live.

Automatically expires keys.

TTL is an expiry set on a key so that Redis automatically deletes it after the configured duration. It is essential for caching: it bounds how stale a cached value can become and ensures memory is reclaimed for entries that are no longer relevant, so the cache does not grow unbounded.

Choosing a TTL is a trade-off between freshness and database load: a short TTL keeps data fresh but causes more misses and database reads, while a long TTL maximises hit ratio but risks serving stale data. Adding a small random jitter to TTLs also helps prevent many keys expiring at the same instant (a cause of cache stampede).

Q37. Redis Eviction Policies

ROI: ★★★★★ Frequency: Senior Favorite

Examples

  • LRU
  • LFU
  • Random

When Redis reaches its memory limit, an eviction policy decides which keys to drop to make room. LRU (Least Recently Used) evicts keys that have not been accessed for the longest time, which suits general caching where recent access predicts future access.

LFU (Least Frequently Used) evicts the keys accessed least often, which is better when some keys are persistently hot regardless of recency. Random eviction is the cheapest but least intelligent. The policy choice matters because the wrong one can evict hot keys and tank the hit ratio.

Senior Engineer Tip

Mention "maxmemory-policy" and the distinction between allkeys-lru and volatile-lru. The "volatile" variants only evict keys that have a TTL set, which lets you protect critical keys (like distributed locks) from eviction while still capping the memory used by cached data. Knowing this configuration detail signals real operational experience with Redis.

SECTION 9 - CACHING STRATEGIES

Q38. Why Cache?

ROI: ★★★★★ Frequency: Extremely High

Reduce database load and improve response time.

A cache stores the results of expensive operations close to the application so repeated requests are served from fast memory instead of recomputing or re-querying the database. This both lowers latency for the user and protects the database, which is usually the hardest tier to scale, from being overwhelmed.

Caching works because most workloads are read-heavy and skewed: a small fraction of data accounts for most reads. By caching that hot data, a high hit ratio offloads the bulk of traffic, letting a modest database handle far more effective throughput. The cost is added complexity around staleness and invalidation.

Q39. Cache Aside Pattern

ROI: ★★★★★ Frequency: Extremely High

Most Common Pattern


Check Cache

↓

Miss

↓

Database

↓

Store In Cache

In cache-aside (lazy loading) the application controls the cache directly: it checks the cache first, and on a miss it reads the database, stores the result in the cache, and returns it. Subsequent reads for the same key then hit the cache. This is the most common pattern because only requested data is cached, keeping memory usage efficient.

Its trade-offs are a first-request penalty (the initial miss pays the full database cost) and the responsibility for invalidation on writes, which the application must handle by evicting or updating the key. It pairs naturally with a TTL so entries eventually refresh even without explicit invalidation.

Q40. Write Through Cache

ROI: ★★★★★ Frequency: High 


Application

↓

Cache

↓

Database

In write-through, every write goes to the cache and the database synchronously as a single operation, so the cache is always consistent with the database. Reads are therefore very likely to hit fresh data, and there is no first-read penalty for recently written keys.

The cost is write latency, because each write waits for both the cache and the database, and the cache may hold data that is never read again, wasting memory. It suits read-heavy workloads where consistency matters and writes are not the dominant operation.

Q41. Write Behind Cache

ROI: ★★★★★ Frequency: High 


Application

↓

Cache

↓

Database Later

In write-behind (write-back) the application writes to the cache and returns immediately, while the cache asynchronously persists to the database later, often batching multiple writes. This gives very low write latency and high write throughput, which is valuable for write-heavy workloads.

The serious trade-off is durability: if the cache fails before flushing, recent writes are lost, and the database is temporarily inconsistent with the cache. It is only appropriate when some data loss is acceptable or when the cache itself is made durable, and it is the most operationally risky of the three patterns.

Q42. Cache Invalidation

ROI: ★★★★★ Frequency: Extremely High

Answer

Removing stale cache entries.

One of the hardest distributed system problems.

Invalidation is the act of removing or refreshing a cached value when the underlying data changes, so clients do not keep reading stale data. The common approaches are TTL-based expiry (simple but allows a staleness window), explicit eviction on write (precise but easy to get wrong), and event-driven invalidation through a message bus.

It is famously hard because in a distributed system there are many caches (per-instance local caches, a shared Redis layer, a CDN) and updates race against reads. Getting it wrong produces either stale data shown to users or thundering herds when too much is invalidated at once. The pragmatic answer is usually TTL plus targeted invalidation on the writes that matter most.

Q43. Cache Stampede

ROI: ★★★★★ Frequency: Senior Favorite

Answer

Many requests hit database simultaneously after cache expiry.

A cache stampede (thundering herd) happens when a popular key expires and, in the brief window before it is repopulated, a flood of concurrent requests all miss the cache and hit the database at once. The database, sized for cache-protected traffic, can be overwhelmed by this sudden surge.

It is especially dangerous for hot keys and after events that expire many keys simultaneously, such as a cache flush or synchronised TTLs. Because it can cascade into a database outage, interviewers treat awareness of it as a senior signal.

Q44. Prevent Cache Stampede

ROI: ★★★★★ Frequency: High

  • Distributed Locks
  • Cache Warming
  • Random TTL

A distributed lock ensures only one request rebuilds an expired key while the others wait for or briefly serve the stale value, so the database sees a single recomputation instead of thousands. This is the most direct defence against a stampede on a single hot key.

Cache warming pre-populates hot keys before they are needed (for example after a deploy), and randomised TTLs (adding jitter) prevent many keys from expiring at the same instant, spreading refreshes over time. Together these techniques keep database load smooth rather than spiky.

SECTION 10 - HIGH AVAILABILITY

Q45. What Is High Availability?

ROI: ★★★★★ Frequency: Extremely High

Answer

System remains operational despite failures.

High availability means the system continues serving users even when individual components fail. It is usually expressed as a percentage of uptime (the "nines"), where, for example, 99.99% availability allows only about 52 minutes of downtime per year. Achieving it requires designing so that no single failure causes an outage.

The core principle is eliminating single points of failure through redundancy: every critical component has a standby, and traffic automatically shifts away from failed instances. Availability is always a trade-off against cost and, in distributed databases, against consistency (the CAP theorem).

Q46. How To Achieve High Availability?

ROI: ★★★★★ Frequency: Extremely High

  • Redundancy
  • Replication
  • Load Balancing
  • Automatic Failover

Redundancy means running multiple instances of every critical component across different machines and ideally different availability zones, so one failure leaves healthy copies. Replication keeps data copied across nodes so the database survives the loss of any single node and reads can be served from replicas.

Load balancing distributes traffic and health-checks instances, removing failed ones from rotation, while automatic failover promotes a standby (for example a replica to primary) without human intervention. Together these turn component failures into non-events from the user's perspective.

Q47. Single Point Of Failure (SPOF)

ROI: ★★★★★ Frequency: Extremely High

Answer

A component whose failure causes system outage.

A single point of failure is any component that has no redundancy, so that its failure takes down the whole system. Identifying and eliminating SPOFs is the central exercise in designing for high availability: you trace every request path and ask "what happens if this one thing dies?"

The fix is always some form of redundancy plus failover: multiple application instances behind a load balancer, a replicated database with automatic promotion, a Redis cluster instead of a single node, and multiple gateways. The goal is that no individual failure is fatal.

Q48. Examples Of SPOF

ROI: ★★★★★ Frequency: High

  • Single Database
  • Single API Gateway
  • Single Redis Node
  • Single Application Server

A single database is the most dangerous SPOF because it usually holds the source of truth; the mitigation is replication with a standby and automatic failover. A single API gateway funnels all traffic through one node, so it needs to be run redundantly behind a load balancer.

A single Redis node, if relied on for sessions or rate limiting, can degrade or break the system on failure, so a replicated cluster is used. A single application server is solved simply by running several behind a load balancer. The recurring pattern is: find the lone component, make it plural, and add automatic failover.

SECTION 11 - DATABASE SCALING IN SYSTEM DESIGN

Q49. Read Scaling Strategy

ROI: ★★★★★ Frequency: Extremely High 


Primary

↓

Replica

Replica

Replica

Read scaling uses read replicas: a primary handles all writes and asynchronously replicates to several read-only replicas, which serve read queries. Because most workloads are read-heavy, distributing reads across replicas dramatically increases read throughput without touching the write path.

The key trade-off is replication lag: replicas are eventually consistent, so a read immediately after a write may return stale data. The design must route reads that require the latest data to the primary (read-your-writes) and accept eventual consistency for the rest. Caching with Redis is the complementary first line of defence before adding replicas.

Q50. Write Scaling Challenges

ROI: ★★★★★ Frequency: High

Much harder than read scaling.

Writes are harder to scale because every write must update the single source of truth consistently, so you cannot simply add more copies the way you do with read replicas. A single primary eventually becomes a write bottleneck, and replicas do not help because they only serve reads.

Scaling writes means splitting the data itself across multiple primaries (sharding/partitioning), which introduces cross-shard queries, rebalancing, distributed transactions, and the risk of hot partitions. This added complexity is why a senior answer exhausts caching and read replicas first and only shards when write volume genuinely demands it.

Q51. Solutions For Write Scaling

ROI: ★★★★★ Frequency: High

  • Sharding
  • Partitioning
  • CQRS

Sharding splits data horizontally across multiple database nodes by a shard key (for example user_id), so writes are spread across independent primaries and each handles a fraction of the load. Partitioning applies the same idea, often within a single database, to keep individual tables and indexes manageable.

CQRS (Command Query Responsibility Segregation) separates the write model from the read model, letting each be optimised and scaled independently, often with the read side served from a denormalised store or search index. The cost of all three is increased complexity: choosing a good shard key, handling cross-shard joins, and managing eventual consistency.

Q52. Hot Partition Problem

ROI: ★★★★★ Frequency: Senior Favorite

Answer

Uneven traffic concentrated on one shard.

A hot partition occurs when the shard key distributes data or traffic unevenly, so one shard receives a disproportionate share of requests while others sit idle. This defeats the purpose of sharding, because the overloaded shard becomes the new bottleneck and effectively a single point of failure for that data.

Common causes are a low-cardinality shard key or a naturally skewed access pattern (one celebrity user, one viral job posting). Mitigations include choosing a higher-cardinality or composite shard key, adding a hashing/salting layer to spread hot keys, and caching the hot entity in front of the database so the shard sees fewer reads.

Senior Engineer Perspective

Shard-key selection is the decision that makes or breaks a sharded system, and interviewers probe it deliberately. A strong answer ties the key to the dominant query pattern (so most queries stay within a single shard) while still distributing load evenly. Getting this wrong is expensive to fix later because it requires re-sharding the entire dataset.

SECTION 12 - SEARCH SYSTEMS

Q53. Why Elasticsearch/OpenSearch?

ROI: ★★★★★ Frequency: Extremely High

Answer

Optimized for searching large text datasets.

Elasticsearch and OpenSearch are built around inverted indexes, a structure that maps each term to the documents containing it. This makes full-text search, relevance ranking, fuzzy and typo-tolerant matching, and faceted filtering fast even across millions of documents, which a relational database cannot do efficiently.

They also scale horizontally by sharding the index across nodes and replicating shards for availability. The standard pattern keeps the relational database as the system of record and treats the search engine as a derived, query-optimised view kept in sync asynchronously.

Q54. Why Not SQL LIKE Queries?

ROI: ★★★★★ Frequency: Extremely High

Poor performance at large scale.

A LIKE query with a leading wildcard (for example %java%) cannot use a standard B-tree index, so the database must scan every row. At small data sizes this is fine, but as the table grows the scan time grows linearly and search latency becomes unacceptable.

LIKE also offers none of the features users expect from search: relevance ranking, typo tolerance, stemming, or scoring. A dedicated search engine handles all of these efficiently through its inverted index, which is why text search is offloaded rather than forced onto SQL pattern matching.

Q55. Search Architecture

ROI: ★★★★★ Frequency: High 


Database

↓

Kafka

↓

Search Index

↓

Search API

The database remains the source of truth. When data changes, an event is published to Kafka, and a consumer applies that change to the search index. The Search API then queries the index rather than the database, so search traffic never touches the primary store.

This event-driven indexing decouples the two systems: the search cluster can be rebuilt, scaled, or briefly unavailable without affecting writes to the database. The trade-off is eventual consistency, since there is a short delay between a write and its appearance in search, which is almost always acceptable for discovery use cases.

Q56. Why Event Driven Index Updates?

ROI: ★★★★★ Frequency: Senior Favorite

Keeps search system decoupled from primary database.

Pushing changes through an event stream rather than writing to the index synchronously means the write path is never blocked by, or coupled to, the search cluster's availability or performance. A slow or down search index produces a harmless backlog of events instead of failed user writes.

It also makes the index rebuildable and additive: you can replay events to reconstruct the index after a schema change, and add new consumers (analytics, recommendations) without modifying the producer. The accepted cost is eventual consistency, which is the standard trade-off for this pattern.

Expected Follow-Up

How do you keep the index in sync if a consumer fails mid-update? The answer is idempotent, replayable consumers: each event is applied in a way that re-processing it produces the same result, and Kafka offsets let the consumer resume from where it stopped. For capturing every change reliably, the Change Data Capture (CDC) pattern (for example via the database's write-ahead log) is the production-grade approach.

SECTION 13 - CDN (CONTENT DELIVERY NETWORK)

CDN questions are common in system design interviews because they touch performance, scalability, and user experience.

Q57. What Is A CDN?

ROI: ★★★★★ Frequency: Extremely High

Answer

A geographically distributed network of servers that cache and serve content closer to users.

A CDN places edge servers in data centres around the world. When a user requests content, they are routed to the nearest edge, which serves a cached copy instead of forcing the request to travel back to a single origin server that might be on another continent. This cuts latency dramatically.

It is essentially a globally distributed cache for static and cacheable content. The origin remains the source of truth, and the CDN handles invalidation through TTLs and explicit purges, trading a small staleness window for major performance and scalability gains.

Q58. Why Use A CDN?

ROI: ★★★★★ Frequency: Extremely High

  • Lower Latency
  • Reduced Origin Load
  • Improved User Experience
  • Higher Availability

Lower latency comes from serving content from an edge physically close to the user, which reduces network round-trip time. Reduced origin load follows because the edge absorbs most requests, so the origin servers and their bandwidth are freed for dynamic work.

This translates directly into a better user experience (faster page loads, smoother media) and higher availability, since the CDN can keep serving cached content even if the origin is temporarily degraded, and it absorbs traffic spikes that would otherwise overwhelm the origin.

Q59. What Content Should Use CDN?

ROI: ★★★★★ Frequency: High

  • Images
  • Videos
  • JavaScript
  • CSS
  • Documents

The ideal CDN candidates are static, cacheable assets that are identical for all users: images, video, JavaScript, CSS, and downloadable documents. These change infrequently and are requested by many users, so caching them at the edge yields a high hit ratio.

Highly dynamic or personalised content (a user's private dashboard) is generally not cached at the CDN, though modern CDNs can cache it briefly or use edge logic. The principle is to push everything static and shared to the edge and let the origin focus on dynamic, per-user responses.

Q60. CDN Flow

ROI: ★★★★★ Frequency: High 


User

↓

Nearest CDN Edge

↓

Cache Hit?

↓

Yes → Return Content

No → Origin Server

The user's request is routed to the nearest edge server. If the edge has the content cached and unexpired (a cache hit), it returns it immediately without contacting the origin. This is the fast, common path that delivers the CDN's latency and load benefits.

On a cache miss, the edge fetches the content from the origin, serves it to the user, and stores a copy locally so subsequent requests in that region become hits. TTLs control how long the edge keeps content before re-validating with the origin, balancing freshness against origin load.

SECTION 14 - API DESIGN

Q61. REST API Design Principles

ROI: ★★★★★ Frequency: Extremely High

  • Resource Oriented
  • Stateless
  • Consistent Naming
  • HTTP Standards
  • Versioning

REST models the API around resources (nouns like jobs and applications) addressed by URIs, with HTTP methods expressing the action. Statelessness means each request carries everything needed to process it, which is what lets any server handle any request and underpins horizontal scaling.

Consistent naming and adherence to HTTP standards (correct status codes, proper use of GET/POST/PUT/DELETE) make the API predictable and self-describing, while versioning protects existing clients when the contract evolves. Together these principles produce APIs that are easy to consume, cache, scale, and change safely.

Q62. Good Endpoint Design

ROI: ★★★★★ Frequency: Extremely High

Good


/api/v1/jobs

/api/v1/jobs/{id}

/api/v1/applications

Bad


/getJobs

/createJob

/deleteJob

Good endpoints are resource-oriented: the URL names a noun (/jobs, /jobs/{id}) and the HTTP method conveys the verb (GET to read, POST to create, DELETE to remove). This keeps the surface small and consistent, and lets infrastructure like caches and gateways reason about requests correctly.

The "bad" examples encode the action into the URL (/getJobs, /createJob), which is RPC-style rather than REST. It bloats the API with one endpoint per action, breaks the uniform interface, and confuses caching since a GET-style action might be issued as a POST. Sticking to nouns plus standard methods is what makes an API intuitive.

Q63. API Versioning Strategies

ROI: ★★★★★ Frequency: High

  • URI Versioning
  • Header Versioning
  • Query Parameter Versioning

Most Common


/api/v1/jobs

URI versioning (/api/v1/jobs) is the most common because it is explicit, easy to route, easy to test in a browser, and unambiguous in logs. Header versioning keeps URLs clean by selecting the version through a custom header, and query-parameter versioning passes it as ?version=1.

Versioning exists so the contract can evolve without breaking existing clients: you introduce v2 alongside v1 and migrate consumers gradually. The trade-off across strategies is cleanliness of the URL versus visibility and ease of routing, and URI versioning usually wins on operability.

Q64. Idempotent HTTP Methods

ROI: ★★★★★ Frequency: Extremely High

Method Idempotent?
GET Yes
PUT Yes
DELETE Yes
POST No

An idempotent method produces the same result whether it is called once or many times. GET only reads, so it changes nothing. PUT replaces a resource with a given representation, so repeating it leaves the same final state. DELETE removes a resource, and deleting it again leaves it deleted.

POST is not idempotent because it typically creates a new resource each time, so two identical POSTs create two records. This matters under retries and network failures: a client can safely retry GET, PUT, and DELETE, but must guard POST with an idempotency key so a retried "apply to job" does not create a duplicate application.

Q65. Pagination API Example

ROI: ★★★★★ Frequency: Extremely High 


/jobs?page=1&size=20

Pagination returns results in bounded pages instead of all at once, which protects both the server and the client from huge responses. Offset pagination (page and size) is simple and shown here, but its weakness is that deep pages get slow, because the database must skip all preceding rows.

For large or frequently-changing datasets, cursor-based (keyset) pagination is preferred: the client passes the last-seen key and the server fetches the next slice using an indexed range, which stays fast at any depth and avoids items shifting between pages as data changes. Mentioning this trade-off is a senior signal.

Q66. Filtering API Example

ROI: ★★★★★ Frequency: High 


/jobs?location=kolkata

/jobs?skill=java

/jobs?salary=1000000

Filtering is expressed through query parameters on the collection resource, keeping the URL readable and the parameters composable (location, skill, and salary can be combined). The server translates these into indexed query conditions, which is why the underlying columns should be indexed to keep filtered reads fast.

At scale, complex multi-field filtering and free-text search are usually pushed to a search engine rather than answered by SQL, as discussed in the search section. For structured filters on a relational store, well-chosen composite indexes keep these endpoints performant.

Start by separating functional requirements (the features above) from non-functional ones (scalability, availability, low-latency search, durability of resumes). Each requirement maps to a design choice made later in this section: search to a dedicated index, resume upload to object storage, notifications to an async event pipeline, and apply-to-job to a transactional constraint. Framing it this way shows the interviewer structured thinking rather than jumping straight to boxes and arrows.

Q68. High-Level Architecture

The frontend talks to a single API Gateway, which handles cross-cutting concerns: authentication, rate limiting, routing, and request aggregation. Behind it sit services split by business domain, each owning its own data and able to scale independently based on its specific load profile.

This layout lets the read-heavy Search Service scale separately from the write-heavy Application Service, and lets the Notification Service work asynchronously off an event stream. For a team of moderate size, the same boundaries can begin life as modules in a single deployable (a modular monolith) and be extracted into separate services as scale and team structure demand.

Q69. Database Choice

ROI: ★★★★★ Frequency: High 


PostgreSQL

Reason: Strong consistency. Relational data. Complex filtering.

The core entities (users, jobs, applications, recruiters) are highly relational and require transactional integrity, for example to guarantee one application per user per job. PostgreSQL provides ACID transactions, foreign keys, mature indexing, and a strong query planner for the multi-condition filters job listings require.

It scales for this domain through read replicas for read traffic and, only if write volume truly demands it, sharding by a sensible key. Reaching for NoSQL here would trade away the consistency and relational guarantees that this data fundamentally needs.

Q70. Resume Storage

ROI: ★★★★★ Frequency: Extremely High 


AWS S3

Store URL in database.

Resumes are large binary files, so they belong in object storage like S3, not in the database. S3 gives high durability, cheap virtually unlimited capacity, and CDN integration, while the database stores only the metadata and the S3 URL/key needed to locate and authorise access to the file.

Uploads should go directly to S3 via pre-signed URLs so the application server never has to stream large files, keeping it lightweight and scalable. This separation of bytes (S3) from references (PostgreSQL) is the standard pattern for any file-handling feature.

Q71. Search Architecture

ROI: ★★★★★ Frequency: Senior Favorite 


Job Created

↓

Kafka

↓

Search Index

↓

OpenSearch

When a job is created or updated, the Job Service publishes an event to Kafka. A consumer reads that event and updates the OpenSearch index, so search queries hit a structure optimised for full-text relevance and filtering rather than the transactional database.

This event-driven indexing decouples search from the write path: OpenSearch can be rebuilt or scaled independently, and an indexing delay only means a new job appears in search a moment later (eventual consistency), which is acceptable. PostgreSQL stays the source of truth while OpenSearch serves discovery.

Q72. Cache Strategy

ROI: ★★★★★ Frequency: High

  • Popular Jobs
  • Featured Jobs
  • Company Profiles

The caching targets are read-heavy, slowly-changing, and widely-shared data: popular and featured jobs and company profiles. Caching these in Redis with the cache-aside pattern and a TTL yields a high hit ratio and removes the bulk of repetitive reads from PostgreSQL.

Because this data changes occasionally, the design relies on TTL-based freshness plus targeted invalidation when a company or featured listing is updated. Personalised or rapidly-changing data (a specific user's application list) is generally not cached, since it would have a low hit ratio and high invalidation churn.

Q73. Prevent Duplicate Applications

ROI: ★★★★★ Frequency: Extremely High 


UNIQUE(user_id, job_id)

A composite unique constraint on (user_id, job_id) enforces one application per user per job at the database level, which is the only reliable place to guarantee it under concurrency. A racing double-click or retried request is rejected by the constraint rather than slipping through an application-level check.

The application catches the unique-violation and returns a clean "already applied" response. This pushes the correctness guarantee down to where consistency is enforced, instead of depending on read-then-write logic that two concurrent requests could both pass.

Q74. Notification Design

ROI: ★★★★★ Frequency: High

↓

↓

Notification Service

↓

Email

SMS

Push

Submitting an application publishes a Kafka event and returns immediately, so the user is never blocked waiting on email or SMS delivery. The Notification Service consumes the event asynchronously and fans it out to the appropriate channels based on user preferences.

Because delivery runs off a durable queue, a provider outage causes a retryable backlog rather than a user-facing error, and the service applies retries with backoff, idempotency to avoid duplicate sends, and a dead-letter queue for messages that repeatedly fail. This keeps the core apply flow fast and reliable.

Q75. Security Design

ROI: ★★★★★ Frequency: Extremely High

  • JWT
  • Refresh Tokens
  • RBAC
  • Rate Limiting
  • HTTPS

JWTs provide stateless authentication so any service can validate a request without a shared session store, which is essential for horizontal scaling. Short-lived access tokens are paired with longer-lived refresh tokens so a leaked access token has a small blast radius while users are not forced to log in constantly.

RBAC enforces that job seekers, recruiters, and admins can only perform their permitted actions, and is checked at the gateway and service layer. Rate limiting protects against abuse and brute-force attempts, and HTTPS encrypts all traffic in transit. Together these form defence in depth rather than relying on any single control.

SECTION 16 - SENIOR DESIGN SCENARIOS

Q76. Job Search Suddenly Becomes Slow

ROI: ★★★★★ Frequency: Senior Favorite

Investigation

  1. Database Queries
  2. Index Usage
  3. Search Cluster Health
  4. Cache Hit Ratio
  5. Network Latency

The value of this question is the structured investigation, not a single root cause. You work from the most likely and cheapest-to-check causes outward: are slow queries appearing, are indexes still being used (or did a query plan regress), is the search cluster healthy and not rebalancing, has the cache hit ratio dropped, and is there a network or downstream latency spike.

A senior engineer also anchors the investigation in observability: dashboards, slow-query logs, APM traces, and metrics tell you where time is actually being spent rather than guessing. The interviewer is checking that you reason methodically under pressure instead of jumping to a fix.

Q77. Redis Goes Down

ROI: ★★★★★ Frequency: Extremely High

Expected Answer

System should continue working.

Performance decreases.

Database becomes fallback source.

Redis is a cache, not the source of truth, so its failure should degrade performance rather than cause an outage. The application falls back to reading from the database, accepting higher latency and load until Redis recovers. This is the graceful-degradation principle interviewers want to hear.

The risk is that with the cache gone, full traffic hits the database at once, which can overwhelm it (a stampede). Mitigations include a replicated Redis cluster to avoid a single point of failure, circuit breakers and timeouts so a slow Redis does not block requests, and load shedding to protect the database while the cache is rebuilt.

Q78. Kafka Goes Down

ROI: ★★★★★ Frequency: High

Answer

Asynchronous features impacted.

  • Notifications Delayed
  • Search Updates Delayed
  • Core APIs Should Continue

Because Kafka backs only the asynchronous pipelines, its failure delays notifications and search index updates but should not break the synchronous core APIs like login, job posting, or applying. This is the benefit of decoupling: the user-facing path does not depend on the event bus being up.

Once Kafka recovers, the backlog is processed and the system catches up, since events were either buffered or are re-published. In production Kafka itself is run as a replicated cluster to make this scenario rare, and producers handle brief outages with retries or local buffering so events are not lost.

Q79. Database Goes Down

ROI: ★★★★★ Frequency: Extremely High

Answer

Critical outage.

Unlike Redis or Kafka, the database holds the source of truth, so its loss is a critical outage: writes fail and most reads cannot be served correctly. This is why the database is the SPOF that most deserves redundancy and why protecting it shapes much of the architecture.

Mitigation

  • Replication
  • Failover
  • Backups
  • Disaster Recovery

Replication keeps synchronised standbys so a replica can be promoted, and automatic failover does that promotion quickly to minimise downtime. Regular, tested backups guard against data corruption and accidental deletion, which replication alone does not, since corruption replicates too.

A disaster recovery plan with a secondary region, plus defined RPO (how much data loss is tolerable) and RTO (how fast you must recover), turns "the database is down" from a catastrophe into a rehearsed procedure. Mentioning RPO/RTO signals senior-level operational maturity.

Q80. System Receives 10x Traffic

ROI: ★★★★★ Frequency: Extremely High

Scaling Plan

  • More Application Instances
  • Redis Scaling
  • Read Replicas
  • CDN
  • Load Balancer

Because the application tier is stateless, the first lever is horizontal scaling: add more instances behind the load balancer, ideally via auto-scaling that reacts to load. A CDN offloads static content at the edge, and Redis absorbs read traffic so the database is shielded from the surge.

On the data tier, read replicas scale read throughput; if writes become the bottleneck, sharding is the next step, with its added complexity. The senior framing is to apply the cheap, high-leverage changes first (scale out, cache, CDN, replicas) and reach for the complex ones (sharding, CQRS) only when the simpler measures are exhausted.

SECTION 17 - ARCHITECTURE TRADEOFF QUESTIONS

Q81. SQL vs NoSQL

ROI: ★★★★★ Frequency: Extremely High

SQL NoSQL
ACID Flexible Schema
Strong Consistency Horizontal Scaling

Job Portal Choice

SQL.

SQL databases offer ACID transactions, strong consistency, and rich relational joins, which fit data with clear relationships and integrity requirements. NoSQL trades some of that away for a flexible schema and easier horizontal scaling, which suits massive scale, high write volume, or loosely-structured data.

For the job portal the choice is SQL, because the data is relational and consistency matters (no duplicate applications, accurate ownership). The honest senior answer is that it is not "SQL is better" but "match the store to the access pattern and consistency needs", and many large systems use both (polyglot persistence) for different parts.

Q82. Monolith vs Microservices

ROI: ★★★★★ Frequency: Extremely High

Interview Gold Answer

Start with modular monolith.

Move to microservices when justified.

Interview Gold Point

A modular monolith gives you clear domain boundaries inside a single deployable, so you get most of the design benefits without the operational cost. Microservices add real overhead: network latency between calls, distributed transactions, service discovery, distributed tracing, and far more deployment and monitoring complexity. The mature answer is to defer that complexity until team size, independent scaling needs, or release cadence genuinely justify it. Saying "I would start with a modular monolith and extract services when there is a concrete reason" signals seniority far more than reflexively reaching for microservices.

Q83. Redis vs Database

ROI: ★★★★★ Frequency: High

Redis Database
Fast Durable
Cache Source Of Truth

Redis and the database are complementary, not competing. Redis is an in-memory cache: extremely fast but volatile, used to serve hot data and reduce load. The database is durable and is the source of truth, holding the authoritative, persistent state the system relies on.

The design principle is that Redis can fail without data loss, because everything in it is reconstructable from the database, whereas the database failing is a true outage. You therefore never treat the cache as authoritative, and you size each for its role: the database for durability and consistency, Redis for speed.

These ten are the vocabulary of almost every system design discussion: a load balancer plus horizontal scaling for the application tier, Redis and a CDN for caching, database replication for read scaling and availability, Kafka for async decoupling, an API gateway for cross-cutting concerns, and rate limiting for protection.

The tenth, trade-offs, is the most important and the one interviewers actually score. Knowing the components is table stakes; explaining why you chose one over another, and what you gave up, is what distinguishes a senior engineer. Every concept above is a lever, and good design is choosing the right levers for the specific constraints.
