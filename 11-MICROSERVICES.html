SECTION 1 - MICROSERVICE FUNDAMENTALS

Q1. What Is A Microservice?

ROI: ★★★★★ Frequency: Extremely High

Answer

A microservice is a small, independently deployable service focused on a specific business capability. Instead of bundling every feature into one large application, the system is split into a set of services that each own one slice of the domain and can be built, deployed and scaled on their own.

The key word is "independently deployable." A service that cannot be released without coordinating with five other teams is not really a microservice, it is a distributed monolith. A true microservice has its own codebase, its own database, and a clear contract (usually a REST or messaging interface) through which the rest of the system talks to it.

Each service is also organized around a business capability rather than a technical layer. We do not create a "controllers service" and a "repositories service"; we create a User Service, a Payment Service, and so on, so that a single team can own a vertical slice end to end.

Q2. Monolith vs Microservices

ROI: ★★★★★ Frequency: Extremely High

Monolith Microservices
Single Deployment Multiple Deployments
Single Codebase Distributed Codebases
Simpler More Complex
Easier Development Better Scalability

A monolith packages all functionality into one deployable unit with a single shared codebase and usually a single database. This makes early development fast: there are no network calls between modules, transactions are simple, and you only deploy one artifact. The cost is that as the codebase grows, build times increase, teams step on each other, and you must scale the entire application even when only one feature is hot.

Microservices trade that simplicity for independence. Each service is its own deployable, which lets teams release on their own schedule and scale individual services to match real load. The price is operational complexity: network latency, partial failures, distributed transactions and harder debugging all become everyday concerns.

The senior framing is that this is a trade-off, not an upgrade. Monoliths optimize for development simplicity; microservices optimize for organizational and runtime scalability. You choose based on team size, traffic, and how independently parts of the system need to evolve.

Q3. Advantages Of Microservices

ROI: ★★★★★ Frequency: Extremely High

  • Independent Deployment
  • Independent Scaling
  • Fault Isolation
  • Technology Flexibility
  • Smaller Teams

Independent deployment means a bug fix in the Notification Service does not require redeploying Payments or Orders, which shortens release cycles and reduces blast radius. Independent scaling lets you run ten instances of a high-traffic service and one instance of a rarely used one, instead of scaling the whole application uniformly.

Fault isolation is one of the strongest arguments: if one service degrades, well-designed boundaries (with circuit breakers and timeouts) prevent that failure from taking down the rest of the system. Technology flexibility lets a team pick the right tool per service, and smaller teams owning smaller services aligns directly with Conway's Law, keeping ownership clear and communication overhead low.

Q4. Disadvantages Of Microservices

ROI: ★★★★★ Frequency: Extremely High

  • Network Complexity
  • Distributed Transactions
  • Monitoring Challenges
  • Deployment Complexity
  • Operational Overhead

Every in-process method call in a monolith becomes a network call in microservices, which introduces latency, serialization cost and a whole class of failures (timeouts, retries, partial failures) that simply did not exist before. Maintaining data consistency across services is genuinely hard because you can no longer wrap everything in a single database transaction.

Operationally, you now run many services, each needing deployment pipelines, monitoring, logging, service discovery and on-call coverage. Debugging a single user request can mean tracing it across five services, which is why distributed tracing and centralized logging stop being optional and become core infrastructure.

Interview Gold Point

Microservices are not automatically better than monoliths. The senior signal here is acknowledging the cost. Strong candidates say "microservices solve organizational and scaling problems, but they introduce distributed-systems complexity," and they only recommend the split when the team size and traffic actually justify it. Reciting only the advantages is a junior tell.

Q5. When Would You Avoid Microservices?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

  • Small Team
  • Simple Product
  • Early Startup Stage
  • Limited Infrastructure

With a small team, the operational overhead of running many services (pipelines, monitoring, on-call, infrastructure) outweighs any scaling benefit. A handful of engineers maintaining fifteen services spend more time on plumbing than on features. Similarly, an early-stage product still discovering its domain boundaries will get those boundaries wrong, and refactoring boundaries across deployed services is far more painful than refactoring inside one codebase.

The recommended path is to start with a modular monolith: one deployable, but with clean internal module boundaries. This gives you the development speed of a monolith while keeping the seams along which you can later split out services once the boundaries and the scaling pressure are clear.

Start with modular monolith.

SECTION 2 - SERVICE COMMUNICATION

Q6. How Do Microservices Communicate?

ROI: ★★★★★ Frequency: Extremely High

Methods

  • REST
  • gRPC
  • Message Queues
  • Kafka

Communication splits into two broad styles. Synchronous request/response (REST, gRPC) is used when the caller needs an answer right now, for example a checkout flow that must confirm payment before responding to the user. Asynchronous messaging (message queues, Kafka) is used when the caller can fire an event and move on, letting downstream services react in their own time.

The senior point is to default to asynchronous, event-driven communication wherever the business flow allows it, because synchronous chains couple services together at runtime: if Service A calls B calls C, then C being slow makes A slow. Choosing the right style per interaction is one of the most important design decisions in a microservice system.

Q7. Synchronous vs Asynchronous Communication

ROI: ★★★★★ Frequency: Extremely High

Synchronous Asynchronous
Waits For Response No Waiting
REST Kafka
Tighter Coupling Looser Coupling

In synchronous communication the caller blocks until it receives a response, so its availability and latency are directly tied to the callee's. This is simple to reason about and gives immediate feedback, but it creates temporal coupling: every service in the chain must be up at the same time, and slowness propagates upstream.

Asynchronous communication decouples services in time. The producer publishes a message or event and continues; consumers process it whenever they are ready. This improves resilience and scalability because a temporarily down consumer just causes a backlog rather than a user-facing failure. The trade-off is that you give up immediate consistency and must design for eventual consistency, ordering and duplicate handling.

Q8. REST vs gRPC

ROI: ★★★★★ Frequency: High

REST gRPC
JSON Protocol Buffers
Human Readable Binary
Slower Faster

REST over HTTP with JSON is the default for external and public APIs because it is human-readable, universally supported, easy to debug with a browser or curl, and loosely coupled to schema. The cost is larger payloads and the overhead of text-based serialization.

gRPC uses Protocol Buffers over HTTP/2, giving compact binary messages, a strongly typed contract, generated client stubs, and support for streaming. It shines for high-throughput internal service-to-service calls where performance matters and both sides are under your control. The practical rule: REST at the edge for clients and partners, gRPC internally where latency and throughput are critical.

SECTION 3 - SERVICE DISCOVERY

Q9. Why Service Discovery?

ROI: ★★★★★ Frequency: Extremely High

Problem

Service instances change dynamically. In a containerized or cloud environment, instances are created and destroyed constantly through auto-scaling, deployments, crashes and restarts. Each new instance gets a different IP address, so the set of addresses backing a given service is never stable. 


user-service

10.0.1.1

10.0.1.2

10.0.1.3

Hardcoding addresses is impossible. If a consumer hardcoded "10.0.1.1", the first time that instance is rescheduled the call would fail. Service discovery solves this by letting consumers look up "user-service" by logical name and get back the current healthy instances, so the system keeps working as instances come and go.

Q10. What Is Service Discovery?

ROI: ★★★★★ Frequency: Extremely High

Answer

Service discovery is the mechanism for locating service instances dynamically by logical name rather than by fixed network address. A service registry keeps an up-to-date list of which instances of each service are currently running and healthy, and consumers query it (or a load balancer that uses it) to find a target.

There are two models. In client-side discovery, the caller queries the registry and picks an instance itself (for example with a Eureka client plus Spring Cloud LoadBalancer). In server-side discovery, the caller hits a fixed endpoint like a load balancer or Kubernetes Service, which consults the registry and routes the request. Both rely on services registering on startup and being removed via health checks when they die.

Q11. What Is Eureka?

ROI: ★★★★★ Frequency: High

Answer

Eureka is Netflix's service registry, widely used in the Spring Cloud ecosystem. Services register themselves with the Eureka server on startup, announcing their name and network location, and they send periodic heartbeats so the registry knows they are still alive.

Consumers fetch the registry (and cache it locally) so they can resolve a logical name like "user-service" to a live instance and load-balance across the available ones. If an instance stops sending heartbeats, Eureka eventually evicts it so traffic stops being routed to a dead node. In Kubernetes environments, the platform's built-in Service and DNS often replace Eureka, but Eureka remains a common interview reference for self-managed Spring Cloud setups.

SECTION 4 - API GATEWAY

Q13. What Is API Gateway?

ROI: ★★★★★ Frequency: Extremely High

Answer

An API Gateway is a single entry point for all client requests into the microservice system. Instead of clients knowing about and calling dozens of individual services, they talk to the gateway, which routes each request to the appropriate backend service.

This solves a real problem: without a gateway, every client would need to know the location of every service, handle authentication against each one, and deal with cross-cutting concerns like rate limiting independently. The gateway centralizes those concerns and presents a clean, stable surface to the outside world, decoupling clients from the internal service topology.

Q14. Why API Gateway?

ROI: ★★★★★ Frequency: Extremely High

  • Routing
  • Authentication
  • Rate Limiting
  • Logging
  • Load Balancing

The gateway is where cross-cutting concerns live so individual services do not each reimplement them. It routes requests to the right service, performs authentication and token validation once at the edge (so downstream services can trust the identity), and enforces rate limiting to protect the system from abuse and traffic spikes.

It is also the natural place for centralized logging, request correlation IDs, load balancing across instances, and response aggregation. Pushing these to the edge keeps the backend services focused purely on business logic, which is exactly the separation of concerns interviewers want to hear.

The client sends every request to the gateway. The gateway authenticates the request, applies filters such as rate limiting and logging, inspects the path or headers, and then forwards it to the matching backend service (User, Job, Notification, and so on). The response travels back through the gateway to the client.

Because the gateway sits on the critical path for every request, it must be highly available and fast. That is why it is typically run as multiple stateless instances behind a load balancer, so it never becomes a single point of failure.

Q16. Spring Cloud Gateway

ROI: ★★★★★ Frequency: High

Answer

Spring Cloud Gateway is the modern Spring API Gateway solution, built on Spring WebFlux and Project Reactor. Being reactive and non-blocking, it handles a large number of concurrent connections efficiently, which matters at the edge where all traffic converges. It replaced the older Netflix Zuul 1.x in the Spring ecosystem.

Common Features

  • Routing
  • Filters
  • Security
  • Rate Limiting

Its model is built around routes (predicates that match a request) and filters (logic applied before and after forwarding). Filters handle authentication, header rewriting, rate limiting, retries and circuit breaking. It integrates cleanly with service discovery and Resilience4j, so you can declaratively configure how the edge routes, secures and protects traffic to your services.

SECTION 5 - LOAD BALANCING

Q17. Why Load Balancing?

ROI: ★★★★★ Frequency: Extremely High

Load balancing distributes incoming traffic across the multiple instances of a service. Since each service runs several instances for scalability and redundancy, something must decide which instance handles each request; that is the load balancer's job.

Beyond spreading load evenly (round-robin, least-connections, and so on), load balancing is also a resilience mechanism: when combined with health checks, it stops sending traffic to unhealthy instances, so a single failing node does not cause user-facing errors. This is what lets a service scale horizontally and survive instance failures transparently.

Q18. Client Side vs Server Side Load Balancing

ROI: ★★★★★ Frequency: High

Client Side Server Side
Service Chooses Instance Load Balancer Chooses
Spring Cloud LoadBalancer Nginx

In client-side load balancing, the calling service holds the list of available instances (from the service registry) and picks one itself before making the call, for example using Spring Cloud LoadBalancer. This removes an extra network hop and lets the client apply smart, application-aware strategies, but it pushes balancing logic into every client.

In server-side load balancing, a dedicated component such as Nginx, an AWS ELB, or a Kubernetes Service sits in front of the instances and chooses the target. Clients stay simple because they just hit one stable address. The trade-off is the extra hop and the load balancer becoming a component you must scale and keep available.

Q19. What Replaced Ribbon?

ROI: ★★★★★ Frequency: High

Answer

Spring Cloud LoadBalancer replaced Netflix Ribbon, which is now in maintenance mode and deprecated in the Spring Cloud ecosystem. Spring Cloud LoadBalancer is the modern, actively maintained client-side load balancer; it is reactive-friendly, integrates with Spring's discovery abstractions, and is the default backing the load-balanced WebClient and RestTemplate.

This is part of a broader pattern: many of the original Netflix OSS components (Ribbon, Hystrix, Zuul 1) have been superseded by newer alternatives such as Spring Cloud LoadBalancer, Resilience4j and Spring Cloud Gateway. Knowing the modern replacements signals that your microservice knowledge is current rather than several years stale.

SECTION 6 - FAILURE IN DISTRIBUTED SYSTEMS

One of the biggest differences between monoliths and microservices: Network failures become normal. Senior interviewers often ask: "What can go wrong when one service calls another?"

Q20. What Types Of Failures Exist In Microservices?

ROI: ★★★★★ Frequency: Extremely High

  • Network Failure
  • Service Down
  • Slow Responses
  • Database Failure
  • Message Queue Failure
  • Partial Failure

In a monolith, a method call either returns or throws. In a distributed system, a call can fail in many more ways: the network can drop or delay packets, the target service can be entirely down, or, most insidiously, it can be slow rather than dead. Slow responses are often worse than outright failures because they tie up the caller's threads and connections while it waits.

Databases and message brokers can also fail independently of the services that use them. The defining characteristic, though, is partial failure: in a distributed system some components keep working while others fail, leaving the overall system in an inconsistent, half-completed state. Designing for these failures with timeouts, retries, circuit breakers and idempotency is the core of resilient microservice design.

Q21. What Is Partial Failure?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

Partial failure is when some services work while others fail. Because each service runs in its own process on its own host, there is no single switch that brings everything down together; instead, one piece can break while the rest keep serving traffic.

Q22. Why Are Distributed Systems Hard?

ROI: ★★★★★ Frequency: High

  • Network Latency
  • Partial Failures
  • Data Consistency
  • Observability Challenges
  • Retry Complexity

Distributed systems are hard because the network is unreliable and not free: every remote call adds latency and can fail or time out. Once data is spread across services and databases, keeping it consistent without a single transaction becomes a design problem in its own right, forcing trade-offs around eventual consistency.

On top of that, understanding what the system is doing is much harder: a single user action spans many services, so you need distributed tracing and centralized logging just to follow one request. Even retries, which sound trivial, are subtle: naive retries can duplicate work or amplify an outage into a retry storm. The classic "fallacies of distributed computing" capture exactly these traps.

SECTION 7 - CIRCUIT BREAKER

Q23. What Is Circuit Breaker Pattern?

ROI: ★★★★★ Frequency: Extremely High

Answer

The Circuit Breaker pattern prevents repeated calls to a failing service. Like an electrical circuit breaker, it monitors calls to a remote dependency and, once failures cross a threshold, it "trips" and short-circuits further calls, failing fast instead of waiting on a service that is clearly unhealthy.

Purpose

The purpose is to avoid cascading failures. Without it, a slow or down dependency causes callers to pile up waiting on it, exhausting their threads and connection pools; those callers then become slow, and their callers pile up in turn, until the failure cascades across the whole system. By failing fast and giving the struggling service room to recover, the circuit breaker contains the damage and keeps the rest of the system responsive.

A circuit breaker breaks this chain. After it detects that B is failing, it stops forwarding calls and immediately returns an error or a fallback, so A's threads are freed instead of being consumed waiting on a dead dependency. This is the single most important resilience pattern for synchronous service-to-service calls.

Q25. Circuit Breaker States

↓

A circuit breaker is a small state machine with three states. It starts CLOSED, letting calls through while monitoring the failure rate. When failures exceed the configured threshold it moves to OPEN, where it blocks calls and fails fast for a cooldown period. After that period it moves to HALF-OPEN, allowing a few trial calls to test whether the dependency has recovered.

From HALF-OPEN, if the trial calls succeed it returns to CLOSED and normal traffic resumes; if they fail it goes back to OPEN and waits again. This cycle lets the system automatically detect recovery and heal without human intervention, which is the whole point of the pattern.

Q26. CLOSED State

ROI: ★★★★★ Frequency: High

In the CLOSED state, requests flow normally to the target service and the breaker simply watches the outcomes, counting failures and successes over a sliding window. This is the healthy, default state.

The breaker is constantly evaluating whether the failure rate (or slow-call rate) exceeds the configured threshold. As long as the dependency behaves, it stays closed and adds essentially no overhead. The moment the failure rate crosses the limit, it transitions to OPEN to protect the caller.

Q27. OPEN State

ROI: ★★★★★ Frequency: High

In the OPEN state, requests are blocked immediately and no remote call is performed. The breaker fails fast, instantly returning an exception or invoking a fallback, which frees the caller's threads and stops piling pressure on the failing dependency.

OPEN lasts for a configured wait duration, giving the downstream service breathing room to recover (restart, clear its backlog, scale up). When that timer expires, the breaker moves to HALF-OPEN to cautiously test whether the dependency is healthy again rather than blindly reopening the floodgates.

Q28. HALF-OPEN State

ROI: ★★★★★ Frequency: High

In the HALF-OPEN state, a small, limited number of trial requests are allowed through to test whether the downstream service has recovered. The rest are still rejected, so the dependency is not hit with full traffic before it has proven itself.

This is the recovery-probing state. If the trial calls succeed, the breaker concludes the service is healthy and transitions back to CLOSED, restoring normal traffic. If they fail, it returns to OPEN and starts another cooldown. This controlled probing is what prevents the breaker from oscillating wildly and from slamming a service that is only partway recovered.

Q29. What Is Resilience4j?

ROI: ★★★★★ Frequency: Extremely High

Answer

Resilience4j is a lightweight fault-tolerance library widely used in Spring Boot microservices, and it is the modern replacement for Netflix Hystrix. It is modular and functional in design: you wrap a call with the specific resilience decorators you need rather than pulling in a heavy framework.

Features

  • Circuit Breaker
  • Retry
  • Rate Limiter
  • Bulkhead
  • Time Limiter

Each feature addresses a specific failure mode: the circuit breaker stops calls to failing dependencies, retry handles transient errors, the rate limiter caps request volume, the bulkhead isolates resources so one slow dependency cannot consume all threads, and the time limiter enforces timeouts. In Spring Boot these are typically applied with annotations or configuration and combined (for example retry inside a circuit breaker), giving you composable, declarative resilience around any remote call.

Q30. Fallback Method

ROI: ★★★★★ Frequency: High

Answer

A fallback is the alternative logic executed when a service call fails or the circuit breaker is open. Instead of propagating an error to the user, the system degrades gracefully by returning something useful, which keeps the overall experience functional even when a dependency is down.

Example


Return Cached Data

Return Default Response

Common fallbacks include returning recently cached data, returning a sensible default or empty result, or queuing the work to be processed later. The key design judgement is that a fallback must be safe for the business: returning a cached product price is fine, but you would never "fall back" by silently approving a payment. A good fallback turns a hard failure into a controlled, degraded response.

SECTION 8 - RETRY PATTERN

Q31. What Is Retry Pattern?

ROI: ★★★★★ Frequency: Extremely High

The Retry pattern automatically retries failed operations, on the assumption that many failures in distributed systems are transient: a brief network blip, a momentary timeout, or a quick GC pause on the target. Rather than failing the whole request on the first error, the caller tries again a limited number of times.

The important constraints are that retries should only apply to transient, retryable errors (timeouts, 503s), not to deterministic ones like a 400 Bad Request, and that the operation being retried should be idempotent so repeating it does not duplicate side effects. Retries are also almost always combined with backoff and a circuit breaker to avoid making a struggling system worse.

Q32. When Is Retry Useful?

ROI: ★★★★★ Frequency: High

  • Temporary Network Issues
  • Transient Failures
  • Short Service Outages

Retries are most valuable when the failure is likely to disappear on its own within a short window: a dropped packet, a connection reset, a momentary spike that briefly overloaded the target, or a few seconds of unavailability during a rolling deployment. In these cases a second attempt a moment later usually succeeds, turning a flaky dependency into a reliable one from the caller's perspective.

The judgement call is that the operation must be safe to repeat. Reads are naturally safe; writes need an idempotency key or a unique constraint so a retried request does not create a duplicate. When those conditions hold, retries dramatically improve reliability with very little code.

Q33. When Is Retry Dangerous?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

Retries are dangerous when the service is already overloaded, because in that case retries amplify the failure. A struggling service is failing precisely because it cannot keep up; sending it the same requests again, multiplied across many clients, adds even more load and can push it from degraded to fully down. This is the classic "retry storm" or "thundering herd."

Retries are also harmful for non-idempotent operations, where repeating the call (for example a payment) can cause duplicate side effects, and for non-transient errors, where retrying a deterministic failure just wastes time and resources. The senior answer is to retry only transient failures, only on idempotent operations, with a small cap, exponential backoff plus jitter, and a circuit breaker that stops retries entirely once the dependency is clearly down.

Exponential backoff means each successive retry waits progressively longer, typically doubling the delay: 1s, then 2s, then 4s, then 8s. Instead of hammering a failing service every few milliseconds, the caller spaces its attempts out, giving the dependency time to recover before the next try.

Benefit

The benefit is that it reduces the retry storm. If thousands of clients fail at the same instant and all retry on the same fixed schedule, they hit the recovering service in synchronized waves and knock it over again. Exponential backoff spreads the load over time, and adding random jitter to each delay desynchronizes the clients so the retries do not align into spikes. This combination is the standard production approach to retrying safely.

SECTION 9 - BULKHEAD PATTERN

Q35. What Is Bulkhead Pattern?

ROI: ★★★★★ Frequency: Increasingly Common

Answer

The Bulkhead pattern isolates resources so a failure in one area does not spread to others. The name comes from ships, which are divided into watertight compartments (bulkheads) so that a breach in one compartment floods only that section instead of sinking the entire vessel.

Applied to services, it means partitioning resources such as thread pools, connection pools or instances per dependency or per feature. If one downstream dependency becomes slow and saturates its dedicated pool, only the calls that use that pool are affected; the rest of the service keeps functioning normally. It is a containment pattern that complements circuit breakers and timeouts.

Bulkheads guarantee that a non-critical, misbehaving dependency cannot starve a critical one of resources.

SECTION 10 - DISTRIBUTED TRANSACTIONS

Q37. Why Are Distributed Transactions Hard?

ROI: ★★★★★ Frequency: Extremely High

Answer

Distributed transactions are hard because the data spans multiple services and databases, each owned independently. In a monolith, a single ACID transaction against one database can atomically update orders, payments and inventory together, and either all of it commits or none of it does. That guarantee disappears once each concern lives in its own service with its own database.

In this flow, three separate services each commit to their own database. There is no shared transaction wrapping them, so you can end up with an order created and payment taken but inventory not reserved. You cannot simply "roll back" the first two because they have already committed in databases the inventory service cannot touch. Solving this requires application-level coordination through patterns like Saga and compensating transactions rather than a single database rollback.

Q38. Why Can't We Use Traditional Transactions?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

A single database transaction cannot reliably span independent services because each service owns a separate database, and a local transaction only controls a single connection to a single database. There is no shared transaction context that the Order, Payment and Inventory databases all participate in, so the usual "begin, commit, rollback" guarantee simply does not extend across service boundaries.

You could in theory use a distributed transaction protocol like two-phase commit, but in modern microservices that is avoided because it requires a coordinator, holds locks across services while waiting, couples their availability together, and scales poorly. The standard answer is to give up on global ACID transactions and instead use eventual consistency with the Saga pattern, where each service commits its own local transaction and failures are undone with compensating actions.

Q39. What Is Two Phase Commit (2PC)?

ROI: ★★★★☆ Frequency: Medium

Answer

Two-Phase Commit is a distributed transaction protocol coordinated by a central transaction manager. In the first phase the coordinator asks all participants to prepare and lock the necessary resources, and each replies whether it can commit. In the second phase, if every participant voted yes, the coordinator tells them all to commit; if any voted no, it tells them all to roll back.

Problem

The problem is that 2PC is slow and complex, and it is not common in modern microservices. Participants must hold locks from the prepare phase until the final commit, which hurts throughput and concurrency. Worse, it is a blocking protocol: if the coordinator crashes after participants have prepared, they are stuck holding locks waiting for a decision, so the coordinator becomes a single point of failure.

Slow. Complex. Not common in modern microservices.

Because of these drawbacks, microservice architectures generally favor the Saga pattern, accepting eventual consistency in exchange for loose coupling and availability instead of trying to force strong, synchronous, cross-service atomicity.

SECTION 11 - SAGA PATTERN

Q40. What Is Saga Pattern?

ROI: ★★★★★ Frequency: Extremely High

One Of The Most Important Senior Questions

The Saga pattern manages distributed transactions using a sequence of local transactions. Instead of one big atomic transaction across services, each service performs its own local transaction and then triggers the next step, either by publishing an event or by being told to by an orchestrator.

The crucial part is failure handling. Because there is no global rollback, if a step fails the saga executes compensating transactions that semantically undo the work already done by earlier steps. This gives you a consistent end state across services without locking them together, at the cost of accepting eventual consistency during the saga's execution.

Q41. Saga Example

ROI: ★★★★★ Frequency: Extremely High 


Create Order

↓

Process Payment

↓

Reserve Inventory

↓

Send Notification

This is a happy-path saga for placing an order. Each step is a local transaction in a different service: the Order service creates the order, the Payment service charges the customer, the Inventory service reserves stock, and the Notification service informs the user. As each step completes, it signals the next, either through an event (choreography) or via a central coordinator (orchestration).

The state moves forward one committed local transaction at a time. The system is only fully consistent once the whole chain completes, which is why the interesting design work is in what happens when one of these steps fails partway through.

Q42. What Happens If Payment Fails?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

If a step like payment fails, the saga executes compensating transactions to undo the work that already succeeded. Since the earlier steps have already committed in their own databases, you cannot perform a database rollback; instead you run new business operations that reverse their effect.

Example


Order Created

↓

Payment Failed

↓

Cancel Order

Here the order was created and committed, but payment failed. The saga responds by issuing a compensating action: Cancel Order, which marks the order as cancelled rather than pretending it never existed. If inventory had already been reserved, a further compensation would release it. The result is a consistent business state reached by deliberately reversing completed steps, which is exactly how sagas substitute for a missing global rollback.

Q43. What Is Compensating Transaction?

ROI: ★★★★★ Frequency: Extremely High

Answer

A compensating transaction is a business-level rollback action: a new operation that semantically undoes the effect of a previously committed step. It is not a database rollback (the original change is already committed) but an explicit "undo" operation defined at the domain level.

Example


Reserve Inventory

↓

Release Inventory

If a step reserved inventory and a later step in the saga fails, the compensating transaction is "Release Inventory," which returns the reserved stock to the available pool. Other examples are refunding a payment to compensate a charge, or cancelling an order to compensate its creation. A subtle but important point is that compensations must themselves be idempotent and should account for the fact that the world may have moved on, which is why designing good compensating transactions is one of the harder parts of saga design.

Q44. Choreography Saga

ROI: ★★★★★ Frequency: High

Answer

In a choreography saga, services communicate through events with no central coordinator. Each service listens for events it cares about, does its local work, and then emits its own event, which the next service reacts to. For example, an OrderCreated event triggers the Payment service, whose PaymentCompleted event triggers the Inventory service, and so on.

This style is highly decoupled and has no single point of control, which fits naturally with event-driven, Kafka-based systems and scales well. The downside is that the overall workflow is implicit, spread across many services' event handlers, which makes it harder to see the end-to-end flow, harder to debug, and harder to reason about compensation when something fails midway.

Q45. Orchestration Saga

ROI: ★★★★★ Frequency: High

Answer

In an orchestration saga, a central orchestrator coordinates the workflow. The orchestrator explicitly tells each service what to do and when ("charge payment," then "reserve inventory"), waits for the result, and decides the next step, including triggering compensating actions if a step fails.

This makes the business process explicit and centralized: the whole flow lives in one place, which is much easier to understand, monitor and modify, and it gives clear control over compensation logic. The trade-off is that the orchestrator becomes a component you must build, scale and keep available, and it introduces some coupling since it knows about all the participating services. Tools like Camunda or Temporal are often used to implement orchestrators.

Q46. Choreography vs Orchestration

ROI: ★★★★★ Frequency: Extremely High

Choreography Orchestration
Event Driven Central Controller
More Decoupled More Visibility
Harder To Trace Easier To Manage

Choreography is event-driven and maximally decoupled: services react to each other's events without anyone being "in charge." That decoupling is its strength and its weakness, because no single place describes the workflow, so tracing and debugging a failed saga means stitching together events across many services.

Orchestration centralizes the workflow in a controller, giving you visibility and straightforward management of the sequence and its compensations, at the cost of some coupling and an extra component to operate. The practical guidance: prefer choreography for simple flows with few steps, and reach for orchestration when the workflow is complex, has many steps, or needs strong visibility and explicit compensation handling.

SECTION 12 - EVENT DRIVEN ARCHITECTURE (EDA)

Modern microservice systems increasingly use events rather than direct synchronous communication. This section is extremely important for senior backend interviews.

Q47. What Is Event Driven Architecture?

ROI: ★★★★★ Frequency: Extremely High

Answer

Event-driven architecture is a style where services communicate through events instead of direct requests. When something significant happens, a service publishes an event describing that fact (for example "OrderCreated"), and any interested service consumes it and reacts. The producer does not know or care who the consumers are.

This inverts the dependency direction compared to synchronous calls. Instead of the Order service calling Payment, Inventory and Notification directly, it simply announces that an order was created, and those services subscribe to that event. The result is loose coupling: you can add a new consumer (say an Analytics service) without touching the producer at all.

Q48. Benefits Of Event Driven Architecture

ROI: ★★★★★ Frequency: Extremely High

  • Loose Coupling
  • Scalability
  • Resilience
  • Independent Services
  • Asynchronous Processing

Because producers do not know their consumers, services are loosely coupled and can evolve independently; adding or changing a consumer does not ripple back to the producer. Asynchronous processing means the producer is not blocked waiting on downstream work, which improves responsiveness and lets each consumer scale to its own load by adding more instances to a consumer group.

Resilience also improves: if a consumer is temporarily down, events accumulate in the broker and are processed once it recovers, instead of the whole flow failing. This buffering and decoupling is what makes event-driven systems handle spikes and partial outages gracefully, which is exactly why large-scale systems lean on them.

Q49. Challenges Of Event Driven Architecture

ROI: ★★★★★ Frequency: Senior Favorite

  • Eventual Consistency
  • Debugging Complexity
  • Ordering Issues
  • Duplicate Events
  • Observability Challenges

The asynchrony that gives EDA its benefits also creates its challenges. The system is only eventually consistent, so reads may briefly see stale data while events propagate. Following a single business flow now means tracing a chain of events across many services, which makes debugging and observability much harder and requires correlation IDs and good tooling.

You also have to handle the realities of messaging: events can arrive out of order, and they can be delivered more than once. That forces consumers to be designed for idempotency and, where order matters, to rely on partition keys to keep related events ordered. Acknowledging these trade-offs, rather than presenting EDA as free, is the senior signal here.

Q50. Eventual Consistency

ROI: ★★★★★ Frequency: Extremely High

Answer

Eventual consistency means the system becomes consistent eventually rather than immediately. After a change, different parts of the system may briefly hold different views of the data, but given a little time and no new updates, they all converge to the same correct state.

SECTION 13 - APACHE KAFKA

Q51. What Is Kafka?

ROI: ★★★★★ Frequency: Extremely High

Answer

Kafka is a distributed event streaming platform. At its core it is a durable, partitioned, replicated commit log: producers append messages to topics, and those messages are persisted on disk and retained for a configured period, so multiple consumers can read them independently and even re-read history.

Unlike a traditional queue that deletes a message once consumed, Kafka keeps messages and tracks each consumer group's position (offset) separately. This log-based design is what makes Kafka both a messaging system and a streaming/replay platform, and it is why it scales to very high throughput.

Uses

  • Messaging
  • Event Streaming
  • Analytics
  • Log Aggregation

These use cases all exploit the same durable log: as a messaging backbone between microservices, as the transport for event-driven architectures and event sourcing, as a pipeline feeding real-time analytics and stream processing, and as a central place to aggregate logs and metrics from many systems.

Q52. Kafka Core Components

ROI: ★★★★★ Frequency: Extremely High

Component Description
Producer Sends Messages
Consumer Reads Messages
Topic Message Category
Broker Kafka Server
Partition Topic Division

A producer publishes messages to a topic, optionally choosing a partition via a key; a consumer subscribes and reads from one or more partitions. A topic is the named category or feed of messages, and it is split into partitions, which are the unit of parallelism and ordering. A broker is a single Kafka server, and a cluster of brokers stores the partitions and replicates them for fault tolerance.

Understanding how these fit together is key: topics give you logical separation, partitions give you scale and ordering, brokers give you durability and availability through replication, and consumer groups (built on top of these) give you horizontal scaling of consumption.

Q53. Kafka Message Flow

ROI: ★★★★★ Frequency: Extremely High 


Producer

↓

Topic

↓

Partition

↓

Consumer

A producer sends a message to a topic. Kafka assigns the message to a specific partition, either by hashing the message key or round-robin if no key is provided, and appends it to that partition's log with a monotonically increasing offset. The message is persisted and replicated across brokers for durability.

Consumers in a consumer group then read from the partitions, each consumer owning a subset of partitions, and they commit their offsets to track progress. Because consumption is offset-based rather than destructive, different consumer groups can read the same topic independently and at their own pace, which is the foundation of Kafka's flexibility.

Q54. What Is A Topic?

ROI: ★★★★★ Frequency: Extremely High

Answer

A topic is a logical channel or category that stores a stream of related messages, similar to a table name or a feed. Producers write to a named topic and consumers subscribe to it, without either side needing to know about the other. Topics are durable and append-only, retaining messages for a configured retention period.

Examples


orders

payments

notifications

users

Naming topics by business event or domain (orders, payments, notifications, users) keeps the system understandable and lets each topic be scaled and consumed independently. Each topic is internally divided into partitions, which is how a single logical channel achieves parallelism and high throughput across a cluster.

Q55. What Is A Partition?

ROI: ★★★★★ Frequency: Extremely High

Answer

A partition is a subdivision of a topic and the actual ordered, append-only log that Kafka stores. A topic with, say, three partitions is physically three separate logs, and a message lands in exactly one of them. Partitions can live on different brokers, which is how Kafka spreads a topic's load across a cluster.

Purpose

  • Parallelism
  • Scalability
  • High Throughput

Partitions are the unit of parallelism: multiple consumers in a group can each read a different partition simultaneously, so you scale consumption by adding partitions and consumers. They are also the unit of ordering, since Kafka only guarantees order within a single partition. This is why the partition count is one of the most important capacity-planning decisions for a topic.

Q56. Why Partitions Matter?

ROI: ★★★★★ Frequency: Senior Favorite

Example


orders-topic

Partition-0

Partition-1

Partition-2

Consumers can process partitions in parallel.

With the orders-topic split into three partitions, a consumer group of three consumers can have each consumer own one partition and process messages fully in parallel, roughly tripling throughput compared to a single log. The number of partitions sets the ceiling on parallelism: you can never have more actively consuming instances in a group than partitions, so extra consumers would sit idle.

The trade-off interviewers probe is ordering. Kafka guarantees order only within a partition, so if all order events for the same customer must be processed in sequence, you must route them to the same partition using a consistent key (such as customerId). More partitions mean more parallelism but weaker global ordering, which is the central tuning decision in Kafka design.

Q57. What Is Consumer Group?

ROI: ★★★★★ Frequency: Extremely High

Answer

A consumer group is a set of consumers that cooperate to share the workload of a topic. Kafka assigns each partition to exactly one consumer within the group, so the partitions are divided among the members and the messages are processed in parallel without duplication inside that group.

Different consumer groups are independent: each maintains its own offsets, so the same topic can be consumed by multiple groups (for example one for billing and one for analytics) that each see the full stream. Within a single group, adding consumers up to the partition count scales throughput, and Kafka automatically rebalances partitions when consumers join or leave, which is how it provides both scalability and fault tolerance.

Q58. Consumer Group Example

ROI: ★★★★★ Frequency: High

Each consumer handles partitions.

With three partitions and three consumers in the group, Kafka assigns one partition to each consumer, giving a clean one-to-one mapping and maximum parallelism. If a fourth consumer joined this group, it would have no partition to read and would sit idle, since a partition can only be owned by one consumer in the group at a time.

If one consumer crashes, Kafka rebalances and reassigns its partition to a surviving consumer, so processing continues automatically. This combination of static assignment and automatic rebalancing is what lets consumer groups scale out and self-heal, and it is why partition count should generally be set with the expected consumer parallelism in mind.

Q59. Ordering In Kafka

ROI: ★★★★★ Frequency: Extremely High

Answer

Kafka guarantees ordering only within a partition, not across an entire topic. Messages within one partition are strictly ordered by offset and delivered in that order, but if a topic has multiple partitions, there is no global ordering guarantee across them because they are consumed independently and in parallel.

This is a direct consequence of the partition-based parallelism that gives Kafka its throughput: you cannot have both unlimited parallelism and total global ordering. The practical implication is that any time order matters for a set of related messages, you must ensure they land in the same partition, which leads directly to the use of partition keys.

Q60. How To Maintain Ordering?

ROI: ★★★★★ Frequency: High

Answer

To maintain ordering for related messages, use a consistent partition key so that all messages with the same key are routed to the same partition. Kafka hashes the key to choose the partition, so identical keys always map to the same partition, and within that partition order is guaranteed.

Example


orderId

Using orderId as the key ensures every event for a given order (created, paid, shipped) goes to the same partition and is therefore processed in the correct sequence, even though events for different orders are spread across partitions and processed in parallel. This gives you per-key ordering without sacrificing overall throughput, which is the standard way to reconcile Kafka's parallelism with business sequencing requirements.

SECTION 14 - DELIVERY GUARANTEES

Q61. At Most Once Delivery

ROI: ★★★★★ Frequency: High

Answer

At-most-once delivery means a message may be lost but is never duplicated. The consumer commits its offset before (or without guaranteeing) processing, so if it crashes mid-processing, the message is skipped on restart because its offset was already advanced. You never get duplicates, but you can lose messages.

This is the weakest guarantee and is only acceptable when occasional loss is tolerable, such as high-volume metrics or logs where a missing data point does not matter. It is rarely used for business-critical data like orders or payments, where losing a message is unacceptable.

Q62. At Least Once Delivery

ROI: ★★★★★ Frequency: Extremely High

Answer

At-least-once delivery means a message may be delivered multiple times but is never intentionally lost. The consumer processes the message first and only then commits its offset, so if it crashes after processing but before committing, the message is redelivered and processed again on restart. Nothing is lost, but duplicates are possible.

Most Common

Real production systems often use this.

At-least-once is the default and most common guarantee in real production systems because it is reliable and far cheaper than exactly-once. The catch is that it shifts responsibility to the consumer: since duplicates can happen, consumers must be idempotent so that processing the same message twice has the same effect as processing it once. This is precisely why idempotency is such a heavily tested microservices topic.

Q63. Exactly Once Delivery

ROI: ★★★★★ Frequency: High

Answer

Exactly-once delivery means a message is processed exactly one time, with no loss and no duplicates. Kafka supports exactly-once semantics within its own boundaries through idempotent producers and transactions that atomically write messages and commit offsets together.

Reality

Complex and expensive.

In practice, true end-to-end exactly-once is complex and expensive, and it only holds cleanly when everything stays inside the transactional system. The moment a consumer has a side effect on an external system (calling a payment API, writing to a different database), you are back to needing idempotency to guard against retries. The pragmatic senior answer is: aim for at-least-once delivery plus idempotent consumers, which achieves "effectively exactly once" with far less cost and complexity than chasing strict exactly-once everywhere.

SECTION 15 - IDEMPOTENCY

Q64. What Is Idempotency?

ROI: ★★★★★ Frequency: Extremely High

Answer

Idempotency means that performing the same operation repeatedly produces the same result as performing it once. Whether a request is processed one time or five times, the end state of the system is identical and there are no duplicate side effects.

In HTTP terms, GET, PUT and DELETE are naturally idempotent while POST usually is not. In distributed systems, idempotency is something you deliberately engineer into write operations, because the environment guarantees that retries and duplicate deliveries will happen. It is the property that lets you retry safely.

Q65. Why Idempotency Matters?

ROI: ★★★★★ Frequency: Extremely High

Retries and duplicate messages happen in distributed systems.

Because networks are unreliable, retries and duplicate messages are unavoidable. A client may not receive a response and retry, even though the original request actually succeeded; a Kafka consumer under at-least-once semantics may reprocess a message after a crash; a load balancer may resend a request. In every case the same operation can hit your service more than once.

Without idempotency, these duplicates cause real damage: a customer charged twice, an order created twice, inventory decremented multiple times. Idempotency is the safety net that makes retries and at-least-once messaging usable, which is why it underpins almost every other resilience pattern (retries, sagas, the outbox pattern) in microservices.

Q66. Example Of Idempotency

ROI: ★★★★★ Frequency: High

Bad


Payment Processed Twice

Good


Unique Transaction ID

↓

Duplicate Ignored

The bad case is a payment endpoint that simply charges the card on every request. If the client retries after a timeout, the customer is charged twice, even though only one payment was intended. This is the textbook failure that idempotency exists to prevent.

The good case attaches a unique transaction or idempotency ID to the request. The service records that ID when it first processes the payment; if a request with the same ID arrives again, it recognizes the duplicate and returns the original result without charging again. The same idea applies to Kafka consumers, which can track processed event IDs to skip reprocessing. The fix is always: make the duplicate detectable, then ignore it.

Q67. How To Implement Idempotency?

ROI: ★★★★★ Frequency: Senior Favorite

  • Idempotency Keys
  • Unique Constraints
  • Processed Event Tables
  • Deduplication Logic

The most common approach is an idempotency key: the client generates a unique key per logical operation and sends it with the request; the server stores keys it has handled and, on seeing a repeat, returns the stored result instead of redoing the work. A complementary technique is a database unique constraint (for example on transaction ID), which lets the database itself reject duplicate inserts atomically.

For event consumers, a processed-event table records the IDs of events already handled, and the consumer checks it before processing so it can skip duplicates; this is essentially deduplication logic backed by durable storage. The key design point an interviewer wants is that the dedup check and the work should be atomic (ideally in the same transaction), otherwise a crash between them reopens the duplicate window.

SECTION 16 - OBSERVABILITY

Q68. What Is Observability?

ROI: ★★★★★ Frequency: Extremely High

Answer

Observability is the ability to understand a system's internal state from its external outputs, primarily its logs, metrics and traces, without having to add new instrumentation each time something goes wrong. A system is observable if you can answer new questions about why it is behaving a certain way using the data it already emits.

It differs subtly from monitoring. Monitoring tells you whether known problems are happening ("is CPU high, is the error rate up"); observability lets you explore unknown problems ("why are checkout requests suddenly slow for one region"). In microservices, where a single request crosses many services, observability is essential because no one service has the full picture on its own.

Q69. Three Pillars Of Observability

ROI: ★★★★★ Frequency: Extremely High

  • Logs
  • Metrics
  • Traces

Logs are discrete, timestamped records of events, ideal for understanding exactly what happened in a specific request or error. Metrics are aggregated numeric measurements over time (request rate, latency percentiles, error counts) that are cheap to store and perfect for dashboards and alerting on trends. Traces follow a single request as it flows across multiple services, showing where time was spent and where it failed.

The three are complementary: metrics tell you something is wrong, traces tell you which service in the chain is responsible, and logs tell you the exact detail of what went wrong there. Used together with correlation IDs that tie them, they let you go from "latency spiked" to "this specific downstream call failed" quickly, which is exactly what you need in a distributed system.

Q70. What Are Metrics?

ROI: ★★★★★ Frequency: High

Examples

  • CPU Usage
  • Memory Usage
  • Request Count
  • Latency

Metrics are numeric measurements captured over time and stored as time series, which makes them compact and cheap to retain at scale. They split roughly into system metrics (CPU, memory, disk, network) that reveal resource pressure, and application metrics (request count, error rate, latency) that reveal user-facing health.

Because they aggregate well, metrics are the foundation of dashboards and alerting. A team typically watches latency percentiles (p95, p99 rather than averages, since averages hide tail problems) and error rates, and configures alerts so that a breach pages the on-call engineer before users feel the impact. Tools like Prometheus scrape and store these, and Grafana visualizes them.

SECTION 17 - CAP THEOREM

This is one of the most frequently asked distributed systems theory questions. Senior interviewers often use it to determine whether a candidate understands the realities of distributed systems.

Q74. What Is CAP Theorem?

ROI: ★★★★★ Frequency: Extremely High

Answer

The CAP theorem states that a distributed system can guarantee at most two of three properties: Consistency, Availability and Partition tolerance.

  • Consistency (C)
  • Availability (A)
  • Partition Tolerance (P)

The common misconception is that you freely pick any two. In reality, network partitions can and will happen in any distributed system, so partition tolerance is not optional. That means the real choice, and the only one that occurs at runtime, is between consistency and availability during a partition. Stating it that way is what separates a textbook answer from an understanding answer.

Q75. What Is Consistency?

ROI: ★★★★★ Frequency: High

Answer

In CAP terms, consistency means every node sees the same data at the same time: after a write completes, any subsequent read from any node returns that latest value. There is a single, up-to-date view of the data regardless of which replica you talk to.

Note this is a stricter, more specific notion than the "C" in ACID, which is about preserving database invariants within a transaction. CAP consistency is really about linearizability across replicas. Choosing consistency means that if a node cannot be sure it has the latest data (for instance during a partition), it would rather refuse to answer than return possibly stale data.

Q76. What Is Availability?

ROI: ★★★★★ Frequency: High

Answer

Availability means the system always responds to requests, returning a non-error response in a reasonable time, even if that response is not guaranteed to reflect the most recent write. Every working node remains able to serve reads and writes.

Choosing availability means that during a network partition, a node keeps serving clients using whatever data it has, accepting that it might be slightly stale and reconcile later. This is the right choice for systems where being responsive matters more than being perfectly current, such as a shopping cart, a social feed, or a product catalog, where a brief staleness is far better than an error page.

Q77. What Is Partition Tolerance?

ROI: ★★★★★ Frequency: Extremely High

Answer

Partition tolerance means the system continues functioning despite network failures that split it into groups of nodes that cannot communicate with each other. A "partition" is exactly that kind of communication breakdown, where messages between some nodes are dropped or delayed indefinitely.

A partition-tolerant system does not simply collapse when the network splits; it keeps operating in some defined way on each side of the split. Since you cannot prevent network failures in a real distributed system spread across machines, racks and data centers, partition tolerance is the property you must always design for, which is why CAP effectively reduces to a C-versus-A choice.

Q78. Why Is Partition Tolerance Mandatory?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

Partition tolerance is mandatory because network failures are unavoidable in distributed systems. Nodes communicate over networks, and networks drop packets, suffer latency spikes, lose links and have switches fail. Over a long enough time, a partition is not a possibility but a certainty.

Real systems must tolerate partitions.

Because of this, you do not get to "choose" to drop P. A system that abandons partition tolerance would mean choosing C and A together, which only works if the network never fails, an assumption that does not hold in practice. So real distributed systems always keep P, and the genuine engineering decision is what to do during a partition: favor consistency (CP) and reject some requests, or favor availability (AP) and serve possibly stale data. Framing CAP as "P is fixed, choose C or A" is the senior-level answer.

Q79. CA vs CP vs AP Systems

ROI: ★★★★★ Frequency: High

Type Behavior
CA Not Practical Distributed System
CP Consistency Preferred
AP Availability Preferred

CA (consistency plus availability without partition tolerance) is essentially only achievable in a single-node or non-distributed setup, because the moment the network can partition, you must sacrifice one of C or A. That is why CA is labeled as not a practical distributed system.

CP systems prefer consistency during a partition: they refuse or block requests on the side that cannot guarantee up-to-date data, accepting reduced availability to never return stale data. Strongly consistent stores and coordination systems (for example traditional RDBMS clusters or ZooKeeper) lean this way. AP systems prefer availability: they keep serving on both sides of a partition with possibly stale data and reconcile afterward; many NoSQL stores like Cassandra and DynamoDB are tuned this way. The right choice is per use case, payments lean CP, a product catalog or feed leans AP.

Q80. Real Interview Answer For CAP

ROI: ★★★★★ Frequency: Senior Favorite

Answer

The crisp interview answer is: when a network partition occurs, a distributed system must choose between consistency and availability, because partition tolerance is not optional.

  • Consistency
  • Availability

Partition tolerance is not optional.

A strong candidate then grounds it in a concrete trade-off: "for a payment or banking flow I would choose consistency (CP) and reject the request rather than risk a double-spend, whereas for a product catalog or social feed I would choose availability (AP) and serve slightly stale data rather than show an error." Tying CAP to a real decision, and mentioning that during normal operation, with no partition, you can have both C and A, shows you understand the theorem operationally rather than as a memorized triangle.

SECTION 18 - DATABASE PER SERVICE

Q81. Why Database Per Service?

ROI: ★★★★★ Frequency: Extremely High

Answer

The database-per-service pattern means each service owns its data independently, with its own private database that no other service may access directly. Other services can only reach that data through the owning service's API or the events it publishes, never by querying its tables.

Benefits

  • Loose Coupling
  • Independent Deployment
  • Independent Scaling

This data ownership is what makes microservices truly independent. Because no one else depends on a service's schema, the team can change its tables, migrate, or even switch database technology without coordinating with anyone, which preserves loose coupling and independent deployment. Each service can also pick and scale the storage that fits its workload (a relational DB here, a document store there).

The cost is that data is now distributed, so cross-service queries and transactions are no longer trivial; you handle them with API composition, events and patterns like Saga and CQRS. But that cost is precisely the price of independence, and giving each service its own database is widely considered foundational to doing microservices properly.

Q82. Why Shared Database Is Dangerous?

ROI: ★★★★★ Frequency: Senior Favorite

Problems

  • Tight Coupling
  • Schema Dependencies
  • Deployment Coordination
  • Ownership Confusion

A shared database is dangerous because it secretly couples services together through the schema. When multiple services read and write the same tables, a schema change to satisfy one service can break the others, so you can no longer change anything safely without checking every consumer. This is the hidden coupling that quietly turns a microservice system into a distributed monolith.

It also forces deployment coordination, since a migration may require several services to be released together, defeating the independence that justified microservices in the first place. And it blurs ownership: when everyone can write to a table, no one is truly responsible for its integrity, validation or invariants. The senior point is that a shared database undermines the core promise of microservices, which is why database-per-service is preferred despite the extra complexity.

Q83. How Do Services Access External Data?

ROI: ★★★★★ Frequency: Extremely High

Approaches

  • API Calls
  • Events
  • CQRS Read Models

Since a service cannot touch another service's database, it gets the data it needs through that service's contract. The simplest way is a synchronous API call to ask the owning service for the data on demand, which is easy but adds coupling and latency and depends on that service being up.

The more scalable, decoupled approach is events: the owning service publishes changes, and interested services subscribe and maintain their own local copy of just the fields they need. A CQRS read model formalizes this by building a query-optimized view (often denormalized and pre-joined) from those events, so a service can answer its queries locally and fast without calling out at request time. The trade-off is eventual consistency, since the local copy lags the source slightly, which is usually acceptable and is the standard pattern for cross-service data access.

SECTION 19 - CQRS

Q84. What Is CQRS?

ROI: ★★★★★ Frequency: High

Full Form

Command Query Responsibility Segregation

Answer

CQRS separates write operations (commands, which change state) from read operations (queries, which return state), using different models and often different data stores for each. Instead of one model serving both reads and writes, you have a write model optimized for correctness and consistency and a read model optimized for fast queries.

The motivation is that reads and writes usually have very different requirements: writes need validation and strong consistency, while reads often need denormalized, query-shaped data and far higher volume. By splitting them, each side can be modeled, scaled and optimized independently, which is hard to do when a single normalized schema has to serve both.

Q85. Why CQRS?

ROI: ★★★★★ Frequency: High

Benefits

  • Independent Scaling
  • Optimized Reads
  • Optimized Writes
  • Complex Domain Support

Most systems read far more than they write, so being able to scale the read side independently (more read replicas, caches, or read services) without touching the write side is a major benefit. The read model can be denormalized and pre-joined so queries are simple and fast, while the write model stays clean and focused on enforcing business rules and invariants.

CQRS also shines in complex domains where the natural way to express business operations (commands) is very different from the many ways the data needs to be viewed (queries). It pairs naturally with event-driven systems and event sourcing, where commands produce events that then update one or more read models. The interviewer wants to hear that you would apply it selectively, where the read/write divergence genuinely justifies it.

ROI: ★★★★★ Frequency: High

In this flow, commands update the write database, which is the source of truth. Each change emits an event, and a separate process consumes those events to update a read database that is shaped specifically for the queries the application needs (denormalized, indexed for fast lookups).

Queries then hit the read database directly and never touch the write side, so heavy read traffic does not contend with writes. The important consequence is that the read store is eventually consistent with the write store: there is a small propagation delay between a write committing and the read model reflecting it, which the system and users must be able to tolerate.

Q87. Drawback Of CQRS

ROI: ★★★★★ Frequency: Senior Favorite

Answer

The main drawback of CQRS is additional complexity. You now maintain two models and often two data stores, plus the event pipeline that keeps the read side synchronized with the write side. That is more moving parts to build, deploy, monitor and reason about than a single model serving both.

Should not be used everywhere.

It also introduces eventual consistency between the write and read sides, so the application and the user experience must tolerate reads that are briefly stale. Because of this overhead, CQRS should not be used everywhere; it is justified only in specific areas with high read/write asymmetry or genuinely complex domains. Applying it blindly to a simple CRUD service adds cost with no benefit, and recognizing when not to use it is itself a senior signal.

SECTION 20 - EVENT SOURCING

Q88. What Is Event Sourcing?

ROI: ★★★★★ Frequency: High

Answer

Event sourcing means storing events instead of current state. Rather than keeping a single row that holds an entity's latest values and overwriting it on each change, you persist an append-only log of every state-changing event that ever happened, and you derive the current state by replaying those events.

For example, instead of storing "account balance = 100," you store the sequence of deposits and withdrawals, and the balance is computed by folding over them. The event log becomes the single source of truth, which means you never lose information about how the entity reached its current state, a property the traditional "update in place" model throws away.

Q89. Traditional Storage vs Event Sourcing

ROI: ★★★★★ Frequency: High

Traditional Event Sourcing
Current State All Events
Update Rows Append Events

Traditional storage keeps only the current state and mutates rows in place, so each update destroys the previous value; you know what the data is now, but not how it got there. Event sourcing instead keeps the full history as an immutable, append-only stream of events, and current state is a projection computed from that stream.

The "append events" model has powerful consequences: because nothing is ever overwritten, you have a perfect audit trail, you can reconstruct the state of any entity at any past point in time, and you can build entirely new read models later by replaying the existing events. The cost is that reading current state requires replaying (mitigated by periodic snapshots), and storage grows continuously, which is the trade-off the next questions explore.

Q90. Event Sourcing Benefits

ROI: ★★★★★ Frequency: High

  • Auditability
  • History Tracking
  • Replay Capability
  • Debugging

Because every change is stored as an immutable event, you get auditability and history tracking for free: you can always answer not just what the current state is but exactly how and when it changed, which is invaluable in regulated domains like finance. Nothing is lost to an overwrite.

Replay capability is the other big win. You can rebuild current state, or construct a brand-new read model, simply by reprocessing the event log, which is great for fixing a bug in a projection or adding a new view of the data after the fact. For debugging, you can replay the exact sequence of events that led to a problem and reproduce it deterministically, which is far more powerful than inspecting only the final corrupted state.

Q91. Event Sourcing Challenges

ROI: ★★★★★ Frequency: Senior Favorite

  • Complexity
  • Storage Growth
  • Replay Cost
  • Schema Evolution

Event sourcing adds significant complexity: it is a different mental model from CRUD, querying current state requires building and maintaining projections, and developers must think in terms of events and replays. Storage also grows continuously because events are never deleted, and rebuilding state by replaying a long history can be slow, which is why snapshots are used to checkpoint state periodically.

The subtlest challenge is schema evolution. Events are stored forever, so old events written with an old structure must still be readable years later; you cannot just migrate them away as you would mutate a table. This forces careful event versioning and upcasting strategies. Because of all this, event sourcing is reserved for domains where the audit trail and history genuinely add value (finance, ledgers, complex workflows), not applied as a default, and saying so signals maturity.

SECTION 21 - OUTBOX PATTERN

Q92. What Is The Dual Write Problem?

ROI: ★★★★★ Frequency: Extremely High

Example


Save Order

↓

Publish Event

What if database succeeds but event publication fails?

The dual-write problem is the danger of writing to two different systems (a database and a message broker) in one operation without a shared transaction. In the example, the service saves the order to its database and then publishes an OrderCreated event to Kafka. These are two separate systems, so there is no single transaction that makes both succeed or both fail atomically.

That gap creates inconsistency. If the database commit succeeds but the Kafka publish fails (or the service crashes in between), the order exists but no event was sent, so downstream services never react, payment and inventory never happen, and the systems drift out of sync. The reverse can also occur: the event is published but the database commit rolls back, so consumers act on an order that does not exist. Naive "save then publish" code is therefore fundamentally unreliable, which is exactly what the outbox pattern fixes.

Q93. What Is Outbox Pattern?

ROI: ★★★★★ Frequency: Extremely High

Answer

The outbox pattern solves the dual-write problem by storing the event in a database table within the same transaction as the business data. Instead of writing to the database and then separately publishing to Kafka, the service writes both the order and an "outbox" row in one atomic local transaction, so either both are persisted or neither is.

Separate process publishes event later.

A separate process then reads unpublished rows from the outbox table and publishes them to the message broker, marking them as sent once acknowledged. Because the event is captured atomically with the business change, it can never be lost; the worst case is that it is published a little later or more than once. This converts an unreliable dual write into a single reliable local write plus an at-least-once publish, which is why the outbox pattern is the standard production answer for reliable event publishing.

ROI: ★★★★★ Frequency: High

The flow shows the atomic part clearly: creating the order and inserting the outbox record happen before a single commit, so they are saved together in one local database transaction. At this point the event is durably recorded even though it has not yet been sent anywhere.

After the commit, a separate publisher (a polling job, or a change-data-capture tool like Debezium reading the database log) picks up the new outbox rows and publishes them to Kafka, then marks them processed. Since publishing is decoupled and retried until it succeeds, a crash anywhere is safe: on restart the unpublished rows are still there and get sent. Because the same event could be published twice (for example if the service crashes after publishing but before marking it sent), consumers must be idempotent, which ties the outbox pattern back to idempotency.

Q95. Why Outbox Pattern?

ROI: ★★★★★ Frequency: Senior Favorite

Avoids data inconsistency caused by dual writes.

The outbox pattern exists to avoid the data inconsistency caused by dual writes: the scenario where the database and the message broker can disagree because they are updated in separate, non-atomic steps. By recording the event in the same transaction as the data, it guarantees that an event exists for every committed business change and that no event exists for a change that rolled back.

The senior framing is that it trades a hard problem (atomic write across two systems, which would otherwise need fragile distributed transactions) for an easy, well-understood one (a single local transaction plus at-least-once delivery with idempotent consumers). This is why it is the go-to solution for reliably publishing events in an event-driven microservice system, and it pairs naturally with the saga, CQRS and idempotency patterns discussed earlier.

Q96. Design Order Processing System

ROI: ★★★★★ Frequency: Senior Favorite

Services

  • Order Service
  • Payment Service

  • Notification Service

Patterns

  • Saga
  • Kafka
  • Outbox

This question ties the whole handbook together. The system is decomposed into services along business capabilities (Order, Payment, Inventory, Notification), each with its own database. The Order service creates an order and, using the outbox pattern, atomically records an OrderCreated event, which is published to Kafka so the flow is reliable from the very first step.

A saga then coordinates the cross-service workflow: payment is charged, inventory reserved, and the customer notified, with compensating transactions (cancel order, release inventory, refund payment) if any step fails, since there is no global transaction across the services. The strong answer also layers in resilience (circuit breakers, timeouts, retries with backoff) and observability (distributed tracing, metrics), showing you can assemble the individual patterns into one coherent, production-grade design.

Q97. API Gateway Goes Down. What Happens?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

If the API Gateway goes down, the entire entry point becomes unavailable: since all client traffic flows through the gateway, a single failed gateway means clients can no longer reach any backend service, even though those services are perfectly healthy. This is the classic single-point-of-failure risk of the gateway pattern.

Solution

  • Multiple Gateway Instances
  • Load Balancer
  • High Availability

The fix is to never run a single gateway. You deploy multiple stateless gateway instances behind a load balancer (or across multiple availability zones), so if one instance fails, the load balancer simply routes traffic to the others and clients never notice. Keeping the gateway stateless is what makes this horizontal scaling and failover possible.

A complete answer goes further: health checks remove dead instances automatically, the deployment spans multiple zones for resilience against a whole-zone outage, and clients use retries with backoff so a transient blip is absorbed. The interviewer is really testing whether you recognize and design away single points of failure, which is a core senior responsibility.

Q98. Kafka Message Processed Twice. What Do You Do?

ROI: ★★★★★ Frequency: Extremely High

Expected Answer

  • Idempotency Keys
  • Deduplication
  • Unique Constraints

The first thing to recognize is that this is normal, not a bug: Kafka's default at-least-once delivery means a consumer that crashes after processing but before committing its offset will reprocess the message on restart. So the correct response is not to try to make Kafka deliver exactly once everywhere, but to make the consumer safe to process the same message twice, that is, idempotent.

Concretely, you attach a unique identifier to each message and track which IDs have already been handled. A processed-event table lets the consumer check-and-skip duplicates, and a database unique constraint on the business key (for example transaction ID) lets the database reject a duplicate insert atomically. The key detail an interviewer listens for is that the dedup check and the work should commit together, otherwise a crash between them reopens the duplicate window. This is the same idempotency thinking that underpins retries, the outbox pattern and at-least-once messaging throughout the system.
