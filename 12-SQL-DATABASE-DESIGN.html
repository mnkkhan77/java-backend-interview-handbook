SECTION 1 - SQL FUNDAMENTALS

Q1. What Is A Primary Key?

ROI: ★★★★★ Frequency: Extremely High

Answer

A primary key uniquely identifies each row in a table. It is the canonical handle the database uses to locate, relate and enforce identity on a record, so no two rows can ever share the same primary key value.

In practice the primary key is far more than a label. Most relational engines automatically build a clustered index (or its equivalent) on the primary key, which physically organises the data and makes lookups by key extremely fast. Foreign keys in other tables point back to this key, so it becomes the backbone of every relationship in the schema.

For senior-level work the recommendation is almost always a surrogate primary key, such as an auto-increment BIGINT or a UUID, rather than a business value like email. Business values can change, but a primary key should be stable and meaningless so that the entire relational graph never has to be rewritten when business data evolves.

Properties

  • Unique
  • Not Null
  • One Per Table
Q2. Primary Key vs Unique Key

ROI: ★★★★★ Frequency: Extremely High

Primary Key Unique Key
One Per Table Multiple Allowed
Not Null Can Allow Null
Main Identifier Additional Uniqueness

Both a primary key and a unique key guarantee uniqueness and are usually backed by an index, so people often confuse them. The real distinction is one of role and constraints rather than mechanism. A table may have exactly one primary key, the single authoritative identity of the row, but it can have many unique keys representing additional business rules.

The NULL behaviour is the most common interview gotcha. A primary key column can never be NULL because identity must always exist, whereas a unique key may allow one (or, depending on the engine, several) NULL values since NULL is treated as "unknown" and is not considered equal to another NULL.

A typical design uses an auto-generated id as the primary key and adds a unique constraint on a natural business field such as email or username. This keeps relationships stable while still enforcing that no two users register with the same email.

Q3. What Is Foreign Key?

ROI: ★★★★★ Frequency: Extremely High

Answer

A foreign key maintains the relationship between two tables by referencing the primary key (or a unique key) of another table. It is the mechanism that turns a collection of independent tables into a connected relational model.

Its main job is referential integrity. The database will reject any attempt to insert a child row pointing to a parent that does not exist, and it can control what happens when the parent is deleted or updated through ON DELETE and ON UPDATE actions such as CASCADE, SET NULL or RESTRICT. This pushes data-consistency rules down into the database rather than relying on application code to enforce them.

A practical senior note: foreign key columns are frequently used in joins and WHERE filters, so they should usually be indexed. Many databases do not automatically index the child side of a foreign key, and a missing index here is a classic cause of slow joins and lock contention in production.

Q4. What Is SQL Join?

ROI: ★★★★★ Frequency: Extremely High

A join combines rows from two or more tables based on a related column between them, typically a foreign key matching a primary key. Because normalized schemas deliberately spread data across many tables to avoid redundancy, joins are how we reassemble that data into a meaningful result.

The type of join controls which rows survive when there is no match: an inner join keeps only matching rows, while outer joins preserve unmatched rows from one or both sides and fill the gaps with NULLs. Choosing the correct join type is a correctness decision, not just a performance one.

From a performance standpoint, the join condition columns should be indexed, and the volume of data on each side strongly influences how the optimizer executes the join (nested loop, hash join or merge join). Poorly written joins on unindexed columns are one of the most common reasons backend APIs slow down under load.

SECTION 2 - SQL JOINS

Q5. INNER JOIN

ROI: ★★★★★ Frequency: Extremely High

Answer

An INNER JOIN returns only the rows where the join condition is satisfied on both tables. If a row in either table has no matching counterpart, it is excluded from the result entirely.

This is the default and most common join in production code. For example, joining users to orders with an inner join returns only users who have actually placed orders; users with no orders simply disappear from the result. That filtering behaviour is exactly why inner joins are sometimes used as an implicit existence filter.

Q6. LEFT JOIN

ROI: ★★★★★ Frequency: Extremely High

Answer

A LEFT JOIN (left outer join) returns all rows from the left table, plus the matching rows from the right table when they exist. Where there is no match on the right side, the right-hand columns are filled with NULL.

This is the join you reach for when the left table is your primary entity and the right table is optional supplementary data. For instance, listing all users together with their latest order, including users who have never ordered, requires a left join so those users are not dropped.

A common senior trap: if you place a condition on the right table in the WHERE clause instead of in the ON clause, you silently convert a left join back into an inner join, because rows with NULL right-side columns get filtered out. Conditions on the optional side belong in the ON clause.

Q7. RIGHT JOIN

ROI: ★★★★☆ Frequency: Medium

Answer

A RIGHT JOIN returns all rows from the right table and the matching rows from the left table, filling unmatched left-side columns with NULL. It is simply the mirror image of a LEFT JOIN.

In real codebases right joins are rare because most engineers find left joins more readable: you can always rewrite a right join as a left join by swapping the table order. Keeping the "main" table on the left and using LEFT JOIN consistently tends to make queries easier to reason about and review.

Q8. FULL OUTER JOIN

ROI: ★★★★☆ Frequency: Medium

Answer

A FULL OUTER JOIN returns all rows from both tables. Where rows match on the join condition they are combined; where there is no match on one side, the missing columns are filled with NULL. It is effectively the union of a left join and a right join.

It is most useful for reconciliation scenarios, such as comparing two datasets to find records that exist on one side but not the other. Note that MySQL historically does not support FULL OUTER JOIN directly, so it is emulated with a UNION of a left join and a right join, whereas PostgreSQL, Oracle and SQL Server support it natively.

Q9. Self Join

ROI: ★★★★★ Frequency: High

Example

A self join is a join in which a table is joined to itself, using table aliases to treat the single table as if it were two. It is the standard technique for querying hierarchical or recursive relationships stored within one table.

The classic example is an employees table where each row has a manager_id that points back to the id of another employee. To list every employee alongside their manager's name, you join the employees table to itself, aliasing one copy as the employee and the other as the manager. The same pattern applies to category trees, organisational hierarchies and comment threads. 


Employee ↔ Manager relationship.
Q10. Most Common Join In Production?

ROI: ★★★★★ Frequency: High

Answer

In day-to-day backend development, INNER JOIN and LEFT JOIN dominate. Inner joins are used whenever you only want records that have a corresponding relationship, and left joins are used whenever the related data is optional but you still need the primary rows.

Right joins and full outer joins appear far less often because they can always be rephrased as left joins or handled with set operations, and reviewers generally prefer the more conventional forms. Being able to justify why you chose inner versus left in a given query signals that you understand the correctness implications, not just the syntax.

SECTION 3 - INDEXES

Q11. What Is An Index?

ROI: ★★★★★ Frequency: Extremely High

Answer

An index is a separate data structure, most commonly a B-tree, that the database maintains to locate rows quickly without scanning the entire table. Instead of reading every row to find matches, the engine can navigate the index in logarithmic time to jump straight to the relevant rows.

Indexes are the single most important lever for query performance. They accelerate WHERE filters, JOIN conditions, ORDER BY sorts and uniqueness checks. The trade-off is that every index must be kept in sync on writes, so they speed up reads at the cost of slightly slower inserts, updates and deletes plus additional storage.

Analogy

Think of the index at the back of a book. Rather than reading every page to find a topic, you look it up in the alphabetical index and jump directly to the right page. A database index works the same way, mapping key values to the physical location of the rows that contain them.

Q12. Why Use Indexes?

ROI: ★★★★★ Frequency: Extremely High

Indexes exist to turn expensive full table scans into targeted lookups. On a large table the difference between scanning millions of rows and seeking a handful through an index is often the difference between a query taking seconds and taking milliseconds.

Beyond raw searching, a well-chosen index can also satisfy sorting and grouping directly, because B-tree indexes store keys in sorted order. A covering index that contains every column a query needs can even answer the query entirely from the index without touching the table at all.

  • Faster Searches
  • Faster Sorting
  • Faster Filtering
Q13. Index Drawbacks

ROI: ★★★★★ Frequency: High

Indexes are not free. Each index consumes additional disk space, and more importantly it must be updated whenever the underlying data changes. Every insert, update or delete that touches an indexed column forces the engine to maintain the index structure as well, which slows down write-heavy workloads.

This is why over-indexing is a real production problem. A table with too many indexes can have excellent read performance but unacceptably slow writes, and the optimizer may also spend more time choosing between them. The senior approach is to index deliberately, based on the actual query patterns, and to periodically drop unused indexes.

  • Extra Storage
  • Slower Inserts
  • Slower Updates
  • Slower Deletes
Q14. Clustered Index

ROI: ★★★★★ Frequency: Extremely High

Answer

A clustered index determines the physical order in which the rows are actually stored on disk. Because the table data itself is arranged according to the clustered key, the index and the data are effectively one and the same structure.

This makes range queries and lookups on the clustered key extremely efficient, since the matching rows are stored physically adjacent to each other. In SQL Server and MySQL's InnoDB engine the primary key is the clustered index by default, which is one reason a sequential, ever-increasing primary key tends to perform better than a random one for inserts.

Important

There can be only one clustered index per table, because the data can only be physically sorted in one way. Choosing the right column for it is a foundational design decision.

Q15. Non-Clustered Index

ROI: ★★★★★ Frequency: Extremely High

Answer

A non-clustered index is a separate structure that stores the indexed key values along with pointers (row locators) back to the actual data rows. The table data itself stays in its existing physical order; only the index is sorted by the indexed columns.

Because it is independent of the physical row order, a table can have many non-clustered indexes, each supporting a different set of queries. The trade-off is the extra lookup step: after finding a match in the index, the engine often has to follow the pointer back to the base table to fetch the remaining columns, unless the index already covers all the columns the query needs.

Q16. Clustered vs Non-Clustered

ROI: ★★★★★ Frequency: Extremely High

The core difference is that a clustered index is the table data sorted in key order, whereas a non-clustered index is a smaller side structure that points back to the data. You can only have one clustered index because the data has a single physical ordering, but you can create many non-clustered indexes for different access paths.

Performance-wise, a lookup by the clustered key is the fastest possible because the row is found in place, while a non-clustered lookup may incur an additional step to retrieve non-indexed columns. A senior tip worth stating in interviews is that in InnoDB every non-clustered (secondary) index entry stores the primary key value as its pointer, so a bloated primary key inflates the size of every secondary index on the table.

Q17. Composite Index

ROI: ★★★★★ Frequency: Extremely High

Answer

A composite index is a single index built on two or more columns together. It is sorted first by the leading column, then by the next column within each value of the first, and so on, much like sorting a phone book by last name and then first name.

Composite indexes are powerful because one index can serve queries that filter on several columns at once, and column order is critical. The order should generally place the most selective or most frequently filtered columns first, and it must match how queries actually filter to be useful.

Example


(first_name,last_name)
Q18. Leftmost Prefix Rule

ROI: ★★★★★ Frequency: Senior Favorite

The leftmost prefix rule explains how a composite index can be used. Because the index is sorted by its columns left to right, the database can only use it efficiently if the query filters on a continuous prefix of the index columns starting from the leftmost one.

With an index on (a, b, c), queries that filter on a, on a and b, or on a, b and c can all use the index. But a query that filters only on b, or only on c, cannot use this index effectively because the leading column a is not constrained, so the values it needs are scattered throughout the index rather than grouped together. This is why the choice and ordering of columns in a composite index must be driven by your actual query patterns.

Q19. When Indexes Are Ignored?

ROI: ★★★★★ Frequency: High

Even when an index exists, the optimizer may decide not to use it. A leading wildcard such as LIKE '%abc' cannot use a B-tree index because the index is ordered from the left and the query does not anchor the start of the string. Wrapping an indexed column in a function, like LOWER(name), hides the raw column value from the index so the engine falls back to scanning.

The optimizer also skips indexes when they would not actually help. On a very small table a full scan is cheaper than the overhead of an index lookup, and on a low-selectivity column (one where a predicate matches a large fraction of rows) reading the table directly is faster than seeking through the index and then fetching most of the rows anyway. Reading the execution plan is how you confirm what the database actually chose.

  • Leading Wildcards
  • Functions On Indexed Columns
  • Small Tables
  • Poor Selectivity
Q20. Example

ROI: ★★★★★ Frequency: High

Bad


WHERE LOWER(name)

Applying a function such as LOWER() to an indexed column in the WHERE clause typically prevents the database from using the index on that column. The index stores the original column values, not the transformed ones, so the engine can no longer perform an ordered seek and falls back to a full scan.

The fix depends on the database. You can create a functional (expression) index on LOWER(name) so the transformed value is itself indexed, or store the data already normalised (for example a lowercased copy column), or in PostgreSQL use a case-insensitive collation or the citext type. The general lesson is to keep indexed columns "bare" on the left side of a predicate.

SECTION 4 - NORMALIZATION

Q21. What Is Normalization?

ROI: ★★★★★ Frequency: Extremely High

Answer

Normalization is the process of organising tables and columns to reduce data redundancy and improve data integrity. It works by breaking large, repetitive tables into smaller related tables and defining relationships between them through keys.

The goal is to ensure that each piece of information is stored in exactly one place, which eliminates update anomalies, where the same fact stored in multiple rows can drift out of sync. By storing a customer's address once and referencing it, for example, you never risk having two different addresses for the same customer.

Normalization proceeds in stages called normal forms (1NF, 2NF, 3NF and beyond), each addressing a specific class of redundancy. In real systems most schemas target 3NF as a sensible balance, then selectively denormalize where performance demands it.

Q22. First Normal Form (1NF)

ROI: ★★★★★ Frequency: High

Rule

First Normal Form requires that every column hold atomic, indivisible values and that there are no repeating groups or arrays within a single column. Each cell contains one value, and each row is uniquely identifiable.

A violation would be storing multiple phone numbers in a single comma-separated field, or having columns like phone1, phone2, phone3. The 1NF fix is to move those repeating values into a separate related table with one row per value, which makes the data queryable and avoids brittle string parsing in application code.

Q23. Second Normal Form (2NF)

ROI: ★★★★★ Frequency: Medium

Second Normal Form builds on 1NF and removes partial dependencies, meaning no non-key column may depend on only part of a composite primary key. It only becomes relevant when a table has a composite (multi-column) primary key.

For example, in an order_items table keyed on (order_id, product_id), storing the product_name in that table violates 2NF because the product name depends only on product_id, not on the whole key. The fix is to move product_name into a products table keyed solely by product_id. This eliminates the redundancy of repeating the product name across every order line.

Q24. Third Normal Form (3NF)

ROI: ★★★★★ Frequency: High

Third Normal Form builds on 2NF and removes transitive dependencies, where a non-key column depends on another non-key column rather than directly on the primary key. Every non-key column should depend on the key, the whole key, and nothing but the key.

A classic violation is storing both a zip_code and the city_name it implies in the same table: city depends on zip, which depends on the key, so it is transitive. The fix is to keep only the zip in the main table and look up the city from a separate reference table. 3NF is the practical target for most transactional (OLTP) schemas because it eliminates the majority of redundancy while keeping the model simple to query.

Q25. Why Denormalize?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

Denormalization is the deliberate introduction of controlled redundancy to improve read performance. A fully normalized schema can require many joins to assemble a single view, and at scale those joins become expensive, so we sometimes trade storage and write complexity for faster reads.

Common techniques include duplicating a frequently displayed column to avoid a join, storing pre-computed aggregates such as an order total or a comment count, and maintaining read-optimised summary tables. Analytical and reporting workloads (OLAP, data warehouses, star schemas) lean heavily on denormalization for exactly this reason.

The cost is that the same fact now lives in more than one place, so writes must update every copy and keep them consistent, often via triggers, application logic or asynchronous jobs. The senior judgement is to normalize first for correctness, then denormalize only where measured query performance justifies the added maintenance burden.

SECTION 5 - ACID PROPERTIES

This is one of the most frequently asked database interview topics. Every backend engineer should know ACID thoroughly.

Q26. What Is ACID?

ROI: ★★★★★ Frequency: Extremely High

Answer

ACID is the set of four guarantees that make database transactions reliable: Atomicity, Consistency, Isolation and Durability. Together they ensure that even in the face of concurrent access, errors and crashes, the database remains in a correct and trustworthy state.

These properties are why relational databases are the default choice for systems where correctness is non-negotiable, such as banking, payments and order processing. When an interviewer asks about ACID they usually want you to explain each letter and, ideally, tie it to a real money-transfer or order scenario.

  • Atomicity
  • Consistency
  • Isolation
  • Durability
Q27. What Is Atomicity?

ROI: ★★★★★ Frequency: Extremely High

Answer

Atomicity means a transaction is all-or-nothing: either every operation within it succeeds and is committed, or if any step fails the entire transaction is rolled back as if it never happened. There is no partial state left behind.

The database achieves this using mechanisms such as undo logs (or rollback segments), which let it reverse partially applied changes when an error or crash occurs mid-transaction. Atomicity is what guarantees you never end up with half a transaction applied.

Example


Debit Account

Credit Account

In a money transfer, debiting one account and crediting another must happen together. If the credit fails after the debit succeeds, atomicity rolls back the debit too, so money is never silently lost or created. If one fails, the entire transaction rolls back.

Q28. What Is Consistency?

ROI: ★★★★★ Frequency: Extremely High

Answer

Consistency guarantees that a transaction moves the database from one valid state to another valid state, never leaving it in a state that breaks the defined rules. Those rules include constraints such as primary keys, foreign keys, unique constraints, check constraints and any triggers.

If a transaction would violate any of these invariants, the database rejects it and rolls back rather than persisting invalid data. In other words, consistency ensures that all the integrity rules you declared are still true before and after every transaction, so the data always makes business sense.

Q29. What Is Isolation?

ROI: ★★★★★ Frequency: Extremely High

Answer

Isolation ensures that concurrently executing transactions do not interfere with one another, so the end result is as if they had run in some serial order. Without isolation, two transactions reading and writing the same rows at the same time could see each other's incomplete work and corrupt the data.

In reality, perfect isolation is expensive, so databases offer a spectrum of isolation levels (READ UNCOMMITTED through SERIALIZABLE) that trade strictness for concurrency. The level you choose decides which concurrency anomalies, dirty reads, non-repeatable reads and phantom reads, are allowed to occur.

Q30. What Is Durability?

ROI: ★★★★★ Frequency: Extremely High

Answer

Durability guarantees that once a transaction is committed, its changes are permanent and survive any subsequent failure, including crashes, power loss or restarts. A committed transaction is never lost.

Databases implement durability with write-ahead logging (WAL): changes are first written to a durable transaction log on disk before the commit is acknowledged, so even if the server crashes immediately afterwards, the log can be replayed on recovery to restore the committed state. This is also why proper disk flushing and reliable storage configuration matter so much in production.

SECTION 6 - TRANSACTIONS

Q31. What Is A Transaction?

ROI: ★★★★★ Frequency: Extremely High

Answer

A transaction is a logical unit of work that groups one or more SQL operations so they execute as a single, indivisible action. The database treats the whole group as one operation that either fully succeeds (commit) or fully fails (rollback), giving it the ACID guarantees.

In a Spring Boot application this is typically expressed declaratively with the @Transactional annotation on a service method. Everything inside that method runs in one transaction, so if an exception is thrown the framework rolls the whole thing back, which is how you keep multi-step business operations consistent.

Q32. Transaction Lifecycle

ROI: ★★★★★ Frequency: High

SQL Operations

↓

COMMIT

or

ROLLBACK

A transaction begins explicitly with BEGIN (or implicitly when the first statement runs), then executes its SQL operations, accumulating changes that are not yet visible to other transactions. It ends in one of two ways: COMMIT makes all the changes permanent and visible, while ROLLBACK discards every change and returns the data to its pre-transaction state.

Keeping this lifecycle short is a key performance principle. The longer a transaction stays open, the longer it holds locks and the more it strains resources, so well-designed code does only the necessary work between BEGIN and COMMIT.

Q33. Commit vs Rollback

ROI: ★★★★★ Frequency: Extremely High

Commit Rollback
Save Changes Undo Changes
Permanent Restore Previous State

COMMIT finalises a transaction, making all its changes durable and visible to other transactions; after a commit there is no going back. ROLLBACK does the opposite, undoing every change made since the transaction began and restoring the data to its prior state.

Rollback is triggered both explicitly (when application code detects an error and calls it) and automatically (when the database hits a constraint violation, deadlock or crash). Designing code so that any failure path leads cleanly to a rollback is what makes operations safe to retry.

Q34. Why Transactions Matter?

ROI: ★★★★★ Frequency: High

Example

Account B

Transactions matter because real business operations almost always involve multiple steps that must succeed or fail together. Transferring 1000 from Account A to Account B is two writes: a debit and a credit. Without a transaction, a crash between the two leaves the system inconsistent.

Without a transaction wrapping both steps, money could be debited from A but never credited to B, effectively disappearing, or vice versa. The transaction guarantees atomicity so the transfer is always complete or completely absent, which is exactly the kind of correctness guarantee that justifies using a relational database.

SECTION 7 - CONCURRENCY PROBLEMS

Q35. What Is Dirty Read?

ROI: ★★★★★ Frequency: Extremely High

Answer

A dirty read occurs when one transaction reads data that another transaction has modified but not yet committed. If that other transaction later rolls back, the first transaction has acted on data that never truly existed.

Dirty reads are only possible at the READ UNCOMMITTED isolation level. Every higher level prevents them, which is why almost all production systems run at READ COMMITTED or above. Reading uncommitted data is dangerous precisely because it can be undone at any moment.

Q36. Dirty Read Example

ROI: ★★★★★ Frequency: High 


Transaction A

Updates Salary

Not Committed

↓

Transaction B Reads Salary

Here Transaction A updates a salary but has not committed yet. Transaction B reads that new, uncommitted salary value and proceeds to use it. This read of unconfirmed data is the dirty read.

The danger becomes clear if Transaction A then rolls back: the salary reverts to its original value, but Transaction B has already made decisions based on a number that was never real. This is why isolation levels above READ UNCOMMITTED simply do not allow this read to happen.

Q37. What Is Non-Repeatable Read?

ROI: ★★★★★ Frequency: Extremely High

Answer

A non-repeatable read happens when a transaction reads the same row twice and gets different values, because another transaction committed an update to that row in between the two reads. The data "changed under its feet" within a single transaction.

This anomaly is prevented at the REPEATABLE READ isolation level and above, which guarantee that any row you read once will return the same value if you read it again within the same transaction. It is distinct from a phantom read, which concerns new rows appearing rather than existing rows changing.

Q38. Non-Repeatable Read Example

ROI: ★★★★★ Frequency: High 


Read Balance

↓

Another Transaction Updates

↓

Read Balance Again

Different Result

In this sequence a transaction reads an account balance, then a second transaction updates and commits that balance, and when the first transaction reads it again it sees a different number. The same query produced two different answers within one transaction.

This matters whenever logic assumes a value is stable for the duration of the transaction, for example reading a balance, performing a calculation, then re-checking the balance. Choosing REPEATABLE READ, or taking an explicit lock on the row, prevents the value from shifting mid-transaction.

Q39. What Is Phantom Read?

ROI: ★★★★★ Frequency: Extremely High

Answer

A phantom read occurs when a transaction re-runs a query with a search condition and finds that new rows have appeared (or existing matching rows have vanished) because another transaction inserted or deleted rows that match the condition and committed in between.

The distinction from a non-repeatable read is subtle but important: a non-repeatable read is about the same row changing value, while a phantom is about the set of rows matching a predicate changing. Only the SERIALIZABLE isolation level fully prevents phantoms in the general case (though databases like PostgreSQL and MySQL InnoDB use range or gap locks to mitigate them at REPEATABLE READ).

Q40. Phantom Read Example

ROI: ★★★★★ Frequency: High 


SELECT Employees

↓

Another Transaction Inserts Employee

↓

SELECT Employees Again

Extra Row Appears

Here a transaction selects all employees matching some criteria, another transaction inserts a new qualifying employee and commits, and when the first transaction repeats the SELECT an extra "phantom" row appears that was not in the original result set.

This is problematic for operations that depend on the count or completeness of a set, such as validating "no more than N bookings exist" before inserting one more. Preventing phantoms requires SERIALIZABLE isolation or explicit range locking so that no new rows can sneak into the queried range.

SECTION 8 - ISOLATION LEVELS

Q41. What Is Isolation Level?

ROI: ★★★★★ Frequency: Extremely High

An isolation level defines the visibility rules between concurrent transactions, specifically which uncommitted or concurrently changing data one transaction is allowed to see from another. It is the dial that controls the trade-off between data correctness and concurrency throughput.

A stricter level prevents more anomalies (dirty reads, non-repeatable reads, phantoms) but reduces concurrency because transactions must hold locks longer or do more work to stay consistent. A looser level allows more parallelism and higher throughput but exposes the application to more concurrency anomalies. Choosing the right level is a deliberate engineering decision based on how much inconsistency the use case can tolerate.

Q42. Isolation Levels In Order

ROI: ★★★★★ Frequency: Extremely High 


READ UNCOMMITTED

↓

READ COMMITTED

↓

REPEATABLE READ

↓

SERIALIZABLE

The four standard isolation levels run from weakest to strongest. READ UNCOMMITTED allows everything including dirty reads; READ COMMITTED prevents dirty reads; REPEATABLE READ additionally prevents non-repeatable reads; and SERIALIZABLE prevents all anomalies including phantoms by making transactions behave as if run one at a time.

As you move down the list, correctness increases but concurrency and throughput decrease. Most real systems sit at READ COMMITTED (PostgreSQL's default) or REPEATABLE READ (MySQL InnoDB's default), reserving SERIALIZABLE for the rare operations that genuinely need it.

Q43. READ UNCOMMITTED

ROI: ★★★★★ Frequency: High

Behavior

READ UNCOMMITTED is the weakest level. It allows a transaction to read data that other transactions have changed but not yet committed, which means dirty reads, non-repeatable reads and phantoms are all possible.

Because it provides essentially no read guarantees, it is rarely used in practice. Its only niche is rough, approximate reporting where occasional incorrect values are acceptable in exchange for never blocking on locks. Notably, PostgreSQL treats READ UNCOMMITTED the same as READ COMMITTED, so dirty reads never actually occur there.

Rarely Used

Q44. READ COMMITTED

ROI: ★★★★★ Frequency: Extremely High

Behavior

READ COMMITTED guarantees that a transaction only ever reads data that has been committed, which eliminates dirty reads. However, each statement sees a fresh snapshot, so the same row can still change between two reads within the transaction (non-repeatable reads) and new rows can still appear (phantoms).

It is the most widely used level in production because it offers a strong balance: no dirty data, good concurrency and minimal locking. It is the default in PostgreSQL, Oracle and SQL Server, and is appropriate for the vast majority of typical web application workloads.

ROI: ★★★★★ Frequency: Extremely High

Behavior

REPEATABLE READ guarantees that any row read during a transaction will return the same value if read again, preventing both dirty reads and non-repeatable reads. It typically achieves this by giving the transaction a consistent snapshot of the data taken at its start.

Under the strict SQL standard, phantoms can still occur at this level, but MySQL's InnoDB engine, where REPEATABLE READ is the default, uses next-key (gap) locking to prevent most phantom inserts as well. It is the right choice when a transaction needs a stable view of existing rows throughout its execution.

Default In


MySQL InnoDB
Q46. SERIALIZABLE

ROI: ★★★★★ Frequency: High

Behavior

SERIALIZABLE is the strictest isolation level. It guarantees that the outcome of running transactions concurrently is identical to running them one after another in some serial order, which eliminates dirty reads, non-repeatable reads and phantoms entirely.

Databases achieve this either by aggressive range locking or, as in PostgreSQL, by Serializable Snapshot Isolation that detects conflicting transactions and aborts one of them. Either way, the application must be prepared to catch serialization-failure errors and retry the transaction.

Cost

The price of this strictness is the lowest concurrency of any level, since transactions either block each other heavily or are forced to abort and retry. SERIALIZABLE is reserved for operations where absolute correctness outweighs throughput, such as financial settlement or inventory reservation under contention.

SECTION 9 - DEADLOCKS

Q48. What Is A Deadlock?

ROI: ★★★★★ Frequency: Extremely High

Answer

A deadlock is a situation where two or more transactions are each waiting for a lock that the other holds, so none of them can ever proceed. Each is blocked forever by the other, forming a cycle of dependencies.

Deadlocks are an inherent risk of any locking system, not a bug per se. Databases detect them automatically using a wait-for graph and break the cycle by aborting one transaction (the victim), allowing the others to continue. The application should be ready to catch the resulting error and retry.

Q49. Deadlock Example

ROI: ★★★★★ Frequency: Extremely High 


Transaction A

Locks Row 1

Waiting For Row 2

↓

Transaction B

Locks Row 2

Waiting For Row 1

Here Transaction A locks Row 1 and then needs Row 2, while Transaction B has already locked Row 2 and now needs Row 1. Each holds the lock the other is waiting for, so neither can advance: this circular wait is the deadlock.

Notice that the root cause is inconsistent lock ordering, the two transactions acquired the same resources in opposite sequences. This insight directly leads to the most effective prevention technique: always acquire locks in a consistent, agreed-upon order.

Q50. How Database Resolves Deadlock?

ROI: ★★★★★ Frequency: High

Answer

The database has a background deadlock detector that periodically looks for cycles in the wait-for graph. When it finds one, it chooses a victim transaction, typically the one that has done the least work or holds the fewest locks, and forcibly rolls it back, releasing its locks so the remaining transactions can proceed.

From the application's perspective, the victim receives a deadlock error. The correct handling is not to treat this as a fatal failure but to catch the specific exception and retry the transaction, since on retry the conflicting transaction has usually already finished. Designing transactions to be idempotent and retry-safe is therefore important.

Q51. How To Prevent Deadlocks?

ROI: ★★★★★ Frequency: Senior Favorite

The most reliable prevention is consistent lock ordering: if every transaction always acquires locks on resources in the same order (for example, always the lower account id before the higher), the circular wait that causes a deadlock can never form. Keeping transactions short minimises the window during which locks are held, reducing the chance of conflict.

Proper indexing matters too, because an unindexed update may lock far more rows (or whole ranges) than necessary, widening the lock footprint and increasing contention. Finally, avoid acquiring locks you do not need and consider lower isolation levels where appropriate. When deadlocks do occur despite this, the application should retry the aborted transaction gracefully.

  • Consistent Lock Ordering
  • Short Transactions
  • Proper Indexing
  • Avoid Unnecessary Locks

SECTION 10 - LOCKING

Q52. Shared Lock

ROI: ★★★★★ Frequency: High

A shared lock (read lock) allows multiple transactions to read the same resource simultaneously. Many transactions can hold a shared lock on the same row or table at once, because reading does not change the data and therefore does not conflict with other reads.

However, a shared lock blocks any transaction that wants an exclusive (write) lock on the same resource until all shared locks are released. This is how databases let concurrent readers proceed in parallel while ensuring a writer waits until the readers are done.

Q53. Exclusive Lock

ROI: ★★★★★ Frequency: High

An exclusive lock (write lock) is held by a single transaction that intends to modify a resource. While it is held, no other transaction can acquire either a shared or an exclusive lock on the same resource, so all other readers and writers must wait.

This guarantees that a write happens in isolation and that no one reads a half-written value. The flip side is that exclusive locks reduce concurrency, which is why writes should be quick and why holding write locks across long transactions is a common cause of contention and timeouts in production.

Q54. Why Long Transactions Are Dangerous?

ROI: ★★★★★ Frequency: Senior Favorite

A long-running transaction holds its locks for the entire duration, blocking other transactions that need the same rows and dramatically increasing the probability of deadlocks. The longer the transaction, the larger the window in which conflicting operations can collide.

Long transactions also hurt the database in less obvious ways. In MVCC systems like PostgreSQL they prevent vacuum from cleaning up old row versions, causing table bloat and degraded performance over time, and they tie up connection-pool resources. The senior guidance is to keep transactions as short and focused as possible, doing slow work such as external API calls outside the transaction boundary.

  • More Locks
  • Deadlocks
  • Reduced Concurrency
  • Resource Consumption

SECTION 11 - QUERY OPTIMIZATION

This section is heavily asked for engineers with 3+ years of experience. Interviewers often present a slow query and ask: "How would you investigate and optimize it?"

Q55. What Is Query Optimization?

ROI: ★★★★★ Frequency: Extremely High

Answer

Query optimization is the practice of improving the performance of a query, reducing its execution time and resource consumption, while keeping its results correct. It spans rewriting the SQL, adding the right indexes, restructuring the schema and tuning the database itself.

The work is always evidence-driven. Rather than guessing, you examine the execution plan to understand how the database is running the query, identify the expensive operation (a full table scan, a large sort, a nested loop over many rows) and address that specific bottleneck. Optimization without measuring is just speculation.

Q56. First Step When Query Becomes Slow?

ROI: ★★★★★ Frequency: Extremely High

Expected Answer

The first step is always to look at the execution plan using EXPLAIN (or EXPLAIN ANALYZE), not to start adding indexes blindly. The plan tells you exactly how the database intends to run the query and where the cost is concentrated, which turns guessing into diagnosis.

From there you check whether existing indexes are being used, analyse the filters and join conditions to see if they are sargable (index-friendly), and consider the data volume involved. A query that is fine on a thousand rows can behave very differently on ten million, so understanding scale is part of the diagnosis.

  1. Check Execution Plan
  2. Check Index Usage
  3. Analyze Filters
  4. Analyze Joins
  5. Review Data Volume
Q57. Common Causes Of Slow Queries

ROI: ★★★★★ Frequency: Extremely High

The majority of slow queries trace back to a small set of recurring causes. Missing indexes force full table scans; selecting or returning very large result sets wastes memory and network; poorly written joins, especially on unindexed columns, explode the work the optimizer must do; and applying functions to columns in the WHERE clause defeats indexes.

Pagination patterns are another frequent culprit, particularly large OFFSET values that make the database scan and discard huge numbers of rows. The N+1 query pattern from ORMs like Hibernate, where one query for a parent triggers a separate query per child, is also a classic cause that you fix with fetch joins or batch fetching.

  • Missing Indexes
  • Full Table Scans
  • Large Result Sets
  • Bad Joins
  • Functions In WHERE Clause
  • Poor Pagination
Q58. SELECT * Problem

ROI: ★★★★★ Frequency: High

Answer

SELECT * retrieves every column from a table even when the application only needs a few. This pulls back unnecessary data, increasing network transfer, memory usage on both the database and the application, and serialization overhead.

It also undermines optimization. A query that lists only the columns it needs may be fully satisfied by a covering index without touching the base table, whereas SELECT * almost always forces the engine to read the full row. On top of that, SELECT * is fragile: adding a column to the table can silently change the query's result shape and break downstream code. Always select only the columns you actually use.

Issues

  • More Network Transfer
  • More Memory Usage
  • Reduced Performance
Q59. Why Avoid Functions On Indexed Columns?

ROI: ★★★★★ Frequency: Extremely High

Bad


WHERE LOWER(email)

When you wrap an indexed column in a function, such as WHERE LOWER(email) = '...', the predicate becomes non-sargable. The index stores the raw email values, not their lowercased form, so the database cannot perform an ordered seek and is forced into a full scan that applies the function to every row.

The remedies are to create a functional index on the expression (for example an index on LOWER(email)), to store the data pre-normalised, or in PostgreSQL to use a case-insensitive type or collation. The same principle applies to date functions, arithmetic and casts on indexed columns: keep the indexed column bare on one side of the comparison.

Problem

May prevent index usage, turning what should be a fast indexed lookup into a slow full table scan.

Q60. EXISTS vs IN

ROI: ★★★★★ Frequency: High

Interview Answer

Both EXISTS and IN are used to test membership against a subquery, and which one performs better depends on the database, the data and the optimizer. Modern optimizers often rewrite both into a semi-join internally, so on recent PostgreSQL or MySQL versions the difference is frequently negligible.

The traditional guidance is that EXISTS tends to be more efficient for large or correlated subqueries because it can stop scanning as soon as it finds the first match, whereas IN may materialise the entire subquery result. A subtle but important correctness point is NULL handling: NOT IN behaves unexpectedly when the subquery can return NULLs (it returns no rows), so NOT EXISTS is usually the safer choice for anti-joins.

SECTION 12 - EXECUTION PLANS

Q61. What Is Execution Plan?

ROI: ★★★★★ Frequency: Extremely High

Answer

An execution plan is the step-by-step strategy the database's query optimizer chooses to execute a SQL statement. It describes which indexes are used, the order tables are accessed, the join algorithms (nested loop, hash join, merge join) and the estimated cost and row counts at each step.

You view it with EXPLAIN, and with EXPLAIN ANALYZE the database actually runs the query and reports real timings and row counts alongside its estimates. Reading the plan is the central skill of query tuning, because it reveals what the database is really doing rather than what you assume it is doing.

Q62. Why Execution Plans Matter?

ROI: ★★★★★ Frequency: Extremely High

Execution plans matter because they turn performance debugging from guesswork into diagnosis. They reveal exactly why a query is slow, whether it is doing a full table scan instead of an index seek, choosing a bad join order, or misestimating row counts because statistics are stale.

A large gap between the optimizer's estimated rows and the actual rows (visible in EXPLAIN ANALYZE) is a strong signal that statistics need updating or the query needs rewriting. Without reading the plan, any optimization is a shot in the dark; with it, you can target the single most expensive operation and fix the real bottleneck.

Q63. What Is Full Table Scan?

ROI: ★★★★★ Frequency: Extremely High

Answer

A full table scan (sequential scan) is when the database reads every single row in a table to find the ones that match, rather than using an index to jump directly to them. On a small table this is cheap, but on a large table it is one of the most expensive operations and a common reason queries become slow under growth.

A full scan in an execution plan is not always wrong, however. When a query needs most of the rows anyway, or the table is tiny, a sequential scan is genuinely faster than an index seek plus row fetches. The red flag is a full scan on a large table for a query that should be returning only a few rows: that almost always indicates a missing or unusable index.

Usually Indicates

  • Missing Index
  • Bad Query
Q64. Index Scan vs Table Scan

ROI: ★★★★★ Frequency: Extremely High

Index Scan Table Scan
Targeted Access Reads Everything
Usually Faster Usually Slower

An index scan uses an index to navigate directly to the rows that match a predicate, reading only a targeted subset of the data, whereas a table scan reads every row sequentially. For selective queries that return a small fraction of the table, the index scan is dramatically faster.

The nuance worth mentioning is selectivity: when a query matches a large percentage of rows, the optimizer may correctly prefer a table scan, because following an index and then fetching most of the rows individually (random I/O) can be slower than reading the whole table in one sequential pass. The "right" choice is therefore data-dependent, which is exactly why the optimizer relies on up-to-date statistics.

Q65. What Is Cardinality?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

Cardinality refers to the number of distinct values in a column. A column like email or user_id has high cardinality because almost every value is unique, while a column like gender, country or a boolean flag has low cardinality because only a handful of distinct values repeat across many rows.

Cardinality drives the optimizer's decisions. It estimates how many rows a predicate will match using column statistics, and that estimate determines whether an index seek or a table scan is cheaper. High-cardinality columns make for selective, effective indexes; low-cardinality columns often do not benefit from a plain B-tree index at all.

Example


country

Low Cardinality

email

High Cardinality
Q66. Why Cardinality Matters?

ROI: ★★★★★ Frequency: High

Cardinality matters because it determines an index's selectivity, how effectively it narrows down the rows. An index on a high-cardinality column lets the database eliminate the vast majority of rows in one seek, which is exactly what makes the index worthwhile.

Conversely, indexing a low-cardinality column often gives little benefit: if a value matches half the table, the optimizer will usually ignore the index and scan instead, because random lookups for that many rows cost more than a sequential read. This is why, when building composite indexes, you generally place the most selective (highest-cardinality) columns first, and why understanding cardinality is key to choosing which columns to index at all.

SECTION 13 - PAGINATION

Q67. Why Pagination?

ROI: ★★★★★ Frequency: Extremely High

Pagination exists to avoid loading huge result sets into memory and transferring them over the network all at once. Instead of returning a million rows to render a list, the application fetches them in manageable pages of, say, twenty or fifty rows.

This keeps both the database and the application responsive, bounds memory usage, and improves the user experience by returning the first page quickly. The way you implement pagination, however, has a major performance impact at scale, which is why the offset-versus-keyset distinction is a senior favourite.

Q68. LIMIT/OFFSET Pagination

ROI: ★★★★★ Frequency: Extremely High 


LIMIT 20

OFFSET 40

LIMIT/OFFSET pagination asks the database to skip a number of rows (OFFSET) and then return the next batch (LIMIT). It is simple, intuitive and works well for the first few pages, which is why it is the most common approach in basic applications.

The hidden cost is that the database must still read and discard all the skipped rows to honour the OFFSET. For shallow pages this is fine, but it degrades badly as the offset grows, because the work scales with how deep into the result set you are, not with the page size.

Q69. Problem With Large OFFSET

ROI: ★★★★★ Frequency: Senior Favorite

Answer

The problem with a large OFFSET is that the database cannot simply jump to the Nth row; it must scan and throw away every row before the offset. With OFFSET 1000000, it reads and discards a million rows just to return the next twenty, making deep pages progressively slower.

This is why infinitely scrollable lists and deep page numbers built on OFFSET degrade so badly in production. The cost grows linearly with the offset value, so the deeper a user navigates, the slower each page becomes, even though every page returns the same small number of rows.

Example


OFFSET 1000000

Very expensive, because the engine reads a million rows only to skip them.

Q70. Keyset Pagination

ROI: ★★★★★ Frequency: Extremely High

Answer

Keyset pagination (also called seek or cursor pagination) replaces OFFSET with a WHERE condition based on the last value seen on the previous page. Instead of skipping rows, it seeks directly to the starting point using an index, then reads forward by the page size.

Because it relies on an indexed column to locate the start, every page costs roughly the same regardless of how deep you are, which makes it the standard solution for large datasets and infinite scroll. The trade-off is that it requires a stable, unique ordering column (or tuple) and does not support arbitrary "jump to page N" navigation as naturally as offset does.

Example


WHERE id > 500

LIMIT 20
Q71. Keyset vs Offset Pagination

ROI: ★★★★★ Frequency: Extremely High

Offset Keyset
Simple Faster Large Datasets
Slow Large Offsets Scalable

Offset pagination is simple and supports jumping to any page, but its cost grows with depth because skipped rows must still be scanned. Keyset pagination uses the last-seen key to seek directly to the next page via an index, so its cost stays constant no matter how deep you go.

The practical guidance is to use offset for small datasets, admin tables and cases where direct page jumps matter, and to switch to keyset for large, frequently-scrolled lists and APIs that feed infinite scroll. Keyset requires a stable unique sort key and sacrifices random page access in exchange for predictable scalability.

SECTION 14 - DATABASE SCALING

Q72. Vertical Scaling

ROI: ★★★★★ Frequency: High

Answer

Vertical scaling, or scaling up, means increasing the resources of a single database server, adding more CPU, RAM, faster disks or higher IOPS. It is the simplest way to handle growth because the application and the data model stay exactly the same; you just give the machine more power.

The appeal is zero architectural change and no consistency complications. The limitation is that there is a ceiling: hardware has a maximum, the cost rises steeply at the high end, and a single server remains a single point of failure. Vertical scaling is usually the right first move, but eventually horizontal strategies become necessary.

Q73. Horizontal Scaling

ROI: ★★★★★ Frequency: Extremely High

Answer

Horizontal scaling, or scaling out, means distributing the load across multiple servers rather than making one server bigger. For databases this typically takes the form of read replicas to spread read traffic, or sharding to spread the data and write traffic across nodes.

It offers far greater scalability and better fault tolerance, since the system no longer depends on a single machine. The cost is significant added complexity: you must deal with replication lag, data distribution, and the harder problem of distributed transactions and cross-node queries. This is why teams usually scale up first and scale out only when they must.

Q74. Replication

ROI: ★★★★★ Frequency: Extremely High

Answer

Replication is the process of copying data from one database (the primary) to one or more other databases (replicas) so that the same data exists in multiple places. The primary handles writes and streams its changes to the replicas, which apply them to stay in sync.

Replication can be synchronous, where the primary waits for replicas to confirm before committing (stronger consistency, higher latency), or asynchronous, where it does not wait (lower latency, but replicas can lag behind). That replication lag is the key trade-off: a read from a replica might briefly return slightly stale data, which the application design must account for.

Q75. Primary Replica Architecture

ROI: ★★★★★ Frequency: High 


Primary

↓

Replica 1

Replica 2

Replica 3

In a primary-replica (formerly master-slave) architecture, a single primary node accepts all writes and propagates them to several read-only replicas. Application read traffic is then distributed across the replicas, which dramatically increases read throughput because reads no longer compete with each other on one machine.

This pattern fits the typical web workload, which is overwhelmingly read-heavy. The caveats are that all writes still funnel through the single primary (so it can become a write bottleneck) and that asynchronous replicas may serve slightly stale data, so writes that must be read back immediately should be routed to the primary.

Q76. Benefits Of Replication

ROI: ★★★★★ Frequency: High

Replication delivers three major benefits. Read scaling lets you spread read queries across many replicas, multiplying read capacity for read-heavy applications. High availability means that if the primary fails, a replica can be promoted to take over, keeping the system online.

Disaster recovery is the third: keeping replicas in a different data centre or region protects against the loss of a single location. Replicas can also offload heavy reporting and analytics queries away from the primary so they do not slow down user-facing traffic.

  • Read Scaling
  • Availability
  • Disaster Recovery
Q77. What Is Sharding?

ROI: ★★★★★ Frequency: Extremely High

Answer

Sharding is horizontal partitioning of data across multiple independent databases, where each shard holds a distinct subset of the rows. Unlike replication, which copies the same data everywhere, sharding splits the data so that each node stores and serves only its portion.

This is how you scale writes and total data volume beyond what a single machine can handle, since both the storage and the write load are divided across shards. The choice of shard key, the column that decides which shard a row lives on, is the most critical and hardest design decision, because a poor key leads to hotspots and uneven load.

Q78. Sharding Example

ROI: ★★★★★ Frequency: High 


Shard A

User 1-1M

Shard B

User 1M-2M

This example shows range-based sharding: users with ids 1 to 1M live on Shard A, and users 1M to 2M live on Shard B. To find a user, the application (or a routing layer) uses the shard key to determine which shard to query.

Range sharding is easy to reason about but risks hotspots if activity clusters in one range (for example, the newest users on the latest shard). The common alternative is hash-based sharding, which distributes rows more evenly by hashing the key, at the cost of making range scans across shards harder.

Q79. Sharding Challenges

ROI: ★★★★★ Frequency: Senior Favorite

Sharding solves scale but introduces hard problems. Queries that need data from multiple shards (cross-shard queries) and joins across shards are slow or impossible to do efficiently, because the data lives on separate machines, so applications often have to denormalize or aggregate in code. Distributed transactions spanning shards are likewise difficult and usually avoided.

Rebalancing is another major pain point: when one shard grows hot or a new shard is added, moving data between shards without downtime is operationally complex. Overall, sharding adds significant complexity to development and operations, so the senior view is to exhaust replication, caching and vertical scaling first and shard only when genuinely necessary.

  • Cross Shard Queries
  • Rebalancing
  • Complexity
  • Joins

SECTION 15 - POSTGRESQL INTERVIEW QUESTIONS

Q80. Why PostgreSQL?

ROI: ★★★★★ Frequency: High

PostgreSQL is a popular choice because it combines being free and open source with enterprise-grade capabilities. It is fully ACID compliant and offers strong consistency, which makes it dependable for systems where data correctness is critical.

Beyond the basics it provides advanced features that set it apart: rich data types including native JSONB, arrays and geospatial support via PostGIS, powerful indexing options (B-tree, GIN, GiST, BRIN), window functions, CTEs and an extension ecosystem. Its MVCC implementation gives excellent concurrency, and its standards compliance and reliability have made it the default relational database for many modern backends.

  • Open Source
  • ACID Compliant
  • Advanced Features
  • Strong Consistency
Q81. PostgreSQL Default Isolation Level

ROI: ★★★★★ Frequency: Extremely High

Answer


READ COMMITTED

PostgreSQL defaults to the READ COMMITTED isolation level. At this level every statement sees only data committed before that statement began, so dirty reads never occur, but non-repeatable reads and phantoms are still possible within a transaction.

This default reflects a deliberate balance of correctness and concurrency suited to typical applications. A useful interview detail is that PostgreSQL implements isolation through MVCC rather than read locks, and it actually treats a request for READ UNCOMMITTED as READ COMMITTED, so dirty reads are impossible even if you ask for them.

Q82. What Is MVCC?

ROI: ★★★★★ Frequency: Extremely High

Full Form

Multi-Version Concurrency Control.

Answer

MVCC lets readers and writers work concurrently without blocking each other by keeping multiple versions of a row. When a row is updated, the database creates a new version rather than overwriting the old one in place, and each transaction sees the version that was current as of its own snapshot.

The practical consequence is that readers never block writers and writers never block readers, which is a huge concurrency win compared to lock-based approaches. The cost is that old, no-longer-visible row versions accumulate and must be cleaned up, which in PostgreSQL is the job of the VACUUM (and autovacuum) process; neglecting it leads to table bloat.

Q83. Why MVCC?

ROI: ★★★★★ Frequency: High

The reason MVCC is used is that it dramatically reduces locking and improves concurrency. Because each transaction reads from a consistent snapshot of its own, a long-running read query does not block writers, and writes do not block readers, which keeps throughput high under heavy mixed workloads.

This is what allows databases like PostgreSQL and Oracle to serve many simultaneous transactions without the heavy read-lock contention seen in simpler locking schemes. The trade-off, again, is the overhead of maintaining and eventually cleaning up multiple row versions, but for most workloads the concurrency benefit far outweighs that cost.

SECTION 16 - MYSQL INTERVIEW QUESTIONS

Q84. Most Common MySQL Storage Engine?

ROI: ★★★★★ Frequency: High

Answer


InnoDB

InnoDB is the default and most widely used storage engine in MySQL. It became the default (replacing MyISAM) because it supports transactions, foreign keys and row-level locking, all essential for reliable transactional applications.

A key architectural detail is that InnoDB is clustered-index based: the table data is physically stored in primary-key order, and every secondary index stores the primary key as its row pointer. This makes primary-key lookups very fast and is why a compact, sequential primary key is recommended for InnoDB tables.

Q85. MySQL Default Isolation Level

ROI: ★★★★★ Frequency: Extremely High

Answer


REPEATABLE READ

MySQL's InnoDB engine defaults to REPEATABLE READ, which is a notable contrast with PostgreSQL's READ COMMITTED default, and is a frequent interview gotcha. At this level a transaction sees a consistent snapshot of the data taken at its first read, so repeated reads of the same row return the same value.

Although the SQL standard permits phantom reads at REPEATABLE READ, InnoDB largely prevents them using next-key (gap) locking, which locks ranges to stop concurrent inserts into the gaps a query has read. Knowing both the default and this gap-locking behaviour signals real familiarity with MySQL internals.

Q86. Why InnoDB?

ROI: ★★★★★ Frequency: High

InnoDB is preferred because it provides the features required for serious transactional workloads. It supports full ACID transactions with commit and rollback, row-level locking for high concurrency (rather than table-level locking that serialises writes), and crash recovery through its redo log so committed data survives failures.

It also enforces foreign-key constraints, giving you referential integrity at the database level. Together these make InnoDB the right default for almost every application, whereas the older MyISAM engine, which lacks transactions and uses table-level locks, is now only suitable for niche read-only scenarios.

  • Transactions
  • Row Locks
  • Crash Recovery
  • Foreign Keys

SECTION 17 - DATABASE DESIGN PRINCIPLES

This section is extremely important for Senior Software Engineer interviews. Interviewers are often less interested in syntax and more interested in: "Can this person design a scalable database?"

Q87. What Are The Main Goals Of Database Design?

ROI: ★★★★★ Frequency: Extremely High

Answer

Good database design balances several competing goals. Data integrity ensures the stored data is accurate and follows the business rules, enforced through keys and constraints. Performance ensures common queries run fast, which is shaped by normalization decisions and indexing. Scalability ensures the design can grow with data volume and traffic.

Maintainability keeps the schema understandable and easy to evolve as requirements change, and consistency ensures the same fact is represented the same way everywhere. These goals often pull against each other, normalization aids integrity but can hurt read performance, so the senior skill is making conscious trade-offs appropriate to the workload rather than chasing a single ideal.

  • Data Integrity
  • Performance
  • Scalability
  • Maintainability
  • Consistency
Q88. How Do You Design A Database?

ROI: ★★★★★ Frequency: Extremely High

Framework

A reliable design process starts from the requirements and works toward a physical schema. First identify the entities (the nouns of the domain such as user, job, order), then the relationships between them (one-to-many, many-to-many) and their cardinality. Next define the keys, choosing primary keys and the foreign keys that wire the relationships together.

Then normalize, usually to 3NF, to remove redundancy and protect integrity. After the logical model is sound, add indexes based on the queries the application will actually run, and finally validate the design against those real query patterns, denormalizing or adjusting indexes where measurements show it is needed. Designing from query patterns, not just entities, is what separates a senior approach.

  1. Identify Entities
  2. Identify Relationships
  3. Define Keys
  4. Normalize
  5. Add Indexes
  6. Validate Queries
Q89. Entity vs Table

ROI: ★★★★★ Frequency: High

Answer

An entity is a conceptual object in the problem domain, something the business cares about, such as a Customer, an Order or a Product, identified during analysis and modelling. A table is the physical implementation of that concept inside the relational database, with concrete columns, data types and constraints.

The mapping is usually but not always one-to-one. A single entity may be split across multiple tables for normalization, and a many-to-many relationship between two entities requires an additional junction table that does not correspond to any single business entity. Keeping the conceptual model and the physical schema distinct in your mind helps you design cleanly before worrying about implementation details.

Q90. Natural Key vs Surrogate Key

ROI: ★★★★★ Frequency: High

Natural Key Surrogate Key
Email UUID/ID
Business Meaning System Generated

A natural key is a primary key drawn from real business data that already uniquely identifies a row, such as an email address, a national ID or an ISBN. A surrogate key is an artificial, system-generated identifier with no business meaning, typically an auto-increment integer or a UUID.

Natural keys carry meaning but are risky as primary keys because business values can change (a person changes their email) or turn out not to be as unique as assumed, and any change forces updates to every referencing foreign key. Surrogate keys are stable, compact and meaningless, so they isolate the relational structure from business churn, which is why they are the usual production preference.

Production Preference

Usually surrogate keys are chosen as the primary key, with a unique constraint placed on the natural key (such as email) to still enforce business uniqueness without coupling relationships to it.

Q96. Prevent Duplicate Applications

ROI: ★★★★★ Frequency: Extremely High

Answer

The robust way to prevent a user applying to the same job twice is a composite unique constraint on (user_id, job_id) in the applications table. This pushes the rule into the database, so even concurrent duplicate requests or retries cannot both succeed; the second insert fails with a unique-violation error.

Relying purely on an application-level "check then insert" is unsafe because of the race condition: two simultaneous requests can both pass the check before either inserts. The database constraint is the authoritative guard. The application then simply catches the violation and returns a friendly "already applied" message, combining correctness with good UX.

An e-commerce schema revolves around the order lifecycle. Users browse products, which belong to categories (often a self-referencing hierarchy). When a user checks out, an orders row is created representing the whole purchase, and order_items rows capture each individual product, quantity and the price at the time of sale.

Q98. Why Separate Order And Order Items?

ROI: ★★★★★ Frequency: High

Answer

An order and its line items are separated because they model a one-to-many relationship: a single order can contain many different products, each with its own quantity and price. Trying to cram multiple products into one orders row would violate first normal form and make the data impossible to query cleanly.

The orders table holds information about the purchase as a whole, customer, shipping address, order status, total, timestamp, while each order_items row holds the per-product detail. This separation keeps the schema normalized, lets each line item be tracked and fulfilled independently, and makes reporting on product-level sales straightforward.

Q99. Inventory Design Consideration

ROI: ★★★★★ Frequency: Senior Favorite

Challenge

The central challenge in inventory design is preventing overselling, allowing more units to be sold than are actually in stock, when many customers buy the same scarce item at the same time. This is a classic concurrency problem: two transactions both read "1 in stock" and both proceed to sell it.

Solutions

There are three common approaches. Optimistic locking adds a version column and only commits the decrement if the version is unchanged, retrying on conflict, which works well when contention is low. Pessimistic locking takes an explicit lock on the inventory row (SELECT ... FOR UPDATE) so concurrent buyers serialise, which is simpler but reduces concurrency on hot items.

A reservation system temporarily holds stock for a customer during checkout (with a timeout), decoupling the reserve step from the final purchase, which is what high-scale platforms use. The right choice depends on contention levels and user-experience requirements, and articulating that trade-off is exactly what an interviewer is probing for.

  • Optimistic Locking
  • Pessimistic Locking
  • Reservation Systems

SECTION 20 - BOOKING SYSTEM DESIGN

Q100. Design Movie Ticket Booking System

ROI: ★★★★★ Frequency: Extremely High

Core Tables


movies

theaters

screens

shows

seats

bookings

payments

A movie-booking schema is naturally hierarchical. Theaters contain multiple screens, each screen has a fixed set of seats, and a show represents a specific movie playing on a specific screen at a specific time. Bookings link a user to the seats they reserve for a particular show, and payments record the transaction for each booking.

The key modelling insight is that availability is per show, not per seat in the abstract: the same physical seat is independently bookable for each different show. So the booking (or a show_seats) table ties a seat to a show, and that combination is what must be protected against double booking. Getting this granularity right is the heart of the design.

Q101. Biggest Challenge?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

The biggest challenge is preventing double booking, two users reserving the same seat for the same show at the same instant. Booking is a high-contention concurrency problem, because popular shows trigger many simultaneous attempts to grab the same limited seats the moment tickets open.

This is fundamentally the same class of problem as inventory overselling: without proper concurrency control, two transactions can both see a seat as available and both book it. Solving it correctly while keeping the system responsive under a flood of concurrent requests is what makes booking systems a favourite senior design question.

Q102. How To Prevent Double Booking?

ROI: ★★★★★ Frequency: Extremely High

The foundation is a unique constraint on the (show_id, seat_id) combination, so the database itself rejects any second booking of the same seat for the same show, guaranteeing correctness even under a race. Wrapping the reserve-and-confirm steps in a transaction ensures the check and the write happen atomically.

For concurrency control you choose between pessimistic locking (SELECT ... FOR UPDATE on the seat rows so concurrent buyers serialise) and optimistic locking (a version column with retry on conflict). High-scale systems typically add a temporary seat-reservation step with a short timeout so a seat is held while the user pays, then released if payment is not completed. Combining the database unique constraint with a locking strategy and a reservation flow is the complete, production-grade answer.

  • Optimistic Locking

SECTION 21 - DATABASE TROUBLESHOOTING

Q103. API Suddenly Becomes Slow

ROI: ★★★★★ Frequency: Extremely High

Investigation Flow

When an API suddenly slows down, the senior approach is a systematic, top-down investigation rather than jumping to a guess. Start with application logs to confirm where the latency is, is it the database calls, an external service or the application itself? Then move to database metrics (CPU, connections, active queries) to see if the database is the bottleneck.

Next, examine the execution plans of the suspect queries and the slow query log to find the specific offenders, check whether indexes are being used (a recently changed query or grown table may now scan), and finally review infrastructure metrics like disk I/O, memory and network. Following this chain isolates the root cause methodically; interviewers care more about this structured process than about naming a single magic fix.

  1. Application Logs
  2. Database Metrics
  3. Execution Plans
  4. Slow Query Logs
  5. Index Usage
  6. Infrastructure Metrics
Q104. Database CPU 100%

ROI: ★★★★★ Frequency: Senior Favorite

Check

A database pegged at 100% CPU is almost always doing too much computation per query. The first things to check are slow queries and missing indexes, because a query forced into a full table scan burns CPU reading and filtering far more rows than necessary, and a sudden traffic increase against such a query can saturate the server.

Also investigate lock contention, where transactions pile up waiting and retrying, and full table scans surfaced by the execution plans. Tools like the database's active-query view (pg_stat_activity in PostgreSQL, SHOW PROCESSLIST in MySQL) reveal what is currently running. The fix is usually to add the right index, rewrite the offending query, or address the workload spike, not simply to throw more hardware at it.

  • Slow Queries
  • Missing Indexes
  • Lock Contention
  • Full Table Scans
Q105. Database Connections Exhausted

ROI: ★★★★★ Frequency: High

Possible Causes

Running out of database connections usually points to one of three causes. Connection leaks, where the application opens connections but fails to close or return them to the pool (often due to missing try-with-resources or unclosed transactions), steadily consume the pool until none remain. Long-running queries hold their connections for a long time, so fewer are available for new requests.

A connection pool that is simply too small for the traffic is the third cause. The investigation looks at active versus idle connections and how long they are held. The fixes are to ensure connections are always released, to optimise the slow queries that hold them, and to size the pool appropriately, remembering that an oversized pool can itself overwhelm the database, so the right size is bounded by what the database can handle.

  • Connection Leaks
  • Long Queries
  • Small Connection Pool
Q106. What Is Connection Pooling?

ROI: ★★★★★ Frequency: Extremely High

Answer

Connection pooling is the technique of maintaining a pool of pre-established database connections that the application reuses, rather than opening and closing a brand-new connection for every request. Establishing a connection is expensive, it involves a network handshake, authentication and server-side setup, so reusing connections eliminates that repeated overhead.

The pool hands an idle connection to a request, takes it back when the request finishes, and caps the total number of connections to protect the database from being overwhelmed. In Spring Boot this is handled by HikariCP out of the box. Tuning the pool size correctly is critical: too small starves the application, while too large can exhaust the database's own connection limit and memory.

Q107. Why HikariCP?

ROI: ★★★★★ Frequency: Extremely High

Answer

HikariCP is the default connection pool in Spring Boot because it is exceptionally fast and lightweight. Its codebase is small and highly optimised, with minimal locking and lean bytecode, which makes it consistently outperform older pools like Apache DBCP and C3P0 in benchmarks for both throughput and latency.

It is also reliable and battle-tested, with sensible defaults and important safeguards such as leak detection and connection-validation timeouts. Because Spring Boot auto-configures it, you simply tune properties like maximum-pool-size and connection-timeout. Being able to explain why HikariCP is the default, and which pool-size and timeout settings matter, signals real production awareness.

  • Fast
  • Lightweight
  • Reliable
  • Spring Boot Default
