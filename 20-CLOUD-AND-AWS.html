20 - CLOUD & AWS INTERVIEW HANDBOOK

Almost every backend role today runs in the cloud, and AWS is the most common platform you will be asked about. For a 4-year Senior Java engineer, interviewers are not testing whether you can recite the AWS certification syllabus — they want to know that you understand the building blocks, can pick the right service for a problem, and appreciate the trade-offs in cost, availability and security. The bar is "can this person design and operate a Spring Boot service in production on AWS without making naive mistakes". Expect questions that connect cloud concepts to things you already know — caching, databases, messaging and deployment. For a Senior backend role you should confidently discuss: * The cloud service models (IaaS / PaaS / SaaS) and the Shared Responsibility Model * Compute choices: EC2 vs containers (ECS/EKS/Fargate) vs Lambda * Storage and databases: S3, EBS/EFS, RDS, DynamoDB, ElastiCache * Decoupling with SQS, SNS and EventBridge * Networking and security basics: VPC, security groups, IAM least privilege, secrets * Operating at scale: CloudWatch, Auto Scaling, load balancers and cost awareness They usually continue with a scenario like "you have a Spring Boot API that is getting slow under load — how would you scale it on AWS?", so be ready to reason out loud across compute, caching, the database tier and load balancing.

SECTION 1 - CLOUD FUNDAMENTALS

Q1. What Is Cloud Computing and Why Use It?

ROI: ★★★★☆ Frequency: High

Short Answer

Cloud computing is renting compute, storage, networking and managed services on demand over the internet, instead of buying and running your own data centre. You pay for what you use and can provision or release capacity in minutes through an API.

The real wins for a backend team are elasticity (scale up for Black Friday, scale down at night), no capital expenditure (turn fixed CapEx into variable OpEx), speed (a new environment in minutes, not weeks of hardware procurement), and managed services that remove undifferentiated heavy lifting — you let AWS run the database, the message queue and the patching while you focus on business logic.

Interview Gold Point

The mature framing is "use managed services so the team spends its time on the product, not on operating infrastructure". Naming the trade-off — you give up some control and risk vendor lock-in in exchange for velocity — signals seniority far more than just listing benefits.

Q2. Explain IaaS vs PaaS vs SaaS

ROI: ★★★★☆ Frequency: Very High

Short Answer

They are layers of abstraction describing how much the provider manages versus how much you manage. As you move from IaaS to SaaS, you give up control but also give up operational burden.

Model You Manage Provider Manages AWS Example
IaaS OS, runtime, app, data Hardware, virtualisation, networking EC2, EBS, VPC
PaaS App code and data only OS, runtime, scaling, patching Elastic Beanstalk, RDS, Lambda
SaaS Just configuration / your data Everything else Gmail, Salesforce, Cognito

Interview Favorite

A clean analogy: IaaS is leasing an empty kitchen, PaaS is a kitchen with the appliances installed, SaaS is ordering a finished meal. Note that real architectures mix all three — e.g. your Spring Boot app on Fargate (PaaS-ish), data in RDS (PaaS), auth via Cognito (SaaS).

Q3. What Are Regions, Availability Zones and the Shared Responsibility Model?

ROI: ★★★★★ Frequency: Very High

Short Answer

A Region is a geographic area (e.g. eu-west-1, Ireland). Each Region contains multiple Availability Zones (AZs) — physically separate data centres with independent power and networking, connected by low-latency links. You deploy across multiple AZs so that the failure of one data centre does not take your service down.

The Shared Responsibility Model defines who secures what: AWS is responsible for security of the cloud (the physical hardware, the hypervisor, the managed-service internals), while you are responsible for security in the cloud (your OS patches on EC2, IAM policies, security group rules, encryption settings, and your application code).

Why It Matters in Design

  • Multi-AZ = high availability within a Region (RDS Multi-AZ failover, ASG spanning AZs).
  • Multi-Region = disaster recovery and lower latency for global users, at higher cost and complexity.
  • Data residency and compliance (e.g. GDPR) often dictate which Region you must run in.

Common Mistake

Candidates assume "it's in the cloud, so AWS secures it". AWS secures the data centre, but a public S3 bucket or an over-broad IAM policy is your mistake. The line shifts with the service: on EC2 you patch the OS; on Lambda or RDS, AWS does — but you always own IAM and your data.

SECTION 2 - COMPUTE

Q4. What Is EC2 and When Do You Use It?

ROI: ★★★★☆ Frequency: Very High

Short Answer

EC2 (Elastic Compute Cloud) gives you virtual machines — you pick an instance type (CPU/memory profile), an AMI (the OS image), and you are responsible for patching, the runtime and the app. It is the IaaS workhorse: maximum control and flexibility.

Instance families are tuned for workloads: t (burstable, cheap, dev/low-traffic), m (general purpose, balanced), c (compute-optimised, CPU-heavy), r (memory-optimised, in-memory caches/JVM heaps). Pricing models: On-Demand (flexible, pay per second), Reserved/Savings Plans (1–3 year commit, big discount for steady load), and Spot (up to ~90% off but can be reclaimed — good for fault-tolerant batch work).

When to Use vs Avoid

  • Use EC2 when you need full control of the OS, special networking, GPUs, or you are lifting-and-shifting a legacy app.
  • Prefer containers/serverless for new stateless services — you usually do not want to be hand-patching VMs in 2026.

Interview Gold Point

Mention the pricing trade-off proactively: "steady baseline on Savings Plans, bursty traffic on On-Demand, and Spot for the batch tier". Cost-awareness is a strong senior signal — see also Q16 on right-sizing.

Q5. ECS vs EKS vs Fargate

ROI: ★★★★☆ Frequency: High

Short Answer

All three run containers (this ties directly to the Docker/Kubernetes chapter). The choice is about orchestration engine and how much infrastructure you manage:

Service What It Is Best For
ECS AWS-native container orchestrator, simpler than K8s Teams all-in on AWS who want low operational overhead
EKS Managed Kubernetes control plane Teams that need the K8s ecosystem or multi-cloud portability
Fargate Serverless launch type for ECS or EKS — no servers to manage Not wanting to manage EC2 worker nodes at all

The key insight: Fargate is not a separate orchestrator — it is a compute mode. You can run ECS or EKS on EC2 nodes (you manage and patch the nodes) or on Fargate (AWS runs the nodes; you just declare CPU/memory per task). Fargate trades a little per-unit cost for zero node management.

Interview Favorite

A crisp summary: "ECS for AWS-native simplicity, EKS when you specifically need Kubernetes, and Fargate on top of either when you do not want to manage worker nodes." If a candidate already knows K8s from the DevOps chapter, EKS is just managed K8s — the concepts (Pods, Deployments, Services) carry straight over.

Q6. What Is AWS Lambda / Serverless, and What About Cold Starts?

ROI: ★★★★☆ Frequency: High

Short Answer

Lambda runs your code in response to events (an HTTP request via API Gateway, an S3 upload, an SQS message) with no servers to manage. You are billed per invocation and per millisecond of execution. It scales automatically from zero to thousands of concurrent executions.

A cold start is the latency penalty the first time a function runs after being idle — AWS has to provision an execution environment and initialise your runtime. For the JVM this hurts: class loading and JIT warm-up can add hundreds of milliseconds to seconds. Mitigations: Provisioned Concurrency (keep warm instances ready), smaller deployment packages, GraalVM native images (e.g. Spring Native / Quarkus), or simply choosing a lighter runtime for latency-critical paths.

When to Use Lambda

  • Good fit: spiky or unpredictable traffic, event-driven glue (process an uploaded file, react to a queue), cron jobs, low-to-moderate steady volume.
  • Poor fit: long-running or very high-throughput workloads (the per-request cost overtakes a steady container), latency-critical APIs where cold starts are unacceptable, or anything needing >15 min execution.

Common Mistake

Saying "serverless is always cheaper". It is cheaper for low/spiky usage but a steady, high-volume service is often cheaper on Fargate or EC2. The honest senior answer weighs request volume, the cold-start sensitivity of the JVM, and operational simplicity — not a blanket rule.

Q7. How Would You Deploy a Spring Boot App on AWS?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

There is no single right answer — the senior move is to give the spectrum and justify a default. Pick based on team maturity and traffic shape:

  • Elastic Beanstalk — upload the JAR, AWS provisions EC2 + load balancer + auto scaling. Fastest path; least control.
  • Container on ECS Fargate (my usual default) — build a Docker image, push to ECR, run it as a Fargate service behind an ALB with an Auto Scaling policy. No servers to patch, clean CI/CD story.
  • EKS — if the org already standardises on Kubernetes.
  • EC2 + ASG — when you need full OS control.

A Concrete Fargate Flow

The pipeline ties the Docker chapter and this one together: 

1. Build a Docker image (multi-stage Dockerfile, slim JRE base)
2. Push it to ECR (Elastic Container Registry)
3. ECS Fargate Service runs N tasks across multiple AZs
4. An Application Load Balancer fronts the tasks (HTTPS via ACM cert)
5. Target Auto Scaling on CPU / request count
6. Config via environment variables; secrets from Secrets Manager
7. RDS (Multi-AZ) for the database, ElastiCache for caching
8. Logs and metrics to CloudWatch; alarms on error rate / latency

Health checks matter: expose Spring Boot Actuator /actuator/health and point the ALB target group and ECS health check at it — this mirrors the readiness probe idea from Kubernetes.

Interview Gold Point

Externalise everything (12-factor): build the image once, then inject config per environment via env vars and pull secrets from Secrets Manager at runtime. Never bake DB passwords into the image. That single point connects deployment, security and the 12-factor mindset from the DevOps chapter.

SECTION 3 - STORAGE & DATABASES

Q8. What Is S3, and When Should You NOT Store Data in a Database?

ROI: ★★★★★ Frequency: Very High

Short Answer

S3 (Simple Storage Service) is object storage: you store files ("objects") in "buckets", addressed by a key, accessed over HTTP. It is virtually unlimited, durable (11 nines — 99.999999999%), and cheap. It is not a filesystem and not a block device — you cannot mount it and update a file in place; you PUT and GET whole objects.

The rule: do not store large binary blobs in a relational database. Images, videos, PDFs, user uploads, backups and logs belong in S3. In the DB you store only the S3 key / URL plus metadata (owner, size, content-type, timestamps). Putting blobs in the DB bloats backups, kills query performance, and wastes expensive transactional storage.

Useful S3 Features

  • Storage classes — Standard, Infrequent Access, Glacier (archival) for cost tiering.
  • Lifecycle policies — auto-transition old objects to cheaper tiers or delete them.
  • Pre-signed URLs — let a client upload/download directly to S3 with a time-limited, scoped URL, so file bytes never pass through your app servers.
  • Versioning + encryption (SSE) — protect against accidental deletes and secure data at rest.

Interview Gold Point

Pre-signed URLs are a favourite follow-up: "how do you let a user upload a 2 GB video without it touching your Spring Boot servers?" Answer: the backend generates a pre-signed PUT URL, the browser uploads directly to S3, then notifies your API of the resulting key. Offloads bandwidth and scales effortlessly.

Q9. EBS vs EFS vs S3

ROI: ★★★☆☆ Frequency: Medium-High

Short Answer

These are the three AWS storage shapes — block, file and object. Picking the right one is a common scenario question.

Type Service Attaches To Use Case
Block EBS One EC2 instance (one AZ) OS disk, database storage — like a hard drive
File EFS Many EC2 instances (NFS, multi-AZ) Shared filesystem across servers
Object S3 Anything over HTTP Files, media, backups, static assets

Rule of thumb: EBS = a disk for one machine; EFS = a shared network drive for many machines; S3 = an API for blobs at internet scale. EBS volumes are AZ-bound, so a Multi-AZ design relies on snapshots or replication, not on the volume itself moving.

Interview Favorite

"Block vs file vs object" in one breath wins points. If asked where a self-managed database's data files go, the answer is EBS (low-latency block storage) — never EFS for a high-throughput OLTP database.

Q10. What Is RDS, and How Do Read Replicas Help?

ROI: ★★★★☆ Frequency: Very High

Short Answer

RDS (Relational Database Service) is managed relational databases — PostgreSQL, MySQL, MariaDB, Oracle, SQL Server, plus Aurora. AWS handles provisioning, patching, backups, and failover, so you operate it through configuration rather than running it on EC2 yourself. The schema, queries and indexing (the SQL chapter material) are still entirely your responsibility.

Two features matter for interviews and they solve different problems:

  • Multi-AZ — a synchronous standby in another AZ for high availability. If the primary fails, RDS automatically fails over. The standby does not serve reads; it is purely for resilience.
  • Read Replicas — asynchronous copies for read scaling. You route read-heavy traffic (reports, dashboards) to replicas to offload the primary. Because replication is async, replicas can lag — so they are eventually consistent.

Common Mistake

Confusing Multi-AZ with read replicas. Multi-AZ = availability (standby, no reads); read replicas = scaling reads (lag allowed). A read replica is not a failover target in the same automatic way, and the standby is not a query target. Knowing the distinction is a classic senior differentiator.

Q11. What Is DynamoDB, and When Do You Choose NoSQL?

ROI: ★★★★☆ Frequency: High

Short Answer

DynamoDB is AWS's fully managed key-value / document NoSQL database. It offers single-digit-millisecond latency at essentially unlimited scale, with no servers to manage and automatic partitioning. You design around access patterns, not around a normalised schema — you pick a partition key (and optional sort key) so your queries hit a single partition.

Choose DynamoDB when you need predictable low latency at massive scale, a flexible schema, and your queries are known and key-based — e.g. user sessions, shopping carts, IoT events, leaderboards. Choose RDS when you need complex joins, ad-hoc queries, strong relational integrity and ACID transactions across many tables.

Aspect DynamoDB (NoSQL) RDS (Relational)
Schema Flexible, schema-less Fixed, normalised
Scaling Horizontal, near-unlimited Vertical + read replicas
Queries Key-based; joins are an anti-pattern Rich SQL, joins, aggregations
Best for Known access patterns, huge scale Complex relationships, ad-hoc reporting

Interview Gold Point

The senior framing: "with DynamoDB you model the data to fit the query, not the query to fit the data." If you cannot define your access patterns up front, that is a signal you probably want a relational database. Avoid the trap of treating DynamoDB like SQL — scanning or joining at scale is exactly what it is bad at.

Q12. What Is ElastiCache (Redis) and How Does It Fit Your Caching Strategy?

ROI: ★★★★☆ Frequency: High

Short Answer

ElastiCache is managed in-memory caching, most commonly Redis (also Memcached). It sits between your application and the database to serve hot data with sub-millisecond latency, reducing load on RDS and cutting response times — this is the System Design caching layer made concrete on AWS.

The most common pattern is cache-aside (lazy loading): the app checks the cache first; on a miss it reads the DB, writes the result into the cache with a TTL, then returns it. Redis also serves distributed sessions, rate-limiting counters, leaderboards, and distributed locks. 

// Cache-aside in a Spring service
public Product getProduct(String id) {
    Product cached = redis.get("product:" + id);
    if (cached != null) {
        return cached;                  // cache hit
    }
    Product p = repository.findById(id) // cache miss -> DB
                          .orElseThrow();
    redis.set("product:" + id, p, Duration.ofMinutes(10)); // TTL
    return p;
}

Interview Favorite

Be ready for the hard parts of caching: invalidation (set sensible TTLs, evict on write), stampede (many misses hammering the DB at once — mitigate with locks or request coalescing), and consistency (the cache can serve stale data). "There are only two hard things: cache invalidation and naming things" usually earns a smile and shows you respect the trade-offs.

SECTION 4 - MESSAGING & INTEGRATION

Q13. What Is SQS, and Explain Visibility Timeout and Dead-Letter Queues

ROI: ★★★★☆ Frequency: High

Short Answer

SQS (Simple Queue Service) is a fully managed message queue used to decouple producers from consumers. A producer drops a message on the queue; one or more workers poll and process it. This smooths traffic spikes (the queue acts as a buffer) and lets a slow downstream task run asynchronously — the same async decoupling theme from the microservices chapter.

Two concepts always come up:

  • Visibility Timeout — when a consumer receives a message, it becomes invisible to other consumers for a set period. The consumer must delete the message after successful processing; if it crashes or the timeout expires first, the message reappears and is retried. This is what gives at-least-once delivery — so consumers must be idempotent.
  • Dead-Letter Queue (DLQ) — after a message fails processing N times (the maxReceiveCount), SQS moves it to a separate DLQ so it stops blocking the main queue. You inspect, alert on, and replay DLQ messages.

Queue types: Standard (high throughput, at-least-once, best-effort ordering) and FIFO (exactly-once processing and strict ordering, lower throughput).

Interview Gold Point

The line that lands: "SQS is at-least-once, so I design consumers to be idempotent — e.g. dedupe on a business key or message ID — and I always configure a DLQ so poison messages do not block the queue." That single sentence shows production scars.

Q14. SNS vs SQS vs EventBridge

ROI: ★★★★☆ Frequency: High

Short Answer

All three move messages between components, but the delivery model differs:

Service Model Use Case
SQS Queue (1 message → 1 consumer group, pull) Decouple and buffer work for async processing
SNS Pub/sub fan-out (1 message → many subscribers, push) Broadcast an event to multiple endpoints at once
EventBridge Event bus with content-based routing rules Event-driven architectures, SaaS & AWS event integration

A classic combination is the SNS-to-SQS fan-out: publish one event to an SNS topic, which fans it out to several SQS queues — each downstream service consumes from its own queue at its own pace, and each queue can have its own DLQ. EventBridge goes further with rich rule-based filtering and routing, and built-in integration with dozens of AWS and third-party SaaS event sources.

Interview Favorite

The crisp distinction: SQS is pull and point-to-point; SNS is push and one-to-many; EventBridge is a smart router for event-driven systems. When asked "notify the billing, email and analytics services when an order is placed", the textbook answer is SNS fan-out into three SQS queues (or EventBridge rules).

SECTION 5 - NETWORKING & SECURITY

Q15. Explain VPC, Security Groups, IAM and Secrets Management

ROI: ★★★★★ Frequency: Very High

Networking Basics (VPC)

A VPC (Virtual Private Cloud) is your private, isolated network in AWS. You carve it into subnets: public subnets (routed to an Internet Gateway — for load balancers) and private subnets (no direct internet inbound — for app servers and databases). A typical layout puts the ALB in public subnets, the Spring Boot tasks and RDS in private subnets, so the database is never reachable from the internet.

Security Groups are stateful virtual firewalls attached to instances/ENIs — you allow traffic by port and source (an IP range or, better, another security group). "Stateful" means a reply to an allowed inbound request is automatically allowed out. A good pattern: the ALB's SG allows 443 from the world; the app SG allows 8080 only from the ALB's SG; the RDS SG allows 5432 only from the app SG.

IAM — Identity and Access

  • Users — long-lived identities for humans (avoid for app workloads).
  • Roles — assumable identities with temporary, auto-rotated credentials — the right way for services to get permissions.
  • Policies — JSON documents granting/denying specific actions on specific resources.
  • Least privilege — grant only the permissions actually needed, nothing more.

The big one: use instance/task roles, not access keys. Attach an IAM role to the EC2 instance or ECS task; the AWS SDK in your Spring Boot app then receives temporary credentials automatically — nothing to store, nothing to leak, and they rotate on their own. Hard-coded access keys in code or env files are how breaches happen.

Secrets Manager vs Parameter Store

Never bake DB passwords or API keys into images or source. Use Secrets Manager (built for secrets — supports automatic rotation, e.g. of RDS credentials) or SSM Parameter Store (great for general config; SecureString parameters cover secrets cheaply). The app reads them at startup using its IAM role — this is the AWS realisation of the 12-factor "config in the environment" rule from the DevOps chapter.

Interview Gold Point

Two phrases signal seniority instantly: "least privilege" and "use IAM roles, not long-lived access keys". Add "the database lives in a private subnet, reachable only from the app's security group" and you have demonstrated defence in depth across network and identity layers.

SECTION 6 - OPERATIONS & SCALING

Q16. Explain CloudWatch, Auto Scaling, Load Balancers (ALB vs NLB) and Cost Right-Sizing

ROI: ★★★★★ Frequency: Very High

CloudWatch — Observability

CloudWatch is the AWS monitoring backbone: metrics (CPU, memory, request count, latency), logs (your app logs streamed to log groups), and alarms that fire when a metric breaches a threshold (e.g. p99 latency > 500 ms or 5xx rate too high). Alarms can page you via SNS or trigger an Auto Scaling action. Point your Spring Boot Micrometer/Actuator metrics at CloudWatch for end-to-end visibility.

Auto Scaling Groups

An Auto Scaling Group (ASG) keeps a desired number of instances running across multiple AZs and replaces unhealthy ones. With a target tracking policy you say "keep average CPU at 50%" and AWS adds/removes instances automatically. This gives elasticity (scale out under load, in when quiet, saving money) and self-healing (a failed instance is replaced) — for containers, ECS/EKS provide the equivalent service auto scaling.

Load Balancers — ALB vs NLB

Aspect ALB (Application LB) NLB (Network LB)
OSI Layer Layer 7 (HTTP/HTTPS) Layer 4 (TCP/UDP)
Routing Path/host-based, headers, WebSockets Pure connection forwarding
Performance Rich features, slight overhead Ultra-low latency, millions of req/s
Use For Web apps & REST APIs (the usual choice) Extreme throughput, non-HTTP, static IP

For a Spring Boot REST API, the answer is almost always an ALB — it does HTTP routing, TLS termination, and health checks against /actuator/health. Reach for an NLB only for raw TCP/UDP, extreme throughput, or when you need a fixed static IP.

Cost & Right-Sizing

  • Right-size — use CloudWatch data to drop oversized instances; do not run an m5.4xlarge at 10% CPU.
  • Buy smart — Savings Plans/Reserved for steady baseline, Spot for fault-tolerant batch.
  • Scale to demand — auto scaling so you are not paying for peak capacity 24/7.
  • Tier storage — S3 lifecycle policies to Glacier; delete unattached EBS volumes and old snapshots.

Interview Gold Point

Tying scaling to cost is the senior signal: "I size from CloudWatch metrics, auto scale to demand rather than provisioning for peak, and commit Savings Plans only for the steady baseline." Engineers who talk about the bill — not just availability — stand out, because in the cloud architecture decisions are cost decisions.

QUICK REVISION SHEET

  • IaaS (EC2, you manage OS) < PaaS (RDS/Lambda, manage app only) < SaaS (use it). Mix all three.
  • Region = geography; AZ = isolated data centre. Deploy Multi-AZ for HA. Shared Responsibility: AWS secures of the cloud, you secure in the cloud.
  • Compute ladder: EC2 (control) → ECS/EKS (containers) → Fargate (no nodes) → Lambda (no servers, watch JVM cold starts).
  • Default Spring Boot deploy: Docker image → ECR → ECS Fargate behind an ALB, RDS + ElastiCache, secrets from Secrets Manager.
  • S3 = object storage (11 nines); store blobs in S3, keep only the key in the DB. Pre-signed URLs for direct client uploads.
  • EBS = disk for one instance; EFS = shared network drive; S3 = blobs over HTTP.
  • RDS Multi-AZ = availability (standby, no reads); read replicas = read scaling (async, can lag). Do not confuse them.
  • DynamoDB = key-value NoSQL at scale; model data to fit the query. RDS for joins, ad-hoc queries and ACID.
  • ElastiCache (Redis) = cache-aside with TTLs; mind invalidation, stampede and staleness.
  • SQS = pull queue, at-least-once → idempotent consumers + DLQ; visibility timeout protects in-flight messages.
  • SNS = push fan-out; EventBridge = rule-based event router. Classic pattern: SNS → multiple SQS queues.
  • Security: VPC with public/private subnets, stateful security groups chained, IAM least privilege, instance/task roles — not access keys.
  • Ops: CloudWatch metrics/logs/alarms, Auto Scaling Groups for elasticity + self-healing, ALB (L7) for APIs / NLB (L4) for raw throughput.
  • Cost: right-size from metrics, scale to demand, Savings Plans for baseline + Spot for batch, lifecycle-tier storage.
