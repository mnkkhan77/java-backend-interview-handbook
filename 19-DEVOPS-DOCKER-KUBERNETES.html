SECTION 1 - DOCKER

Q1. What Is Docker and Why Use It?

ROI: ★★★★★ Frequency: Extremely High

Docker packages an application together with its dependencies, runtime and configuration into a single portable image that runs identically anywhere a container engine exists. It solves the classic "works on my machine" problem.

Unlike a virtual machine, a container shares the host OS kernel and isolates only the process — so it starts in milliseconds and uses a fraction of the memory. That lightweight isolation is what makes containers ideal for microservices.

Aspect Container Virtual Machine
Isolation Process-level, shared kernel Full OS per VM
Startup Milliseconds Seconds to minutes
Size MBs GBs

Interview Gold Point

The key distinction: VMs virtualise hardware; containers virtualise the operating system. Containers are not "lightweight VMs" — they are isolated processes, which is why density and speed are so much higher.

Q2. What Is the Difference Between an Image and a Container?

ROI: ★★★★★ Frequency: Very High

An image is a read-only template (the blueprint) built from a Dockerfile. A container is a running instance of an image — you can start many containers from one image, just as you create many objects from one class.

Interview Favorite

The class-vs-object analogy lands well: image = class, container = object. Images are immutable and versioned by tag; containers are ephemeral and should be treated as disposable (any state worth keeping goes in a volume or external store).

Q3. Walk Through a Dockerfile for a Spring Boot App

ROI: ★★★★★ Frequency: Very High

A Dockerfile is the recipe for building an image. Each instruction describes a step: 

FROM eclipse-temurin:21-jre        # base image (Java 21 runtime)
WORKDIR /app                       # working directory inside image
COPY target/app.jar app.jar        # copy the built jar in
EXPOSE 8080                        # document the port
ENTRYPOINT ["java","-jar","app.jar"]  # command run on start

Interview Gold Point

Know the difference between ENTRYPOINT and CMD: ENTRYPOINT sets the fixed executable, CMD provides default arguments that are easy to override at docker run. Also use a slim jre base, not the full jdk, for a smaller production image.

Q4. What Is a Multi-Stage Build and Why Use It?

ROI: ★★★★☆ Frequency: High

A multi-stage build uses one stage to compile the app (with the full JDK and build tools) and a second, clean stage that copies only the resulting artifact into a tiny runtime image. The heavy build tooling never ships to production. 

# Stage 1: build
FROM maven:3.9-eclipse-temurin-21 AS build
WORKDIR /src
COPY . .
RUN mvn -q clean package -DskipTests

# Stage 2: runtime (small)
FROM eclipse-temurin:21-jre
WORKDIR /app
COPY --from=build /src/target/app.jar app.jar
ENTRYPOINT ["java","-jar","app.jar"]

Interview Favorite

The payoff is a dramatically smaller, more secure final image — less attack surface, faster pulls, no compilers or source code shipped to production. This is a standard senior-level best practice.

Q5. How Do Docker Image Layers and Caching Work?

ROI: ★★★★☆ Frequency: Medium-High

Each Dockerfile instruction creates a cached layer. On rebuild, Docker reuses cached layers up to the first instruction that changed, then rebuilds everything after it. Ordering instructions from least- to most-frequently-changed maximises cache hits.

Interview Gold Point

The practical tip: copy your dependency descriptor (pom.xml) and download dependencies before copying source code. Then a code change reuses the cached dependency layer instead of re-downloading the world — much faster builds.

Q6. What Is docker-compose?

ROI: ★★★★☆ Frequency: Medium-High

docker-compose defines and runs multi-container applications from a single YAML file — for example your Spring Boot service plus a Postgres database and Redis — started together with one command (docker compose up).

Interview Favorite

Compose is ideal for local development and integration tests (it pairs naturally with Testcontainers). It is not a production orchestrator — for production scale and self-healing you move to Kubernetes.

SECTION 2 - KUBERNETES

Q7. What Is Kubernetes and What Problem Does It Solve?

ROI: ★★★★★ Frequency: Very High

Kubernetes (K8s) is a container orchestrator. Once you have many containers across many machines, you need something to schedule them, restart failed ones, scale them up and down, roll out new versions safely, and route traffic to them. K8s does all of this declaratively.

You describe the desired state ("I want 3 replicas of this image") and Kubernetes continuously works to make reality match — restarting crashed pods and rescheduling them if a node dies. This self-healing is its core value.

Interview Gold Point

The phrase to use is "declarative desired-state reconciliation". You do not tell K8s how to do things step by step; you declare the end state and its control loop converges toward it.

Q8. Explain Pod vs Deployment vs Service

ROI: ★★★★★ Frequency: Very High

Object Role
Pod Smallest unit — one or more containers sharing network/storage
Deployment Manages a set of identical pods: replicas, rollouts, self-healing
Service Stable network endpoint + load balancing across the pods

Pods are ephemeral and get new IPs when recreated, so you never talk to a pod directly. A Service gives a stable DNS name and load-balances across whichever pods are currently healthy.

Interview Favorite

A common follow-up: "why not just run a container directly?" — because a bare pod is not recreated if it dies. A Deployment guarantees the declared number of replicas and handles rolling updates and rollbacks.

Q9. Liveness vs Readiness Probes

ROI: ★★★★★ Frequency: High

Both are health checks, but they trigger different actions:

  • Liveness probe — "is the app alive?" If it fails, Kubernetes restarts the container (recovers from deadlocks/hangs).
  • Readiness probe — "is the app ready to serve traffic?" If it fails, K8s stops routing requests to that pod but does not restart it.

Interview Gold Point

Spring Boot Actuator exposes /actuator/health/liveness and /actuator/health/readiness out of the box — tie this straight back to the Actuator content in your Spring Boot chapter. Getting the two probes mixed up causes restart loops or traffic sent to a not-yet-ready app, so the distinction matters in production.

Q10. What Are ConfigMaps and Secrets?

ROI: ★★★★☆ Frequency: Medium-High

Both externalise configuration out of the image so the same image runs in every environment. A ConfigMap holds non-sensitive config (URLs, feature flags). A Secret holds sensitive data (passwords, API keys, tokens) and is handled more carefully.

Interview Favorite

Be honest about Secrets: by default they are only base64-encoded, not encrypted. Production setups enable encryption-at-rest and often integrate an external vault (e.g. HashiCorp Vault, AWS Secrets Manager). Saying this shows real-world awareness.

Q11. How Does a Rolling Update Work?

ROI: ★★★★☆ Frequency: Medium-High

A rolling update replaces old pods with new ones gradually — spinning up new-version pods, waiting for their readiness probes to pass, then terminating old pods a few at a time. The service stays available throughout, and if the new version is unhealthy you can roll back.

Interview Gold Point

This is why readiness probes matter: K8s only shifts traffic to a new pod once it reports ready, guaranteeing zero-downtime deploys. kubectl rollout undo reverts to the previous version if something goes wrong.

Q12. Rolling vs Blue-Green vs Canary Deployments

ROI: ★★★★☆ Frequency: Medium-High

Strategy How It Works
Rolling Gradually replace old pods with new ones
Blue-Green Run two full environments; switch traffic all at once
Canary Send a small % of traffic to the new version, then ramp up

Blue-green gives instant rollback (just switch back) at the cost of double the infrastructure. Canary limits blast radius by exposing the new version to a few users first and watching metrics before a full rollout.

Interview Favorite

Match the strategy to risk: canary for high-risk changes where you want real-traffic validation, blue-green when you need an instant, clean cutover, rolling as the sensible low-cost default.

SECTION 3 - CI/CD & BUILD

Q13. What Is CI/CD?

ROI: ★★★★★ Frequency: Very High

CI (Continuous Integration) means every code push is automatically built and tested, so integration problems surface within minutes instead of at a painful end-of-sprint merge. CD (Continuous Delivery/Deployment) automates releasing that validated build to environments — delivery stops at a manual approval, deployment goes all the way to production automatically.

Typical Pipeline Stages

Commit -> Build -> Unit Tests -> Package (Docker image)
       -> Integration Tests -> Push to Registry
       -> Deploy to Staging -> Deploy to Production

Interview Gold Point

Name tools you have touched — Jenkins, GitHub Actions, GitLab CI. The core value to articulate: fast feedback, repeatable releases, and smaller, safer changes shipped frequently rather than big risky batches.

Q14. What Is the 12-Factor App / Externalised Configuration?

ROI: ★★★★☆ Frequency: Medium-High

The 12-Factor App is a set of principles for building cloud-native services. The one interviewers probe most is config in the environment: never hard-code environment-specific values (DB URLs, credentials) in the build — inject them at runtime via environment variables or config services.

This is what lets the same immutable image run in dev, staging and production, differing only by injected config. Other key factors: stateless processes, treat backing services as attached resources, and disposability (fast startup/graceful shutdown).

Interview Gold Point

Connect to Spring: profiles and application.yml overridden by environment variables, or Spring Cloud Config for centralised config, are exactly this principle in practice. "Build once, deploy anywhere with injected config" is the phrase to land.

Q15. Maven vs Gradle

ROI: ★★★☆☆ Frequency: Medium

Aspect Maven Gradle
Config XML (pom.xml), declarative Groovy/Kotlin DSL, flexible
Speed Slower Faster (incremental + build cache)
Learning curve Simpler, rigid lifecycle Steeper, more powerful

Both manage dependencies and the build lifecycle. Maven's rigid, convention-driven lifecycle (validate → compile → test → package → install → deploy) is predictable; Gradle trades some predictability for speed and scripting flexibility.

Interview Favorite

Know dependency scopes (Maven) / configurations (Gradle): compile/implementation, provided/compileOnly, test, runtime. A frequent follow-up is how transitive dependency conflicts are resolved and how to exclude a transitive dependency.

QUICK REVISION SHEET

  • Container = isolated process (shares kernel); VM = full OS. Image = class, container = object
  • Dockerfile builds an image; ENTRYPOINT (fixed) vs CMD (overridable args)
  • Multi-stage build → small, secure runtime image (no build tools shipped)
  • Order Dockerfile steps for layer cache hits; copy deps before source
  • docker-compose = local multi-container; Kubernetes = production orchestration
  • K8s = declarative desired-state reconciliation + self-healing
  • Pod (smallest unit) < Deployment (replicas/rollout) + Service (stable endpoint)
  • Liveness probe restarts; readiness probe gates traffic
  • ConfigMap = config, Secret = sensitive (base64, not encrypted by default)
  • Deploys: rolling (default), blue-green (instant switch), canary (gradual %)
  • CI = auto build+test on push; CD = automated release pipeline
  • 12-factor: config in the environment → build once, deploy anywhere
