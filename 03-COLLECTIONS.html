SECTION 1 - COLLECTION FRAMEWORK OVERVIEW

Q1. What Is Collection Framework?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

The Collection Framework is a set of interfaces and classes used to store and manipulate groups of objects.

Rather than every project inventing its own list, set or map, Java provides a unified, well-tested set of data structures with a consistent API. This lets you swap one implementation for another (for example moving from an ArrayList to a LinkedList) with minimal code changes, because they share the same interfaces.

Main Interfaces


Collection

 ├── List

 ├── Set

 └── Queue

Map
(separate hierarchy)

A key detail interviewers like to confirm is that Map is intentionally NOT a sub-interface of Collection. A Collection stores individual elements, whereas a Map stores key-value pairs, so the two have fundamentally different contracts and sit in separate hierarchies. List, Set and Queue all extend Collection and inherit its element-based operations.

Benefits

  • Reusable Data Structures
  • Standardized APIs
  • Optimized Implementations
  • Reduced Development Effort

Because the implementations are written and tuned by the JDK team, you inherit decades of optimization for free. The standardized API also means that generic algorithms (sorting, searching, iteration) work across any conforming collection, which is exactly why code written against the interfaces rather than the concrete classes is more flexible and maintainable.

A List is an ordered sequence that allows duplicates and supports positional (index-based) access, which makes it the right choice when order matters or the same value can legitimately appear more than once. A Set models a mathematical set: it rejects duplicates and, depending on the implementation, may be unordered (HashSet), insertion-ordered (LinkedHashSet) or sorted (TreeSet). A Map stores associations between unique keys and their values, so it is the structure you reach for whenever you need fast lookup by some identifier.

SECTION 2 - ARRAYLIST

Q3. How ArrayList Works Internally?

ROI: ★★★★★ Frequency: Extremely High

Short Answer

ArrayList is backed by a dynamic array. It stores elements contiguously in an internal Object[] and grows that array automatically as more elements are added.

Internal Structure


Object[]

Process

  1. Array Created
  2. Elements Added
  3. Capacity Exhausted
  4. New Larger Array Created
  5. Elements Copied

When the backing array fills up, ArrayList allocates a new, larger array (in modern Java roughly 1.5x the old size), copies every existing element across using System.arraycopy, and discards the old array. This growth-and-copy step is what makes a single add() occasionally expensive, but because it happens rarely the cost is spread out across many additions, giving the amortized O(1) append that the framework guarantees.

Because elements live in a contiguous block, ArrayList supports true O(1) random access by index: the position is computed by simple pointer arithmetic. The trade-off is that inserting or removing in the middle requires shifting all subsequent elements, which is O(n).

Interview Gold Point

ArrayList default capacity is 10 (Java 8+ allocates lazily). The empty array is shared until the first element is added, at which point the backing array is created with capacity 10. This lazy allocation avoids wasting memory on ArrayLists that are created but never populated, which is a subtle but real optimization in large applications that create many short-lived lists.

Q4. ArrayList vs LinkedList

ROI: ★★★★★ Frequency: Extremely High

Feature ArrayList LinkedList
Storage Dynamic Array Doubly Linked List
Random Access Fast O(1) Slow O(n)
Insertion Middle Costly Better
Memory Usage Lower Higher

ArrayList stores elements in a contiguous array, so it excels at index-based access and is cache-friendly because neighbouring elements sit together in memory. LinkedList stores each element in a separate node holding pointers to the previous and next node, so it can splice in or remove an element in O(1) once you already hold a reference to the position, but reaching that position still requires walking the list in O(n).

LinkedList uses more memory because every element carries the overhead of two node pointers, and its scattered nodes cause more cache misses. In practice this means even LinkedList's theoretical insertion advantage rarely materialises unless you are inserting via an iterator already positioned at the spot.

Interview Favorite

Which one is used more often? Answer: ArrayList. Most applications perform more reads than middle insertions, and ArrayList's contiguous memory layout makes it faster in real-world workloads even when Big-O might suggest otherwise. A strong candidate adds that LinkedList is mainly justified when it is used as a Deque or Queue (frequent add/remove at the ends), not as a general-purpose list.

SECTION 3 - HASHMAP

Q5. How HashMap Works Internally?

ROI: ★★★★★ Frequency: Interviewer's Favorite Question

Short Answer

HashMap stores data using buckets determined by hash values. Internally it is an array of buckets (the table), and the bucket for a key is chosen from that key's hash code, which is what gives HashMap its average O(1) lookup.

Insertion Process

Bucket Selection

↓

Store Entry

Internal Structure

Bucket 1

Bucket 2

Bucket 3

...

On put(), HashMap calls the key's hashCode(), then applies an internal spreading function that mixes the high bits into the low bits (this reduces clustering when hash codes differ mainly in their upper bits). The result is masked against the table length to compute a bucket index. The entry is then stored in that bucket, and if the key already exists, its value is overwritten rather than duplicated.

On get(), the same calculation locates the bucket, and then equals() is used to find the exact matching key within that bucket. This is why both hashCode() and equals() must be implemented correctly and consistently for any object used as a key.

Java 8 Improvement

Large collision chains are converted into Red Black Trees. Before Java 8, all entries that collided in the same bucket were kept in a linked list, so a pathological case (many keys hashing to one bucket) degraded lookups to O(n). From Java 8 onward, once a bucket exceeds a threshold (8 entries, with the table at least 64 in size) the chain is converted into a balanced red-black tree, improving worst-case bucket lookup from O(n) to O(log n). If the bucket later shrinks, it reverts back to a linked list.

Q6. Explain HashMap put() Operation

ROI: ★★★★★ Frequency: Extremely High

Step By Step

  1. Compute hashCode()
  2. Calculate bucket index
  3. Locate bucket
  4. Check existing key
  5. Update or Insert

The key's hashCode() is computed and spread, then reduced to a bucket index using index = (table.length - 1) & hash. HashMap walks the entries already in that bucket and compares each one with the incoming key using both hash equality and equals(). If a matching key is found, only the value is replaced and the old value is returned; otherwise a new entry is appended. After insertion, if the size crosses the resize threshold, the table is rehashed.

Expected Follow-Up

How is bucket calculated? How are collisions handled? When does rehashing happen?

These three follow-ups almost always come together. Be ready to say: the bucket is calculated as (n - 1) & hash where n is the table length; collisions are handled by chaining entries in a linked list that converts to a red-black tree past the treeify threshold; and rehashing happens when size exceeds capacity multiplied by the load factor (default 0.75).

Q7. What Is Hash Collision?

ROI: ★★★★★ Frequency: Extremely High

Answer

A hash collision occurs when two different keys produce the same bucket index. Because the space of possible hash codes is mapped down onto a small fixed number of buckets, collisions are unavoidable in any real hash table; the goal of a good hashCode() is simply to make them rare and evenly distributed.

Example


Key A

↓

Bucket 3

Key B

↓

Bucket 3

Collision occurs. Both keys land in Bucket 3, so they must coexist in the same bucket. HashMap keeps them distinct by storing both entries there and using equals() to tell them apart on lookup. A high collision rate is what turns HashMap's average O(1) into something slower, which is why collision handling is so heavily probed in interviews.

Handling

  • Linked List
  • Red Black Tree (Java 8+)

Small buckets keep colliding entries in a simple linked list. From Java 8, once a single bucket grows beyond the treeify threshold (8 entries, table size at least 64), that bucket is restructured into a red-black tree so even a heavily collided bucket stays at O(log n) instead of degrading to a linear scan.

Q8. What Is Load Factor?

ROI: ★★★★★ Frequency: Extremely High

Answer

The load factor determines when a HashMap should resize. It is the ratio of stored entries to bucket capacity at which the map decides it is getting too full and grows its table. It is a deliberate trade-off between memory usage and lookup speed: a lower load factor wastes more memory but reduces collisions, while a higher one packs entries more tightly at the cost of slower lookups.

Default Value


0.75

The default of 0.75 is chosen as a sweet spot that keeps collision probability low while still using memory reasonably. It is rarely worth changing in practice unless you have profiled a specific need.

Example


Capacity = 16

Threshold =

16 * 0.75

= 12

After 12 entries, resize occurs. With the default capacity of 16, the resize threshold is 16 multiplied by 0.75, which is 12. The thirteenth insertion triggers a resize that doubles the table. Understanding this is what lets you pre-size a map correctly to avoid mid-population resizing.

Q9. What Is Rehashing?

ROI: ★★★★★ Frequency: Extremely High

Answer

Rehashing is what happens when the load factor threshold is crossed:

  1. Capacity doubled
  2. New buckets created
  3. Existing entries redistributed

The table capacity is doubled (it is always a power of two), a new bucket array is allocated, and every existing entry is moved into the larger table. Because the bucket index depends on the table length, each entry's index must be recomputed against the new size, which is why this is called rehashing.

Performance Impact

Rehashing is expensive. It is an O(n) operation that touches every entry, and if it happens repeatedly while a map is being filled, the cumulative cost can dominate. This is exactly why pre-sizing matters for large, predictable datasets.

Senior Engineer Tip

If the expected size is known, initialize capacity appropriately so the map never has to resize while being populated. Note the subtlety: passing the expected number of elements directly is not quite right because of the 0.75 load factor; to truly avoid a resize you size the map for expectedSize / 0.75. Pre-sizing is a cheap, common production optimization for bulk loads. 


new HashMap<>(1000)
Q10. Why Is HashMap Not Thread Safe?

ROI: ★★★★★ Frequency: Extremely High

Answer

HashMap is not thread safe because multiple threads can modify its internal structure simultaneously, and none of its operations are synchronized. If two threads call put() at the same time and both trigger structural changes (such as a resize), they can interleave in ways the class was never designed to handle.

This can lead to:

  • Lost Updates
  • Data Corruption
  • Race Conditions

Beyond lost updates, the classic danger in older Java versions was an infinite loop during concurrent resizing, where the linked-list rehashing logic could form a cycle and peg a CPU core at 100 percent. The safe options are to use ConcurrentHashMap for genuine concurrent access, or Collections.synchronizedMap for coarse external locking. You should never share a plain HashMap across threads without external synchronization.

Q11. Can HashMap Have Null Keys?

ROI: ★★★★★ Frequency: High

Answer

Yes. One null key is allowed, and multiple null values are allowed.

HashMap handles the null key as a special case: because calling hashCode() on null would fail, it simply assigns the null key to bucket 0 rather than computing a hash. This is a deliberate convenience of HashMap and a frequent contrast point with ConcurrentHashMap and Hashtable, both of which forbid null keys and values entirely. Knowing which collections permit null is a common quick-fire interview check.

Q12. Why Override hashCode() And equals()?

ROI: ★★★★★ Frequency: Extremely High

Answer

Hash-based collections depend on both methods. hashCode() decides which bucket an object goes into, and equals() decides whether two objects in the same bucket are actually the same key. If you use a custom object as a key or store it in a set, both methods must agree on what "equal" means.

Incorrect implementation can break:

  • HashMap
  • HashSet
  • Caching
  • Duplicate Detection

The contract is strict: if two objects are equal according to equals(), they must return the same hashCode(). Violate it and you get the infamous bug where you put an object into a HashMap and then cannot get it back, because the map looks in the wrong bucket. The same flaw silently allows duplicates into a HashSet and corrupts any cache keyed on the object.

Interview Trap

Override equals() but forget hashCode(). This is wrong. It is the single most common mistake interviewers probe for. With equals() overridden and hashCode() left as the default identity-based implementation, two logically equal objects produce different hash codes, land in different buckets, and the map fails to find them. Always override the two together, using the same set of fields in both.

SECTION 4 - HASHSET

Q13. How HashSet Works Internally?

ROI: ★★★★★ Frequency: Extremely High

Answer

HashSet internally uses a HashMap. It is essentially a thin wrapper: every element you add to the set becomes a key in a backing HashMap, which is why HashSet inherits all of HashMap's performance characteristics and its rules around hashCode() and equals().

Conceptually


HashMap

The value is a dummy object (a single shared constant named PRESENT in the JDK source), and the key stores the actual data. Because the map only cares about keys for uniqueness, the value carries no information; it merely satisfies the map's key-value structure. This design is a neat example of reuse: rather than reimplement hashing, the set leans entirely on the already-optimized HashMap.

Duplicate Prevention

hashCode() and equals() determine uniqueness. When you add an element, the backing map's put() is called, and the map's own collision logic checks whether an equal key already exists. If it does, the put is treated as an update of an existing key rather than a new entry, so the set's size does not grow, which is how duplicates are silently rejected.

Q14. Why Does HashSet Not Allow Duplicates?

ROI: ★★★★★ Frequency: High

Answer

Before insertion, HashSet checks:

  • hashCode()
  • equals()

If a matching element exists, the insertion is rejected. Concretely, add() delegates to the backing HashMap's put(): the element's hashCode() locates the bucket, and equals() compares it against any element already there. A match means the map sees the same key, so it overwrites the (dummy) value instead of adding a new entry, and add() returns false to tell you nothing was inserted.

This is why duplicate detection is only as reliable as your hashCode() and equals() implementations. If a custom class implements them inconsistently, the set will happily store what you consider duplicates, because the collection's notion of equality comes entirely from those two methods.

SECTION 5 - TREEMAP

Q15. What Is TreeMap?

ROI: ★★★★★ Frequency: Very High

Short Answer

TreeMap is a sorted Map implementation based on a Red-Black Tree. Unlike HashMap, which stores keys in an unpredictable order, TreeMap keeps its keys continuously sorted, either by their natural ordering or by a Comparator you supply.

Characteristics

  • Keys Sorted Automatically
  • No Null Keys
  • O(log n) Operations
  • Based On Red-Black Tree

Because it is a self-balancing binary search tree, all core operations (get, put, remove) cost O(log n) rather than HashMap's O(1), which is the price you pay for ordering. It also rejects null keys, because the map must compare every key to position it in the tree and there is no sensible way to compare null. In return you gain ordered navigation methods like firstKey(), lastKey(), floorKey(), ceilingKey() and subMap(), which are impossible on a HashMap.

Example


TreeMap map =
new TreeMap<>();

map.put(3,"C");
map.put(1,"A");
map.put(2,"B");

Output


1=A
2=B
3=C

Note that the entries were inserted in the order 3, 1, 2, yet iteration returns them sorted as 1, 2, 3. The tree positions each key on insertion, so iteration always reflects sorted order regardless of how the data arrived.

Q16. HashMap vs TreeMap

ROI: ★★★★★ Frequency: Extremely High

Feature HashMap TreeMap
Ordering No Sorted
Structure Hash Table Red Black Tree
Performance O(1) O(log n)
Null Key Yes No

The fundamental trade-off is ordering versus speed. HashMap gives average constant-time access but no ordering guarantee at all, while TreeMap guarantees sorted keys at the cost of logarithmic-time operations. HashMap permits a single null key; TreeMap permits none because every key must be comparable. Choose based on whether you actually need ordered traversal or range queries.

Interview Favorite

When should TreeMap be used? Answer: when sorted data is required. More precisely, reach for TreeMap when you need keys kept in order, or when you need navigation and range operations such as "all entries between X and Y" or "the closest key not greater than X." If you only need fast key lookups and ordering is irrelevant, HashMap is the better default.

SECTION 6 - TREESET

Q17. What Is TreeSet?

ROI: ★★★★★ Frequency: High

Answer

TreeSet is a sorted Set implementation, internally backed by a TreeMap. Just as HashSet wraps a HashMap, TreeSet wraps a TreeMap, storing each element as a key with a shared dummy value. This is why it shares TreeMap's ordering behaviour and logarithmic performance.

Properties

  • No Duplicates
  • Sorted
  • Red Black Tree
  • O(log n)

A TreeSet keeps its elements unique (set semantics) and continuously sorted (tree semantics), and like TreeMap it rejects null and requires elements to be comparable, either through Comparable or a supplied Comparator. It also exposes navigation methods such as first(), last(), ceiling() and floor(), which make it ideal when you need an ordered, duplicate-free collection with range queries.

Q18. HashSet vs TreeSet

ROI: ★★★★★ Frequency: High

HashSet TreeSet
Unordered Sorted
O(1) O(log n)
HashMap TreeMap

HashSet offers average constant-time add, remove and contains but makes no promise about iteration order, while TreeSet keeps elements sorted at the cost of logarithmic-time operations. The choice mirrors the HashMap versus TreeMap decision: use HashSet when you only need uniqueness and fast membership tests, and TreeSet when you also need the elements ordered or need range-based navigation. HashSet additionally allows a single null element, whereas TreeSet does not.

SECTION 7 - QUEUE AND DEQUE

Q19. What Is Queue?

ROI: ★★★★★ Frequency: High

Answer

A Queue follows FIFO: First In, First Out. The element that has been waiting the longest is the next one to be removed, which models any real-world line where items are served in arrival order.

Example


offer()

poll()

peek()

The Queue interface deliberately provides two styles of methods. offer(), poll() and peek() return a special value (false or null) when the operation cannot be performed, whereas their counterparts add(), remove() and element() throw an exception instead. The non-throwing variants are usually preferred in production because they let you handle an empty or full queue with normal control flow rather than exception handling.

Real Project Example

Request Processing, Task Scheduling, and Messaging Systems. Queues are the backbone of asynchronous processing: incoming requests are buffered and consumed at a controlled rate, background jobs are scheduled and drained by worker threads, and message brokers like Kafka or RabbitMQ are essentially distributed queues. In-memory, a BlockingQueue is the standard way to hand work safely between producer and consumer threads.

Q20. What Is PriorityQueue?

ROI: ★★★★★ Frequency: High

Answer

In a PriorityQueue, elements are ordered by priority rather than insertion order. The head of the queue is always the smallest element according to natural ordering or a supplied Comparator, so polling repeatedly returns elements in priority order rather than the order they were added.

Internal Structure


Binary Heap

It is implemented as a binary heap stored compactly in an array, which is why it can find the highest-priority element so cheaply while still keeping inserts efficient. Note that a PriorityQueue is only ordered with respect to its head: iterating over it does not produce sorted order, a detail interviewers occasionally test.

Performance

Insert O(log n)
Remove O(log n)
Peek O(1)

Inserting and removing the head each cost O(log n) because the heap must sift the affected element up or down to restore the heap property, while peeking at the head is O(1) since it is simply the array's first element. PriorityQueues are commonly used for scheduling, Dijkstra-style shortest-path algorithms, and any "process the most important item next" workload.

Q21. What Is Deque?

ROI: ★★★★☆ Frequency: Medium

Answer

A Deque is a Double Ended Queue, where insertion and removal are possible from both ends. This makes it a flexible structure that can behave as a FIFO queue (add at one end, remove from the other) or as a LIFO stack (add and remove at the same end), which is why ArrayDeque is the recommended modern replacement for the legacy Stack class.

Implementations


ArrayDeque

LinkedList

ArrayDeque is backed by a resizable circular array and is generally the faster, more memory-efficient choice for both stack and queue use. LinkedList also implements Deque via its node structure and is useful when you specifically need the List and Deque behaviours together. In most new code, prefer ArrayDeque over both Stack and LinkedList for end-based operations.

SECTION 8 - ITERATORS

Q22. Iterator vs ListIterator

ROI: ★★★★★ Frequency: High

Iterator ListIterator
Forward Only Both Directions
All Collections List Only
No Add Add Supported

Iterator is the general-purpose cursor available on any Collection; it traverses forward only and supports reading and removing elements via next() and remove(). ListIterator is a more powerful variant available only on List implementations: it can move in both directions with next() and previous(), report element indices, and additionally add() or set() elements during traversal. In short, ListIterator is the right tool when you need bidirectional movement or in-place modification while iterating a list.

Q23. What Is Fail Fast Iterator?

ROI: ★★★★★ Frequency: Extremely High

Answer

A fail-fast iterator throws an exception if the collection is structurally modified during iteration. It works by tracking a modification counter (modCount) on the collection; each call to next() checks that this counter has not changed since the iterator was created, and if it has, the iterator fails immediately rather than risking unpredictable behaviour.

Example


List list =
new ArrayList<>();

for(String s : list){

   list.add("Java");

}

Result


ConcurrentModificationException

Modifying the list inside an enhanced for-loop changes modCount, so the next iteration step detects the mismatch and throws ConcurrentModificationException. The correct way to remove during iteration is to use the iterator's own remove() method, or to collect changes separately and apply them after the loop. Note the exception name is misleading: this is fail-fast behaviour on a single thread too, not strictly a concurrency feature.

Q24. What Is Fail Safe Iterator?

ROI: ★★★★★ Frequency: Extremely High

Answer

A fail-safe iterator works on a copy (or a stable snapshot) of the collection, so modification during iteration does not throw an exception. Because the iterator traverses its own view of the data, structural changes made to the live collection simply do not affect the ongoing iteration.

Example


CopyOnWriteArrayList

CopyOnWriteArrayList and ConcurrentHashMap provide fail-safe iteration. The trade-off is that the iterator may not reflect the most recent updates (it sees the snapshot taken when iteration began) and, in the copy-on-write case, every write allocates a fresh array. This is why fail-safe collections are best for concurrent, read-heavy scenarios where avoiding exceptions matters more than seeing live updates.

Q25. Fail Fast vs Fail Safe

ROI: ★★★★★ Frequency: Extremely High

Fail Fast Fail Safe
Throws Exception No Exception
Original Collection Copy Used
Faster More Memory

Fail-fast iterators (ArrayList, HashMap, HashSet) operate directly on the original collection and throw ConcurrentModificationException the moment they detect concurrent structural change, which surfaces bugs early and keeps overhead low. Fail-safe iterators (CopyOnWriteArrayList, ConcurrentHashMap) iterate over a separate copy or snapshot, so they never throw but consume more memory and may show slightly stale data. The practical guidance: fail-fast is the default for single-threaded code, while fail-safe collections are designed for concurrent access where you cannot afford an exception mid-iteration.

SECTION 9 - CONCURRENTHASHMAP

Q26. Why Was ConcurrentHashMap Introduced?

ROI: ★★★★★ Frequency: Extremely High

Answer

HashMap is not thread safe, so it cannot be shared across threads without external synchronization. The older alternative, Hashtable, is thread safe but synchronizes the entire structure on a single lock, meaning only one thread can read or write at a time even if they are touching completely unrelated keys. This makes Hashtable a severe bottleneck under concurrency.

ConcurrentHashMap was introduced to provide thread safety with far better concurrency. Instead of locking the whole map, it locks only the small portion being modified, allowing many threads to read and write different parts of the map simultaneously. This gives it near HashMap-level throughput while remaining safe, which is why it is the standard choice for concurrent maps in modern applications.

Q27. HashMap vs Hashtable vs ConcurrentHashMap

ROI: ★★★★★ Frequency: Extremely High

Feature HashMap Hashtable ConcurrentHashMap
Thread Safe No Yes Yes
Performance Fastest Slow High
Null Key Yes No No

HashMap is the fastest because it does no locking, but it is unsafe for concurrent use. Hashtable is a legacy class that achieves safety by synchronizing every method on one lock, so it is correct but slow and effectively single-threaded. ConcurrentHashMap is the modern thread-safe option: it locks at a fine granularity (individual buckets) so it stays fast under heavy concurrency. Both Hashtable and ConcurrentHashMap forbid null keys and null values, whereas HashMap permits one null key. In new code, prefer HashMap for single-threaded use and ConcurrentHashMap for concurrent use; Hashtable is essentially obsolete.

Q28. How ConcurrentHashMap Works Internally?

ROI: ★★★★★ Frequency: Senior Interview Favorite

Java 7

Java 7 used segment-based locking. The map was divided into a fixed number of independent segments (16 by default), each guarding its own portion of the table with its own lock. Two threads writing to different segments could proceed in parallel, so concurrency was limited by the number of segments rather than reduced to a single global lock. 


Segment 1

Segment 2

Segment 3

Java 8+

Java 8 onward replaced segments with fine-grained locking at the bucket level. Writes use CAS (compare-and-swap) operations for empty buckets and synchronize only on the first node of a specific bucket when a collision chain exists. This pushes concurrency far higher because the effective lock granularity is now a single bucket rather than a whole segment, and reads are generally lock-free.

  • Fine-Grained Locking
  • CAS Operations
  • Synchronized Buckets

Interview Gold Point

Most candidates still answer using Java 7 segmentation, and modern JVMs use a different implementation. If you confidently explain the Java 8 shift to CAS plus per-bucket synchronization, you immediately stand out from the majority who only memorised the older segment model. Mentioning that reads are essentially lock-free and that treeification also applies inside ConcurrentHashMap signals genuine depth.

Q29. Why ConcurrentHashMap Does Not Allow Null?

ROI: ★★★★★ Frequency: High

Answer

The reason is ambiguity in a concurrent environment. Consider a lookup: 


map.get(key)

If get() returns null, does it mean the key is absent, or that the key is present with a null value? In a single-threaded HashMap you can disambiguate by calling containsKey(), but in a concurrent map another thread could insert or remove the key between your get() and your containsKey() call, making the check unreliable. Forbidding null entirely removes this race condition: a null return can only ever mean "key absent," which keeps the contract unambiguous and safe under concurrency. The same reasoning is why null values are also disallowed.

SECTION 10 - COPYONWRITEARRAYLIST

Q30. What Is CopyOnWriteArrayList?

ROI: ★★★★★ Frequency: High

Answer

CopyOnWriteArrayList is a thread-safe List implementation in which every write operation creates a new copy of the underlying array. Readers always work against an immutable snapshot, so they never need a lock and never see a partially updated state, while each add, set or remove quietly swaps in a freshly copied array under a write lock.

Advantages

  • Safe Iteration
  • No ConcurrentModificationException

Because iterators traverse the snapshot that existed when iteration began, concurrent modifications by other threads do not disturb them and never trigger ConcurrentModificationException. This makes iteration completely safe without external synchronization.

Disadvantages

  • Memory Heavy
  • Slow Writes

The cost of this safety is that every single write copies the entire backing array, which is O(n) per write and produces garbage for the collector. For a list that changes frequently this is prohibitively expensive in both time and memory.

Best Use Case

Read-heavy systems. CopyOnWriteArrayList shines when reads vastly outnumber writes, such as maintaining a list of event listeners or a rarely-changing configuration that is iterated constantly. If writes are frequent, a different concurrent structure or external locking is a better fit.

SECTION 11 - PERFORMANCE ANALYSIS

Q31. Big O Complexity Of Common Collections

ROI: ★★★★★ Frequency: Extremely High

Collection Search Insert Delete
ArrayList O(1) O(1)* O(n)
LinkedList O(n) O(1) O(1)
HashMap O(1) O(1) O(1)
TreeMap O(log n) O(log n) O(log n)

A few nuances behind the table are worth knowing. ArrayList "Search" is O(1) only for access by index; searching by value (contains) is O(n). LinkedList insert and delete are O(1) only once you are already positioned at the node, since reaching that node is itself O(n). HashMap's O(1) is an average case that assumes a good hash distribution; with heavy collisions a bucket degrades to O(log n) after treeification. TreeMap's consistent O(log n) reflects the balanced red-black tree underneath.

*Amortized Complexity

The asterisk on ArrayList insert reflects amortized cost: most appends are O(1), but the occasional resize is O(n). Averaged over many insertions, the per-insert cost is still effectively constant, which is the standard meaning of amortized complexity.

Q32. Which Collection Would You Choose?

ROI: ★★★★★ Frequency: Senior Favorite

Scenario 1

Need Fast Lookup.

Answer


HashMap

When the only requirement is fast key-based retrieval and order does not matter, HashMap's average O(1) get and put make it the clear default.

Scenario 2

Need Sorted Data.

Answer


TreeMap

When you need keys kept in sorted order or need range and navigation queries, TreeMap's red-black tree maintains order automatically at O(log n) per operation.

Q33. Why Is HashMap Performance Sometimes Poor?

ROI: ★★★★★ Frequency: High

Reasons

  • Bad hashCode()
  • High Collisions
  • Frequent Rehashing
  • Large Buckets
  • Poor Capacity Planning

The common thread among these causes is poor hash distribution and avoidable resizing. A weak hashCode() (for example one that returns a constant) funnels every key into the same bucket, collapsing the map into a glorified linked list and destroying its O(1) guarantee. Frequent rehashing during bulk inserts adds repeated O(n) copy work, and oversized buckets caused by collisions slow every lookup that touches them.

The practical fixes are to implement a well-distributed hashCode() (and a consistent equals()), and to pre-size the map when the expected number of entries is known so it never resizes mid-load. These are exactly the kinds of concrete, root-cause answers senior interviewers want rather than a vague "HashMap can be slow."

Q34. Why Might ArrayList Consume More Memory Than Expected?

ROI: ★★★★☆ Frequency: High

Answer

The usual cause is unused internal capacity. ArrayList often allocates more space than is currently needed, because it grows its backing array in chunks (roughly 1.5x) rather than one element at a time. After a growth step the array may be significantly larger than the number of elements actually stored, and that extra slack sits idle as wasted memory.

This is amplified when many large lists are held in memory, or when a list grew large temporarily and then most elements were removed; the backing array does not shrink automatically. If memory matters, you can call trimToSize() to release the surplus, or construct the list with an accurate initial capacity so it never over-allocates in the first place.
