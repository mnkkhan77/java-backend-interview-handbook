SECTION 1 - SECURITY FUNDAMENTALS

Q1. What Is Spring Security?

ROI: ★★★★★ Frequency: Extremely High

Answer

Spring Security is the de-facto security framework for the Spring ecosystem. It plugs into the request lifecycle through a chain of servlet filters and provides a complete, configurable model for both authentication (proving who you are) and authorization (deciding what you can do). Instead of writing security logic by hand in every controller, you declare your rules once and the framework enforces them consistently across the whole application.

Spring Security is a framework for:

  • Authentication
  • Authorization
  • Session Management
  • Password Security
  • Protection Against Common Attacks

In real backend systems it is rarely used in isolation. It typically combines a custom UserDetailsService for loading users, a PasswordEncoder such as BCrypt for credential storage, and either session cookies or a stateless JWT filter for carrying identity between requests. For senior roles, interviewers care less about which annotations you know and more about whether you understand the filter-based architecture underneath, which is why almost every follow-up drills into the filter chain.

Q2. Authentication vs Authorization

ROI: ★★★★★ Frequency: Extremely High

Authentication Authorization
Who Are You? What Can You Access?
Identity Verification Permission Verification

Example

Login = Authentication

Access Admin API = Authorization

The order matters: authentication always happens first, and only an authenticated principal can be authorized. In Spring Security this is reflected directly in the filter chain. Authentication filters run early and populate the SecurityContext with the current user and their granted authorities. Authorization decisions then run later, comparing those authorities against the rules you configured for the endpoint or method.

A common interview mistake is treating these as the same concept. A user can be perfectly authenticated and still be denied access, for example a regular user who successfully logs in but is forbidden from calling an admin-only endpoint. Keeping the two clearly separate is what lets you reason about 401 (not authenticated) versus 403 (authenticated but not permitted) responses correctly.

Q3. Common Authentication Methods

ROI: ★★★★★ Frequency: High

  • Username Password
  • JWT
  • OAuth2
  • SSO
  • API Keys

Each method fits a different context. Username and password (typically backed by sessions) is the simplest model and works well for traditional server-rendered web apps. JWT is preferred for stateless REST APIs and microservices where the server should not hold session state. OAuth2 and SSO are used when you want to delegate identity to a trusted provider such as Google or an enterprise identity server, so users do not create yet another password. API keys are common for service-to-service or machine clients where there is no interactive human login.

In a senior interview, the value is in explaining the trade-offs rather than listing the names. Sessions are easy to revoke but harder to scale horizontally; JWTs scale beautifully but are hard to revoke before expiry; OAuth2 removes password handling but adds integration complexity and redirect flows.

SECTION 2 - SPRING SECURITY INTERNALS

Q4. How Does Spring Security Work?

ROI: ★★★★★ Frequency: Extremely High

High Level Flow


Request

↓

Security Filter Chain

↓

Authentication

↓

Authorization

↓

Controller

Every incoming HTTP request first passes through a single entry-point filter (the DelegatingFilterProxy, which delegates to the FilterChainProxy). That proxy routes the request into the appropriate SecurityFilterChain, which is an ordered list of filters. Each filter has one responsibility, for example extracting credentials, validating a token, or enforcing authorization rules.

Authentication filters attempt to establish identity and, on success, store an Authentication object in the SecurityContext. Authorization then runs against that established identity. Only if both succeed does the request reach the Spring MVC DispatcherServlet and finally your controller. If authentication or authorization fails, the chain short-circuits and returns a 401 or 403 before the controller ever executes.

The key mental model for interviews is that security in Spring is filter-based and sits in front of the MVC layer, which is exactly why it can protect requests before any application code runs.

Q5. What Is Security Filter Chain?

ROI: ★★★★★ Frequency: Extremely High

Answer

A chain of security filters that process requests before reaching controllers.

The chain is ordered, and that order is deliberate. Filters such as SecurityContextPersistenceFilter, the authentication filters (for example UsernamePasswordAuthenticationFilter or your custom JWT filter), the ExceptionTranslationFilter, and finally the AuthorizationFilter each run in sequence. A request must survive every relevant filter before it is dispatched to a controller.

Understanding this is what separates someone who "uses" Spring Security from someone who understands it. When you add a JWT filter, you are inserting your own filter at a specific position in this chain (typically before UsernamePasswordAuthenticationFilter) so that the token is parsed and the SecurityContext is populated before authorization decisions are made.

Q6. Why Filters Instead Of Interceptors?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

Filters execute before Spring MVC.

Security must happen before controller execution.

Servlet filters are part of the servlet container lifecycle and run before the request ever reaches the DispatcherServlet. Spring MVC interceptors, by contrast, run inside the MVC layer, after dispatch and handler mapping. If security were implemented as an interceptor, the request would already be partway through the framework before being rejected, which is both wasteful and risky.

Placing security at the filter level also means it can protect non-MVC resources such as static files or other servlets, and it can reject malicious requests as early as possible. This is exactly why Spring Security is built on filters rather than interceptors, and naming this distinction is a reliable way to signal senior-level understanding.

Q7. What Is AuthenticationManager?

ROI: ★★★★★ Frequency: High

Answer

Main interface responsible for authentication.

The AuthenticationManager receives an unauthenticated Authentication object (for example a username and password) and either returns a fully populated, authenticated object or throws an AuthenticationException. It is the central coordination point for the authentication process.

In practice it usually delegates to a ProviderManager, which holds a list of AuthenticationProvider instances. A typical provider is the DaoAuthenticationProvider, which uses your UserDetailsService to load the user and a PasswordEncoder to verify the password. Understanding this delegation chain matters because in custom JWT setups you often expose the AuthenticationManager as a bean and call it manually inside your login endpoint.

Q8. What Is UserDetailsService?

ROI: ★★★★★ Frequency: Extremely High

Answer

Loads user information from database or another source.

It is a single-method strategy interface. Spring Security calls it during authentication to fetch the user that matches the supplied identifier, and you implement it to read from wherever your users actually live, most commonly a database via a JPA repository, but it could equally be an LDAP directory or an external service.

Example


loadUserByUsername()

The method returns a UserDetails object, or throws UsernameNotFoundException if no matching user exists. The returned object carries the stored (hashed) password and the user's authorities, which the DaoAuthenticationProvider then uses to verify credentials and build the authenticated principal. Keeping this lookup separate from password verification is a clean separation of concerns: your code decides where users come from, and the framework decides how they are authenticated.

Q9. What Is UserDetails?

ROI: ★★★★★ Frequency: High

Represents authenticated user information.

UserDetails is the contract Spring Security uses to describe a user once they have been loaded. It is the bridge between your domain model and the framework: your application entity is converted into a UserDetails so that Spring can read credentials, authorities and account flags in a standard way.

Contains

  • Username
  • Password
  • Authorities
  • Account Status

The account-status flags (enabled, account non-expired, account non-locked, credentials non-expired) are important in production. They let you disable a user, lock an account after repeated failed logins, or force a password reset without deleting the record. A common pattern is to implement UserDetails directly on a wrapper around your JPA entity so these flags map to real database columns.

Q10. What Is SecurityContext?

ROI: ★★★★★ Frequency: Extremely High

Answer

Stores authenticated user information for current request.

The SecurityContext holds the Authentication object for the current execution. Once an authentication filter succeeds, it places the authenticated principal here, and every later component, authorization checks, controllers, and method-security annotations, reads from it to know who is making the request.

By default it is bound to the current thread via a ThreadLocal, which is why it is naturally scoped per request. This is also a common source of bugs in asynchronous or multi-threaded code: the context does not automatically propagate to new threads, so background tasks may see an empty context unless you explicitly carry it across.

Q11. SecurityContextHolder

ROI: ★★★★★ Frequency: High

Purpose

Provides access to current authenticated user.

SecurityContextHolder is the static access point to the SecurityContext. Anywhere in your code you can ask it for the current authentication, which is how a service method discovers the logged-in user without having it passed explicitly through every layer.

Example


Authentication auth =

SecurityContextHolder
.getContext()
.getAuthentication();

By default it uses a ThreadLocal storage strategy, but this is configurable. For example, MODE_INHERITABLETHREADLOCAL propagates the context to child threads, which is sometimes needed for asynchronous processing. In production code it is usually cleaner to inject the principal via a controller method argument or @AuthenticationPrincipal, and reserve the direct SecurityContextHolder call for service-layer code where no request parameter is available.

SECTION 3 - PASSWORD SECURITY

Q12. Why Not Store Plain Text Passwords?

ROI: ★★★★★ Frequency: Extremely High

Answer

Massive security risk.

Database compromise exposes all passwords.

If passwords are stored in plain text, a single database breach instantly hands the attacker every user's actual password. Because people reuse passwords across services, this does not just compromise your application, it likely compromises those users' email, banking and other accounts too. This is why storing plain-text passwords is treated as a serious negligence issue and violates virtually every security standard and regulation.

The correct approach is to store a one-way hash produced by a slow, salted algorithm such as BCrypt. When a user logs in, you hash the submitted password and compare it to the stored hash; the original password is never recoverable from the database. Even your own administrators should not be able to read user passwords.

Q13. What Is PasswordEncoder?

ROI: ★★★★★ Frequency: Extremely High

Answer

Responsible for hashing passwords.

PasswordEncoder is the Spring Security abstraction for one-way password hashing. It exposes encode() to hash a raw password before storage and matches() to verify a raw password against a stored hash. Because it is an interface, you can switch the underlying algorithm without changing the rest of your authentication code.

A recommended choice is DelegatingPasswordEncoder (the Spring default), which prefixes each hash with the algorithm id such as {bcrypt}. This lets an application support multiple encoders at once and migrate old hashes to a newer algorithm gradually, which is valuable when an application lives for years and security recommendations evolve.

Q14. Why BCrypt?

ROI: ★★★★★ Frequency: Extremely High

Answer

  • Salt Included
  • Adaptive Cost Factor
  • Resistant To Rainbow Tables

BCrypt is designed specifically for password hashing rather than for speed. It automatically generates a random salt for every password and embeds it in the resulting hash, so two users with the same password still produce different hashes. This defeats precomputed rainbow-table attacks because the attacker cannot reuse a single table across accounts.

Its adaptive cost factor (the work factor or "strength") makes hashing deliberately slow, and you can increase it over time as hardware gets faster. This is critical: a fast hash like plain SHA-256 can be brute-forced billions of times per second on a GPU, whereas BCrypt is tuned so each guess is expensive, which dramatically slows offline cracking if the database is ever leaked.

Most Common Encoder


BCryptPasswordEncoder
Q15. Hashing vs Encryption

ROI: ★★★★★ Frequency: Extremely High

Hashing Encryption
One Way Two Way
Passwords Sensitive Data

The defining difference is reversibility. Hashing is a one-way transformation: given a hash you cannot mathematically recover the original input, which is exactly what you want for passwords because you only ever need to compare, never to read them back. Encryption is two-way: with the correct key you can decrypt the ciphertext back to the original value.

That is why the two are used in different places. Passwords are hashed because you never legitimately need the original. Data that you do need to read back later, such as a stored API secret, a payment token, or PII that must be displayed, is encrypted so it can be decrypted when required. Confusing the two is a classic interview trap: "encrypting" passwords is wrong precisely because encryption is reversible.

SECTION 4 - JWT FUNDAMENTALS

Q16. What Is JWT?

ROI: ★★★★★ Frequency: Extremely High

Answer

JSON Web Token.

Compact token used for stateless authentication.

A JWT is a self-contained, digitally signed token that carries a set of claims about a user, such as their id, username and roles. Because the identity information lives inside the token itself, the server can verify the token's signature and trust its contents without looking anything up in a session store. This is what makes it "stateless".

That property is exactly why JWT is popular in REST APIs and microservices. Any service that knows the signing secret or public key can independently validate the token, so authentication does not require a shared session database and the system scales horizontally with ease. The trade-off, which interviewers love to probe, is that this same self-contained nature makes tokens hard to revoke before they expire.

Q17. JWT Structure

ROI: ★★★★★ Frequency: Extremely High 


Header

.

Payload

.

Signature

A JWT is three Base64Url-encoded parts joined by dots. The header describes the token type and signing algorithm, the payload carries the claims, and the signature binds the first two parts to a secret or key so any tampering is detectable. The dots are literal separators, which is why a JWT always visually appears as three blocks of text.

A crucial point for interviews is that the header and payload are only encoded, not encrypted. Anyone holding the token can decode and read them. The signature does not hide the contents; it only guarantees that the contents have not been altered since the token was issued.

Q18. JWT Header Contains?

ROI: ★★★★★ Frequency: High

Examples

  • Algorithm
  • Token Type

The header is a small JSON object describing how the token is signed. The alg field names the signing algorithm (for example HS256 for HMAC with SHA-256, or RS256 for RSA), and the typ field declares the token type, usually JWT. The validating server reads the header to know which algorithm to use when verifying the signature.

This is also where one of the most important JWT security rules lives: a server must never blindly trust the algorithm declared in the header. Historically, attackers exploited libraries that accepted an alg of none or that allowed switching from RSA to HMAC, so a hardened implementation pins the expected algorithm rather than letting the token dictate it.

Q19. JWT Payload Contains?

ROI: ★★★★★ Frequency: Extremely High

Examples

  • User ID
  • Username
  • Roles
  • Claims
  • Expiration

The payload holds the claims, which are statements about the user and the token. These include registered claims defined by the standard such as sub (subject), iat (issued-at), and exp (expiration), plus any custom claims you add such as roles or a tenant id. The validating service reads these claims to build the authenticated principal and to enforce expiry.

Because the payload is readable by anyone who has the token, you should never put sensitive data here, no passwords, no secrets, no full personal records. Keep it to identifiers and authorization data, and keep it small, since the token is sent on every request and bloated payloads add overhead to every call.

Q20. JWT Signature Purpose

ROI: ★★★★★ Frequency: Extremely High

Answer

Ensures token integrity.

Detects tampering.

The signature is computed over the encoded header and payload using a secret key (for HMAC algorithms) or a private key (for RSA/ECDSA). When the server receives a token, it recomputes the signature and compares it. If even a single character in the header or payload was changed, the recomputed signature will not match and the token is rejected.

This is what lets a stateless server trust a token it did not store. An attacker can read the payload, but they cannot change their role from USER to ADMIN and produce a valid signature without the signing key. Protecting that signing key is therefore the single most important operational concern with JWT: if it leaks, an attacker can mint valid tokens for any user.

Q21. Is JWT Encrypted?

ROI: ★★★★★ Frequency: Interview Trap

Answer

No.

JWT is typically encoded and signed.

Not encrypted.

A standard signed JWT (a JWS) is Base64Url-encoded and signed, but its contents are fully readable to anyone who intercepts the token. Encoding is not encryption; it only converts bytes into a URL-safe text form. The signature protects integrity, not confidentiality. If you genuinely need the contents to be unreadable, you must use JWE (JSON Web Encryption), which is a separate construct and far less common in typical web APIs.

Interview Gold Point

Anyone can decode payload.

Do not store sensitive data.

Interviewers ask this specifically to catch candidates who assume a signed token is private. The correct senior answer is that a signed JWT guarantees the data has not been tampered with, but it does nothing to hide it. Never place passwords, secrets, or sensitive personal data in the payload, and always transmit tokens over HTTPS so they cannot be sniffed on the wire.

SECTION 5 - JWT AUTHENTICATION FLOW

This is arguably the most frequently asked Spring Security topic. Most interviewers will ask: "Explain the complete JWT flow."

Q22. Explain JWT Authentication Flow

ROI: ★★★★★ Frequency: Extremely High

Flow


User Login

↓

Username Password Validation

↓

JWT Generated

↓

Token Returned

↓

Client Stores Token

↓

Client Sends Token

↓

JWT Validation

↓

Authentication Created

↓

Request Allowed

The flow has two distinct phases. In the login phase, the client posts credentials to a login endpoint, the server authenticates them through the AuthenticationManager and UserDetailsService, and on success it generates a signed JWT containing the user's id, roles and an expiry. That token is returned to the client, which stores it.

In the access phase, the client attaches the token to every subsequent request, normally in the Authorization: Bearer <token> header. A custom JWT filter intercepts the request, validates the signature and expiry, extracts the claims, builds an Authentication object, and places it in the SecurityContext. From that point the request is treated as authenticated and authorization rules apply.

The senior-level insight is that the server stores nothing between these phases. Each request is authenticated solely from the token, which is what makes the design stateless and horizontally scalable, but also why revocation needs extra machinery.

Q23. Where Should JWT Be Stored?

ROI: ★★★★★ Frequency: Extremely High

Options

  • HttpOnly Cookie
  • Memory Storage
  • Local Storage

Each storage location trades off against a different attack. Local storage is easy to use but is fully readable by JavaScript, so a single XSS vulnerability lets an attacker steal the token. In-memory storage (a JavaScript variable) is safer against XSS persistence because it is cleared on refresh, but it is lost on page reload and still exposed if the page is compromised while running.

Senior Answer

HttpOnly Cookies are generally safer against XSS.

A cookie marked HttpOnly cannot be read by JavaScript at all, which neutralises token theft via XSS. Combined with the Secure flag (HTTPS only) and SameSite (to limit cross-site sending), it is the strongest default. The catch is that cookies are sent automatically by the browser, which reintroduces CSRF as a concern, so you typically pair cookie-based tokens with CSRF protection or a strict SameSite policy. The honest senior answer is that there is no perfect option: you choose based on whether XSS or CSRF is the bigger risk for your application.

Q24. What Happens On Every Request?

ROI: ★★★★★ Frequency: Extremely High

Answer

  1. Token Extracted
  2. Signature Verified
  3. Expiration Checked
  4. Claims Parsed
  5. Authentication Created
  6. SecurityContext Updated

These six steps happen inside your JWT filter, typically a OncePerRequestFilter placed before the username-password authentication filter. It pulls the token out of the Authorization header, verifies the signature against the signing key, rejects it if it has expired, and then reads the claims to identify the user and their authorities.

Only after all checks pass does the filter build an Authentication token and set it on the SecurityContext, after which the request is allowed to continue. If any step fails, the filter does not populate the context and the request is later rejected by the authorization layer. Because this runs on every request, the validation must be fast and must not hit the database in the normal path, which is the whole performance argument for stateless JWT.

Q25. Why Is JWT Stateless?

ROI: ★★★★★ Frequency: Extremely High

Answer

Server does not need session storage.

Authentication data exists inside token.

Statelessness means the server keeps no per-user session record between requests. All the information needed to authenticate, the user id, roles and expiry, is carried in the token itself and verified cryptographically each time. The server only needs the signing key, not a session store.

This has direct architectural consequences. Any instance behind a load balancer can serve any request without sticky sessions, and any microservice can validate the token independently. That is why JWT is the default choice for distributed systems. The price you pay is revocation: because nothing is stored server-side, you cannot simply delete a session to log someone out, which leads directly to the blacklist and refresh-token strategies discussed next.

Q26. Session Authentication vs JWT

ROI: ★★★★★ Frequency: Extremely High

Session JWT
Stateful Stateless
Server Stores Session Client Stores Token
Scaling Harder Microservice Friendly

With session authentication the server creates a session record on login and returns only a session id (in a cookie). The id is meaningless on its own; every request requires the server to look up the session, which means the server holds state. This makes revocation trivial, just delete the session, but complicates horizontal scaling because every instance needs access to the same session store or sticky routing.

JWT inverts this. The client holds a self-describing token and the server holds nothing, so scaling across many instances and services is effortless, but invalidating a token before it expires requires extra infrastructure. The honest interview position is that neither is universally "better": sessions suit traditional monoliths where easy logout matters, while JWT suits stateless APIs and microservices where scale matters.

Q27. Can JWT Be Revoked?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

Not naturally.

JWT remains valid until expiration.

This is the most important limitation of JWT and a favourite senior follow-up. Because validation is purely cryptographic and stateless, a token stays valid for its entire lifetime even if the user logs out, changes their password, or is banned. The server has no built-in way to "tear up" a token it never stored.

Common Solutions

  • Blacklist
  • Redis Token Store
  • Short Expiry
  • Refresh Tokens

The practical answer is that you reintroduce a small amount of state to regain control. A blacklist (often stored in Redis with a TTL matching the token expiry) lets you explicitly reject revoked tokens. Keeping access tokens short-lived limits the damage window so a leaked or stale token is only usable for a few minutes. Refresh tokens then let users stay logged in without long-lived access tokens, and the refresh token, which is stored server-side, can be revoked to truly end a session. Most real systems combine short expiry plus refresh tokens, and add a blacklist only for high-security cases.

SECTION 6 - REFRESH TOKENS

Q28. Why Refresh Tokens?

ROI: ★★★★★ Frequency: Extremely High

Problem

Long-lived access tokens are risky.

If an access token is valid for hours or days, a single leak gives an attacker a long window of access that you cannot easily cut short. But making access tokens short-lived would force users to log in every few minutes, which is a poor experience. Refresh tokens resolve this tension.

Solution

Short-lived access token.

Long-lived refresh token.

The access token is kept short-lived (for example 15 minutes) so a stolen one expires quickly. The refresh token is long-lived but is used only to obtain new access tokens, never to call business APIs, and it is stored more securely (typically in an HttpOnly cookie and recorded server-side). This separation gives you both convenience and control: users stay logged in, while the highly exposed access token has a small blast radius, and the refresh token can be revoked on the server to end the session.

Q29. Access Token vs Refresh Token

ROI: ★★★★★ Frequency: Extremely High

Access Token Refresh Token
Short Lifetime Long Lifetime
API Access Generate New Access Token

The two tokens have deliberately different jobs and exposure profiles. The access token is sent on every API call, so it is the most exposed credential; keeping it short-lived limits the damage if it is intercepted. The refresh token is sent rarely, only to the token-refresh endpoint, so it is far less exposed and can safely live longer.

A strong production pattern is refresh-token rotation: each time a refresh token is used, the server issues a new one and invalidates the old. If an old (already-used) refresh token is ever presented again, that signals theft, and the server can revoke the entire token family. This turns the refresh mechanism into both a convenience feature and a breach-detection mechanism.

Q30. Refresh Token Flow

ROI: ★★★★★ Frequency: Extremely High 


Login

↓

Access Token

+

Refresh Token

↓

Access Token Expires

↓

Refresh Token Used

↓

New Access Token

On login the server issues both tokens together. The client uses the access token for normal API calls until it expires. Instead of forcing a full re-login, the client then silently presents the refresh token to a dedicated refresh endpoint, which validates it (often checking it against a server-side store) and issues a fresh access token.

This loop continues until the refresh token itself expires or is revoked, at which point the user must log in again. The key design point is that only the refresh endpoint accepts the refresh token, and the refresh token never travels with ordinary API requests, which keeps the long-lived credential off the most exposed path.

Q31. Where Should Refresh Tokens Be Stored?

ROI: ★★★★★ Frequency: High

Recommended

Secure HttpOnly Cookie.

A refresh token is a high-value, long-lived credential, so it deserves the strongest available storage. An HttpOnly cookie cannot be read by JavaScript, which protects it from XSS-based theft, and adding the Secure flag ensures it is only sent over HTTPS while SameSite limits cross-site exposure.

On the server side, refresh tokens are typically also persisted (hashed) in a database or Redis so they can be validated, rotated and revoked. Storing them this way is what gives you a real logout and a way to terminate a compromised session, which a pure stateless access token cannot provide. Never store a refresh token in local storage, since that is precisely the location most vulnerable to XSS.

SECTION 7 - ROLE BASED ACCESS CONTROL (RBAC)

Q32. What Is RBAC?

ROI: ★★★★★ Frequency: Extremely High

Answer

Access control based on roles.

RBAC assigns permissions to roles rather than to individual users, and then assigns roles to users. A user inherits all the permissions of their roles, so you manage access by managing roles, not by editing each user. This centralises authorization logic and makes it far easier to audit and maintain as the system grows.

Example


ADMIN

RECRUITER

USER

In a job-portal style application these three roles map cleanly onto real behaviour: an ADMIN moderates the platform, a RECRUITER manages job postings, and a USER applies to jobs. When requirements change, for example a new "premium recruiter" capability, you adjust the role definition once rather than touching every affected user. The main limitation, which interviewers like to push on, is that pure RBAC struggles with fine-grained, data-dependent rules (such as "a recruiter can only edit their own postings"), which is where attribute-based checks complement it.

Q33. Role vs Authority

ROI: ★★★★★ Frequency: Extremely High

Role Authority
High Level Fine-Grained Permission
ADMIN CREATE_USER

In Spring Security both roles and authorities are represented by the same GrantedAuthority type, the distinction is mainly conventional. A role is a coarse grouping such as ADMIN or USER, while an authority is usually a granular permission such as CREATE_USER or DELETE_JOB. A common design is for each role to map to a set of authorities.

The practical reason this matters is flexibility. Checking against fine-grained authorities lets you change what a role can do without changing the code that enforces it, you simply re-map the role's authorities. The framework even reflects the convention internally: roles are stored with a ROLE_ prefix, which is the source of the hasRole versus hasAuthority difference covered next.

Q34. How Are Roles Stored In Spring Security?

ROI: ★★★★★ Frequency: High

Example


ROLE_ADMIN

ROLE_USER

Spring Security stores roles as authorities with a ROLE_ prefix. So the role "ADMIN" is actually held internally as the authority ROLE_ADMIN. This convention is what lets the framework distinguish a role-style authority from an arbitrary permission, and it is wired into the helper methods you use in configuration.

The practical consequence is that you must be consistent. When you load authorities for a user from the database, a role should be stored or mapped as ROLE_ADMIN, not just ADMIN, otherwise hasRole("ADMIN") checks will silently fail to match. Many "my authorization isn't working" bugs trace back to a missing or doubled ROLE_ prefix.

Q35. hasRole() vs hasAuthority()

ROI: ★★★★★ Frequency: Extremely High

hasRole hasAuthority
ROLE_ Prefix Added Exact Match

The difference is entirely about the prefix. hasRole("ADMIN") automatically prepends ROLE_ and therefore checks for the authority ROLE_ADMIN. hasAuthority("ROLE_ADMIN") performs an exact string match with no prefix added, so you must spell out the full authority yourself.

This is a frequent source of subtle bugs. If you write hasRole("ROLE_ADMIN") by mistake, Spring effectively looks for ROLE_ROLE_ADMIN and the check never passes. The senior takeaway is to pick one convention: use hasRole with the bare role name for role checks, and reserve hasAuthority for fine-grained permission strings, and to ensure your stored authorities line up with whichever you choose.

SECTION 8 - METHOD SECURITY

Q36. What Is Method Security?

ROI: ★★★★★ Frequency: High

Answer

Security enforced directly on methods.

Method security applies authorization rules at the level of individual service or controller methods using annotations, rather than only at the URL level in your security configuration. It is enabled with @EnableMethodSecurity and works through Spring AOP, wrapping secured methods in a proxy that checks the rule before the method body runs.

The value is defence in depth and locality. URL-based rules can drift out of sync with the code as endpoints are refactored, but a @PreAuthorize sitting directly on a service method travels with that method wherever it is called from. It also lets you protect business logic that is reused across multiple entry points, so the rule is enforced no matter which controller or scheduled job invokes it.

Q37. Common Method Security Annotations

ROI: ★★★★★ Frequency: High

  • @PreAuthorize
  • @PostAuthorize
  • @Secured

@PreAuthorize evaluates a SpEL expression before the method runs and is the most flexible and widely used, since it can reference method arguments and the current principal. @PostAuthorize runs after the method completes and can inspect the returned object, which is useful for "you may read this record only if it belongs to you" checks. @Secured is the older, simpler annotation that only supports plain role names and no expressions.

In modern code, @PreAuthorize is the default choice because of its expressiveness. @PostAuthorize is reserved for cases where the authorization decision genuinely depends on the result, and you should be mindful that it still executes the method body first, so it is unsuitable when the method itself has side effects.

Q38. Example Of @PreAuthorize

ROI: ★★★★★ Frequency: Extremely High 


@PreAuthorize(
"hasRole('ADMIN')"
)

This annotation tells Spring to allow the method only if the current authenticated user holds the ADMIN role. The expression is written in SpEL, so it can be far richer than a single role check, for example combining conditions with and/or, or referencing arguments as in @PreAuthorize("#userId == authentication.principal.id") to enforce that a user can only act on their own data.

Because the check happens before the method executes, an unauthorized call is rejected with an AccessDeniedException and the business logic never runs. This makes @PreAuthorize the natural place to express both role-based and ownership-based rules in one readable line that lives right next to the code it protects.

Q39. Why Method Security?

ROI: ★★★★★ Frequency: High

Protects business logic even if endpoint security changes.

URL-level security and method-level security operate at different layers, and using both gives you defence in depth. If someone refactors a controller, exposes a new endpoint, or invokes a service method from a scheduled job or another service, the method-level rule still applies because it is attached to the method itself rather than to a URL pattern.

This locality also makes the rules easier to read and audit: the authorization requirement sits directly above the code it guards, so a reviewer can see at a glance who is allowed to run it. The trade-off is that scattering security rules across many methods can become hard to manage, so teams usually combine a coarse URL-level policy with targeted method security on the most sensitive operations.

SECTION 9 - CSRF

Q40. What Is CSRF?

ROI: ★★★★★ Frequency: Extremely High

Full Form

Cross Site Request Forgery

Answer

Attack that tricks authenticated user into performing unwanted actions.

CSRF exploits the fact that browsers automatically attach cookies to requests aimed at a site, regardless of which site initiated the request. If a user is logged into your application (with a session cookie) and then visits a malicious page, that page can silently fire a request to your application, and the browser will include the user's cookie, making the request look legitimate.

The crucial detail is that the attacker never needs to read any data or steal the cookie; they only need the user's browser to send an authenticated request on their behalf. That is why CSRF specifically targets state-changing operations such as transferring money, changing an email, or deleting a resource, rather than simple reads.

Q41. Typical CSRF Scenario

ROI: ★★★★★ Frequency: High 


User Logged In

↓

Visits Malicious Site

↓

Hidden Request Sent

↓

Application Executes Request

Walking through the scenario: the user is authenticated to your bank-like application, with a valid session cookie in their browser. While still logged in, they open a malicious or compromised page in another tab. That page contains a hidden form or image that auto-submits a request to your application's transfer endpoint.

Because the request goes to your domain, the browser dutifully attaches the user's session cookie, and your server sees what appears to be a fully authenticated, legitimate request and executes it. The user never clicked anything meaningful. This is exactly why the defence is a CSRF token: a secret value the attacker's page cannot know or read, proving the request originated from your own application.

Q42. How Does Spring Prevent CSRF?

ROI: ★★★★★ Frequency: Extremely High

Answer

CSRF Tokens.

Server validates token before processing request.

Spring Security uses the synchronizer token pattern. When it renders a page or starts a session, it generates a unique, unpredictable CSRF token tied to that user. Legitimate state-changing requests must echo this token back (in a hidden form field or a request header), and the server compares it against the expected value before processing the request.

This works because a malicious cross-site page cannot read the token, the same-origin policy prevents it from accessing your page's contents or your token cookie's value in a usable way. Without the correct token the forged request is rejected. By default Spring enables CSRF protection and applies it to state-changing methods (POST, PUT, DELETE, PATCH) while leaving safe, read-only methods like GET untouched.

Q43. Why Is CSRF Often Disabled In JWT APIs?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

JWT APIs usually use Authorization headers.

Not cookie-based session authentication.

Traditional CSRF attack surface is reduced.

CSRF depends on the browser automatically attaching a credential, which is exactly what cookies do. When a JWT is sent in an Authorization: Bearer header instead of a cookie, the browser does not attach it automatically. A malicious cross-site page cannot read or set that header for your domain, so it cannot forge an authenticated request, which is why header-based JWT APIs are largely immune to classic CSRF and commonly disable the protection.

Senior Engineer Perspective

Be careful with the nuance here, because interviewers probe it. The "CSRF is irrelevant for JWT" claim only holds when the token travels in a header, not when it is stored in a cookie. If you decide to store your JWT in a cookie (a common choice for XSS protection), the browser will auto-send it and CSRF risk returns, so you must keep CSRF protection or rely on a strict SameSite cookie policy. The mature answer is: disable CSRF only after confirming your tokens are not carried in cookies.

SECTION 10 - CORS

Q44. What Is CORS?

ROI: ★★★★★ Frequency: Extremely High

Full Form

Cross Origin Resource Sharing

CORS is a browser mechanism that controls whether a web page loaded from one origin is allowed to make requests to a different origin. It works through HTTP headers: the server responds with headers such as Access-Control-Allow-Origin that explicitly tell the browser which origins are permitted to read the response.

A common misconception is that CORS is a server-side security feature that blocks attacks. It is the opposite, it is a relaxation mechanism enforced by the browser. By default the browser forbids cross-origin reads (the same-origin policy), and CORS is how a server opts in to allowing specific trusted origins. It protects users, not the server, and it is irrelevant to non-browser clients like Postman or backend services.

Q45. Why Does CORS Exist?

ROI: ★★★★★ Frequency: Extremely High

Answer

Browsers restrict cross-origin requests for security.

The underlying rule is the same-origin policy, which prevents JavaScript on one site from freely reading responses from another site. Without it, a malicious page you visit could quietly make authenticated requests to your bank or email and read the responses, since your browser holds your sessions. The same-origin policy stops that by default.

CORS exists because legitimate applications frequently do need controlled cross-origin access, for example a single-page app served from one domain calling an API on another. CORS provides a safe, explicit way for the API to declare "these specific origins are allowed", so the browser can permit genuine cross-origin use while still blocking everything else. It is a carefully scoped exception to the same-origin policy, not a removal of it.

Q46. What Is Origin?

ROI: ★★★★★ Frequency: High 


Protocol

+

Host

+

Port

An origin is the combination of three parts: the scheme/protocol (http or https), the host (domain), and the port. Two URLs share an origin only if all three match exactly. So https://app.example.com and http://app.example.com are different origins because the protocol differs, and https://example.com:443 and https://example.com:8443 differ by port.

This precise definition is why developers hit CORS errors so often in local development. A frontend on localhost:3000 and a backend on localhost:8080 share the host and protocol but differ in port, which makes them distinct origins and triggers the browser's cross-origin restrictions.

Q47. Common CORS Interview Question

ROI: ★★★★★ Frequency: Extremely High

Question

Frontend on: 


localhost:3000

Backend on: 


localhost:8080

Why blocked?

Answer

Different origins.

CORS required.

Even though both run on localhost, the ports differ (3000 versus 8080), and port is part of the origin, so the browser treats them as two different origins. When the React app at localhost:3000 calls the API at localhost:8080, the browser enforces the same-origin policy and blocks the script from reading the response unless the backend explicitly allows it.

The fix is to configure CORS on the backend, in Spring this is typically done with a global CorsConfigurationSource bean or @CrossOrigin, listing http://localhost:3000 as an allowed origin along with the permitted methods and headers. In production you allow your real frontend domain rather than a wildcard, and you avoid allowedOrigins("*") together with credentials, since the spec forbids that combination for good security reasons.

SECTION 11 - OAUTH2

OAuth2 is one of the most misunderstood interview topics. A common mistake: "OAuth2 is authentication." Not exactly. OAuth2 is primarily an authorization framework.

Q48. What Is OAuth2?

ROI: ★★★★★ Frequency: Extremely High

Answer

OAuth2 is an authorization framework that allows applications to access resources on behalf of users without sharing passwords.

The core idea is delegated authorization. Instead of giving a third-party application your password, you authenticate directly with a trusted provider (like Google), and that provider issues the application a limited-scope access token. The application uses the token to access only what you consented to, and it never sees your actual credentials.

This solves a real problem from the early web, where apps used to ask for your email password to "import your contacts". OAuth2 replaces that dangerous pattern with scoped, revocable tokens. The senior nuance, and a classic interview trap, is that OAuth2 is fundamentally about authorization (what an app may access), not authentication (who you are); proving identity is the job of OpenID Connect built on top of it.

Example


Login With Google

Login With GitHub

Login With Facebook
Q49. Why OAuth2?

ROI: ★★★★★ Frequency: Extremely High

Problem

Applications should not receive user passwords.

Handing your password to a third-party app is dangerous: that app could store it insecurely, use it beyond what you intended, or be breached, exposing credentials that often unlock many other accounts. There is also no way to grant limited access or to revoke it cleanly once a password has been shared.

Solution

Delegate authorization to trusted provider.

OAuth2 puts a trusted authorization server in the middle. The user authenticates there, approves a specific scope of access, and the provider hands the application a token rather than the password. The token is limited (it grants only the approved scopes), time-bound (it expires), and revocable (the user can withdraw access without changing their password). This is why "Login with Google" is both safer for users and simpler for applications, which no longer have to store or protect passwords at all.

Q50. OAuth2 Roles

ROI: ★★★★★ Frequency: Extremely High

Role Description
Resource Owner User
Client Application
Authorization Server Issues Tokens
Resource Server Protected API

These four roles describe every OAuth2 interaction. The resource owner is the human user who owns the data and grants access. The client is the application that wants access on the user's behalf. The authorization server (for example Google's identity service) authenticates the user and issues tokens. The resource server is the API holding the protected data, which accepts and validates those tokens.

Understanding this separation is what makes the flows make sense. The authorization server and resource server are sometimes the same system and sometimes separate, but conceptually keeping them distinct clarifies why the client talks to one service to get a token and a different service to use it. Naming these roles correctly is a quick way to show you understand OAuth2 beyond just "login with Google".

Q51. OAuth2 Authorization Code Flow

ROI: ★★★★★ Frequency: Extremely High

Most Important OAuth Flow


User

↓

Client App

↓

Authorization Server

↓

Authorization Code

↓

Access Token

↓

Resource Server

The authorization code flow works in two exchanges to keep the access token off the browser address bar. First, the client redirects the user to the authorization server, where the user logs in and consents. The authorization server then redirects back to the client with a short-lived, single-use authorization code in the URL.

Second, the client's backend exchanges that code, together with its client secret, directly with the authorization server (a server-to-server call) to obtain the actual access token. The client then presents the access token to the resource server to fetch protected data. The reason for the two-step dance is security: the code that travels through the browser is useless on its own, since exchanging it for a token requires the client secret that only the backend holds.

Q52. Why Authorization Code Flow?

ROI: ★★★★★ Frequency: Senior Favorite

Answer

Most secure flow for web applications.

Access token is not directly exposed during initial exchange.

The flow is considered the most secure because the access token is never placed in the browser URL or browser history. Only the short-lived authorization code travels through the front channel, and that code is single-use and exchanged for the token on the back channel using the client secret. Even if the code is intercepted, it cannot be redeemed without the secret.

This is in deliberate contrast to the old implicit flow, which returned the access token directly in the URL fragment and is now discouraged. For modern single-page and mobile apps that cannot safely hold a client secret, the authorization code flow is extended with PKCE to retain the same protection without a stored secret, which is why authorization code (with PKCE) is the recommended default today.

Q53. What Is PKCE?

ROI: ★★★★★ Frequency: Increasingly Common

Full Form

Proof Key for Code Exchange

Purpose

Protects Authorization Code Flow from interception attacks.

PKCE adds a dynamically generated secret to the authorization code flow so that public clients (mobile apps, SPAs) can use it safely without a fixed client secret. The client creates a random code_verifier, sends a hashed version (the code_challenge) when requesting the authorization code, and then must present the original verifier when exchanging the code for a token. The authorization server checks that they match.

This defeats authorization-code interception: even if an attacker steals the code (for example via a malicious app intercepting a redirect on a mobile device), they cannot exchange it for a token because they do not possess the original verifier. PKCE effectively binds the code to the specific client instance that started the flow.

Interview Gold Point

PKCE is now considered standard for public clients.

Modern guidance (OAuth 2.1) recommends PKCE for all clients, not just public ones. Saying "we use authorization code flow with PKCE" is the answer that signals you are current with security best practices, whereas mentioning the implicit flow as a serious option marks you as out of date.

Q54. What Is Access Token?

ROI: ★★★★★ Frequency: Extremely High

Token used to access protected resources.

In OAuth2 the access token is the credential the client presents to the resource server to call protected APIs, usually in an Authorization: Bearer header. It represents the specific permissions (scopes) the user consented to and is intentionally short-lived to limit damage if it leaks. The resource server validates it on each call, either by checking a signature (if it is a JWT) or by introspecting it with the authorization server.

Because it is a bearer token, anyone who holds it can use it, which is why it must always travel over HTTPS, be kept short-lived, and never be logged or stored carelessly. When it expires, the client obtains a new one using a refresh token rather than prompting the user to log in again.

Q55. What Is Scope?

ROI: ★★★★★ Frequency: High

Answer

Defines permissions granted to a token.

Scopes are the mechanism OAuth2 uses to enforce least privilege. When the client requests authorization, it asks for specific scopes, and the user sees and approves exactly what they are granting. The issued token then carries only those scopes, and the resource server checks them before serving a request, so a token granted read cannot be used to perform a write.

Examples


read

write

profile

email

This granularity is what makes delegated access safe. A calendar app might request only profile and calendar read access, and the user can grant that without exposing email or write permissions. Designing sensible, narrow scopes is an important security responsibility: overly broad scopes recreate the very over-sharing problem that OAuth2 was built to prevent.

SECTION 12 - OPENID CONNECT (OIDC)

Q56. What Is OpenID Connect?

ROI: ★★★★★ Frequency: Extremely High

Answer

Identity layer built on top of OAuth2.

OpenID Connect (OIDC) extends OAuth2 to solve the problem OAuth2 alone does not address: proving who the user is. OAuth2 gives an application an access token to call APIs, but that token says nothing reliable about the user's identity. OIDC adds a standardized identity layer, most importantly the ID token, plus a /userinfo endpoint and standard claims, so the application can actually authenticate the user.

Interview Gold Point

OAuth2 = Authorization

OIDC = Authentication + Identity

This one-line distinction is exactly what interviewers are listening for. "Login with Google" works because Google implements OIDC on top of OAuth2: OAuth2 handles the delegated access, and OIDC supplies the verified identity via the ID token. Candidates who say "OAuth2 is for login" without mentioning OIDC reveal a shallow understanding.

Q57. What Is ID Token?

ROI: ★★★★★ Frequency: High

Answer

Token containing authenticated user identity.

The ID token is the defining feature of OIDC. It is a JWT issued by the authorization server that asserts the user's identity, carrying claims such as sub (a stable unique user id), name, email, who issued it (iss), and who it was issued for (aud). The client validates its signature and these claims to trust that the user really authenticated.

The important contrast is purpose: the ID token is for the client to learn who the user is and is not meant to be sent to resource servers, whereas the access token is for calling APIs and is opaque to the client. Mixing them up, for example sending an ID token to call an API, is a common implementation mistake worth being able to call out.

Q58. OAuth2 vs OIDC

ROI: ★★★★★ Frequency: Extremely High

OAuth2 OIDC
Authorization Authentication
Access Token ID Token

The relationship is best stated as: OIDC is OAuth2 plus identity. OAuth2 answers "what is this application allowed to do?" and delivers an access token for that purpose. OIDC answers "who is this user?" and delivers an ID token for that purpose, while still using OAuth2's flows underneath.

They are not competitors; OIDC is a profile built on OAuth2. In a real "sign in with Google" implementation you get both tokens from the same flow: the ID token authenticates the user to your app, and the access token (if requested) lets your app call Google APIs on the user's behalf. Being able to explain that they coexist, rather than choosing one over the other, is the senior-level answer.

SECTION 13 - SINGLE SIGN-ON (SSO)

Q59. What Is SSO?

ROI: ★★★★★ Frequency: Extremely High

Answer

Single Sign-On allows users to authenticate once and access multiple applications.

With SSO, a central identity provider handles authentication, and once a user logs in there, they can move between many independent applications without logging in again. Each application trusts the identity provider and accepts the identity it asserts, typically via protocols such as OIDC or SAML, rather than maintaining its own separate login.

This is the model used across enterprises: log in once to your corporate identity provider and gain access to email, the HR portal, the ticketing system and internal tools. The applications never see the user's password; they receive a signed assertion or token from the trusted provider, which is what makes SSO both convenient and, when done right, more secure than dozens of independent logins.

Q60. Benefits Of SSO

ROI: ★★★★★ Frequency: High

  • Better User Experience
  • Centralized Authentication
  • Improved Security
  • Simplified User Management

The user-experience benefit is obvious, one login instead of many, but the security and operational benefits are what interviewers value. Centralizing authentication means password policies, multi-factor authentication, and account lockout are enforced in one place rather than reimplemented (often poorly) in every application. Fewer passwords also means users are less likely to reuse or write down credentials.

Centralized user management is equally important: when an employee leaves, disabling their single identity instantly revokes access to every connected application, which is critical for security and compliance. The trade-off to acknowledge is that the identity provider becomes a single point of failure and a high-value attack target, so it must be highly available and very well protected.

Q61. Common SSO Providers

ROI: ★★★★★ Frequency: High

  • Google
  • Microsoft Entra ID
  • Okta
  • Keycloak
  • Auth0

These split into a few categories worth knowing. Google and Microsoft Entra ID (formerly Azure AD) are large platform providers, common when an organization already lives in Google Workspace or Microsoft 365. Okta and Auth0 are dedicated identity-as-a-service products used to centralize authentication across many apps. Keycloak is a popular open-source, self-hosted option, valuable when teams want full control or cannot use a cloud provider.

For backend interviews the useful point is that you rarely build SSO from scratch; you integrate with one of these via OIDC or SAML. In a Spring application this typically means using spring-security-oauth2-client and configuring the provider, rather than handling raw protocol details yourself.

SECTION 14 - SECURITY DESIGN QUESTIONS

Q62. Design Secure Login System

ROI: ★★★★★ Frequency: Senior Favorite

Expected Answer

  • BCrypt Passwords
  • JWT Access Tokens
  • Refresh Tokens
  • Role-Based Access
  • Rate Limiting
  • Audit Logging
  • MFA Support

A strong answer walks through the lifecycle. Credentials are stored as BCrypt hashes, never plain text, and all traffic runs over HTTPS. On successful login the system issues a short-lived JWT access token plus a refresh token (stored in an HttpOnly cookie and tracked server-side for revocation). Authorization is enforced through roles and, where needed, fine-grained authorities and method security.

Around that core you add defensive layers: rate limiting and account lockout to blunt brute-force attacks, audit logging of login attempts and sensitive actions for traceability and incident response, and optional MFA for an extra factor beyond the password. The senior signal is discussing these as defence in depth, no single control is enough, and explicitly handling the unhappy paths like failed logins, token theft, and logout/revocation, rather than only the happy login path.

Q63. How To Secure Microservice APIs?

ROI: ★★★★★ Frequency: Extremely High

Answer

  • JWT Validation
  • API Gateway
  • TLS Everywhere
  • Least Privilege
  • Rate Limiting
  • Audit Logging

In a microservices architecture, authentication is usually centralized at an API gateway, which validates the incoming JWT once and forwards the request inward. Each downstream service still independently validates the token (or a propagated identity) so that security does not depend solely on the gateway, this is "zero trust" between services rather than assuming the internal network is safe.

TLS everywhere, including service-to-service traffic (often via mTLS or a service mesh), prevents token sniffing on internal links. Least privilege ensures each service and each token can do only what it needs, limiting blast radius if one service is compromised. Rate limiting protects against abuse and cascading overload, and centralized audit logging with correlation ids makes it possible to trace a request across many services during an incident. The maturity signal is recognizing that the internal network is not a trust boundary.

Q64. What Is Principle Of Least Privilege?

ROI: ★★★★★ Frequency: High

Answer

Users receive only permissions necessary to perform their tasks.

The principle of least privilege says every user, service, and process should be granted the minimum access required to do its job, and nothing more. It applies not just to human users but to database accounts, service credentials, cloud IAM roles, and token scopes. The goal is to shrink the attack surface so that any single compromise is contained.

The practical payoff appears during incidents. If a service account is breached but it only had read access to one table, the damage is limited; if it had broad admin rights "for convenience", the breach becomes catastrophic. This is why mature systems avoid wildcard permissions, scope tokens narrowly, and periodically review and revoke access that is no longer needed.

Q65. Why Use MFA?

ROI: ★★★★★ Frequency: High

Answer

Provides additional security layer beyond passwords.

Multi-factor authentication requires more than one type of evidence to log in, typically combining something you know (a password) with something you have (a phone or hardware key) or something you are (a fingerprint). Because an attacker would need to compromise two independent factors, MFA dramatically reduces the risk from stolen or guessed passwords.

This matters because passwords are the weakest link: they get phished, reused across sites, and leaked in breaches. With MFA enabled, a leaked password alone is no longer enough to take over an account. It is widely regarded as one of the highest-impact, lowest-cost security controls, which is why it is standard for admin accounts and increasingly mandatory for all users in security-conscious organizations.

SECTION 15 - COMMON ATTACKS

Q66. What Is XSS?

ROI: ★★★★★ Frequency: Extremely High

Full Form

Cross Site Scripting

Answer

Injection of malicious scripts into web pages.

XSS occurs when an attacker manages to inject JavaScript that runs in another user's browser in the context of your site. This typically happens when user-supplied input (a comment, a profile field, a query parameter) is rendered back into a page without proper encoding, so the browser executes it as code rather than displaying it as text.

The danger is that the injected script runs with the victim's privileges on your domain: it can read cookies and tokens accessible to JavaScript, make authenticated requests, capture keystrokes, or deface the page. This is precisely why storing tokens in HttpOnly cookies matters, it keeps them out of reach of any injected script, and why XSS is consistently ranked among the most serious web vulnerabilities.

Q67. How To Prevent XSS?

ROI: ★★★★★ Frequency: High

  • Output Encoding
  • Input Validation
  • Content Security Policy
  • HttpOnly Cookies

The single most important defence is context-aware output encoding: whenever user data is written into HTML, attributes, or JavaScript, it must be escaped so the browser treats it as data, not code. Input validation helps by rejecting clearly malicious or malformed input early, but it is a complement to, not a replacement for, output encoding.

A Content Security Policy adds a strong second layer by restricting which scripts the browser is allowed to execute, so even an injected script may be blocked from running. HttpOnly cookies limit the impact of any XSS that does slip through by keeping session tokens unreadable to JavaScript. The senior framing is defence in depth: assume a single control can fail and stack multiple protections.

Q68. What Is SQL Injection?

ROI: ★★★★★ Frequency: Extremely High

Answer

Malicious SQL inserted into queries.

SQL injection happens when untrusted input is concatenated directly into a SQL statement, allowing an attacker to alter the query's meaning. A classic example is a login form where supplying ' OR '1'='1 turns a credential check into a query that always returns true, bypassing authentication entirely.

The consequences can be severe: attackers can read or dump entire tables, modify or delete data, escalate privileges, and in some configurations even execute commands on the database host. It remains one of the most damaging vulnerabilities precisely because it targets the data store directly, which is why parameterized queries are treated as non-negotiable.

Q69. How To Prevent SQL Injection?

ROI: ★★★★★ Frequency: Extremely High

  • Prepared Statements
  • Parameterized Queries
  • JPA/Hibernate
  • Input Validation

The definitive fix is to never build SQL by string concatenation. Prepared statements and parameterized queries send the query structure and the user data separately, so the database treats input strictly as values and never as executable SQL. This neutralizes injection regardless of what the attacker types into a field.

Using JPA/Hibernate with bound parameters gives you this protection by default, which is one reason ORMs reduce SQL injection risk, though you can still reintroduce it by concatenating into native or JPQL queries, so vigilance is required. Input validation and least-privilege database accounts add further layers, limiting both what malicious input can look like and how much damage a successful injection could do.

Q70. What Is Brute Force Attack?

ROI: ★★★★★ Frequency: High

Answer

Repeated login attempts to guess credentials.

A brute-force attack systematically tries many username and password combinations until one works. Variants include credential stuffing, where attackers replay username/password pairs leaked from other breaches (exploiting password reuse), and dictionary attacks that try common passwords first. Automation lets attackers attempt thousands of combinations quickly.

These attacks are effective precisely because weak and reused passwords are common. The defence is to make each guess expensive and to cap the number of attempts, which is why slow password hashing (BCrypt), rate limiting, account lockout, and MFA together form the standard countermeasure set.

Q71. How To Prevent Brute Force Attacks?

ROI: ★★★★★ Frequency: High

  • Rate Limiting
  • Account Locking
  • Captcha
  • MFA

Rate limiting throttles how many login attempts an IP or account can make in a window, sharply reducing how fast an attacker can guess. Account locking (often temporary, with exponential backoff) stops repeated attempts against a single account, though it must be tuned carefully to avoid enabling denial-of-service by deliberately locking out legitimate users. CAPTCHA inserts a human-verification step that defeats simple automated scripts.

MFA is the strongest backstop: even if an attacker eventually guesses the correct password, they still cannot log in without the second factor. Combined with slow BCrypt hashing (which makes offline cracking of a leaked database impractical) and monitoring/alerting on suspicious login patterns, these layers make brute force economically unattractive. As always, the senior answer stresses combining controls rather than relying on any single one.
